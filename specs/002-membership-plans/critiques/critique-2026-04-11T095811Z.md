## Executive Summary

The remediation from the first critique round applied cleanly across all seven artefacts: US7 is deferred with per-file consistency, the cross-cutting `tenants` module is introduced, the Neon `SET LOCAL` verification plan is scheduled as a pre-implementation gate, the `DEBUG_RLS_STATE` dev assertion is designed, the per-plan currency column is dropped, the seed script is split into two idempotent stages, the F10 resolver migration path is documented, the probe detection strategy is revised to avoid `BYPASS RLS` in request-path code, the test-isolation helper is specified with UUID-suffixed slugs, the observability runbook is scoped with a <5 min triage SLA, and the audit-log extension plan aligns with the empirically-verified F1 `audit_event_type` pgEnum. All 15 actionable items from round 1 are genuinely resolved in the spec + plan + design artefacts — I verified each one by spot-grepping the changed files.

However, an independent re-read surfaces **two new Must-Address items that the remediation itself introduced or left unresolved**, plus three recommendations. Both Must-Address items are small structural fixes (minutes of spec/plan editing), not architectural rethinks:

1. **R1 (🎯 — data integrity)**: The new `PATCH /api/fee-config` contract allows admins to change `currency_code` with a confirmation flag, stating the system "re-denominates every plan's money field to the new currency". This is a silent data-corruption vector at currency boundaries — e.g., switching from THB (2 decimal places) to JPY (0 decimal places) reinterprets `3_600_000` minor units from 36,000 THB to 3,600,000 JPY, a ~100× price jump with no intermediate safety net. The correct behaviour is to **block `currency_code` changes in F2 when any non-deleted plan exists for the tenant** (it's a tenant-onboarding-time concern, handled by F10).
2. **R2 (🎯 — cross-file consistency)**: The remediation updated the migration filename from `0007_audit_event_types_f2.sql` → `0007_audit_log_f2_extension.sql` in data-model.md, research.md, and quickstart.md, but **missed plan.md** which still carries the old name in three places. plan.md also says "9 new event types" while data-model.md correctly counts 10 (the +1 is `plan_not_found`). Both are inconsistencies `/speckit.tasks` would propagate into conflicting tasks.

Neither finding is a rethink — both are bug-fix-level edits to plan.md plus one contract clarification. After applying these, F2 is clear for `/speckit.tasks`.

### Security & Privacy

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| R1 | 🎯 | **`PATCH /api/fee-config` currency-change is a silent data-corruption vector** (introduced by round-1 remediation). Round 1's P3 simplification dropped per-plan `currency_code` columns and made `tenant_fee_config.currency_code` the single source of truth for the tenant's catalogue currency. The updated contract (plans-api.md § 13, line 352) then added a `422 currency_change_requires_confirmation` error with the semantic: *"changing `currency_code` re-denominates every plan's money field to the new currency … values are interpreted as the new currency's minor units from that point onward"*. This is **catastrophically wrong at currency boundaries with different decimal places**. Example: SweCham's Premium plan stored as `annual_fee_minor_units = 3_600_000` means 36,000.00 THB (THB has 2 decimals). An admin who changes `currency_code` from `THB` to `JPY` (0 decimals) with confirmation → the same integer now means 3,600,000.00 JPY ≈ 800,000 THB equivalent (a 100× mispricing). Silent, irreversible, audit-visible only as a single `fee_config_updated` event. The admin intended to "switch currency"; they received "revalue catalogue by ~100×". Constitution Principle I (data privacy + integrity) + Principle IV (Payment Security, no floats in money → but the bug is semantic, not floating-point) + Principle VIII (Reliability / Data Integrity) are all implicated. This IS a blocking bug. | **Block `currency_code` change in F2 entirely when any non-deleted plan exists for the tenant.** Three changes: (1) Spec.md — add a new edge case: *"Changing the tenant currency code after plans exist is not supported in F2. The admin must delete or soft-delete all plans first (or wait for F10 multi-tenant onboarding to introduce a proper currency-migration wizard). Attempting to change `currency_code` via `PATCH /api/fee-config` while any non-deleted plans exist for the tenant returns `422 currency_change_blocked_plans_exist` with the plan count."* (2) Spec.md FR-016 — add "…except `currency_code` MUST NOT be changeable after any plan exists for the tenant." (3) Contracts/plans-api.md § 13 — replace the confirmation-flag narrative with: *"`currency_code` is NOT editable via PATCH in F2. The field is fixed at tenant-creation time (seed script for F2, F10 onboarding UI for future tenants). Attempts to change it return `422 currency_code_immutable_in_f2` with a pointer to the delete-all-plans workflow."* (4) Contracts error-code table — replace `422 currency_change_requires_confirmation` with `422 currency_code_immutable_in_f2`. Rationale: YAGNI + safety. No F2 tenant has a currency-change workflow on the roadmap — SweCham is THB-only. The "what if" scenario belongs to F10 where it can be handled properly (price migration with FX rate, admin approval, etc.). |

### Performance & Scalability

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| P8 | 💡 | **Palette lazy-load 100 ms budget on cold connection** (round 1's E7 solved the wasted-bandwidth concern but introduced a cold-start concern). SC-008 says command palette opens within 100 ms p95. Lazy-loading on first `⌘K` means the first open incurs DNS lookup + TLS handshake + HTTP round-trip for `/api/plans/search`. On a warm-connection same-origin fetch this is ~30–50 ms; on a fresh tab with no prior API contact it can be 100–300 ms. First-open p95 may exceed the budget. | Two low-cost mitigations: (1) Add `<link rel="preconnect" href="/api">` (or similar) in the admin shell root so DNS + TLS are warm by the time the palette opens — zero network on mount but the socket is primed. (2) Relax SC-008 to: *"p95 first-open < 300 ms, p95 subsequent-open < 100 ms"* — subsequent-opens hit the `unstable_cache` 30s TTL and are genuinely fast. Both measures together give a realistic cold path + a tight warm path. Recommended: apply both. |
