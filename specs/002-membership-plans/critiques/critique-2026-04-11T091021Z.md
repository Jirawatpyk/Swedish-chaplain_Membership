## Executive Summary

F2's spec and plan are fundamentally sound: they correctly identify the feature as the load-bearing schema for every F3+ commerce feature, translate the authoritative PDF-sourced SweCham 2026 tier data into a typed schema, resolve the five key clarifications (Q1–Q5) with durable decisions, and satisfy all ten Constitution v1.4.0 principles including the new Tenant Isolation NON-NEGOTIABLE clause. Enterprise UX is woven in as a first-class concern, not a retrofit. Testing strategy (contracts + integration + E2E + cross-tenant probe) is comprehensive, and the research document correctly anticipates the pgBouncer / `SET LOCAL` interaction.

However, four **must-address** items surface under closer scrutiny: (1) `TenantContext` is currently placed inside the `plans` bounded context but is genuinely cross-cutting and will cause ESLint-boundary pain the moment F3 lands — fix by extracting it to a dedicated `tenants` module before `/speckit.tasks` freezes the source tree; (2) Neon serverless + pgBouncer + `SET LOCAL` interaction is *described* in research but *not yet empirically verified* — write a smoke test against live Neon before committing to the implementation path; (3) the RLS policy's "NULL equals zero rows" semantics is correct but creates a silent debugging trap — add a dev-mode assertion to `src/lib/db.ts` that fails loudly when a query runs without `app.current_tenant` set; and (4) keeping US6 (Command Palette) and US7 (Inline Edit + Bulk) in F2 full-scope is a judgment call worth re-validating — the user explicitly chose it at `/speckit.specify` but the value math on 9 rows is thin and F3 is the natural home for both primitives. On balance, F2 is green-light after the two small structural fixes (E1/X2 + E5) and the empirical verification (E3); X1 is escalated to the user as an explicit decision.

## Engineering Lens Findings 🔬

### Architecture Soundness

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E1 | 🎯 | **`TenantContext` branded type is placed in `src/modules/plans/domain/tenant-context.ts`** (plan.md Project Structure + research.md § 1) but is a **cross-cutting type** that every F3+ bounded context will need. Per the ESLint `no-restricted-imports` rule F2 extends, deep imports into `@/modules/plans/domain/*` from other modules are **blocked**. This means F3 members, F4 invoices, etc., would be forced to either (a) import `TenantContext` through the `@/modules/plans` barrel — architecturally wrong because plans does not own the concept — or (b) bypass the barrel via an ESLint exception — violates Principle III. | **Create a new `src/modules/tenants/` cross-cutting domain module** containing only `TenantContext` + its constructor + validators. No database table (YAGNI — no `tenants` table in F2). This module is a pure Domain types library that every tenant-scoped bounded context imports through `@/modules/tenants`. Update `plan.md` Project Structure, `data-model.md` § 2.5, and `research.md` § 1 accordingly. 5-minute fix that saves F3 a painful retrofit. |
| E2 | 💡 | **F10 migration path for the resolver is not explicit in research.md**. When real multi-tenant onboarding ships, the resolver must read `tenant_id` from the session. F1's `auth` module has no tenant claim on the session cookie today. How does the resolver know the user's tenant without coupling `auth` to `tenants`? | Add a subsection to research.md § 1 titled "F10 migration path" that documents: (a) F1 session cookie gains an optional `active_tenant_id` claim via an additive schema change; (b) the resolver reads it; (c) users with memberships in multiple tenants use a future `/switch-tenant` UI that updates the claim and re-issues the cookie. Makes the forward path concrete without coupling F1/F2 today. |

### Failure Mode Analysis

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E3 | 🎯 | **pgBouncer + Neon serverless + `SET LOCAL` interaction is described but not empirically verified.** Research.md § 2 asserts "`SET LOCAL` works correctly with transaction-mode pooling because it is scoped to a transaction" — this is correct for vanilla pgBouncer, but **Neon's serverless driver** wraps queries differently (connection-per-request with implicit transaction, or reuses a pooled connection with its own transaction semantics). If a serverless function runs two queries outside an explicit `db.transaction()`, the second query may land on a different pooled connection and see `app.current_tenant` as NULL → RLS returns zero rows silently. This is a real, observed failure mode on Neon + Vercel Functions. | **Before `/speckit.tasks` commits any implementation tasks**, write a throwaway smoke test against live Neon Singapore: (a) set `app.current_tenant = 'swecham'` inside a transaction, run SELECT — expect N rows; (b) run the same SELECT *outside* that transaction — expect 0 rows (proves session-level leakage is impossible); (c) wrap both reads in a single `db.transaction()` — expect N rows for both. Document the result in a new `research.md § 2.4 Neon verification`. If Neon's behaviour is unexpected, fall back to: every plans use case opens its own `db.transaction()` so `SET LOCAL` is always live. This may already be the case via `runInTenant()` — but the smoke test empirically proves it. |
| E4 | 💡 | **Bulk undo + stale state interaction is undefined**. Plan says "bulk action = one transaction, all-or-nothing" AND "undo is a reverse-action API call, not a rollback". Subtle interaction: admin bulk-activates 3 plans → immediately soft-deletes one of them → clicks Undo on the activate toast within 10s → the reverse bulk-deactivate batch hits the soft-deleted plan and the all-or-nothing FR-036 rule rejects the whole undo. User sees: "Undo failed: plan state has changed." | Add an edge case to spec.md: *"Undo on a bulk action that has become partially stale (one or more targets moved to a different state since the original action) refuses with a clear explanatory toast. Best-effort partial undo is NOT supported — state divergence means the admin must manually reconcile."* Implement the check server-side in `POST /api/plans/bulk` — if any target's current state does not match the expected-pre-state, return `422 undo_state_drift`. |

### Security & Privacy

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E5 | 🎯 | **Silent "zero rows" when `app.current_tenant` is unset is a debugging trap**. `current_setting('app.current_tenant', TRUE)` returns NULL when unset; the policy `USING (tenant_id = NULL)` evaluates to NULL and RLS treats NULL as "no match" → zero rows returned silently. This is secure by default (good!) but it means a developer who forgets to wrap a query in `runInTenant()` sees "no data" and debugs it as a data issue, not as a security-policy issue. First-party time-to-diagnose could be 20+ minutes per incident. | Add to `src/lib/db.ts`: a `DEBUG_RLS_STATE=1` dev-mode flag. When set, the helper wraps every tenant-scoped query in a pre-check that runs `SELECT current_setting('app.current_tenant', TRUE)` and **throws a loud developer-facing error with a stack trace** if the result is NULL. Production mode disables the pre-check (the silent-zero-rows behaviour is the correct production default — no error leak). Document in quickstart.md § 8.1 "Troubleshooting" alongside the existing entry. 10-minute implementation, hours of debug-time saved over F2-F9. |
| E6 | 💡 | **`plan.cross_tenant_probe` detection in request-path code is logically tricky**. Contracts/plans-api.md § 2 says a GET that 404s "additionally logs a `plan.cross_tenant_probe` when the plan_id exists in another tenant". But the 404-never-403 rule + RLS means the request path gets 0 rows and cannot legitimately tell whether the plan exists elsewhere. Detecting it would require a `BYPASS_RLS` sibling query — which itself is a privilege-escalation vector in request-path code. | **Revise the detection strategy**: request-path logs every 404 from an authenticated admin as a `plan.not_found` info-severity event (actor + tenant + requested plan_id). A separate **periodic super-admin scan** (F13 future or a scheduled cron) correlates `plan.not_found` events against the platform-wide plan inventory and escalates to `plan.cross_tenant_probe` + high-severity when a match in another tenant is found. Keeps request-path code read-only w.r.t. RLS. Update contracts/plans-api.md § 2 + data-model.md § 2.6 accordingly. |
