## Update Log — fixture fixes applied 2026-05-20T05:01:00Z

### F1 Fix — `cancel-scheduled-plan-change-tenant-isolation.test.ts:172-184`

Pre-fix: test passed BOTH a non-UUID `scheduledChangeId='arbitrary-non-empty-id'` AND a non-UUID `memberId='not-a-uuid'`. R3-C2 (Batch 4a) tightened `scheduledChangeId` to `z.string().uuid()` which is validated FIRST in the zod schema declaration order (`cancel-scheduled-plan-change.ts:38-50`). The use-case's first-issue translation at lines 75-80 then returned `scheduledChangeId` as the failing field, contradicting the test's `expect(field === 'memberId')` assertion.

Post-fix: pass `randomUUID()` for `scheduledChangeId` so the boundary test isolates the `memberId` rejection as originally intended.

Re-run: 3/3 cases pass on live Neon Singapore in 12s.

### F2 Fix — `soft-delete-with-members.test.ts:184-202` (`seedMember` helper)

Pre-fix: when called with `status='archived'`, the helper inserted a row WITHOUT setting `archived_at`. F3 schema enforces the status↔archived_at invariant via DB CHECK `members_archived_at_iff_archived` (status='archived' ⇔ archived_at IS NOT NULL). PostgreSQL rejected the row with `23514 check_violation`.

Post-fix: conditional spread `...(status === 'archived' ? { archivedAt: new Date() } : {})` pairs the archived flag with a current timestamp. Non-archived statuses retain the default null.

Re-run: 4/4 scenarios pass (including Scenario 4 — soft-delete succeeds when only attached members are archived).
