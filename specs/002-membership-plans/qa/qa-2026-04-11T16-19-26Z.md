## Failures Detail

### TC-000: CRITICAL — `LockWrapper` button-in-button hydration error (fixed in-session)

**Status**: ❌ → ✅ (fixed during QA run)
**Severity**: **High** (React hydration error + 4 console errors on every prior-year edit page render + silent DOM warning in prod)

**Steps to Reproduce** (pre-fix):
1. Sign in as admin on `pnpm dev` port 3100
2. Navigate to `/admin/plans/2025/qa-prior-year/edit` (any prior-year plan)
3. Observe console: 4 errors including "Hydration failed because the server rendered HTML didn't match the client"

**Expected**: Clean render with 0 console errors; locked fields disabled with lock icon tooltip.

**Actual** (pre-fix): 4 console errors:
```
[ERROR] In HTML, %s cannot be a descendant of <%s>.
This will cause a hydration error. <button> button
  ... <TooltipTrigger> → <button ...>
        <SelectTrigger> → <button disabled=true ...>
[ERROR] <%s> cannot contain a nested %s. button <button>
[ERROR] Encountered a script tag while rendering React component.
Error: Hydration failed because the server rendered HTML didn't match the client.
```

**Root cause**: `src/components/plans/plan-edit-form.tsx` `LockWrapper` wrapped its children in a BaseUI `<TooltipTrigger>` (which renders a native `<button type="button">`) for hover-tooltip support. When the locked children included their own interactive element (a `<Select>` that renders `<SelectTrigger>` which also renders a native `<button>`), React hit the invalid `<button>` → `<button>` nesting path and threw a hydration mismatch on first render. Works fine client-side after regeneration but wastes CPU + spams console.

**Fix** (commit pending):
`src/components/plans/plan-edit-form.tsx` — replaced the Radix Tooltip wrapper with a plain `<div title={...}>` + `aria-hidden` lock icon. The browser's native `title` attribute provides the same tooltip behaviour on keyboard focus + mouse hover with zero HTML-nesting concerns. The banner at the top of the form already explains the rule in full, so the per-field tooltip is purely a secondary UX hint.

```diff
- import { Tooltip, TooltipContent, TooltipProvider, TooltipTrigger } from '@/components/ui/tooltip';
  ...
- return (
-   <TooltipProvider>
-     <Tooltip>
-       <TooltipTrigger>
-         <div className="relative">
-           <div className="pointer-events-none opacity-60">{children}</div>
-           <div className="absolute right-2 top-2 ...">
-             <Lock className="h-4 w-4" />
-           </div>
-         </div>
-       </TooltipTrigger>
-       <TooltipContent>{t('fieldTooltip')}</TooltipContent>
-     </Tooltip>
-   </TooltipProvider>
- );
+ return (
+   <div className="relative" title={t('fieldTooltip')}>
+     <div className="pointer-events-none opacity-60">{children}</div>
+     <div aria-hidden="true" className="absolute right-2 top-2 ...">
+       <Lock className="h-4 w-4" />
+     </div>
+   </div>
+ );
```

**Why automated tests missed it**:
- Vitest runs in jsdom without SSR hydration — the button-in-button HTML nesting is valid enough for jsdom's parser to accept silently
- Unit component tests mount with `@testing-library/react` which also doesn't trigger Next.js SSR → hydration mismatch
- Only a real browser rendering a server-side-rendered page with client hydration can catch it

**Lesson**: Any Phase 5+ UI component that wraps `<Select>` / `<Button>` / `<DialogTrigger>` / other shadcn interactive primitives inside a `<TooltipTrigger>` or similar interactive wrapper should be reviewed for button nesting. A future F3 lint rule or dev-time React warning could catch this preemptively.
