## QA Summary

Systematic QA of F2 US1 against the running dev server + live Neon Singapore. Scope: the 9 SweCham 2026 plans, the list page, the detail page, the search endpoint, RBAC guards, i18n coverage, and audit-log reconstruction. Three lanes of coverage: (1) **automated suites** — 142 unit+contract + 27 integration tests, (2) **live Neon inspection** — direct SQL reads of the seeded catalogue + audit trail, (3) **running-server smoke** — curl the four public endpoints against `pnpm dev` on port 3100.

**Critical finding**: the first curl against `/api/plans?year=2026` returned **HTTP 500** with a Turbopack "Module not found: @node-rs/argon2-wasm32-wasi" error. Root-caused to `src/modules/plans/domain/policies.ts` runtime-importing `canAccess` + `Role` from the `@/modules/auth` barrel, which transitively chain-pulled the heartbeat use case → `auth-deps` → `@node-rs/argon2` (Node-only native binding) into the client bundle the moment any client component re-exported the plans barrel. **Fix**: refactored `plans/domain/policies.ts` to inline the admin/manager/member rules with a **type-only** import of `Role` from the deep path `@/modules/auth/domain/role` (erased at compile time, zero bundler impact). Same fix applied to `search-plans.ts`. After the fix, every endpoint returns the expected status code (401 for unauth API, 307 for unauth page, zero bundle errors in the response body). **All 59 role policies tests still green** (6 F2 plans-policies + 53 F1 role-policies).

No automated coverage gap would have caught this — Vitest bundles everything together without Next.js client/server splitting, and the existing E2E specs require seeded credentials to run against a real browser. The QA flow caught it in the curl step, which is exactly the purpose of `/speckit.qa`.

## Failure Details

### TC-000: CRITICAL — `@node-rs/argon2` chain-pull into client bundle (fixed in-session)

**Status**: ❌ → ✅ (fixed during QA run)
**Severity**: **Critical** (100% of `/admin/plans` visits returned HTTP 500)
**Steps to Reproduce** (pre-fix):
1. Run `pnpm dev` on port 3100
2. `curl http://localhost:3100/api/plans?year=2026`

**Expected**: HTTP 401 `{error: "no-session"}` (unauth guard) OR HTTP 200 JSON envelope (if authed)
**Actual** (pre-fix): HTTP 500 with a Turbopack compile error:
```
./node_modules/.pnpm/@node-rs+argon2@2.0.2/node_modules/@node-rs/argon2/browser.js:1:1
Module not found: Can't resolve '@node-rs/argon2-wasm32-wasi'
> 1 | export * from '@node-rs/argon2-wasm32-wasi'

Import trace:
  Client Component Browser:
    ./node_modules/@node-rs/argon2/browser.js
    ./src/modules/auth/infrastructure/password/argon2-hasher.ts
    ./src/lib/auth-deps.ts
    ./src/modules/auth/application/heartbeat.ts
    ./src/modules/auth/index.ts
    ./src/modules/plans/domain/policies.ts
    ./src/modules/plans/index.ts
    ./src/components/plans/locale-text-display.tsx
    ./src/components/plans/plans-table.tsx
    ./src/app/(staff)/admin/plans/page.tsx
```

**Root cause**: `plans/domain/policies.ts` did `import { canAccess, type Role } from '@/modules/auth'`. The barrel's `heartbeat` use case re-exports pulled `auth-deps.ts` → `argon2-hasher.ts` → `@node-rs/argon2` (Node-only native binding) into the client bundle the moment `<LocaleTextDisplay>` ('use client') re-exported `plans/domain/policies.ts` via the `@/modules/plans` barrel.

**Fix** (commit pending):
1. `src/modules/plans/domain/policies.ts` — removed runtime `canAccess` delegation; inlined the admin/manager/member rules (~8 lines of trivial boolean logic); kept `Role` as a **type-only** import from the deep path `@/modules/auth/domain/role` (framework-free, ESLint-exempt because `src/modules/plans/**` is whitelisted in the cross-module boundary rule).
2. `src/modules/plans/application/search-plans.ts` — same fix (type-only `Role` from `@/modules/auth/domain/role`).

**Why automated tests missed it**:
- Vitest bundles everything through Vite without Next.js client/server splitting — the transitive argon2 import never triggers the browser bundler error in unit/contract/integration tests.
- E2E Playwright specs (`plans-list.spec.ts` etc.) were authored RED with a `test.skip` guard on missing `E2E_ADMIN_*` env vars; they do not run against the dev server in the default suite.
- Typecheck passes because TypeScript types through the transitive imports but does not evaluate the Turbopack bundle boundary.
- Lint passes because the `no-restricted-imports` rule whitelists `src/modules/plans/**` from the "no deep auth imports" check — and even the barrel-level import is legal according to the rule.

**Lesson**: Domain → Domain module imports should be **type-only** or duplicated inline. Runtime cross-module barrel imports in client-reachable code are a bundler hazard even when ESLint allows them. Adding a post-build check to `pnpm check:bundle-client` that fails if any `@node-rs/*` appears in a client chunk would be a good F3 follow-up.
