## Findings Table

| ID | Severity | File | Line(s) | Description | Recommendation |
|----|----------|------|---------|-------------|----------------|
| W-01 | 🟡 Warning | `src/lib/db.ts` | 132 | `sql.raw()` with string interpolation for `SET LOCAL app.current_tenant`. PostgreSQL GUC parameters don't accept bind parameters, so this uses template literal interpolation. Mitigated by `TenantContext` branded-type validation (`[a-z0-9-]{1,63}`) and the fact that only `asTenantContext()` can construct the type — but the defence relies entirely on the validation boundary holding. | Add a defensive runtime assertion at line 132: `if (!/^[a-z0-9-]{1,63}$/.test(ctx.slug)) throw new Error('invariant')` as a belt-and-suspenders guard. This is low-cost and eliminates any future risk from type-system bypasses (`as` casts). |
| W-02 | 🟡 Warning | `src/modules/plans/application/update-plan.ts` | 119 | `JSON.stringify()` for deep comparison of `before` vs `after` values in audit diff computation. Works correctly for current field types but is fragile for: (a) object property ordering changes, (b) Date objects vs ISO strings, (c) future circular references. | Replace with a dedicated `deepEqual` utility (e.g., reuse `lodash.isEqual` or write a 10-line recursive comparator). For `benefit_matrix` JSONB, parse both sides to normalized form before comparison. Low priority — current data shapes are safe. |
| S-01 | 🟢 Suggestion | `src/modules/plans/application/clone-plans-to-year.ts` | 118 | Fallback error message `'unknown clone error'` in the `server_error` branch is vague. If `planRepo.cloneYear()` returns an unhandled error type in the future, this silently maps to 500 with no diagnostic. | Log the actual `cloneResult.error` object before returning `server_error` for operational debuggability. |
| S-02 | 🟢 Suggestion | `src/modules/plans/application/soft-delete-plan.ts` | — | No check for plan-to-plan references (`includes_corporate_plan_id`). If a corporate plan is soft-deleted while a partnership plan still references it, the reference becomes a dangling pointer. | Document this as an F3 concern: when F3 adds the real `MemberAttachmentChecker`, also add a "referenced by partnership plans" check to the soft-delete guard. For F2, the admin manages ≤9 plans manually — low risk. |
| S-03 | 🟢 Suggestion | `src/modules/plans/application/update-fee-config.ts` | 64 | `patch: unknown` input type — safe because zod re-validates, but less ergonomic for callers. | Define `FeeConfigPatchInput` type matching the zod schema and use it instead of `unknown`. Non-blocking. |
