## CRITICAL Issues (block any F3+ work that depends on F2 patterns)

### C1. `MemberAttachmentChecker` stub still wired in production after F3 shipped
- **Where**: `src/modules/plans/infrastructure/members/stub-member-attachment-checker.ts:1-32` + `src/modules/plans/plans-deps.ts:70`
- **What**: stub always returns 0; `softDeletePlan`'s `has_active_members` guard (FR-010) is a permanent no-op
- **Why critical**: F3 (`005-members-contacts`) has shipped a real `members` table + `DrizzleMembersRepository.countActivePlanMembers` but `plans-deps.ts` still injects the stub. The FR-010 guard is dead on live SweCham tenant
- **Sources**: comment-analyzer #1, simplify #6, tests #6 (I6)

### C2. `tenant-context.ts` silently falls back to deployed tenant on header parse failure
- **Where**: `src/lib/tenant-context.ts:71-84`
- **What**: try/catch around `headers.forEach()` swallows any exception and falls back to `env.tenant.slug` with only a `logger.warn`
- **Why critical**: Constitution v1.4.0 Principle I forbids tenant-isolation fallbacks. Future F10 proxy or signed-claim parse errors would silently route requests to the default tenant
- **Sources**: errors C-2

### C3. `reserveIdempotencyRecord` fire-and-forget in POST/PATCH/DELETE handlers
- **Where**: `src/app/api/plans/route.ts:197`, `src/app/api/plans/clone/route.ts:123`, `src/app/api/plans/[year]/[planId]/route.ts:212+396`
- **What**: reservation called without awaiting its Result; on Upstash quota exhaustion or Redis down, request proceeds without idempotency protection вЖТ duplicate plan/clone on retry
- **Why critical**: hides Upstash quota exhaustion (already seen in `reference_rate_limit_test_failure.md` memory), Redis flap, race in reservation
- **Fix**: `reserveIdempotencyRecord` returns Result; route returns 503 on reservation failure
- **Sources**: errors C-3

### C4. `getPlan` discards audit Result via `void auditResult`
- **Where**: `src/modules/plans/application/get-plan.ts:84`
- **What**: `void auditResult;` after `deps.audit.record(...)` for `plan_not_found`; no logger.error on persist_failed
- **Why critical**: F13 escalates `plan_not_found` вЖТ `plan_cross_tenant_probe` (high-severity security event). Sustained `persist_failed` (audit table RLS drift, immutable trigger drift, DB flap) silently disables the entire cross-tenant probe correlation pipeline
- **Fix**: `if (!auditResult.ok) logger.error({ err: auditResult.error }, 'get-plan: audit_failed on plan_not_found вАФ F13 correlation pipeline at risk')`
- **Sources**: errors C-1, code suggestion #4

### C5. US5 fee-config orphaned after R8 / migration 0029
- **Where**: `src/modules/plans/domain/audit-event.ts:34,62,214` + `src/modules/plans/infrastructure/audit/plan-audit-adapter.ts:70` + `src/modules/plans/domain/fee-config.ts` + `src/modules/plans/domain/policies.ts:67-79` (`canReadFeeConfig` / `canMutateFeeConfig`)
