## Findings

| ID | Severity | File | Line(s) | Description | Recommendation |
|----|----------|------|---------|-------------|----------------|
| W1 | 🟡 Warning | `src/modules/auth/application/create-user.ts` | 173–183 | **Silent enqueue failure is log-only — no metric, no surfaced signal.** When `deps.enqueueInvitation` returns an error (DB connection loss, unique-violation race, Drizzle failure), the use case returns `ok({ user, invitationId })` to the admin. The admin sees success but the invitation email will **never be sent** (the dispatcher only drains rows that made it into `notifications_outbox`). No audit event, no metric, no resend mechanism. Comment at line 18 says "admin can resend via a future 'resend invitation' admin action" — that action is not yet shipped. | Add a metric counter (`authMetrics.invitationEnqueueFailed(role)`) AND either (a) raise an alert via `email_dispatch_failed` audit with a `reason: 'enqueue_failed'` payload (cross-tenant log-only since `tenant_id IS NULL` here), or (b) surface a warning in the route handler so the admin UI shows a "invitation created but email queuing failed — please retry" toast. Minimum: metric + log is already present; add the metric. |
| S1 | 🟢 Suggestion | `src/app/api/cron/outbox-dispatch/route.ts` | 183–208, 284–299 | **Null-tenant permanent-failure drops audit row entirely.** `auditLog.tenantId` is nullable in schema (`src/modules/auth/infrastructure/db/schema.ts:256`), but the dispatcher chooses log-only when `row.tenantId === null`. For F1 invitation failures this is acceptable (cross-tenant platform event), but it means permanent-failure events for the F1 invitation flow never land in the append-only audit table — compliance evidence lives only in pino logs. | Either (a) insert the audit row with `tenantId: null` since the column allows it, or (b) document this trade-off in `observability.md` § email dispatch failure runbook so the ops team knows to check pino logs for null-tenant drops. |
| S2 | 🟢 Suggestion | `src/app/api/cron/outbox-dispatch/route.ts` | 389 | `void sql;` silences the unused import. The `sql` import from `drizzle-orm` appears unused after the round-1 refactor. Dead import — no runtime cost but confuses readers ("why is sql here?"). | Remove the `import { ..., sql }` and the `void sql;` line. |
| S3 | 🟢 Suggestion | `src/app/api/cron/outbox-dispatch/route.ts` | 344–347 | **Dev-mode CRON_SECRET bypass is implicit.** If `env.isDevelopment === true` AND `CRON_SECRET` is unset, any unauthenticated GET drains the outbox. Intended for local dev but carries risk if dev-mode ever leaks into a non-dev deploy (e.g., mis-set `NODE_ENV`). Low likelihood given env zod validation but worth defense-in-depth. | Add an additional guard: require either a `CRON_SECRET` header OR an IP check against `127.0.0.1`/`::1` in dev. Alternative: log a loud warning on each unauthenticated drain so accidental dev-mode deploys are immediately visible in aggregated logs. |
