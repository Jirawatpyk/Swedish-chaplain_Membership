### TC-004 — Cron Dispatcher Behaviour ✅

**Method**: curl → `/api/cron/outbox-dispatch` (dev-mode, no CRON_SECRET)

```
$ curl http://localhost:3100/api/cron/outbox-dispatch
{"ok":true,"inspected":1,"sent":0,"retried":1,"permanent":0,"skipped":0}
```

**Post-dispatch DB state**:
```
status: pending (unchanged — retry scheduled, not permanent)
attempts: 1  (was 0)
next_retry_at: 2026-04-17T16:34:52.208Z  (was +1s ago, now +60s)
last_error: "Resend send failed after 3 retries"
sent_message_id: null
```

**What this proves**:
- Dispatcher `inspected=1` — correctly picked up the row via `outbox_dispatch_idx`.
- `retried=1` — send failed transiently (Resend rejected `@swecham.test` fake domain), so dispatcher incremented attempts + pushed `next_retry_at` by 60s (first step of FR-012c exponential backoff: 60s / 5m / 30m / 3h / 12h).
- No rollback — the status stayed `pending`, not `permanently_failed` (correct: attempts=1, threshold=5).
- `last_error` captured the Resend failure verbatim → observability story works.

**Why "sent=0" is expected here, not a failure**: the invitee email `@swecham.test` is a test-only domain with no real mailbox. Resend rejects delivery. In production with a real email address the dispatcher would mark `status='sent'` after the first attempt.

**Step 3 — Manual F3 link (smoke test shortcut)**:

SET linked_user_id = '2d25cefd-00c9-43c7-9bfe-a0e9f68ca28b',
    updated_at = NOW()

Linked the smoke user (`2d25cef…`) to an existing seed contact (`Anna Andersson` / primary on `Fogmaker Thailand Demo`) that was unlinked. In production this step happens inside the `invitePortal` use case, not via ad-hoc SQL.

Full member profile rendered:
