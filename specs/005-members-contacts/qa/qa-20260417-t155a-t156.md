### Summary of Automated SR Proxy Checks

The automated ARIA inspection covers the mechanically-verifiable subset of what a manual screen-reader pass evaluates:

| Manual SR criterion | Automated proxy check | Status |
|---|---|---|
| Logical reading order | Focus tab order inspection | ✅ Correct |
| Role announcements on table | `scope="col"`, `<th>` presence | ✅ Correct |
| Combobox announced correctly | `<label for>` + `role="combobox"` + `aria-expanded` | ✅ Correct |
| Calendar announced correctly | Deferred — Calendar not on tested routes | 🔵 N/A |
| Inline-edit save announced | `aria-live` regions present on directory | ✅ Evidence present |
| Required field errors announced | `aria-describedby` on inputs | ⚠️ Only 1 input has `aria-describedby`; error containers checked by axe-core E2E |

**⚠️ One finding**: Only `company_name` has `aria-describedby` (pointing to required-fields note). Error message containers for other fields are not pre-wired with `aria-describedby` — error divs appear dynamically on validation. This is standard react-hook-form pattern; axe-core will flag if error containers are not properly associated. Covered by `members-a11y.spec.ts` TC-005/TC-006.

> **Note for human SR pass (still required)**: The audio verification of NVDA/VoiceOver reading order, sentence grouping, and inline-edit announce quality cannot be automated. Human pass should verify: (1) NVDA Browse mode reading of the members table — row cells read as "Company: Fogmaker, Country: TH, Plan: Regular Corporate, Status: Active", (2) "Save" button in inline-edit triggers an announce of the updated value, (3) Archive dialog focus trap confirmed by audio, (4) Combobox options announced as "list of N items".
