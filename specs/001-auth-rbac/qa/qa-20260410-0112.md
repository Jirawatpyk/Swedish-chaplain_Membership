## Failure Details

### TC-005b — AS5 "return to original URL after successful sign-in"

**Status**: ⚠️ PARTIAL (feature intentionally deferred to Phase 10)

**Steps to Reproduce**:
1. Sign out (or visit in a new incognito window)
2. Navigate directly to `/admin/users` (a protected URL deeper than `/admin`)
3. Expected: redirect to `/admin/sign-in?returnTo=%2Fadmin%2Fusers` or similar
4. Actual: redirect to `/admin/sign-in` (no `returnTo` param)
5. After successful sign-in, user lands on `/admin` (the default), not the original `/admin/users`

**Expected** (spec AS5): *"they are redirected to the sign-in page and returned to the original page after a successful sign-in."*
**Actual**: first half works (redirect); second half (return to original) not implemented.
**Evidence**: `src/lib/auth-session.ts:requireSession()` uses a fixed
redirect target (`/admin/sign-in` or `/portal/sign-in`) with no `returnTo`
parameter. `src/components/auth/sign-in-form.tsx:onSubmit` hard-codes the
post-success destination to `data.redirect` from the API response, which is
always `/admin` or `/portal`.
**Severity**: **Low** — the happy path for an admin going directly to
`/admin/sign-in` works perfectly; the edge case only surfaces when an admin
bookmarks a deeper URL and clicks it while signed out. Phase 10 T171 adds
the `return-after-signin.spec.ts` E2E test that would cover this once the
code lands.

No other failures.
