## Failures Detail

No failures. The QA pipeline is 100% green.

### Observations (non-failures, worth documenting)

1. **Short-circuit ordering on reset-password**: spot-check 5
   (`POST /api/auth/reset-password` with a valid-format but non-existent 64-char
   token + deliberately-weak password "short") returned `410 link-invalid` rather
   than `400 weak-password`. This is **correct and intended**: the use case checks
   the token BEFORE running the HIBP + policy check, so an attacker cannot probe
   the password policy via invalid-token requests. Contract test T089 covers the
   `weak-password` branch by mocking the use case — the 400 path is reachable
   only when a valid live token is paired with a policy-failing password.

2. **The "weak-password after valid token" live path** was not exercised directly
   during the QA run because creating a live token requires either (a) running
   the forgot-password flow against the bootstrap admin and burning 1 of 3/h
   rate-limit slots, or (b) connecting directly to Neon to insert a row.
   Integration test T090 covers the same assertion end-to-end against a
   throwaway test user, so the contract is proven by automated tests even
   though the live spot-check skipped it. If Phase 10 polish wants a live
   negative for SC-002 parity, a dedicated test-only helper could be added.

3. **Retry-After header on 429**: not directly spot-checked (would require 11+
   forgot-password hits to force the rate limit, which would pollute the shared
   bucket). Covered by contract test T088 `429 with Retry-After when rate-limited`
   and by the integration brute-force test which uses the same Upstash limiter
   implementation.
