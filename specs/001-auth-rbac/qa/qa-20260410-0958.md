## QA Summary

Ran 176 automated tests (141 unit/contract + 35 integration) against a running
Next.js dev server on port 3100 with a live Neon Postgres database and the
Upstash Redis rate-limit cache. Also executed 7 HTTP spot-checks via `curl`
against the live dev server to verify Phase 7 + Phase 8 wire contracts and
security header posture.

**Phase 7 (US5 Member portal)** — Member sign-in end-to-end verified against
live DB. `portal='member'` creates a session with `role='member'`; cross-portal
attempts (member → staff, admin → member) are rejected with the same generic
`invalid-credentials` slug (FR-016 no portal leak). Member layout's
cross-portal defence (redirecting admin/manager visitors to `/admin`) is
verified by source review — impossible to exercise live without a real
session. The unauthenticated `GET /portal` → 307 redirect now carries the
`returnTo=%2Fportal` query param so the member lands back on their portal
after sign-in (T171 behaviour from Phase 10).

**Phase 8 (US6 Change password)** — Signed-in change-password flow verified
against live DB. Happy path rotates the current session (delete ALL + create
fresh), revokes every other session, and emits both `password_changed` and
`concurrent_sessions_revoked` audit events in the same transaction (SC-021
two-session scenario). Wrong-current-password short-circuits without side
effects and without emitting any audit row. Same-password guard fires BEFORE
the HIBP policy check so the user never sees the "breached" error for their
own current (otherwise-valid) password.

**Lint regression found & fixed** — the initial QA run surfaced 1 warning
in `tests/contract/change-password.test.ts` (`'_id' is defined but never used`
from the `@typescript-eslint/no-unused-vars` rule). Fixed by replacing the
underscore-prefixed parameter with an explicit `void id` body. Lint is now
fully clean before the verdict.

**Prior-phase regression** — All 120 tests from Phases 1-6 still pass
alongside the new 28 Phase 7-8 tests; no pre-existing failures were disturbed.

## Failures Detail

No failures. 176/176 tests green, 7/7 spot-checks match the contract.

### Observations (non-failures)

1. **Defense-in-depth short-circuit ordering** (TC-P8-08): the change-password
   route handler checks `getCurrentSession()` BEFORE parsing the JSON body,
   so an unauthenticated request with malformed JSON returns `401 no-session`
   rather than `400 invalid-input`. This is intentional and correct — it
   prevents an attacker from probing the request-body validator without first
   holding a valid session. The 400/invalid-input path is still reachable
   with a valid session and is covered by contract test T147
   `400 on missing currentPassword` + `400 on missing newPassword` cases.

2. **Lint warning fixed during QA**: the initial `pnpm lint` flagged
   `tests/contract/change-password.test.ts:34` for an unused `_id`
   parameter. The underscore prefix convention doesn't exempt it from
   `@typescript-eslint/no-unused-vars` in this config. Fixed by making
   the mock body explicit: `vi.fn(async (id: string) => { void id; })`.
   Re-ran lint → clean. This fix should be committed with the rest of
   Phase 8 work — it's a one-line test-file edit, not a behaviour
   change.

3. **Member sign-in integration runs via signIn() use case, not the
   route handler**: because the route handler is the same `/api/auth/sign-in`
   as staff (the portal is a body field), there's no member-specific API
   entry point to test. The use case coverage in T138 is the correct
   level of abstraction.
