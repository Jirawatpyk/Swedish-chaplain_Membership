Ran 167 automated tests (114 unit/contract + 53 integration) against a live
Next.js dev server on port 3100 with live Neon Postgres and Upstash Redis.
Executed 10 HTTP spot-checks via `curl` against the dev server to verify

**Phase 9 (US7 Audit trail completeness + Resend webhook + lockout cron)** —
All 16 `AUDIT_EVENT_TYPES` round-trip through `auditRepo` successfully. The
logger redaction test (T158) exercises every `REDACT_PATHS` entry and
confirms zero secret leakage in serialized output. The Resend webhook
implements manual Svix HMAC-SHA256 signature verification without the
`svix` npm dependency; all 7 contract cases pass including forward-compat
handling of unknown event types. The lockout cleanup cron is idempotent
and correctly handles the dev-mode bypass when `CRON_SECRET` is unset.

**🚨 Production blocker found & fixed during QA**: initial spot-checks
against the dev server revealed that **POST /api/webhooks/resend** and
**POST /api/cron/lockout-cleanup** were being blocked by the CSRF Origin
middleware with `403 csrf-rejected, reason=missing-origin`. This is a
production blocker because:

  - Resend webhooks are server-to-server calls and never send an Origin
    header → every webhook delivery would have 403'd
  - Vercel Cron calls its scheduled endpoints without an Origin header
    → every scheduled run would have 403'd

**Fix**: added an `EXEMPT_PATH_PREFIXES` allow-list to `src/lib/csrf.ts`
covering `/api/webhooks/` and `/api/cron/`. These paths authenticate via
their own mechanisms (HMAC signature / Bearer token) and don't need
Origin-based CSRF defence. Added `tests/unit/lib/csrf.test.ts` with 11
cases including 5 regression guards for the exemption behaviour.
Re-ran spot-checks: cron POST now returns `200 {ok:true, cleared:0}`,
webhook POST now reaches the signature check and returns `401
invalid-webhook-signature` as expected.

**Prior-phase regression** — All 141 tests from Phases 1-8 still pass
alongside the new 26 Phase 9 tests + 11 CSRF tests; no flakiness observed.

## Failures Detail

### Initial failure — CSRF middleware blocks server-to-server POSTs

**Severity**: Production blocker (would have broken every Resend webhook
delivery and every Vercel Cron run once Phase 9 was deployed)

**Steps to reproduce** (on commit `2407237`):
1. Start dev server on :3100
2. `curl -X POST http://localhost:3100/api/webhooks/resend`
3. Observe: `HTTP 403` body `{"error":"csrf-rejected","reason":"missing-origin"}`
4. Same for `curl -X POST http://localhost:3100/api/cron/lockout-cleanup`

**Expected**: The request should reach the route handler, where the
signature verification (webhook) or Bearer token check (cron) is the
real authentication.

**Actual**: The middleware rejected the request at the CSRF Origin check
(before it even entered the route handler) because server-to-server
callers don't send the `Origin` header.

**Root cause**: `src/lib/csrf.ts::checkCsrf` enforced the Origin allow-list
uniformly for every state-changing POST under `/api/`. The original
`/api/auth/*` routes (sign-in, sign-out, change-password, etc.) are
browser-driven and legitimately need the Origin defence. Webhooks and cron
are NOT browser-driven and authenticate by other means, but they were
subject to the same check.

**Fix** (committed in the post-QA commit):
```typescript
// src/lib/csrf.ts
const EXEMPT_PATH_PREFIXES = ['/api/webhooks/', '/api/cron/'];

// ...checkCsrf() adds this step BEFORE the Origin check:
for (const prefix of EXEMPT_PATH_PREFIXES) {
  if (pathname.startsWith(prefix)) {
    return { action: 'pass', reason: 'exempt-path' };
  }
}
```

**Regression tests** added in `tests/unit/lib/csrf.test.ts` (11 cases):
- All safe methods (GET/HEAD/OPTIONS) still pass regardless of path
- POST on non-API paths still pass
- POST /api/auth/* still enforces Origin allow-list
- POST /api/webhooks/* passes WITHOUT an Origin header (new)
- POST /api/cron/* passes WITHOUT an Origin header (new)
- POST /api/webhooks/* passes even with a HOSTILE Origin (signature is
  the real defence)
- Look-alike path `/api/authwebhooks/foo` is still rejected (the exempt
  prefix requires a trailing slash — no sneaky prefix match)

**Evidence after fix**:
- Cron POST: `curl -X POST /api/cron/lockout-cleanup` → `200 {"ok":true,"cleared":0}`
- Webhook POST no sig: `curl -X POST /api/webhooks/resend` → `401 {"error":"invalid-webhook-signature"}` (reaches handler)
- Webhook POST wrong sig: same → `401 invalid-webhook-signature`

### Observations (non-failures)

1. **This bug was LATENT in Phase 9 commit `2407237`**: integration tests
   call the use cases directly, bypassing the route handler + middleware.
   Contract tests mock `auth-session`, not middleware. Only a live HTTP
   spot-check exercised the full stack, which is exactly what the QA
   process caught. Lesson: **every new API route that takes POST should
   have a live-server spot-check in the QA run**, even if the use case
   layer is proven by integration tests.

2. **`.env.local` does not set `CRON_SECRET`**: the dev bypass therefore
   fires (the route logs "no_secret_configured" at error level in prod
   but accepts unauth in dev). This is the correct default for dev but
   **operators must set CRON_SECRET in Vercel prod** before scheduling
   the cron — otherwise the dev-bypass clause `!env.isDevelopment`
   triggers a 401. The runbook (Phase 10 T183) should call this out.
