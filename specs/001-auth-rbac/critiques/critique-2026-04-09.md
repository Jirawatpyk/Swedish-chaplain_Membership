### 4b. Failure Mode Analysis

**🎯 E1 MUST-ADDRESS — No documented retry / fallback strategy for Resend (email)**

Password reset and invitation flows depend on email delivery. If Resend is down or slow:
- User requests password reset → success toast → but email never arrives → user assumes it worked → blocked
- Admin invites a new staff member → success → invitation email never arrives → new staff can't sign in → unknown failure
- **No retry mechanism, no operational alert, no dead-letter queue** is specified in `plan.md` or `research.md § 6`.

The spec (FR-005, FR-009) implicitly assumes the email is always delivered. The research mentions Resend webhooks are "out of F1 scope" — but webhook-based failure tracking IS the fallback mechanism for async email.

**Suggestion**: Add to `research.md § 6` or `plan.md`:
1. **Resend webhook integration** for delivery events (`email.sent`, `email.delivered`, `email.bounced`, `email.complained`) — in F1 scope, small handler
2. **Retry policy**: if Resend API call itself fails at the API layer, retry 3 times with exponential backoff (1s / 2s / 4s), then log a pino error + return the API response anyway (the token is still valid; the user can try again)
3. **Operational alert**: if Resend webhook reports a bounce or a non-delivery, log a pino warning with the target email so an on-call engineer can follow up manually for important invitations
4. **User recovery**: the "forgot password" page MUST have a visible "didn't receive the email?" link that re-initiates the request (rate-limited per existing § 5 rules)

This is a small addition (one webhook route + one retry loop) that materially reduces the blast radius of email provider hiccups.

**💡 E2 Recommendation — No documented "fail open vs fail closed" policy for Upstash Redis**

`research.md § 5` specifies rate limiting via Upstash, but does not say what happens if Upstash itself is unreachable:
- **Fail open** (allow the request) = DDoS / brute force risk during outage
- **Fail closed** (deny the request) = legitimate users locked out during outage
- **Correct answer**: fail **open** for convenience endpoints (sign-in during Upstash outage) but apply a **short-circuit lower limit in memory** (e.g., 20 requests per minute per IP) as a defence-in-depth fallback

**Suggestion**: Add to `research.md § 5`:
> **Upstash outage behaviour**: rate-limit checks that throw (network timeout, 5xx) MUST fail **open** — the request continues — to avoid lockouts during provider outages. A per-function-instance in-memory fallback (Map) caps the absolute throughput to ~20 req/min per IP during the outage. Recovery: once Upstash responds normally again, the fallback map is cleared.

**💡 E3 Recommendation — No documented circuit-breaker pattern for Neon Postgres**

Similar to E2. If Neon is slow (P99 > 3s), auth requests pile up on the Vercel function instances and cold-starts explode. A simple timeout + 503 response would be kinder than a dead instance.

**Suggestion**: Add to `plan.md` Technical Context:
> **Database request timeout**: 5 seconds. Connection acquisition timeout: 3 seconds. On either, return 503 `service-unavailable` to the client immediately rather than queueing, and log a pino error for alerting.

### 4c. Security & Privacy Review

**🎯 E4 MUST-ADDRESS — No explicit CSRF protection strategy for `/api/auth/*` routes**

Next.js 16 App Router **automatically** protects Server Actions against CSRF via the built-in `Next-Action` header check and SameSite cookie. But explicit `/api/**` Route Handlers (which is what `contracts/auth-api.md` defines for F1) get **no such protection** — a malicious site could POST to `/api/auth/sign-in` from user's browser if the session cookie is `SameSite=Lax` (the default we chose).

In practice, `SameSite=Lax` blocks cross-site POST cookies for most attack vectors, but:
- Subdomain attacks are possible if the domain hosts user content
- Top-level navigation POST via form submission IS allowed by `Lax`
- State-changing GET routes would not be protected at all (we don't have any, but the rule matters)

**Suggestion**: Document the CSRF strategy explicitly in `research.md § 4` (session mechanism) or a new section:

```
### 4.1 CSRF protection for API routes

For state-changing endpoints under /api/auth/* the server MUST enforce one
of the following on every POST:

1. Double-submit cookie pattern: a random "csrf" cookie (non-HttpOnly) plus
   a matching "x-csrf-token" header on the request. Client sets both; server
   verifies they match.
2. Origin header check: reject the request unless Origin is in an allow-list
   (the app's own domain). Safe because browsers send Origin on state-changing
   requests and scripts cannot forge it.

Recommendation: adopt option 2 (Origin check) — zero client-side work,
trivial to implement in middleware, and covers all /api/* routes uniformly.
Document the allow-list in src/lib/env.ts as APP_ALLOWED_ORIGINS.

All state-changing POST routes in contracts/auth-api.md MUST return 403
csrf-rejected if the Origin header is absent or not in the allow-list.
```

This is mandatory for a security-sensitive feature. Not a big change — ~30 lines in middleware.

**🎯 E5 MUST-ADDRESS — No explicit threat model document**

Constitution Principle I says security is non-negotiable but the spec/plan don't capture a **threat model**. For auth specifically, the standard threats are:
- Credential stuffing / brute force (partially addressed by FR-013 lockout + § 5 rate limiting)
- Phishing (user problem, out of scope)
- Session hijacking (addressed by HttpOnly + Secure)
- CSRF (see E4)
- Session fixation (addressed by session rotation on sign-in + password change)
- Enumeration via sign-in / reset (addressed by FR-016 generic messages)
- Enumeration via response timing (NOT addressed — a 401 that takes 50ms vs 5ms leaks whether the user exists)
- Token predictability (addressed — 32-byte crypto random)
- Privilege escalation via role change race (not explicitly tested)
- Weak password via paste-from-leak (FR-006 says "rejection of known-compromised passwords" but the implementation is deferred)
- Email account takeover (out of F1 scope — user problem)
- Insider abuse via audit log tampering (addressed — append-only DB grants)
- Rainbow table attack (addressed — argon2id with salt)
- Side channel on password comparison (addressed — argon2id is constant-time; BUT the "user exists?" lookup is not)

**Suggestion**: Create `specs/001-auth-rbac/security.md` (or add as a new section in `research.md`) with:
1. A threats table (threat, mitigation, test reference)
2. An explicit note on timing-side-channel: "The sign-in handler MUST perform a full argon2id verify even when the email doesn't exist (using a pre-computed dummy hash) so that response times are indistinguishable between valid and invalid emails."
3. An explicit note on password breach check: "FR-006 is implemented via HaveIBeenPwned k-anonymity API (POST the first 5 chars of SHA1(password), check if the suffix appears in the response)."
4. A threat-to-test mapping: every threat has at least one test that attempts to exploit it

This is ~100 lines of documentation + ~20 lines of code (dummy hash + timing-constant path + HIBP check). Essential for a security review gate sign-off.

### 4f. Operational Readiness

**💡 E7 Recommendation — No feature flag / kill switch for the new auth**

For a security-critical launch, the usual production safety net is a feature flag that can disable the new auth and fall back to the old system if something goes wrong. But SweCham's "old system" is Excel — there's nothing to fall back to.

**Options**:
1. **Vercel rollback** (current plan) — instant, but rolls back ALL code, not just auth
2. **Feature flag** — more granular but adds complexity
3. **Read-only mode** — emergency lever that disables all writes while allowing reads; auth stays up but no one can create/edit/disable users until the kill switch is released

**Suggestion**: Add a `READ_ONLY_MODE` environment variable that, when `true`, rejects all state-changing POST endpoints with 503 `read-only-mode`. Set via Vercel dashboard, no code deploy needed, reversible in 30 seconds. Takes ~15 minutes to implement. Documented in `quickstart.md` § 7.3 (Rollback).

**💡 E8 Recommendation — DB migration reversibility**

The audit_log grants migration (`drizzle/migrations/000x_audit_log_grants.sql`) creates roles and revokes grants. If a migration fails or needs rollback, reversing this isn't straightforward because:
- Dropping a role requires REVOKE on every object first
- Recreating grants after a DOWN migration requires the role to still exist

**Suggestion**: Add a paired `down` SQL file that explicitly reverses the grants and drops the roles in the correct order, OR document in `research.md § 7` that this particular migration is intentionally forward-only and any rollback requires manual DB work. Document the decision so it's not a surprise.

One coordinated change.

### X2 💡 Cross — Metrics dashboard is both a product feedback loop (P5) AND an engineering observability gap (parallel to E7)

P5 (no post-launch metrics dashboard) and the lack of a defined operational signal for "F1 is working in production" are the same gap viewed from two angles. The product team wants engagement metrics; the engineering team wants SLO burn-rate metrics. They can share a single dashboard.

**Combined suggestion**: Create a single `docs/observability.md` in the repo (cross-feature) that lists which metrics matter, who consumes them, and what the alert thresholds are. F1 populates the auth section; F2+ add their own sections.
