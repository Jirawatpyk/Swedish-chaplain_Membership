## Notable Remediation Details

### W-02 — Last-admin DB trigger (highest-risk remediation)

The trigger enforcement is the most invasive change in this review. Three notes:

1. **The 0004 migration is a bug fix on 0003.** The initial trigger function unconditionally returned `NEW`, which is a subtle Postgres gotcha: BEFORE DELETE triggers that return NULL (and `NEW` is NULL during DELETE) **silently cancel the delete**. This was discovered during integration test cleanup when a bulk DELETE returned 0 rows affected with no error. Migration 0004 fixes the return-row logic to explicitly `RETURN OLD` on DELETE. The comment in 0004 documents the gotcha so future maintainers don't regress.

2. **The trigger is narrowly scoped.** It fires on `BEFORE UPDATE OR DELETE` but short-circuits at the `IF` check for any mutation that does NOT remove an active admin (role change admin→non-admin, status change active→non-active, or DELETE of an active admin). For every other UPDATE on `users` (sign-in, lock, clear-fail-count, etc.) the trigger body runs but does nothing. Overhead is negligible.

3. **Error translation is double-guarded.** `src/lib/db-errors.ts::isLastAdminTriggerError()` checks BOTH `error.code === '23514'` (check_violation SQLSTATE) AND `error.message.includes('last-admin-protection')`. A future column constraint emitting SQLSTATE 23514 with a different message will correctly fall through to the re-throw branch — no silent swallowing.

### S-01 — Public barrel boundary

The `src/modules/auth/index.ts` barrel exposes Application use cases + Domain types + brand constructors. It **does not** export Infrastructure singletons (`userRepo`, `sessionRepo`, `rateLimiter`, `argon2Hasher`, etc.) or Drizzle schema. The ESLint rule blocks deep imports from outside `src/modules/auth/**` and `src/lib/**`. Client components have per-line `eslint-disable-next-line` with rationale comments explaining why the barrel can't be used (it transitively pulls Node-only Infrastructure modules, which fail client SSR).

Five legitimate escape-hatch sites remain (page pre-validation, cron, webhook) — each is marked with an `eslint-disable` comment explaining the design rationale (no Application use case wraps the read-only query surface).

### B-02 — Upstash fail-open test

The test file correctly mocks `@upstash/ratelimit` with a class whose `.limit()` throws, forcing every `rateLimiter.check()` call into the catch branch at `upstash-rate-limiter.ts:133`. All four required assertions are present (success on fail, cap enforcement, metric emission, independent bucket isolation). The test uses per-test unique keys so the in-memory fallback map doesn't leak state across test cases.
