## Summary

F5 delivers the **fifth Chamber-OS business feature** and the Phase 2 opener: signed-in members pay their F4-issued membership invoices online with a Thai-preferred rail (PromptPay QR) and a globally-preferred rail (credit/debit card via Stripe Elements). On successful settlement F5 invokes the existing F4 `markPaid` use-case so the invoice state machine, receipt PDF, auto-email delivery, and audit trail all reuse F4's proven atomic transaction — F5 adds the **payment surface** and the **webhook-driven reconciliation** but **does not re-implement** F4's state-machine, tax-document numbering, or PDF engine.

F5 is the first F-stream feature carrying **🔒 PCI (Constitution Principle IV — NON-NEGOTIABLE)** sensitivity. The entire design preserves **SAQ-A scope**: no raw card data ever touches the Chamber-OS server, no self-hosted card form, no card-field parameter in any of our API routes. Card capture is exclusively Stripe Elements (hosted fields); PromptPay QR payloads are generated by Stripe's PaymentIntent API.

**Scope confirmed from spec** (6 clarifications resolved across 2 sessions): 6 user stories (US1–US6; US1+US2 both **P1** — Q1 ship-together), **26 functional requirements** (FR-001…FR-024 + 4 amendments FR-011a, FR-011b, FR-016a, FR-025, FR-026), 13 success criteria, 15 named audit events, 4 new DB tables + 1 new bounded context + 2 new npm dependencies, and 2 post-MVP forward-compat seams (`allow_anonymous_paylink` flag for F5.1 pay-link + `out_of_band_refund_detected` detection for post-MVP reconciliation escape hatch).

**Technical approach**: Reuse the F1+F2+F3+F4 stack unchanged — Next.js 16 App Router + React 19 + TypeScript 5.7 strict + Drizzle ORM on Neon Postgres + Postgres RLS via `runInTenant(ctx, fn)` + shadcn/ui + Tailwind v4 + next-intl + Vitest + Playwright + pino + @vercel/otel + Resend (via F4 outbox reuse for refund-confirmation emails). Add **one new bounded context** `src/modules/payments/` housing `Payment` + `Refund` + `TenantPaymentSettings` + `ProcessorEvent` aggregates and the `ProcessorGateway` + `WebhookVerifier` ports. Add **two new npm dependencies**: `stripe` (server SDK) and `@stripe/stripe-js` + `@stripe/react-stripe-js` (client SDK) — API version pinned via `STRIPE_API_VERSION` env var per Q5 / FR-026. Add **one new API route family**: `/api/payments/*` (member-initiated), `/api/refunds/*` (admin-initiated), `/api/webhooks/stripe/*` (processor-initiated). Payment UX is a shadcn `Sheet` drawer embedded in the existing `/portal/invoices/[id]` F4 route — NO new Next.js route (FR-025 / Q4). Refund UX is an admin dialog embedded in the existing `/admin/invoices/[id]` F4 route. F5 extends F4's public barrel with `markPaidFromProcessor`, `issueCreditNoteFromRefund`, and `getInvoiceForPaymentById` wrappers so the cross-module contract is narrow and test-covered. Enterprise UX per `docs/ux-standards.md`; WCAG 2.1 AA on every surface; SV+EN+TH at release; ≥5-year audit retention, 10-year shared retention where events touch F4 tax documents.

**Storage**:

- Primary: PostgreSQL via Neon `ap-southeast-1` Singapore — unchanged. Adds **four new tables**: `payments`, `refunds`, `tenant_payment_settings`, `processor_events`. Extensions to `audit_log` (new event types only — reuses F2/F3/F4 `payload jsonb` column + `tenant_id`).

  - `refunds(tenant_id, payment_id, status)` — remaining-refundable calculation

- Session / rate-limit cache: Upstash Redis (Singapore) — unchanged. F5 adds **three new token buckets**:
  - `POST /api/payments/initiate` — **10 initiations / 5 min per `(tenant_id, actor_user_id)`** (prevents accidental retry storms + friends-abuse)
  - `POST /api/refunds/initiate` — **20 refunds / 5 min per `(tenant_id, actor_user_id)`** (admin-only; loose limit — real abuse of admin seat is a bigger problem than a rate limit)
  - `POST /api/webhooks/stripe` — **600 events / min per source IP** (generous upper bound; Stripe's expected traffic for SweCham is < 10/min; limit exists to protect against webhook-replay abuse from a compromised secret). Rate-limit bypass is signature-verified requests from Stripe's documented IP ranges (best-effort allowlist reloaded daily).

- **Forbidden fields in logs** (extends F1+F3+F4 pino redact list): `card_number`, `card_cvc`, `card[*]`, `stripe_secret_key`, `stripe_webhook_secret`, `Authorization`, `Stripe-Signature`, full webhook payload bodies (redacted to event-id + event-type + `api_version` only), PAN patterns matched by regex `^(3[47]|4|5[1-5]|6(?:011|5))\d+$` (defense-in-depth). **Scale/Scope**:

- Today: 1 live tenant (SweCham), ~131 members × 1 invoice/year × ~70% online-adoption target = ~92 online payments/year. Peak: annual renewal cycle may settle 50–100 payments in a single week (fully within Stripe's per-account default quotas; no contract-plan upgrade needed).
- Webhook events: each payment produces 3–5 events (`payment_intent.created`, `.processing`, `.succeeded` — plus `charge.succeeded` + `charge.updated`). Refunds add 2–3 events. 100 payments/week = ~500–800 events/week. Trivial.
- 5-year SaaS target: ~15-20 tenants × ~1,000 members × 0.7 online adoption × 1.3 events/payment = ~200 events/day platform-wide (still trivial for Postgres).
- Refund frequency: historical SweCham rate is < 1 refund / 100 invoices. Partial refunds are rare (~1 / 400).
- Concurrent payment attempts on the same invoice (two tabs, two devices): the `processor_payment_intent_id` UNIQUE index + the `payments` row lifecycle collapse duplicates to one PaymentIntent via resume-on-reopen (FR-003 / US1 AS4).
- Admin concurrency on refunds: realistically < 2 per tenant; row-level lock on `payments` row is sufficient for FR-011b serialisation.

### NON-NEGOTIABLE gates (any FAIL blocks the plan; no waivers)

- [x] **I. Data Privacy & Security — including v1.4.0 Tenant Isolation clauses**
  - **PII surfaces introduced**: the Payment row stores only non-PII processor metadata (PaymentIntent id, Charge id, last-4 + brand + expiry for cards — NO cardholder name, NO PAN, NO CVV). Refund row stores refund id + amount + reason (reason is admin-entered free text — advised redaction of PII in runbook). TenantPaymentSettings stores only business metadata (not PII). ProcessorEvent stores the raw webhook payload SHA-256 (not the body itself — the body is redacted beyond event-id + event-type before persistence).
  - **Lawful basis**: "legal obligation" under Thai Revenue Code + PDPA §26 (financial records) + GDPR Art. 6(1)(c) for EU member data — same basis F4 already established for tax documents. Retention ≥5 years (default) or 10 years (where the event touches an F4 tax document — shared). Erasure under PDPA §33 / GDPR Art. 17 is refused on "legal obligation" basis (same F4 pattern).
  - **Purpose limitation**: Payment + Refund records used ONLY for payment reconciliation, audit response, and downstream F4 tax-document generation. Not used for marketing, analytics, or cross-tenant features.
  - **RBAC**: `member` = initiate payment on own company's invoices only; view own payment history; `admin` = initiate refund, view all payment timelines + reconciliation filters, update tenant payment settings; `manager` = read-only on all F5 surfaces (view timeline + reconciliation, no refund button, no settings mutation). Enforced by extending F1 `rbac-guard.ts` with `payments:*`, `refunds:*`, `payment-settings:*` resource families.
  - **Tenant Isolation — two-layer defence-in-depth (Constitution v1.4.0 Principle I clauses 1-5):**
    1. **Application layer (clause 1):** every payments use case in `src/modules/payments/application/**` takes a `TenantContext` as an explicit dependency parameter. Forgetting to pass it is a TypeScript compile error. `TenantContext` imported from `@/modules/tenants` — F5 does NOT redefine.
    2. **Database layer (clause 2):** all four F5 tables have `ENABLE ROW LEVEL SECURITY` + `FORCE ROW LEVEL SECURITY` + the standard `USING (tenant_id = current_setting('app.current_tenant', TRUE))` policy. `runInTenant(tenantCtx, fn)` reused unchanged. `DEBUG_RLS_STATE=1` dev-mode safety net inherited. **Webhook exception**: signature verification + processor-events row insert initially run under a bypass context because `tenant_id` is not known until the event is parsed + resolved via the Stripe account id → tenant mapping. Immediately after resolution, the transaction re-binds `app.current_tenant` via `runInTenant` for all downstream Payment/Refund/audit writes. This is the narrowest possible bypass window (pre-tenant-resolution only) and is justified in Complexity Tracking.
    3. **Test enforcement (clause 3):** `tests/integration/payments/tenant-isolation.test.ts` — Review-Gate blocker, fails the gate if missing or red.
    4. **Audit (clause 4):** Cross-tenant probes return 404 (never 403/401) and emit `payment_cross_tenant_probe` immediately at high severity. Alert threshold: 1 event / 5 min (alarm), 5 events / hour (incident).
    5. **Super-admin impersonation (clause 5):** not applicable — no super-admin console yet (F13). When F13 lands, impersonated payment initiation MUST be explicitly scoped + audit-logged with the super-admin identity (same pattern as F4 plan § clause 5).
  - **OWASP Top 10 coverage** (delta vs F1+F2+F3+F4 for the F5 surface):
    - **A01 Broken Access Control** — RBAC + RLS + member-to-company ownership check on `POST /api/payments/initiate` + admin-only on `POST /api/refunds/initiate` + webhook signature is the sole authz for `POST /api/webhooks/stripe`.
    - **A02 Cryptographic Failures** — at-rest AES-256 (Neon) + TLS 1.2+ + Stripe secret keys in Vercel env only + `Stripe-Signature` HMAC-SHA256 verified via Stripe SDK's `webhooks.constructEvent` (timing-attack-resistant). Inherited platform TLS. Stripe publishable key is client-visible by design (SAQ-A pattern).
    - **A03 Injection** — Drizzle parameterised queries; zod on every API boundary; Stripe event payload parsed by `webhooks.constructEvent` (throws on malformed JSON); no dynamic SQL in webhook handler; no interpolation of `Stripe-Signature` or secret into strings.
    - **A04 Insecure Design** — SAQ-A by construction. PaymentIntent creation + webhook verification + F4 `markPaidFromProcessor` invocation is the single most-designed surface in F5. Row-level lock on `payments` serialises concurrent refunds (FR-011b). Idempotency: (a) `Idempotency-Key` header on Stripe calls, (b) unique `processor_events.id`, (c) unique `processor_payment_intent_id` on `payments`, (d) unique `processor_refund_id` on `refunds`.
    - **A05 Security Misconfiguration** — `FEATURE_F5_ONLINE_PAYMENT` env flag gates every route; Stripe test-vs-live keys in disjoint env vars; webhook endpoint per environment; CSP extended to allow Stripe iframes + script.
    - **A07 Identification & Authentication Failures** — webhook signature is the only authN for webhook route; signature verification refuses before body parse; failure emits `webhook_signature_rejected` at high severity.
    - **A08 Software & Data Integrity** — append-only audit log extended (15 new event types); `processor_events.payload_sha256` stored for tamper detection (full body redacted); webhook signature binds body to Stripe's origin.
    - **A09 Logging Failures** — `payment_cross_tenant_probe`, `webhook_signature_rejected`, `webhook_api_version_mismatch`, `payment_environment_mismatch`, `payment_auto_refunded_stale_invoice`, `out_of_band_refund_detected` are high-severity audit events with alert routing.
    - **A10 SSRF** — no outbound HTTP from user input. Stripe SDK is the only outbound client; endpoint URLs are pinned to `api.stripe.com` by the SDK (not user-controllable).
  - **TLS 1.2+** + **at-rest AES-256** — inherited from F1+F2+F3+F4 unchanged.

- [x] **IV. Payment Security (PCI DSS) — NON-NEGOTIABLE and ACTIVE**

- [x] **VIII. Reliability (Error Handling + Data Integrity + Audit Trail)** — heavy principle for F5 given external service + money movement.
  - Every error path returns a typed `Result<T, E>`.
  - **Transactional boundaries** (each = one Postgres transaction):
    - **Initiate payment** (FR-001–FR-003): authz check → settings-read → resume-check (existing pending Payment for this invoice? reuse clientSecret) → Stripe `paymentIntents.create` (outside tx — idempotency-key `inv-{invoice_id}-{attempt_seq}`) → insert `payments` row with `status='pending'` → insert audit `payment_initiated` → commit. If Stripe call fails with transient error, retry with same idempotency key (SDK handles); if permanent error, surface to UI, no `payments` row.
    - **Process webhook event** (FR-007–FR-010): verify signature (throws on mismatch → 401 + audit `webhook_signature_rejected`) → check environment match (FR-010 → 200 + `payment_environment_mismatch` audit, no state change) → check API version match (FR-026 → 200 + `webhook_api_version_mismatch` audit, no state change unless exact match) → upsert `processor_events` (on-conflict-do-nothing for idempotency FR-008; if row already existed → 200 no-op) → resolve tenant via Stripe account id mapping → enter `runInTenant(ctx, ...)` → branch on event type (see below) → commit.
    - **Confirm payment** (webhook `payment_intent.succeeded` branch): `SELECT … FOR UPDATE` on `payments(processor_payment_intent_id)` → if not found, return 200 (event for an intent we didn't create — rare; log warning) → check target invoice state via F4 `getInvoiceForPayment` → if invoice NOT `issued`/`overdue`, auto-refund via `processorGateway.createRefund` + audit `payment_auto_refunded_stale_invoice` + commit; if invoice OK, update `payments.status='succeeded'` → insert audit `payment_succeeded` → invoke F4 `markPaidFromProcessor(tenantCtx, invoiceId, {...})` within the same tx → F4 writes its own receipt + auto-email-outbox (FR-004 atomicity with F4's existing transactional pattern) → commit.
    - **Fail payment** (webhook `payment_intent.payment_failed` branch): `SELECT … FOR UPDATE` on `payments` → `UPDATE status='failed'`, `failure_reason_code=<Stripe code>` → insert audit `payment_failed` → commit. No F4 invocation.
    - **Cancel payment** (webhook `payment_intent.canceled` branch): `SELECT … FOR UPDATE` on `payments` → `UPDATE status='canceled'` → insert audit `payment_canceled` → commit. No F4 invocation.
    - **Issue refund** (FR-011 + FR-011b): authz check → `SELECT … FOR UPDATE` on `payments(id)` (serialises concurrent admin refund attempts — FR-011b) → read existing refunds, compute `remaining = amount_satang − Σ(succeeded refund amounts)` → zod-validate `0 < refund_amount ≤ remaining` (reject pre-flight if exceed) → insert `refunds` row with `status='pending'` → `processorGateway.createRefund` with idempotency-key `rfnd-{payment_id}-{seq}` (outside tx because external call) → on success, `UPDATE refunds.status='succeeded', processor_refund_id=…` inside new tx → invoke F4 `issueCreditNoteFromRefund(tenantCtx, invoiceId, {refundId, amountSatang, reason})` within the same tx (F4 handles credit-note PDF + invoice-status transition + audit `credit_note_issued` + `invoice_credited|partially_credited`) → update Payment.status to `partially_refunded` or `refunded` (FR-011b) → insert audit `refund_initiated` + `refund_succeeded` → F4 outbox handles refund-confirmation email via the existing credit-note email path → commit. On Stripe failure, update `refunds.status='failed'`, insert audit `refund_failed`, NO F4 side-effect.
    - **Detect out-of-band refund** (webhook `charge.refunded` branch, FR-011a): for each refund in the event payload, look up `refunds(processor_refund_id) = <event refund id>` → if NOT found, the refund was initiated outside our app → write `out_of_band_refund_detected` audit + emit `out_of_band_refund_rejected_total` metric + alert → NO F4 credit-note, NO state transition; if found, correlate with existing in-app refund row (no-op — the refund is already being tracked by the in-app flow).
    - **Update tenant payment settings** (FR-015): authz (`admin` only) → zod-validate → `UPDATE tenant_payment_settings` → insert audit `tenant_payment_settings_updated` → commit. Toggling `online_payment_enabled` emits `online_payment_toggled` as a separate audit for traceability.
  - **Idempotency**:
    - `processor_events.id` PK (Stripe event id is globally unique across both test and live) — on-conflict-do-nothing is the idempotency primitive.
    - `payments.processor_payment_intent_id` UNIQUE — `paymentIntents.create` with idempotency-key-per-attempt; retries return the same Intent.
    - `refunds.processor_refund_id` UNIQUE — `refunds.create` with idempotency-key; retries return the same Refund.
    - `POST /api/payments/initiate` accepts an optional `Idempotency-Key` header; if omitted, the body is hashed + tenant+invoice used. Same for `POST /api/refunds/initiate`.
  - **Audit-log entries**: 15 new event types enumerated in spec FR-020. Retention 5 years minimum; 10 years where tied to F4 tax documents (shared retention).
  - **Failure modes catalogued** per the `payment_intent.*` + `charge.*` event surface (research.md § 3).

**Client-component boundary** (post-critique E10, 2026-04-23): every component under `src/app/(member)/portal/invoices/[id]/_components/pay-sheet/**` MUST carry the `'use client'` directive. The `@stripe/react-stripe-js` primitives (`<Elements>`, `<PaymentElement>`, `useStripe`, `useElements`) require client-side runtime + `useEffect`/`useState`. The parent invoice detail page MAY remain server-rendered; the Sheet drawer trigger crosses the `'use client'` boundary at the `pay-now-button.tsx` boundary. This pattern is friendly to Next.js 16 Cache Components — server-rendered shell + client-only payment surface — and is enforced by an ESLint rule that errors on missing `'use client'` directives in `pay-sheet/**`.

**Stripe SDK loading pattern** (post-critique R2-E1 + R2-E6, 2026-04-23 — verified against React + Next.js best practices `bundle-dynamic-imports` + `rerender-derived-state-no-effect` + Stripe singleton docs): the pattern combines Next.js `next/dynamic` for component-level lazy loading (saves bundle weight per `bundle-dynamic-imports`) with a module-level singleton cache for `loadStripe` (Stripe-recommended; cache is keyed by publishable key for F11 SaaS multi-tenant compatibility). Two-file split:

```tsx
// File 1: src/app/(member)/portal/invoices/[id]/_components/pay-sheet/pay-now-button.tsx
// (Always loaded with the invoice-detail page — small footprint)
'use client';
import dynamic from 'next/dynamic';
import { useState } from 'react';
import { Button } from '@/components/ui/button';
import { SheetSkeleton } from './sheet-skeleton';

// next/dynamic is the canonical Next.js lazy-load pattern (rule: bundle-dynamic-imports).
// ssr:false because Stripe Elements is client-only (uses iframe + window.Stripe).
const PaySheet = dynamic(() => import('./pay-sheet-internal'), {
  loading: () => <SheetSkeleton />,
  ssr: false,
});

export function PayNowButton({ invoice, publishableKey }: Props) {
  const [open, setOpen] = useState(false);
  return (
    <>
      <Button onClick={() => setOpen(true)}>Pay now</Button>
      {open && (
        <PaySheet
          invoice={invoice}
          publishableKey={publishableKey}
          onClose={() => setOpen(false)}
        />
      )}
    </>
  );
}

// File 2: src/app/(member)/portal/invoices/[id]/_components/pay-sheet/pay-sheet-internal.tsx
// (Lazy-loaded chunk — Stripe SDK lives here)
'use client';
import { Elements } from '@stripe/react-stripe-js';
import { loadStripe, type Stripe } from '@stripe/stripe-js';

// Module-level singleton cache, keyed by publishable key.
// Required for F11 SaaS where different tenants use different Stripe accounts;
// `loadStripe` is called at most once per unique publishable key per browser session.
// Stripe-documented pattern: https://docs.stripe.com/stripe-js#loadstripe
const stripeInstanceCache = new Map<string, Promise<Stripe | null>>();

function getStripeInstance(publishableKey: string): Promise<Stripe | null> {
  let cached = stripeInstanceCache.get(publishableKey);
  if (!cached) {
    cached = loadStripe(publishableKey);
    stripeInstanceCache.set(publishableKey, cached);
  }
  return cached;
}

export default function PaySheetInternal({ invoice, publishableKey, onClose }: Props) {
  // Derived directly from props during render (rule: rerender-derived-state-no-effect).
  // No useState + useEffect needed because Map.get is synchronous and the result is stable
  // across renders for the same publishableKey.
  const stripePromise = getStripeInstance(publishableKey);

  return (
    <Sheet open onOpenChange={onClose}>
      <Elements
        stripe={stripePromise}
        options={{ clientSecret: '...', locale: '...' }}
      >
        {/* PaymentElement, PromptPay tab, etc. */}
      </Elements>
    </Sheet>
  );
}
```

**Why this pattern**:
- `next/dynamic` with `ssr: false` defers the entire Stripe SDK chunk (~50KB gzipped of `@stripe/stripe-js` + `@stripe/react-stripe-js`) until the member clicks Pay-now. Members who only view their invoice never download the SDK (rule: `bundle-dynamic-imports` + `bundle-conditional`).
- Module-level `Map` cache acts as a singleton-per-publishable-key, which is correct for both F5 single-tenant SweCham (one key, called once) AND F11 SaaS multi-tenant (one key per tenant; each called once across the browser session).
- No `useState`/`useEffect` ladder — `getStripeInstance` is sync from React's perspective, derived from props (rule: `rerender-derived-state-no-effect`).
- No re-renders caused by stripePromise changes — same Promise reference returned for the same publishableKey.

**Forbidden anti-patterns** (caught by code review):
- Calling `loadStripe` inside JSX (`<Elements stripe={loadStripe(key)}>`) — re-creates Stripe instance on every render.
- Calling `loadStripe` at module top-level with a hard-coded env-var key — breaks F11 multi-tenant; works only for single-tenant SweCham.
- Wrapping in `useState` + `useEffect` — adds re-render cycle for no benefit (Map.get is sync); violates `rerender-derived-state-no-effect`.
- Awaiting `loadStripe` synchronously before render — blocks paint; the SDK is async by design and `<Elements>` accepts a Promise.

## Complexity Tracking

The following deviations survived the Constitution Check and are justified below. No unjustified violation exists; the Constitution Check is GREEN.

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| **Webhook endpoint runs on Node.js runtime (not Edge/Fluid default)** — Principle VII prefers edge for interactive endpoints | Stripe signature verification requires raw request body bytes. Next.js App Router + Edge runtime historically mangle raw bodies across framework upgrades; Node.js runtime is the Stripe-documented recommendation and matches our `tests/integration/payments/webhook-signature.test.ts` fixtures. Single endpoint impact; latency hit is ≤ 100ms cold start (Fluid Compute warm-pool mitigates). | Edge runtime with manual `arrayBuffer()` read: works today but (a) fragile across framework versions, (b) constantly required re-verification on every Next.js minor bump.
