## Review Pass 1 — Correctness & Logic

Spot-checked the highest-risk paths. No new defects found.

- **Webhook ordering invariant** (`route.ts:234-294`): `stripe-signature` header read FIRST, content-length pre-guard at 64 KiB (Threat F-16 defence), raw-body read, length re-check post-realisation, HMAC verify — only then is the event JSON consulted. The "verify-before-parse" assertion from `tests/integration/payments/webhook-signature.test.ts` (T044) is upheld at `route.ts:280` — the verifier is the first code that touches the body for parsing purposes; everything before is pre-parse.
- **Livemode + API-version gates** (`route.ts:326-358`): correctly return 200 (not 401 — to avoid Stripe retry storms on legitimate misconfigurations) but persist a `processor_events` row with `outcome='rejected_*'`, audit the rejection, and abort dispatch. Exactly matches FR-010 + FR-026.
- **Direct-event fallback** (`route.ts:312-324`): when SweCham creates a PaymentIntent without `Stripe-Account` header (single-tenant / platform-owner case), `evAccount` falls back to `env.stripe.accountIdSwecham`. Multi-tenant Connect events carry their own `account` and bypass the fallback. Documented inline; correct semantics.
- **Concurrent-initiate guard** (`initiate-payment.ts:329` + `drizzle-payments-repo.ts:156-171`): three-layer defence intact:
  1. Partial unique index `payments_processor_payment_intent_id_uniq` (migration scope).
  2. `lockForUpdateByPaymentIntentId(tx, paymentIntentId, tenantId)` — explicit `tenantId` parameter (CR-2 + R3 M-4 review fix; matches plan.md § Recent Changes).
  3. `pg_advisory_xact_lock(hashtextextended('payments:' || tenantId || ':' || invoiceId, 0))` — per-tenant-per-invoice channel; `payments:` prefix is disjoint from F4 `invoicing:` namespace so no contention with §87 sequential-numbering allocators. Auto-released at tx end.
- **Stripe idempotency key** = `inv-{invoiceId}-attempt-{n}` — repeatable across SDK retries; combined with the advisory lock closes the TOCTOU window without serialising unrelated invoices.
- **Result<T,E> discipline**: `process-webhook-event.ts` returns Result; route translates Result.err → 500 (Stripe retries), Result.ok → 200; `revalidatePath` is best-effort wrapped in try/catch so a Next.js cache miss does NOT bubble out and force a 24-hour Stripe retry loop on already-committed `markProcessed` (route.ts:592-608).
- **Process-charge-refunded** dispatch covers both branches (FR-011a out-of-band detection + in-app refund finalisation idempotent on `processor_refund_id`); per-tenant + per-env `out_of_band_refund_rejected_total` metric emits before audit append, matching observability.md § 21.3 alert.
- **Error-message redaction** (`route.ts:638-647`): catch-all logs `e.constructor.name` only, NOT `e.message` / `e.stack` — explicitly to avoid SQL parameter / Stripe SDK lib-path leakage. Correct PCI defence-in-depth.

No correctness issues found at the spot-check level.
