## Executive Summary

T166 (async receipt PDF) is well-engineered as a design — the core invariants are intact: Thai Revenue Code §87 no-gap sequential numbering is preserved (receipt number pre-allocated under the `paid` flip tx; worker reads `receipt_document_number_raw` instead of re-allocating); tenant isolation holds (`runInTenant` re-binding on every worker invocation); state machine transitions are guarded; SAQ-A scope is **untouched** (the worker handles invoice/member identity snapshots only — zero card/Stripe charge metadata flows through it); and the measured async p95 = 859 ms vs legacy sync 1657 ms (48 % p95 reduction) closes the prior R001 🔴 Blocker on succeeded-branch latency.

**However, the implementation has shipping blockers**:

1. 🔴 **Migrations 0056 + 0060 are NOT registered in `drizzle/migrations/meta/_journal.json`** (journal stops at idx=55). Running `pnpm drizzle-kit migrate` on production will SILENTLY SKIP both migrations. T166 will crash on every paid invoice because the `receipt_pdf_status` column won't exist. **This is the single highest-risk finding in the review** — production deploy would be a complete F5 outage.
2. 🔴 **Stuck-pending recovery path has no test coverage**. The reconcile cron (`receipt-pdf-reconcile/route.ts`) only targets `status='failed'` — a row stuck at `pending` (e.g. worker crashed mid-render before any failure write) is invisible to it. If kill-switch is flipped while pending rows are in flight, those rows are orphaned indefinitely (chamber-os-architect R6, senior-tester F-03).
3. 🔴 **Performance SLO assertion is gated behind `RUN_PERF=1` and never runs in CI**. Test ceiling is 1500 ms but shipped SLO-F5-002b dev budget is 1000 ms — so even when run locally, a regression up to 1500 ms passes (performance-slo-guardian F002, F008). Comment vs SLO mismatch (cites old 500 ms budget).
4. 🟠 **Pay sheet cancel-after-success race**: cleanup at unmount fires `firePaymentCancel('user_navigated_away')` AFTER successful payment when the success setState hasn't committed yet (UX D1). Risk: spurious `payment_canceled` audit events post-`payment_succeeded` for the same paymentId. Maintainer should grep prod logs to confirm.

The remaining 24 warnings/suggestions span documentation drift (CLAUDE.md / observability.md / saq-a-attestation.md still need T166 paragraph), minor reliability gaps (Phase B silent catch in `confirm-payment.ts:640`, optional `clock` injection), 4 PDPA documentation gaps (Stripe cross-border §28, privacy disclosure on pay sheet, retention map divergence vs data-model), and a handful of ux/i18n polish items.
