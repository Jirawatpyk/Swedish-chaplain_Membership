## Findings

| ID | Severity | File | Line(s) | Summary | Recommendation |
|----|----------|------|---------|---------|----------------|
| S5 | 🟢 Suggestion | `src/modules/payments/infrastructure/repos/drizzle-refunds-repo.ts` | 108–117 | **`updateStatus` lacks a `status='pending'` guard for the sweep call.** The sweep does a read tx (lists pending rows), then a separate write tx per row that calls `updateStatus(..., nextStatus: 'failed')`. The `WHERE` clause filters only `(tenant_id, refund_id)` — not status. **Today this race is unreachable**: the only writer to a stuck-pending refund row is the sweep itself, because (a) Phase B catch flips → `failed` synchronously after Phase B throws, and (b) the F5 webhook `charge.refunded` branch is NOT YET wired to the real `RefundsRepo` (see `di.ts` `makeUnimplementedRefundsRepo` comment, lines 53–66). **When the webhook branch lands**, a delayed `charge.refunded` event arriving exactly between the sweep's read and write tx could finalise the row to `succeeded`, and the sweep would then overwrite that to `failed` — money was returned (Stripe + F4 CN exist) but the F5 row says `failed`, causing audit divergence and possible double-refund if a customer asks. | Add a status-guarded `markStaleAsFailed(tx, refundId, tenantId, completedAt)` repo method that issues `UPDATE ... WHERE status = 'pending' RETURNING ...` — zero rows returned → throw → per-row tx rolls back → skip. Or, simpler: extend `updateStatus` to accept an optional `expectedCurrentStatus` param and add `eq(refunds.status, expectedCurrentStatus)` to the WHERE. Track this as a **prerequisite** for landing the webhook charge.refunded → real adapter migration; until then, file as known-future-risk. |
