## HIGH (6) — Batch 2 must address before flag-flip

### H-1 — Bridge `0n` clamp does NOT route to `not_payable`; sends `amount=0` to Stripe
**Convergent finding**: `code-reviewer HIGH-1` + `silent-failure-hunter HIGH-1` independently flagged.
**File**: `src/modules/payments/infrastructure/invoicing-bridge.ts:54-60` (R3v2 H-2 fix)
**Comment claim**: "will hit the 'invoice not payable' branch with status=paid/credited rather than mysteriously 500"
**Reality**: F5 `initiate-payment.ts:322` only short-circuits on `invoice.status === 'paid'`. Corrupt-total `issued`/`overdue` invoices flow past the gate into `createPaymentIntent({ amountSatang: 0n })`. Stripe rejects 0-amount card PIs (min ~50¢) and PromptPay (min THB 20) as `IntegrationError` → surfaces as `processor_unavailable` → retry storm + `payment_initiated` audit row written with wrong amount.
**Fix**: replace zero-fallback with typed `err({ code: 'corrupted_total', invoiceId: v.id })`. Add `corrupted_total` to `GetInvoiceForPaymentBridgeError` discriminated union. Propagate as `InitiatePaymentError.invoice_data_corrupt` so route handler returns a deterministic 422, not a Stripe round-trip.

### H-2 — Gateway full-refund + Stripe drift = silent $0 refund report
**Convergent finding**: `code-reviewer HIGH-2` + `silent-failure-hunter HIGH-3` independently flagged.
**File**: `src/modules/payments/infrastructure/stripe/stripe-gateway.ts:569` (R3v2 H-3 fix)
**Bug**: `refundAmount = input.amountSatang ?? asSatang(0n)` — when caller requested full refund (`input.amountSatang === undefined`) AND Stripe drift returns malformed amount, the gateway returns `amountSatang: 0n`. Refund row persists with `amount_satang = 0` against a Stripe refund that actually moved money. Breaks `succeededSumSatang` invariant → next refund attempt sees full remaining capacity → over-refund.
**Fix**: for full-refund + malformed-Stripe-amount path, return `err({ kind: 'permanent', code: 'processor_response_amount_invalid', refundId: refund.id, rawAmount: refund.amount })`. Existing out-of-band-refund sweep cron reconciles. Never persist a known-wrong value.

### H-3 — Three new catch blocks log at `warn` not `error`; missing tenantId/event.id context
**Source**: `silent-failure-hunter HIGH-2`
**Files**:
- `src/modules/payments/infrastructure/stripe/stripe-webhook-verifier.ts:93` — missing `event.id`, `stripeAccount`, `objectType`, `livemode`
- `src/modules/payments/infrastructure/stripe/stripe-gateway.ts:557` — missing `input.amountSatang.toString()`
- `src/modules/payments/infrastructure/invoicing-bridge.ts:58` — **no logger call at all**, only counter increment
**Issue**: These are invariant violations at trust boundaries (Stripe SDK type-contract break, F4 data corruption). Constitution Principle X observability classifies these as `error` not `warn`. `warn` historically = expected anomaly (retry, deprecation); `error` = should-never-happen. Counter-only emit makes Sentry triage impossible (you see the spike, can't find a sample payload).
**Fix**: Convert all 3 to `logger.error`. Add missing context fields. Bridge MUST emit structured log line.

### H-4 — C-1 omit-amountSatang cascade falsely trips refund-mismatch + writes wrong dispute audit value
**Source**: `silent-failure-hunter HIGH-4`
**Files**: `src/modules/payments/infrastructure/stripe/stripe-webhook-verifier.ts:88` ↔ `src/modules/payments/application/use-cases/process-webhook-event.ts:616, 669`
**Bug**: When C-1 catch fires, envelope omits `amountSatang`. Downstream:
1. `process-charge-refunded.ts:215` does `if (existing.amountSatang > input.amountSatang)` — with `input.amountSatang ?? 0n` defaulting to `0n`, EVERY pending refund trips `refund_amount_mismatch_detected` + skips status flip. **One fuzzed/drifted webhook = DoS-via-mismatch-audit-storm.**
2. `process-webhook-event.ts:669` writes `amount_satang: '0'` into `dispute_created` audit row — Thai RD §87 + GDPR Art. 6(1)(c) tax-adjacent audit with KNOWN-WRONG value, retained 10 years.
**Fix**: when C-1 catches, set `envelope.dataObject.amountProjectionFailed = true`. Gate `process-charge-refunded` mismatch comparison on `amountProjectionFailed === false`. Write `'projection_failed'` sentinel into dispute audit instead of `0`.

### H-5 — H-3 stripe-gateway promises `+ counter` in comment but emits NO `paymentsMetrics.*`
**Source**: `comments-reviewer H2`

### H-6 — `asSatangUnchecked` has zero direct unit tests
**Source**: `tests-reviewer HIGH-1`
**File**: `src/lib/money.ts:89-91` (new export from B-1)
**Bug**: Grep confirms no test file imports `asSatangUnchecked`. ### M-5 — Forensic vs trusted `Satang` conflated at type level
**Source**: `type-design MED-1` + `code-reviewer MED-2`
