## Executive Summary

Round 1 remediation + Q1/Q3 best-practice revisions landed cleanly across 5 files (17 traceable `post-critique` markers). The fixes are **substantively correct**, but Round 1's E15 fix (introducing `audit_log.retention_years SMALLINT NOT NULL DEFAULT 5`) **inadvertently introduces a regression in F4's compliance posture**: F4's already-shipped audit events (`invoice_paid`, `invoice_voided`, `credit_note_issued`, etc.) need the F4-tax-document subset backfilled to 10 years; without backfill, F9's future purge job would prematurely delete records the Thai Revenue Code requires retained. This is a **🎯 Must-Address** before `/speckit.tasks` because the migration is part of F5 scope and the gap was not caught in Round 1.

The remaining 10 findings are **💡 Recommendations** — none block tasks generation but several would meaningfully harden the implementation. One **🤔 Question** for stakeholder input concerns Thai-tax-compliance of partial-refund credit notes (an F4 inheritance question, not a new F5 design issue).

**Net**: Round 1 remediation worked; one new compliance regression to fix; recommend a 5-minute Round 2 remediation to add the backfill step + a few small tightenings, then `/speckit.tasks`.

## Engineering Lens Findings 🔬

### Architecture Soundness

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| R2-E1 | 💡 | **Stripe Elements `loadStripe` lifecycle**: client-side `loadStripe(publishableKey)` should be called once per browser session at module top-level (Stripe SDK best practice). If our pay-sheet component calls `loadStripe` inside `useEffect` (or worse, on every render), the SDK creates duplicate instances + memory leak. Plan documents `'use client'` boundary but not the singleton `loadStripe` pattern. | Add to plan § Project Structure note under `pay-sheet/`: "`loadStripe(publishableKey)` MUST be called ONCE at module top-level in `pay-sheet/index.tsx` (outside the React component), wrapped in a memoised getter that handles the publishableKey-from-server async load. Pattern: `const stripePromise = useMemo(() => publishableKey ? loadStripe(publishableKey) : null, [publishableKey]); return stripePromise ? <Elements stripe={stripePromise} ...> : <SkeletonShimmer />;`" |
| R2-E2 | 💡 | **Stripe Elements `locale` prop format mismatch with next-intl**: Plan says "Stripe Elements supports TH locale via the `locale` prop on the `Elements` provider — wired to `useLocale()` from next-intl". But Stripe accepts ISO codes like `'th'`, `'en'`, `'sv'`; next-intl typically returns full BCP-47 codes like `'th-TH'`, `'en-US'`, `'sv-SE'`. Direct passing causes silent fallback to English. | Add to plan § Constraints: "Wire next-intl locale → Stripe Elements via a 2-character truncation: `<Elements stripe={stripe} options={{locale: useLocale().split('-')[0] as 'th'\|'en'\|'sv'}}>`. Verify TH locale renders Thai labels in Stripe Elements card form via a Playwright snapshot test." |
