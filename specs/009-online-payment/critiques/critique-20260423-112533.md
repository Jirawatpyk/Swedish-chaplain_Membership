# Critique Report: F5 — Online Payment (Stripe + PromptPay)

**Date**: 2026-04-23 11:25:33 ICT
**Feature**: [spec.md](../spec.md)
**Plan**: [plan.md](../plan.md)
**Verdict**: ⚠️ **PROCEED WITH UPDATES**

---

## Executive Summary

F5's spec + plan are **comprehensive, constitution-aligned, and PCI-disciplined**. The 6-clarification round (3 in `/speckit.specify`, 3 in `/speckit.clarify`) meaningfully de-risked the design before plan, and the plan reuses F4's atomic-transaction pattern instead of inventing parallel state machines — the right architectural call. The Constitution Check is GREEN with 8 well-justified Complexity Tracking entries, security.md covers 16 STRIDE threats with mapped mitigations, and the SAQ-A attestation is the right level of formality for a PCI surface.

**Primary concerns**: (a) **operational gap — no plan for stale `pending` Payment rows when Stripe webhooks give up after 3-day retention** (post-MVP risk, but should be acknowledged in spec or planned for); (b) **F4 merge-ordering is implicit** — F5 implementation cannot start until F4 ships, but no explicit gate exists in tasks; (c) **manual SR pass for Stripe Elements iframe** is buried in plan prose rather than promoted to a ship checklist; (d) **a handful of FR ambiguities** (FR-018 admin-impersonate-pay scope, retention-policy implementation locus, Cache Components × Stripe Elements client-component boundary) that could bite during `/speckit.implement`. None are show-stoppers; all are "tighten before tasks" items.

**Net**: solid foundation. Apply ~6 recommended spec/plan tweaks, then `/speckit.tasks`.

### Failure Mode Analysis

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E3 | 💡 | **Stripe webhook giveup after 3-day retention**: if Chamber-OS webhook is unreachable for >3 days, Stripe permanently abandons the event. Result: a `payments` row stays in `pending` forever even though Stripe processed (or failed) the charge. Plan acknowledges webhook backlog alert > 5 min but doesn't define cleanup for the 3-day-giveup case. | Add to plan § Reliability: "Stale-pending sweep (post-MVP if metric warrants): if `payments(status='pending', initiated_at < now() - 24h)` count > 0 for any tenant, an admin alert fires (no auto-action in MVP — admin manually reconciles via Stripe dashboard + scripts/dev/replay-stripe-event.ts). MVP acceptance: rare (<1/month projected); defer scheduled sweep job to F5.0.1 if metric exceeds 1/month." Add a metric `payments.stale_pending_count{tenant}` gauge. |
| E4 | 💡 | **Stripe outage scenarios**: plan covers initiation failure ("UI shows unavailable") but not "Stripe webhooks delayed by hours". Members would see "Payment processing — we'll email your receipt within 5 minutes" indefinitely — not just temporarily. | Add to plan § Operational: "Stripe degraded-status-API check: a daily cron polls `https://status.stripe.com/api/v2/status.json`. You may close this page.' Banner persists for the duration of the degraded status." | E5 | 💡 | Webhook signature `tolerance` parameter is not explicitly configured in plan. Stripe SDK default is 5 minutes.
