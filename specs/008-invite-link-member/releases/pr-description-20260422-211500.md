## Summary

Closes the F1 spec gap at `specs/001-auth-rbac/spec.md:672-678` — the admin invite dialog at `/admin/users` now accepts an optional **"Link to member company"** picker. When an admin invites a role=`member` user and selects a member org, the backend atomically creates a pending user + invitation + contact row + link + audit, with **Hybrid A+B duplicate-email handling** to prevent orphan F1 users on any deterministic duplicate case.

Also adds a **reverse entry point** on the member detail page (`/admin/members/[memberId]`) so admins working in member context don't have to navigate back to `/admin/users` and re-search the picker.

## Implementation

### Clean Architecture

- New use case **`inviteUserForMember`** lives in `src/modules/members/application/use-cases/` — the auth module stays clean of member/contact concerns. The API route at `src/app/api/auth/invite/route.ts` is the only orchestration seam; it branches on the presence of `memberId` and calls either F1 `createUser` (standalone) or the new members-module use case (linked).
- `CreateUserPort` adapter keeps the dependency direction correct: members depends on the port, not on the F1 module.

### Hybrid A+B duplicate handling

Pre-tx lookup with `contactRepo.findByEmail` decides the path before `createUser` runs, so no F1 orphan is ever created for deterministic duplicates:

| Lookup result | Behavior | HTTP |
|---|---|---|
| not found | Create new contact + link user (path A) | 201 |
| found, same member, `linked_user_id = NULL` | Link existing contact (skip addInTx); emit `contact_linked_to_user` audit | 201 |
| found, same member, `linked_user_id != NULL` | Reject `contact_already_linked` | 409 |
| found, different member in same tenant | Reject `email_belongs_to_other_member` | 409 |

### Tenant isolation (Constitution Principle I)

- `findByEmail` runs under `runInTenant` with explicit app-layer `tenantId` filter (two-layer isolation)
- Defensive tenant-mismatch guard in the use case (comment cites Principle I as a Review-Gate blocker so future readers don't simplify it away)
- `member_cross_tenant_probe` audit emitted **before** `createUser` on cross-tenant memberId, so cross-tenant probes never orphan a user

### Concurrency / reliability

- `linkUserInTx` uses **atomic UPDATE with `isNull(linkedUserId)` guard** — eliminates the SELECT-then-UPDATE TOCTOU race. `0 rows updated` → targeted follow-up probe distinguishes `repo.not_found` from `repo.conflict`.
- `listLinkedUserIdsForMemberInTx` wrapped in try/catch; returns `[]` on infra failure with pino logger output so the archive-cascade tx is never blocked by an observability failure.
- Route-level exhaustiveness check (`assertNever`) on the error switch — adding a new `InviteUserForMemberError` variant is a compile-time error at the handler, preventing silent 500 responses.
