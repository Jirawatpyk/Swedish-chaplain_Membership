## Findings — 🟡 Warnings (7)

### W-01 — `POST /generate-secret` has no rate limit (Security DoS hardening)

| File:line | Pillar |
|---|---|
| `src/app/api/admin/integrations/eventcreate/generate-secret/route.ts:21-86` | Principle I — Security defence-in-depth |

**Issue**: `rotate-secret/route.ts:42-55` and `test-webhook/route.ts:46-59` both apply per-`(tenant, actor)` Upstash rate-limit gates (3/hr and 10/hr respectively). `generate-secret/route.ts` is the only mutating wizard endpoint with NO rate-limit gate. The 409 `secret_already_exists` idempotency bound limits DB writes after the first call, but every additional invocation still consumes Upstash budget that other endpoints share — an attacker who compromises a single admin session can hammer the endpoint to exhaust the auth-rate-limit budget for legitimate admin traffic, while also incurring a SELECT (conflict probe) per hit.

**Impact**: Medium — admin-only endpoint, attack bar is high (requires compromised admin session). 409-after-first call limits damage to Upstash budget exhaustion rather than DB write amplification.

**Recommended fix**: Add a `generateSecretRateLimitCheck(tenantSlug, actorUserId)` factory in `src/lib/events-admin-integration-deps.ts` mirroring `rotateSecretRateLimitCheck` (3/hour per `(tenant, actor)` should suffice — fresh tenant onboarding is a one-shot action). Gate it in the route before `runGenerateWebhookSecret`. ~15 LoC + 1 unit test.

### W-02 — `runRunTestWebhook` outbound fetch has no AbortController timeout

| File:line | Pillar |
|---|---|
| `src/lib/events-admin-integration-deps.ts:~493` (composition adapter `httpFetch` wiring) | Principle VIII — Reliability |

**Issue**: The `runTestWebhook` use-case is pure Application; the composition adapter wires it to the global `fetch`. No `signal: AbortSignal.timeout(N)` is attached. If the webhook receiver hangs (advisory-lock contention, cold-start of Vercel function targeting itself, Neon connectivity glitch), the admin route blocks until the Vercel platform timeout (~30s) before returning, with no actionable error code surfaced to the admin.

**Impact**: Medium UX degradation — admin sees a generic timeout instead of an early "receiver unreachable" hint. Functional reliability is fine (the error eventually surfaces) but pre-flag-flip polish.

**Recommended fix**:
```ts
// events-admin-integration-deps.ts in the runTestWebhook wiring
httpFetch: async (url, init) => {
  return fetch(url, { ...init, signal: AbortSignal.timeout(10_000) });
}
```
Add a corresponding unit test that injects a slow `httpFetch` and asserts `failureCategory === 'network_error'` after 10s. ~5 LoC + 1 test.

### W-03 — Recent-deliveries panel uses `<ul>/<li>` for tabular data (WCAG 1.3.1)

| File:line | WCAG |
|---|---|
| `src/components/events/recent-deliveries-panel.tsx:~157-186` | 1.3.1 Information and Relationships (Level A) |

**Issue**: The panel renders timestamp + request-ID + signature-outcome badge + processing-outcome badge per row as `<li>` elements with CSS flexbox. The data is genuinely tabular (4 columns). Assistive technology users get rows announced as undifferentiated list items with no column header context — when a screen reader reads the "verified" badge, it does not know which column the badge belongs to.

**Impact**: WCAG 2.1 AA Level A violation on a Phase 5 surface. Constitution Principle VI requirement.

**Recommended fix**: Replace `<ul>/<li>` with `<table>` + `<thead><tr><th scope="col">...</th></tr></thead> + <tbody>` semantics. Each badge column should carry a visible or `<span class="sr-only">` header so SRs announce "Signature: Verified" rather than "Verified" in isolation. Keep current visual design — only the underlying semantics change.

| File:line | WCAG |
|---|---|
| `src/components/events/rotate-secret-dialog.tsx:~119-151` | 2.4.3 Focus Order + 3.3.4 Error Prevention (rotation is destructive) |
