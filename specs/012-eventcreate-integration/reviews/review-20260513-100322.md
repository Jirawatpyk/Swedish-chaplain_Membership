## Findings — 🟡 WARNINGS (2)

### R2-A — Skeleton inline `<span aria-hidden>` may collapse block layout (UX-05 from agent)

| Severity | File:line | WCAG |
|---|---|---|
| 🟡 WARN | `src/app/(staff)/admin/events/loading.tsx:18` + `src/app/(staff)/admin/events/[eventId]/loading.tsx:21` | layout regression risk (no WCAG criterion violated; aesthetic) |

**Issue**: W11 wrapped `<Skeleton className="h-7 w-72" />` in `<span aria-hidden>` inside the PageHeader title slot. `<span>` is `display: inline` by default. The inner `Skeleton` (block-display via shadcn) may collapse to 0 height inside an inline parent on some browsers because an inline container does not establish a block-formatting context.

**Risk**: Cosmetic — the shimmer phase is ≤500ms typically; even if collapsed, the page-shift settles when real content arrives. But the original CLS-0 contract may not hold during the shimmer.

**Fix**: `<span aria-hidden="true" className="block">` (or use `<div aria-hidden>`) on both loading.tsx files. ~5 LoC.

---

### R2-B — `aria-busy` referenced in comment but not present in JSX (UX-06 from agent)

| Severity | File:line | WCAG |
|---|---|---|
| 🟡 WARN | `src/app/(staff)/admin/events/[eventId]/loading.tsx:11-13` (comment) | 4.1.3 (Status Messages, A) — gap |

**Issue**: Inline comment claims `aria-busy=true` is applied to the container, but no `aria-busy` attribute appears in the JSX. AT users currently get silence during the shimmer (acceptable improvement over the original VoiceOver "heading level 1 with no label" issue), but the promised positive loading signal is absent.

**Fix**: Add `aria-busy="true"` to the outermost `<DetailContainer>` and `<TableContainer>` wrappers. Verify the container components forward unknown props (likely already true via standard React pass-through). ~2 LoC + 1 container-component prop verification.

## Findings — 🟠 MEDIUM (3)

### R2-C — redactStack missing `/srv/`, `/data/`, `/run/` container paths

| Severity | File:line |
|---|---|
| 🟠 MED | `src/lib/redact-stack.ts:35-37` |

**Issue**: The redactStack alternation covers `/var/`, `/usr/`, `/home/`, `/opt/`, `/tmp/`, `/root/`, `/users/`, `/private/`, `/node_modules/`. Modern container conventions also use `/srv/` (Docker app dirs), `/data/` (Kubernetes PVC mounts), `/run/` (systemd runtime). Vercel sin1 uses `/var/task/` so current deployments are unaffected, but a future containerisation change (e.g., self-hosted) could expose these prefixes.

**Fix**: Extend regex group: `/...(var|usr|home|opt|tmp|root|users|private|node_modules|srv|data|run).../`. 1 LoC.

**Status**: HARDENING — not a current vulnerability; defensive addition.

### R2-D — Migration 0136 atomicity (CLOSED on verification)

| Severity | File:line |
|---|---|
| 🟢 CLOSED | `drizzle.config.ts` + `drizzle/migrations/0136_*.sql` |

### R2-E — Missing alert rule for `webhook_ingest_precondition_failed` (SEC-14 + REL-01 duplicate)

| Severity | File:line |
|---|---|
| 🟠 MED | `docs/observability.md § 24.3 Alerts` |

**Issue**: W5 added the new audit event type `webhook_ingest_precondition_failed`, distinct from `webhook_rolled_back`. The taxonomy split is correct — but the observability section's alert table does NOT define a P2 alert for this new event type. Sustained Neon-connectivity blips causing repeat config-load failures would silently accumulate in `audit_log` until eventually surfacing through the weaker `f6_audit_fallback_double_failure` signal.

**Fix**: Add one row to `docs/observability.md § 24.3`:
```
| f6_webhook_precondition_burst | rate(audit_event_type='webhook_ingest_precondition_failed') > 2/min per tenant for 5 min | P2 | docs/runbooks/f6-webhook-precondition-burst.md (Phase 10) |
```
1 markdown row, no code change.

## Findings — 🟢 LOW (2)

### R2-F — B7 audit emit should use `safeEmitStandalone` helper for pattern consistency

| Severity | File:line |
|---|---|
| 🟢 LOW | `src/app/api/admin/events/[eventId]/route.ts:195-226` |

**Issue**: The B7 cross_tenant_probe audit emit uses a bare `try/catch` rather than the established `safeEmitStandalone` helper. Functionally identical (audit failure does not block 404), but the helper adds structured `logEvent` / `logMsg` fields + consistent error reporting pattern that other F6 standalone emits use (`route.ts:451` for `webhook_ingest_precondition_failed`, `_lib/role-violation-audit.ts` for FR-035).

**Fix**: Refactor B7 emit to call `safeEmitStandalone(makeStandaloneAuditDeps(), entry, { tenantSlug, logEvent: '...', logMsg: '...' })`. ~10 LoC refactor.

### R2-G — Admin route X-Request-ID not capped before audit payload

| Severity | File:line |
|---|---|
| 🟢 LOW / 🟠 MED | `src/app/api/admin/events/[eventId]/route.ts:212` |

**Issue**: W1 applied `.slice(0, 200)` to `X-Request-ID` in the webhook route (public, unauthenticated). The admin detail route's B7 audit emit reads `request.headers.get('x-request-id')` directly without the same cap — a crafted oversized header from an authenticated admin actor would write unbounded text into `audit_log.payload.requestId` (JSONB column, no DB-level size limit).

**Risk**: Lower than W1 (requires authenticated admin session, raising the attack bar significantly), but the inconsistency is a minor defect.

**Fix**: Apply `.slice(0, 200)` to the header read in admin detail route line 212. 1 LoC, same pattern as W1.
