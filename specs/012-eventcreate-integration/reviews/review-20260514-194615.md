## Findings — 🟢 Suggestions (2)

### SUGG-R5-1 — WARN-4 Test 2 (Jan 1 FY 2027) has conditional FY assertion (vacuous on over-quota fallback)

| Severity | File:line | Principle |
|---|---|---|
| 🟢 SUGG | `tests/integration/events/quota-accounting.test.ts:515-563` | II (Test-First — assertion tightness) |

**Issue**: The second FY boundary test asserts `expect(payload.fiscalYear).toBe(2027)` only **conditionally** — wrapped in `if (String(row.eventType) === 'quota_cultural_decremented')`. The comment at L515-521 explains: "If quota is exhausted, a `quota_over_quota_warning` carries the scope='cultural'. Either path is acceptable; we only assert the FY when a decrement audit was emitted."

This is defensible for a real reason — the `Helen Premium` member shared with the earlier AS3 test may have consumed cultural quota in the prior 2 tests (AS3 + WARN-4 Test 1 both ran first, both at FY 2026, 2/2 allotment consumed). But it creates a vacuous-assertion failure mode: if `deriveFiscalYear` regresses and returns **2026 instead of 2027** for the Jan 1 BKK input, the use-case sees consumed=2/2 for FY 2026 → over-quota path fires → no FY assertion runs → test PASSES despite the bug.

The intent of WARN-4 was specifically to lock the off-by-one-year guard via js-joda. The test as written would not catch a UTC-vs-BKK regression in the over-quota path.

**Recommendation** (≤30 LoC):
Either (a) use a fresh tenant + fresh member for Test 2 (so the FY 2027 allotment is genuinely unconsumed → decrement path is the only valid outcome → tighten assertion to `expect(matchingAudits[0].eventType).toBe('quota_cultural_decremented')` + `expect(payload.fiscalYear).toBe(2027)`), OR (b) keep the shared member but use a Premium-12 plan for the second test so FY 2027 is unambiguously fresh.

Option (a) is cleaner — adds ~20 LoC of beforeEach setup but produces a tighter regression guard.

**Status**: Test passes today on live Neon; the assertion gap is a soft-spot for a specific class of future regression. Non-blocking.

### SUGG-R5-2 — Metric dispatcher only wired on `emit()`, silently skipped on `emitStandalone()`

| Severity | File:line | Principle |
|---|---|---|
| 🟢 SUGG | `src/modules/events/infrastructure/pino-audit-port.ts:65-121, 283, 367-385` | VII (Observability) — future-proofing |

**Issue**: `emitMatchingQuotaMetric(entry)` is invoked from `emit()` (line 283) but NOT from `emitStandalone()` (line 367+). Today this is correct: `emitStandalone` is only used by `webhook_signature_rejected` from the route handler (per its docblock at L367-373) — never for a quota event. The metric dispatcher's `default: break` would have ignored it anyway.

But a **future developer** adding a standalone quota emission (e.g., a manual data-recovery script that emits `quota_credit_back_relink` outside a use-case tx) would silently miss the OTel counter increment. The dashboard would under-count without any signal.

**Recommendation** (≤5 LoC):
Move the `emitMatchingQuotaMetric(entry)` call into the shared insert-row success path, OR add the call to `emitStandalone()` after `insertAuditRow` succeeds at line 384, mirroring the `emit()` pattern. Either change keeps the metric/audit-row coupling consistent regardless of which port method emits the event.

**Status**: No current emit site triggers this gap. Pre-emptive hardening for F6.1 (relink + manual recovery surfaces).
