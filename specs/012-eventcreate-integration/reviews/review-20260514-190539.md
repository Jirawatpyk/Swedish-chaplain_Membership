### WARN-2 — Drizzle quota-accounting adapter lacks explicit `tenantId` filter (single-layer enforcement)

| Severity | File:line | Principle |
|---|---|---|
| 🟡 WARN | `src/modules/events/infrastructure/drizzle-quota-accounting-adapter.ts:82-98` | I (Tenant Isolation, NON-NEG, v1.4.0 clause 2) |

**Issue**: The members×plans lookup at L82-98 uses `WHERE eq(members.memberId, input.memberId)` with NO explicit `eq(members.tenantId, input.tenantId)`. The `TenantTx` executor has `SET LOCAL app.current_tenant = ...` set, so RLS+FORCE on `members` (migration `0008_members_rls.sql`) enforces tenant scope at the DB layer.

Constitution v1.4.0 Principle I (NON-NEG) clause **2** mandates **two-layer** tenant isolation: application-layer (explicit `WHERE tenant_id = ?`) **plus** database-layer (RLS). The F2/F3 precedent in `plan-lookup-adapter.ts:38-62` and `drizzle-member-repo.ts` applies BOTH. The F6 quota adapter currently relies on RLS alone, leaving the app layer single-line-of-defense.

The architect agent passed this on the strength of RLS + composite FK constraint — which is correct as a runtime safety guarantee. The reliability of an RLS GUC being `SET LOCAL` correctly is high (transaction-scoped). However, defense-in-depth would catch a future regression where someone forgets to wrap a call in `runInTenant` — the explicit filter would still scope the query, the RLS-only path would return rows.

**Recommendation**: Add `eq(members.tenantId, input.tenantId)` to the WHERE clause (~1 line):
```ts
.where(and(
  eq(members.tenantId, input.tenantId),
  eq(members.memberId, input.memberId),
))
```
Plus parallel `eq(membershipPlans.tenantId, ...)` is already on the LEFT JOIN condition (L91) — adapter is half-right already; just complete the symmetry on the outer WHERE.

**Status**: Defense-in-depth. **Should close before per-tenant flag-flip**.

## Findings — 🟢 Suggestions (4)

### SUGG-1 — `archive-event.ts` step order: snapshot-then-setArchived ordering nit

| Severity | File:line | Principle |
|---|---|---|
| 🟢 SUGG | `src/modules/events/application/use-cases/archive-event.ts:150-175` | VIII (Reliability — defense-in-depth) |

**Issue**: Current order is `listForRequota` (step 2, line 153) BEFORE `setArchived` (step 3, line 171). Concurrent ingests for the same `(member, event)` are blocked by the advisory lock acquired inside `applyQuotaEffect`, so a new counted-true row cannot land between the snapshot and the archive UPDATE — this is the actual safety mechanism today. If a future refactor changed lock semantics (non-blocking `tryLock` or timeout), the implicit "no new counted row possible" invariant would silently shift to "post-archive snapshots are inherently empty for new counted rows" (which would be the stronger invariant).

**Recommendation**: Swap step 2 ↔ step 3 — `setArchived` first, `listForRequota` second. Update docstring. ~5 LoC + 2 comment lines. This makes the invariant explicit at the code level rather than implicit via lock semantics.

**Status**: Optional. Current behaviour is correct.

## Top Strengths

**1. `LockKey` branded type prevents silent namespace-collision class of bugs.**
The `asLockKey` smart constructor + `LOCK_KEY_REGEX = /^[a-z0-9-]+(:[A-Za-z0-9._-]+){1,4}$/` + 256-char cap + control-char rejection (\n/\r/\t/\x00) is a defensive measure that F4/F5/F7/F8 do NOT have. The unit test `as-lock-key.test.ts` (192 LOC, 15 tests) covers every documented boundary including the canonical typo `eventcreate_quota:` (underscore) vs `eventcreate-quota:` (hyphen). This is the most defensive advisory-lock primitive in the codebase.

**2. Discriminated `wrapLockFailure` separates retry-eligible from programmer-error.**
Splitting `lock_acquisition_failed` (transient, retry-eligible) from `lock_key_invariant_violation` (programmer error, page on-call, never retry) in `error-wrappers.ts:87-106` plus non-Error throw normalisation is operationally critical. The prior common-bucket pattern (which F4/F5/F7/F8 still use) would cause an SRE retry runbook to loop forever on `InvalidLockKeyError`. Combined with the route handler `err:` key (auto-serialised by pino) replacing the prior `cause:` key (which serialised as `{}` for non-Error throws), this closes a quietly-broken log-quality gap.

**3. Pool-deadlock fix is sound, documented, and verified.**
`drizzle-quota-accounting-adapter.ts:14-32,82-98` keeps ALL three reads (members JOIN plans + 2 COUNT) on the single TenantTx executor that holds the advisory lock. The comment block explicitly documents why opening a second connection (via the obvious `buildPlansDeps(ctx).planRepo.findOne(...)` path) would deadlock at 10 concurrent workers given the postgres-js pool default of ~10. The `quota-concurrency.test.ts` provides behavioural verification under deterministic 10-worker concurrent ingest. This is the kind of subtle systems-level decision that would not be obvious to a reviewer without the comment — well-engineered defence-by-design.
