# Critique Report: F6 — EventCreate Integration

**Date**: 2026-05-12
**Feature**: [spec.md](../spec.md)
**Plan**: [plan.md](../plan.md)
**Verdict**: ⚠️ **PROCEED WITH UPDATES**

---

## Executive Summary

F6 is a **well-scoped, low-novelty integration feature** that re-uses the F1–F8 stack with discipline (zero new npm dependencies, ~35 audit events, 3 tables, 8 migrations). The spec is one of the tighter ones in the repo — 7 user stories with clear independence, 12 success criteria, 7 clarifications resolved across 2 sessions. The technical approach is sound at the macro level: Zapier-mediated HMAC webhooks, strict-transactional ingest, differentiated PDPA/GDPR retention, F8 port adapter swap, RBAC matrix.

However, the validation pass surfaced **2 critical schema-layer bugs** (F1 audit_log uses a Postgres enum that F6 plan didn't budget a migration for; `severity` envelope field has no backing column) plus **3 internal contradictions** (quota concurrency model, RBAC 403/404 drift, F8 port method name) that would land as confusion in `/speckit.tasks` and rework in `/speckit.implement`. From a product perspective, the feature scope is well-bounded but two MVP-scope decisions deserve a second look (CSV fallback as US5 P2; manual relink as US6 P3) — both add tasks/code surface for arguably-deferrable value.

**Net assessment**: solid foundation, fixable issues. Suggested 4–6 hours of spec/plan refinement closes every must-address finding; the recommendations + questions are non-blocking and can be parked for `/speckit.staff-review`.

### Failure Mode Analysis

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E3 | 🎯 | **`webhook_rolled_back` audit in separate post-rollback tx** assumes the DB is available for the separate tx. If the rollback was caused by DB unavailability (the most common cause), the audit emission also fails — and observability is permanently lost for that delivery. Zapier retries, so the data is recoverable, but the **first failure is invisible** until the retry succeeds. | Add a fallback path: when the separate-tx audit emit fails, write to `stderr` via `pino.fatal(...)` (which Vercel captures as runtime log). Document in research.md R6 that "failed-delivery audit has two layers — DB audit + structured stderr log". Pino logs are durable enough to detect the issue even without DB. |
| E4 | 💡 | What happens if Zapier delivers a webhook for a tenant whose `tenant_webhook_configs` row was deleted (admin offboarded the tenant)? FR-001 says endpoint exists per tenant slug; FR-006 says tenant resolved from URL path. But no spec line covers "tenant config row missing" → presumably 401 (no secret to verify). | Add an FR or explicit case: when `tenant_webhook_configs` row is missing OR `enabled = FALSE`, return 401 (NOT 503 — 503 says "retry later", we want "go away"); audit `webhook_signature_rejected` with note `config_missing`. Or 503 if `enabled=FALSE` per FR-033. Clarify the distinction. |
| E5 | 💡 | CSV import handler runs inline within one function execution. Vercel Fluid Compute has a 300s timeout default but **memory is 1 GB**. A 5 MiB CSV with 5,000 rows + per-row Drizzle ORM round-trip + per-row audit emit could spike memory. No memory budget mentioned. | Add a memory measurement task to plan.md Phase 9 polish: "Profile CSV import at 1k + 5k rows; report peak heap; fail-fast at 500 MiB". Currently the implicit assumption that 5 MiB raw → bounded heap may not hold. |

### Security & Privacy

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E6 | 🎯 | **`audit_log.event_type` is a Postgres enum** (F1 schema, `auditEventTypeEnum`) — extended by F4 via `ALTER TYPE … ADD VALUE … ON CONFLICT DO NOTHING`. F6 plan migration 0132 says "no CHECK constraint; taxonomy enforced by TS union" — **false**. Without ALTER TYPE migrations, INSERT into audit_log will fail with `invalid input value for enum` on every F6 audit emit. | Replace migration 0132's "descriptive comment only" with 35 × `DO $$ BEGIN ALTER TYPE audit_event_type ADD VALUE 'webhook_receipt_verified'; EXCEPTION WHEN duplicate_object THEN NULL; END $$;` pattern (F4 precedent). Each ALTER must be its own DO-block (Postgres limitation on running enum additions in tx). |
| E7 | 🎯 | **`severity` column in `AuditEnvelope`** (contracts/audit-port.md:21) doesn't exist in `audit_log` schema. F1 schema has: id, timestamp, eventType, actorUserId, targetUserId, sourceIp, summary, requestId. No subsequent migration (F4/F5/F7/F8) adds severity. F6's emit will either fail or silently drop the field. | Choose: (a) add migration to extend `audit_log` with `severity` column + backfill default `info`, OR (b) drop `severity` from envelope and pack it inside `payload` JSONB. Recommend (b) — minimal surface change; `payload.severity` is queryable via Postgres `->>'severity'` for alert rules. Update contracts/audit-port.md accordingly. |
| E8 | 💡 | HMAC verify path uses `crypto.timingSafeEqual` (research.md R2). The Node implementation throws on length mismatch, not returns false. If the signature header is missing the `sha256=` prefix or is truncated by Zapier, the verify path crashes (HTTP 500, not the documented 401). | In `verify-webhook-signature.ts`, normalise + length-check before passing to `timingSafeEqual`. Wrap in try/catch and treat any thrown error as signature rejection. Add unit test for: (a) missing `sha256=` prefix, (b) truncated signature, (c) signature too long, (d) malformed hex. |
| E9 | 💡 | The `webhook_secret_active` is stored as plaintext TEXT in `tenant_webhook_configs.webhook_secret_active`. While Neon encrypts at rest (AES-256), an SQL-injection or DB-credential-leak attacker reads the raw secret. Industry pattern (Stripe, GitHub) stores webhook secrets at rest as **hashed**, with a fresh-rotation flow to allow access only at one-time-reveal moment. | Reconsider: hashing the secret prevents replay-from-DB-leak but breaks HMAC verify (we need the raw value to compute HMAC). Reasonable v1 trade-off is to keep raw + accept the risk. But document the threat model in security: "DB-credential leak → tenant secret leak → attacker can sign fake webhooks until rotation". Mitigated by short rotation window + signature audit alerts. |
| E10 | 💡 | PDPA/GDPR pseudonymisation is deterministic salted hash. Re-identification via brute-force (the attacker hashes every email they know and tries) is possible because the salt is per-environment, not per-record. For non-member attendee emails, this matters. | Document this in research.md R9 explicitly: "Pseudonymisation is GDPR Art. 4(5) pseudonymisation (not anonymisation). Re-identification by brute-force-against-known-email-list is mitigated by (a) the per-tenant salt being kept secret, (b) the salt rotation policy at security-incident or every-N-years cadence." Currently the security trade-off is undocumented. |

### Performance & Scalability

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E11 | 🎯 | **Quota concurrency model contradicts itself**. research.md R5 simultaneously says (a) `FOR UPDATE on the counter row` and (b) "Boolean flag + computed counter (not stored counter)". A computed-on-read counter has no row to lock; two concurrent writers will both observe "1 slot remaining" and both write `counted=true` → 7 against a 6-allotment race. | Pick one strategy and document it: (1) `pg_advisory_xact_lock(hashtext('eventcreate-quota:' || tenant_id || ':' || matched_member_id || ':' || event_id))` at the start of `apply-quota-effect.ts` — matches F5's TOCTOU guard pattern; **namespace `eventcreate-quota:` is disjoint** from F4 `invoicing:` / F5 `payments:` / F7 `broadcasts:` so no contention; (2) introduce a stored counter (members table column) with `FOR UPDATE` row lock. Recommend (1) — fewer schema changes. Update research.md R5 + data-model.md § 5 + add to plan.md migration list if (2). |
| E12 | 💡 | Hand-rolled Levenshtein at <2k-member scale (research.md R4) is fine, but the design envelope (plan.md Scale/Scope) targets 5,000 members × multiple tenants. At 5k × 5k pairwise (worst case for a fuzzy search), the inner-loop count is 25M iterations per import; in JS, that's ~250ms per attendee × 500 attendees per event = 125s. **At medium-chamber scale, the hand-rolled fuzzy may breach SC-003's 300ms p95 webhook budget.** | Measure first: add a perf benchmark task in plan.md Phase 10 — "Levenshtein-against-members benchmark at 5k members; assert <50ms p95". If it fails, swap to: (a) pre-computed normalised names with simple equality check + only-fuzzy-on-domain-personal-email cases (smaller set), or (b) Postgres pg_trgm fuzzy search (mentioned as rejected alternative — reconsider). |
| E13 | 💡 | Webhook handler is 60 req/min rate-limited per tenant (FR-005). But Zapier's free-tier polling is "every 15 min" — so a tenant has at most 4 deliveries per hour per Zap. The 60 req/min limit is 900× over what Zapier could realistically deliver. Either the rate limit is mis-sized (should be lower) or there's an attack vector I'm not seeing (someone replaying with valid signature — but `X-Request-ID` idempotency catches that). | Lower FR-005 to 10 req/min/tenant. It's still 150× Zapier free-tier capacity and gives operational headroom for paid-tier Zapier (1-min cadence = 60/hr per Zap = 1 req/min). Keeps the rate-limit-burst alert (research.md R10 alert #4) at a meaningful threshold. |
