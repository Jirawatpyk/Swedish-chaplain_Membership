## Executive Summary

Round 1 closed 6 must-address + 18 recommendations + 5 deferred questions, materially improving the spec/plan. Round 2 verifies the remediation **and uncovers two NEW critical issues introduced by the round-1 fixes themselves** — both are schema-coordination errors where the round-1 text references project artefacts (a column, a table) under the wrong name or non-existent path. These would land as runtime SQL errors during `/speckit.implement` if not corrected.

Specifically:

- **M1 (CRITICAL)**: The E7 fix described F6's structured audit payload as "encoded into the existing `summary` TEXT column (≤500 chars)" — but F2 migration 0007 already added a proper `audit_log.payload jsonb` column. F4/F5/F7/F8 all use it. F6 must use `payload jsonb`; the round-1 fix's "store in summary" text is **wrong** and would lead to a lossy + non-queryable implementation.
- **M2 (CRITICAL)**: Throughout F6 plan + data-model + contracts, the idempotency strategy claims to **reuse an F5-introduced `idempotency_receipts` table** with a new `source = 'eventcreate_webhook'` discriminator. **This table does not exist in F5.** F5's idempotency is `processor_events`, which is Stripe-specific (PK is the Stripe event_id; columns shaped for Stripe). The "reuse F5's idempotency_receipts" claim is fabricated; F6 needs to introduce its own idempotency table.

Otherwise, the spec is in strong shape: the round-1 fixes for E6 (audit_event_type enum), E11 (advisory lock pattern), E16 (canonical method name), E17 (RBAC), and E3 (stderr fallback) all landed correctly. The 18 recommendations are integrated cleanly, and the 5 deferred questions are resolved with concrete commitments.

Round 2 also surfaces 5 lighter recommendations (R1–R5 below) — all non-blocking, none introduced by round 1.

### Failure Mode Analysis

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| R2 | 💡 | **Advisory lock + RLS context-binding ordering**. Research.md R5 specifies the advisory lock pattern but doesn't state the SQL execution order vs. `SET LOCAL app.current_tenant` (which must happen at the start of the tx for RLS to bind correctly). The wrong order (`pg_advisory_xact_lock` BEFORE `SET LOCAL`) would acquire the lock under a default tenant context, possibly leaking lock semantics if the SET fails. | Document the canonical order in research.md R5 as a code-shaped snippet: <br>```sql<br>BEGIN;<br>SET LOCAL app.current_tenant = $tenantId; -- RLS bind FIRST<br>SELECT pg_advisory_xact_lock(hashtextextended('eventcreate-quota:' || $tenantId || ':' || $memberId || ':' || $eventId, 0)); -- THEN advisory lock<br>-- ... query plan, compute consumed count, insert registration, emit audit ...<br>COMMIT; -- both lock + tenant binding released atomically<br>```<br>Note: per [Postgres docs](https://www.postgresql.org/docs/current/explicit-locking.html#ADVISORY-LOCKS), advisory locks are session-scoped (not tenant-scoped at the DB level), so `runInTenant(ctx, fn)` opening a fresh connection per tenant call is the correct primitive — adversarial cross-tenant lock probes would fail tenant-resolution earlier. |
