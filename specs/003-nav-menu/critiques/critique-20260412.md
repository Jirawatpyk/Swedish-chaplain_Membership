# Critique Report: Navigation Menu (003-nav-menu)

**Date**: 2026-04-12  
**Spec**: [spec.md](../spec.md) | **Plan**: [plan.md](../plan.md)  
**Verdict**: ⚠️ **PROCEED WITH UPDATES**

## Executive Summary

The spec and plan are solid for a well-scoped UI infrastructure feature. No constitution violations or fundamental architecture flaws. Three recommendations warrant updates before proceeding: (1) a hydration mismatch risk for sidebar collapse state that could cause visible layout shift (CLS), (2) an inaccurate i18n key count in the plan, and (3) the Dashboard active-state matching needs an explicit edge-case rule. No must-address blockers — all items are resolvable with targeted edits.

## Engineering Lens Findings

### 4a. Architecture Soundness

### 4b. Failure Mode Analysis

| ID | Finding |
|----|---------|
| E1 | **Hydration mismatch / layout shift (CLS)**: Sidebar collapse state is read from `localStorage` (client-only). Server-side rendering has no access to `localStorage`, so the server will render the sidebar in its default state (expanded). On hydration, the client reads `localStorage` and may flip to collapsed — causing a visible layout shift. This violates SC-007 ("no flash of incorrect state") and potentially Principle VII (CLS < 0.1). **Mitigation**: Use a cookie to sync collapse preference to the server, OR render a CSS-only skeleton that defers sidebar width until client hydration completes, OR accept the default-expanded SSR with a client-side transition. |
| E2 | **localStorage unavailable (private browsing / Safari ITP)**: If `localStorage` throws (quota exceeded, private browsing in older Safari), the sidebar should gracefully default to expanded without crashing. The spec mentions this implicitly but the plan doesn't document a fallback strategy. |

## Verdict: ⚠️ PROCEED WITH UPDATES

No must-address blockers found. All 7 findings are 💡 Recommendations — the spec and plan are fundamentally sound. The highest-impact items are:

1. **E1/X1 — Hydration CLS**: Add a note to the plan about using a cookie for SSR sidebar state sync (or accepting the tradeoff)
2. **E3 — i18n count**: Fix the inaccurate number in plan.md
3. **P2/X2 — Dashboard exact-match**: Add explicit edge case to spec

### Suggested Changes

**spec.md** — Add to Edge Cases:
```
- What happens when the user is on the Dashboard (`/admin`) page? Dashboard must be active only on exact `/admin`, not on sub-paths like `/admin/plans`. The active-state matching must treat Dashboard as an exact-match exception or use "deepest prefix wins" to resolve correctly.
```

**plan.md** — Fix Scale/Scope:
```
**Scale/Scope**: ~12 new/modified files. ~15 new i18n keys × 3 locales (~45 entries). 0 database migrations.
```

**plan.md** — Add to Technical Context after Performance Goals:
```
**SSR State Sync**: Sidebar collapse preference is synced to the server via a lightweight cookie (`sidebar:state`) to prevent hydration mismatch / layout shift (CLS). The cookie is read in the server layout to render the correct initial sidebar state. Graceful fallback: if localStorage or cookie is unavailable, default to expanded.
```
