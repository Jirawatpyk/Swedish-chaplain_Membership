## Executive Summary

Round 1's 3 🎯 Must-Address findings all resolved cleanly:
- **P1 (production-data gap)** → Strategy B split shrank scope from 8→3 USs; P2/P3 deferred to F7.1b
- **E13 (F3 backfill scale ceiling)** → US3 deferred with F7.1b; the migration that breaks at scale is no longer in F7.1a
- **X1 (US5 low-value + high-complexity)** → US5 deferred to F7.1b

US7 (Multi-template library) was promoted BACK into F7.1a after maintainer committed to writing starter content directly — net result is a tighter 3-US scope with clear engineering economy (US7 reuses 100% F7 MVP infra) and high product confidence (starter library makes US7 valuable from day 1).

The i18n-translation-reviewer pass surfaced 5 must-fix + 3 recommendations on `starter-templates.md` — all 8 applied. Templates are now ship-ready in EN+TH+SV with correct genitive grammar (SV), consistent greeting register (TH), and accurate semantic content (T3 sponsorship-feature CTA was originally a mistranslation; now correct).

Two new 🎯 Must-Address findings emerge from the round 2 scope:

1. **Engineering 🎯 Must-Address (E1)**: **Template variable substitution semantics are undefined.** `starter-templates.md` uses `{{chamber_name}}`, `{{member_name}}`, `{{event_name}}`, `{{month_year}}`, etc. but neither spec.md nor plan.md nor research.md defines WHO resolves these variables, WHEN, and HOW. Three open questions: (a) Where does `{{member_name}}` come from when a broadcast targets a SEGMENT (no single member context)? (b) Are values HTML-escaped before substitution (XSS risk if member supplied)? (c) What happens when a variable doesn't resolve — empty string, literal `{{var}}` rendered, or error? Without explicit rules, the implementation will pick something at random and a security/UX bug ships.

2. **Engineering 🎯 Must-Address (E2)**: **`starter-templates.md` ↔ migration 0134 single-source-of-truth ambiguity.** `starter-templates.md` is the human-readable content source, but the actual seed migration (`0134_f71a_default_template_seed.sql`) needs the same content embedded as SQL INSERT values. If a maintainer edits `starter-templates.md` after migration 0134 ships, the file and DB drift silently — no CI check catches it. The cleanest solution: either (a) migration 0134 is auto-generated from `starter-templates.md` via a script committed alongside (`scripts/generate-template-seed-migration.ts`), OR (b) `starter-templates.md` is marked "FROZEN — edit the migration directly" and a CI script verifies it matches migration 0134 content.

Lesser concerns: template snapshot timing UX (member-locale vs admin-locale vs draft-time-locale choice), variable-substitution test coverage gap, template-rollback rows orphaning post-rollback, picker UX scaling at >50 templates, and a documentation gap on tenant-locale template-default ordering.

The plan addresses Constitution Principles I, II, III (NON-NEGOTIABLE) + V–X (Core) with rigour. Principle IV (PCI DSS) is N/A. The 3 deviations in plan.md § Complexity Tracking remain justified post-scope-cut. Round 1's 19 audit findings remain closed.

### 4a. Architecture Soundness

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| **E1** | 🎯 | **Template variable substitution semantics undefined.** `starter-templates.md` uses 7 variable types (`{{chamber_name}}`, `{{member_name}}`, `{{event_name}}`, `{{month_year}}`, `{{featured_company_name}}`, `{{spokesperson_name}}`, `{{spokesperson_title}}`). Neither spec nor plan nor research nor any contract defines: (1) WHO resolves variables (Application use-case? Domain value-object? Infrastructure renderer?), (2) WHEN (template-snapshot-to-draft? broadcast-submit? broadcast-send?), (3) WHAT happens when variable doesn't resolve (empty string? literal `{{var}}`? error?), (4) HOW values are HTML-escaped before substitution (XSS risk if any variable derives from member-supplied content). Without these rules, two implementations of the same spec will produce different behavior — and one of them will probably ship a bug. | Add new section to `contracts/broadcast-template.md` § 5 "Variable resolution semantics": (a) **WHO**: `snapshotTemplateToDraft` use-case substitutes ONLY `{{chamber_name}}` at draft-start (tenant-scoped, safe constant); ALL other variables are LEFT AS-IS in draft body (members edit them manually like `[bracketed]` placeholders). This is the simplest model and aligns with FR-019 snapshot semantics. (b) **WHEN**: at `snapshotTemplateToDraft` only; broadcast dispatch performs ZERO variable substitution (broadcasts are static HTML at send time). (c) **NOT-RESOLVED**: any `{{var}}` other than `{{chamber_name}}` is preserved literally — member must replace before submit OR sanitiser strips at submit (decide one). | E3 | 💡 | **`broadcast_templates.locale` enum vs F7 MVP `broadcasts.locale` consistency.** F7 MVP `broadcasts` table doesn't have a locale column (broadcasts are locale-agnostic; sender decides). F7.1a adds `broadcast_templates.locale`. Is the picker filter by template-locale × broadcast-locale? Are there cases where a tenant wants to send a TH template to SV-locale recipients (translated by admin)? | Document explicitly in `contracts/broadcast-template.md` that `broadcast_templates.locale` is the **content** locale (the language the body is written in), NOT the **send** locale. Broadcasts don't have a locale — they have body content in whatever language the member composed. The picker filter is a UX convenience, not an isolation invariant. |
