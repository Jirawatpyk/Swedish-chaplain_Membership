### 🔴 H-NEW-1: Erasure runbook Vercel CLI syntax is wrong — silent DSR failure risk

**Severity**: BLOCKER — direct GDPR Art. 17 / PDPA §30 compliance risk.
**File**: `docs/runbooks/f6-manual-erasure.md` § F6.1 (the new section added by R1 fix commit, ~line 252 in the cascade section)
**Root cause**: The runbook uses:
```bash
vercel blob del "<error_csv_blob_url>" --token "$BLOB_READ_WRITE_TOKEN"
```

Two errors:
1. The Vercel CLI subcommand is `vercel blob delete`, not `del`. With wrong subcommand, the CLI prints an error and exits non-zero — but a DPO running this in a hurry during a DSR window may not notice.
2. The `--token` flag on `vercel` CLI commands refers to a CLI auth token (`VERCEL_TOKEN`), NOT the `BLOB_READ_WRITE_TOKEN` which is an SDK env var. Mixing them produces an authentication failure, not a successful delete.

**Impact**: A DPO follows the runbook during a real GDPR Art. 17 request, executes the command, sees an error (or success-looking output), believes the blob was deleted, ticks off the DSR ticket. Reality: blob persists for up to 30 days until natural sweep cron clears it. PDPA §37 breach notification not triggered (no leak yet), but the chamber is non-compliant with the data subject's Art. 17 right.

**Fix required**: Replace the CLI command with one of:
- **Option A**: A small `scripts/erase-error-blob.ts` that imports `del` from `@vercel/blob` and uses `process.env.BLOB_READ_WRITE_TOKEN`. Document `pnpm tsx scripts/erase-error-blob.ts <url>` invocation.
- **Option B**: Vercel Dashboard navigation path (Settings → Blob → Delete) with screenshots.

Whichever is chosen, the runbook must document the EXACT command + expected output so the operator can confirm success.

### 🟡 M-NEW-6: `attendee_fingerprint` Art. 11 erasure exemption legally borderline

**Severity**: MEDIUM — DPO legal opinion required before invoking.
**File**: `docs/runbooks/f6-manual-erasure.md:303` (cascade section note)
**Issue**: Runbook claims Art. 11 GDPR pseudonymisation exemption for the SHA-256 first-16-hex fingerprint. This is contestable: for a TARGETED DSR where the controller already knows the subject's email, reconstructing the fingerprint costs O(1) — there is no "disproportionate effort". EDPB guidance distinguishes general-public inference from controller-assisted inference.

**Fix**: Expand the note to acknowledge the legal uncertainty:
> "The fingerprint is not reversible by a third party, but the chamber (as controller) can re-derive it from the known email. Upon a targeted DSR, DPO should consult legal counsel on whether to zero-out `csv_import_records.attendee_fingerprint` alongside the event_registrations erasure. The 30-day natural sweep is the lowest-risk path."
