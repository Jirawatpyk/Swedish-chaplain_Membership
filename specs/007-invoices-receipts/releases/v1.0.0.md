# F4 Release Notes — v1.0.0

**Feature**: Membership Invoicing & Thai-Tax Receipts
**Branch**: `007-invoices-receipts`
**Scope**: 7 user stories (US1–US7), 43 functional requirements, 5 DB tables, 16 audit event types
**Rollout order**: F1 (Auth) → F2 (Plans) → F3 (Members) → **F4 (this release)**

## Who this is for

- **Chamber admins** — can draft, issue, preview, and mark paid a Thai-tax-compliant invoice end-to-end, with bilingual (TH+EN) PDF, sequential document numbering, VAT split, amount-in-words, and auto-email on issue.
- **Chamber members** — see their own-company invoices on `/portal/invoices`, download the same PDF the admin sent, and see billing status on their portal landing card.
- **Chamber managers** — read-only view on every F4 surface for finance oversight (no write path).

## Feature summary

| Story | Surface | What it unlocks |
|---|---|---|
| US1 | `/admin/invoices/new` → issue | Admin drafts + issues a Thai-tax invoice with sequential number `SC-YYYY-NNNNNN`, bilingual PDF, auto-email |
| US2 | `/admin/invoices/[id]/pay` | Admin records payment → status flips to paid → receipt PDF generated + emailed |
| US3 | `/portal/invoices` | Member views own-company invoices list + downloads PDFs (byte-identical to admin copy) |
| US4 | `/admin/settings/invoicing` | Admin configures VAT rate, legal identity, numbering, logo, pro-rate policy; FR-010 bootstrap gate enforces "no invoice without settings" |
| US5 | `/admin/invoices/[id]/void` | Admin voids an unpaid invoice with reason → VOID-stamped PDF re-rendered → cancellation email (Phase 9, tracked separately) |
| US6 | `/admin/invoices/[id]/credit-notes/new` | Admin issues partial or full credit note (ใบลดหนี้) with own sequence, proportional VAT, status transition to partially_credited / credited |
| US7 | `/admin/members/[id]` | Invoices section on member detail + 6 F4 audit events chronologically integrated into F3 timeline |

## Thai compliance

- **§86/4 tax invoice fields**: tenant legal identity + tax ID, customer legal identity, bilingual label "ใบกำกับภาษี / Tax Invoice", CE issue date (BE shown in parens on TH side), bilingual line-item descriptions, subtotal + VAT + total, amount-in-words TH + EN.
- **§87 no-gaps sequence**: Postgres advisory lock per `(tenant_id, document_type, fiscal_year)` guarantees monotonic allocation under concurrent issue. 8 chaos scenarios tested (PDF fail, Blob fail, commit fail, lock contention, year boundary, missing seq row, audit fail, idempotency replay).
- **Asia/Bangkok fiscal-year boundary**: `@js-joda/core` + `@js-joda/timezone` handles Dec 31 23:59:59 UTC = next-day BKK correctly.
- **Snapshot immutability (FR-011)**: once issued, `vat_rate_snapshot`, `tenant_identity_snapshot`, `member_identity_snapshot` are frozen on the invoice row via a `BEFORE UPDATE` trigger. Later changes to tenant settings or member records never retroactively alter issued documents.

## Security + multi-tenancy

- **Constitution Principle I clauses 3 + 4**: two-layer tenant isolation (application `runInTenant` + Postgres RLS with FORCE) on all 5 F4 tables. Tenant-isolation integration test (14 scenarios) is a Review-Gate blocker and is green.
- **PCI SAQ-A preservation**: F4 does not touch card data. Payment recording accepts free-form `payment_method` + `payment_reference` string only; Stripe integration is **F5 Online Payment** — a planned feature, not in this release.
- **PDPA + GDPR**: member identity snapshots copied into invoice rows are bounded + audit-logged; archival of a member does not delete their invoices (FR-029, FR-030) to preserve statutory 5-year retention.
- **Logo upload hardening**: MIME whitelist (PNG/JPEG), size ≤ 1 MB, dimension bounds 200×100..2000×500, EXIF strip via sharp re-encode, 50-logo-per-tenant history cap, optional `Idempotency-Key` replay.

## Known limitations / deferred

- **F5 Online Payment (Stripe)** — planned feature, not in this release; admin records payment via free-form method string.
- **Throwaway-tenant per-test E2E infra** — admin mutating fixmes covering settings-wide changes (e.g. VAT 7→10 on SweCham) are still `test.fixme` because mutating real SweCham state would cross-contaminate other tests. Dedicated-member + high-sequence-block seeders unlock the per-member mutating fixmes.
- **Idempotency-Key persistence for logo upload** — uses the shared Upstash-backed `src/lib/idempotency.ts` infra (24 h TTL, fail-open on Redis outage). First-time/replay/conflict semantics match F2+F3 mutating endpoints.

## Env vars added

- `BLOB_READ_WRITE_TOKEN` (required) — Vercel Blob store token for PDF + logo uploads.
- `CRON_SECRET` (required) — protects overdue-invoice sweeper cron from public invocation.
- `FEATURE_F4_INVOICING` (boolean, default true) — kill-switch. Setting false returns 503 on all F4 routes while keeping F1–F3 alive.

## Migration order for a new environment

```
F1 → F2 (0006_plans + 0007_fee_config) → F3 (0009_members + 0010_contacts)
    → F4 (0010_invoicing_tables + 0011_audit_log_f4_extension)
```

Point-in-time restore to pre-`0010` snapshot rolls back F4 cleanly; the audit event types added by `0011` are additive (Postgres does not support ALTER TYPE DROP VALUE) and harmless if left in place.

## Rollout steps

1. Apply `drizzle/migrations/0010_invoicing_tables.sql` + `0011_audit_log_f4_extension.sql` in staging.
2. Run `scripts/seed-f4-invoice-settings.ts` for the target tenant (idempotent).
3. Set `FEATURE_F4_INVOICING=true` + add `BLOB_READ_WRITE_TOKEN` + `CRON_SECRET` in Vercel env (Production + Preview).
4. Smoke-test in staging per `quickstart.md § 5`.
5. Promote to production.
6. Manual: admin issues a real (test-mode) invoice → PDF Thai-RD spot check → admin records payment → receipt spot check.

## Rollback

- Immediate: `FEATURE_F4_INVOICING=false` + redeploy → 503 `read_only_mode` on all F4 routes; existing rows remain, UI is hidden.
- Full: point-in-time Neon restore to pre-`0010` snapshot; audit event types remain but are unused.
