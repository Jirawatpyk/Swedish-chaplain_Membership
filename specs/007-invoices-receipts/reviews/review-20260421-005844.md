## 3. Findings

### 🟡 F-01 — Reserved idempotency record never cleared on validation-error path

**File**: `src/app/api/tenant-invoice-settings/logo/route.ts`
**Lines**: 102–121, 132–147

The Idempotency-Key flow calls `reserveIdempotencyRecord()` **before** `uploadTenantLogo()` runs (line 120). When the use-case returns an error (mime/size/dims/decode/cap), the route returns 4xx **without** calling `rememberIdempotentResponse()`. The reservation persists for 24 h with `response: null`, which `classifyIdempotencyRequest()` treats as **`conflict`** on replay (`src/lib/idempotency.ts` §170–181).

**Impact**: a client that retries the same Idempotency-Key after a validation error (e.g. user picks a too-large file, fixes it, re-clicks submit without regenerating the key) gets a 409 conflict instead of a fresh attempt. User-visible UX bug; not a data-integrity issue.

**Recommendation**: on the failure path, either (a) call `rememberIdempotentResponse()` with the 4xx body so the replay serves the same error verbatim (idiomatic per HTTP idempotency semantics), or (b) don't reserve until after validation passes. Option (a) is simpler and matches the "replay returns same response" contract.

**Classification**: Warning — pattern shared with other F2/F3 mutating endpoints that use the same shared infra. A fix here should probably be applied centrally in `src/lib/idempotency.ts` with a helper. Flagged but not a merge blocker.

### 🟡 F-02 — RBAC resource inconsistency across tenant-invoice-settings routes

**Files**:
- `src/app/api/tenant-invoice-settings/route.ts` — `GET` uses `resource: 'tenant_invoice_settings'` (line 110), `PATCH` uses `resource: 'invoice'` (line 150).
- `src/app/api/tenant-invoice-settings/logo/route.ts` — `POST` uses `resource: 'invoice'` (line 33).

The `src/modules/auth/domain/policies.ts` (v1.4.0) defines `'tenant_invoice_settings'` as a distinct resource precisely so tenant-settings access can be audited + policy-gated separately from invoice CRUD. PATCH + POST /logo still use the legacy `'invoice'` resource from the R7-B2 era.

**Impact**: role audit trails for settings PATCH + logo upload are tagged under the wrong resource category. Admin audits that filter by resource will miss these events. No authorization bug — admin has write on both — but the audit-schema integrity degrades.

**Recommendation**: change both PATCH (line 150) and POST /logo (line 33) from `resource: 'invoice'` → `resource: 'tenant_invoice_settings'`. 2-line change, no migration, no policy-table update (policy already covers it).

**Classification**: Warning — discoverable by the post-audit reviewer filtering by resource; fix before ship.

### 🟢 F-03 — T091 FR-011 test is weaker than spec intent

**File**: `tests/integration/invoicing/settings-form.test.ts` lines 155–195

The original T091 scope was "FR-011 snapshot immutability after issue" — seeding an issued invoice, mutating tenant VAT 7→10, and asserting the issued invoice's `vat_rate_snapshot` stays at 7. My implementation simplified to "changing VAT 10→5 updates what `getForIssue` returns (next issue snapshots 5%)" because hand-crafting an issued-invoice row hits `invoices_non_draft_has_snapshots` + `invoices_draft_has_no_number` CHECK constraints that `issueInvoice` normally satisfies.

**Impact**: the snapshot-freeze guarantee is enforced by the DB trigger `f4_invoices_immutability()` (schema), not tested at the Application layer by Phase 8. The trigger is implicitly covered by `audit-coverage.test.ts` (which issues real invoices via `issueInvoice`) + `tenant-isolation.test.ts` (which also issues real invoices). So FR-011 IS tested end-to-end, just not in the Phase 8 test file.

**Recommendation**: extend T091 to call `issueInvoice` (the real use-case) against a seeded plan + member, then mutate VAT and re-read the invoice row — gives a direct FR-011 assertion without duplicating snapshot-field boilerplate. Defer to Phase 10 polish; functional coverage is already present.

**Classification**: Suggestion — not a regression; documented inline in the test comment.

### 🟢 F-04 — T115s seeder accumulates stale invoice rows under the 990000 block

**File**: `scripts/seed-f4-e2e-admin-fixtures.ts` lines 84–106, 268–285

The seeder's `hasUnpaidIssuedInvoice()` only checks for status=`issued`. When a test pays the pay-target invoice, the row transitions to `paid` and stays. On re-run, the seeder creates a fresh issued target at the next free seq. Over many CI runs, the 990000–999999 slot (10 000 reserved rows) gradually fills with stale paid rows. `findNextAvailableSeq` detects exhaustion and throws, but no purge path is documented.

**Impact**: practically none for years of normal CI volume (~hundreds of mutations per year). Cleanup can be tacked onto `clear-test-data.ts` when the slot nears capacity.

**Recommendation**: add a companion sweeper (`scripts/cleanup-f4-e2e-admin-fixtures.ts`) or extend `clear-test-data.ts` to purge rows where `sequence_number BETWEEN 990000 AND 999999` AND `member_id = <E2E Mutation Co>`. Defer to Phase 10.

**Classification**: Suggestion — budget-exhaustion date is >10k test runs away.

### 🟢 F-05 — T092 MIME-mismatch test accepts format-confusion as pass

**File**: `tests/integration/invoicing/logo-upload-security.test.ts` lines 130–155

Test: "rejects declared-MIME vs detected-format mismatch (claims PNG, bytes are JPEG)". The use-case **accepts** this case (declared=PNG passes whitelist; detected=JPEG also in whitelist; re-encode uses detected format). The test asserts `result.ok === true` and key ends in `.jpg`.

**Impact**: this is intentional per `upload-tenant-logo.ts` N3(b) comment (detected format authoritative for encoding + contentType). The test documents the behavior — it isn't a "rejects mismatch" test despite its description. Test name + assertion drifted.

**Recommendation**: rename the test to "accepts valid image regardless of declared/detected MIME mismatch and re-encodes to detected format". Clarifies intent.

**Classification**: Suggestion — cosmetic test-name drift; assertion is correct.

### 🟢 F-06 — `BlobStoragePort.list()` contract doesn't specify pagination semantics

**File**: `src/modules/invoicing/application/ports/blob-storage-port.ts` lines 52–60

The doc says implementations SHOULD cap the result at a bounded page size but doesn't specify **what cap** callers should assume. `vercel-blob-adapter.list()` uses `limit: 60`. A future caller asking for, say, the invoice-pdf history under `invoicing/<tenant>/invoices/` would silently miss rows beyond 60.

**Impact**: none today — only `uploadTenantLogo` uses `list()`, and 60 > 50 cap. A future caller might misuse this.

**Recommendation**: tighten the port contract — either (a) add `limit?: number` parameter with documented default, or (b) rename method to `listFirstN(prefix, n)` to force callers to name the bound. Defer to first future caller.

**Classification**: Suggestion — acceptable for single-caller today.
