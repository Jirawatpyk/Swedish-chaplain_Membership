## 5. New Finding (minor)

### 🟢 F-09 — Idempotency-Key-replay on logo-upload has no route-level integration test

**File**: `tests/integration/invoicing/logo-upload-security.test.ts` + `tests/e2e/invoice-settings.spec.ts`

The F-01 fix routes the `rememberIdempotentResponse` on error path. Neither T092 (Application-layer with fake blob) nor T097 E2E tests the route-layer idempotency replay flow directly. The shared `src/lib/idempotency.ts` has its own tests covering the classify/reserve/remember state machine, and the route-level glue is exercised by F2 plans endpoints — so the coverage gap is indirect, not absent.

**Recommendation**: add a route-level integration test that (a) POSTs a valid logo twice with the same `Idempotency-Key` + same bytes → expects identical 201 body, (b) POSTs once valid, then again with a mutated byte → expects 409 `idempotency_conflict`, (c) POSTs a too-large file twice with the same key → expects identical 413 body. Deferrable to Phase 10.

**Classification**: Suggestion — adds confidence without blocking ship.
