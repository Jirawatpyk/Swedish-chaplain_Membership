### 🟢 R18-02 — `expected_pdf_sha256` type validation is weak

**Severity**: 🟢 Suggestion.

**File**: `src/app/api/cron/outbox-dispatch/route.ts:271-272`

```ts
const expectedSha =
  typeof ctx.expected_pdf_sha256 === 'string' ? ctx.expected_pdf_sha256 : null;
```

The check only verifies the field is a string. A malformed value (e.g. `""`, `"not-hex"`, a base64 digest) would still pass the typeof check and then fail the equality comparison → trigger `attachment_sha_mismatch` permanent-fail. That's **fail-safe** (better to fail than ship wrong bytes), but it masks a different underlying bug (malformed enqueue) as an integrity violation.

**Recommendation**: tighten to `RE_SHA256.test(expectedSha)` before the comparison. If malformed, log a distinct warning `expected_pdf_sha256.malformed` and clear `expectedSha` — the dispatcher then falls through to the pre-R17-02 behaviour (ship bytes without verification), which is the same attack surface we had before the fix. Lesser-evil: fail permanently with a distinct reason `expected_pdf_sha256_malformed` so ops can diagnose. Post-ship polish.
