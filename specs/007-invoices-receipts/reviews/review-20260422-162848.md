## Incidental findings (🟢 Suggestions, all post-ship polish)

### R20-01 — `ImageReEncodePort` returns `format: 'unknown'` with empty bytes

**File**: `src/modules/invoicing/application/ports/image-reencode-port.ts:36-41`

```ts
readonly format: 'png' | 'jpeg' | 'unknown';
...
bytes: Uint8Array;  // caller must branch on format !== 'unknown' before using
```

The `'unknown'` case returns `new Uint8Array(0)` and the use-case correctly gates on it before using `bytes`. This is **safe but fragile** — a future caller could forget the branch and pass the empty buffer downstream, silently creating a 0-byte Blob.

**Recommendation**: convert to a sum type so the bytes field is only present on the success variant:

```ts
type ImageReEncodeResult =
  | { format: 'png' | 'jpeg'; width: number; height: number; bytes: Uint8Array }
  | { format: 'unknown'; width: number; height: number };  // no bytes field
```

Post-ship polish. TS narrowing would then make the bug structurally impossible.

### R20-02 — Sharp error messages may leak sandbox paths

**File**: `src/modules/invoicing/infrastructure/adapters/sharp-image-reencode-adapter.ts:84-88`

```ts
return {
  ok: false,
  error: { code: 'decode_failed', reason: e instanceof Error ? e.message : String(e) },
};
```

Some `sharp` failure paths (especially `libvips` crashes) include the absolute file path of a temp file or the Vercel function's sandbox directory. That reason string is then logged by the use-case (logger redacts nothing under `reason` key). Not a blocker — reason is logged as `warn` to internal log aggregator, not returned to the client.

**Recommendation**: truncate `reason` to e.g. 200 chars and strip anything that looks like a path (matches `[/\\][a-zA-Z0-9]`). Same pattern as `sanitiseErrorReason()` in `void-invoice.ts:113-116`. Post-ship polish.

### R20-03 — a11y primary-lang span always carries `lang` even when it matches html-root

**File**: `src/app/(member)/portal/invoices/[invoiceId]/page.tsx:322-340` (R19 change)

```tsx
<span lang={primaryLang} className="...">{primary}</span>
```

When `userLocale === 'th'` and the page's `<html lang="th">` is set, the primary span's `lang="th"` is redundant (inherits from root). Screen readers won't pronounce it differently. Not wrong — just marginally noisy markup.

**Recommendation**: render `lang` attribute only when it differs from the page locale. `primaryLang === userLocale ? undefined : primaryLang`. Post-ship polish. WCAG 3.1.2 is already satisfied with explicit lang on both spans.
