### 🟡 R17-01 — `FORCE ROW LEVEL SECURITY` missing on all five F4 tables

**Severity**: 🟡 Warning (pre-existing inherited from F3 pattern; Constitution drift but not a live leak under the current `chamber_app` NOBYPASSRLS role).

**File**: `drizzle/migrations/0019_invoicing_tables.sql` (5 tables: `invoices`, `invoice_lines`, `credit_notes`, `tenant_invoice_settings`, `tenant_document_sequences`)

**Evidence**:

```bash
$ grep -c "FORCE ROW LEVEL SECURITY" drizzle/migrations/0019_invoicing_tables.sql
0

$ grep -n "FORCE ROW LEVEL" drizzle/migrations/*.sql
drizzle/migrations/0006_plans_and_fee_config.sql:166:ALTER TABLE "membership_plans" FORCE ROW LEVEL SECURITY;
drizzle/migrations/0006_plans_and_fee_config.sql:175:ALTER TABLE "tenant_fee_config" FORCE ROW LEVEL SECURITY;
drizzle/migrations/0007_audit_log_f2_extension.sql:140:ALTER TABLE "audit_log" FORCE ROW LEVEL SECURITY;
```

Constitution v1.4.0 Principle I clause 2 (see `plan.md:143`): *"all five F4 tables have `ENABLE ROW LEVEL SECURITY` + `FORCE ROW LEVEL SECURITY` + the standard `USING (tenant_id = current_setting('app.current_tenant', TRUE))` policy."* The migration file header comment at `0019_invoicing_tables.sql:25` claims *"RLS + FORCE + FOR ALL TO chamber_app policy on every table"* — but the SQL body only calls `ALTER TABLE … ENABLE ROW LEVEL SECURITY` (5 hits) + `CREATE POLICY … TO chamber_app` (5 hits), and no `FORCE`.

F3's `0009_members_contacts.sql` has the same drift; F4 copied the F3 precedent. Both were plausibly missed during their respective R1 reviews because the policy IS attached to `chamber_app` and the app connects as that NOBYPASSRLS role, so the `tenant-isolation.test.ts` suite passes (17/17) on the practical happy path.

**Impact**:

- **Current risk**: low. Production Neon connections use the `chamber_app` role which has `NOBYPASSRLS` — the policy fires on every query. `FORCE` only matters for table OWNERS, and the app does not run as owner.
- **Defense-in-depth risk**: a human operator opening a Neon console session as a different role (admin, migration user) bypasses the policy. Any script that accidentally connects with a non-chamber_app role touches tenant data unrestricted.
- **Compliance risk**: literal reading of Constitution v1.4.0 clause 2 is "ENABLE + FORCE". A Review-Gate auditor could block ship on this.

**Recommendation**: ship migration 0032 (next available number):

```sql
-- 0032_f4_force_rls.sql
ALTER TABLE "tenant_invoice_settings"  FORCE ROW LEVEL SECURITY;
ALTER TABLE "tenant_document_sequences" FORCE ROW LEVEL SECURITY;
ALTER TABLE "invoices"                 FORCE ROW LEVEL SECURITY;
ALTER TABLE "invoice_lines"            FORCE ROW LEVEL SECURITY;
ALTER TABLE "credit_notes"             FORCE ROW LEVEL SECURITY;
-- While here: close the F3 drift too
ALTER TABLE "members"                  FORCE ROW LEVEL SECURITY;
ALTER TABLE "contacts"                 FORCE ROW LEVEL SECURITY;
```

Then extend `tests/integration/rls-coverage.test.ts` to assert `pg_class.relforcerowsecurity = true` on every tenant-scoped table — not just `ENABLE`. This prevents future feature branches from repeating the drift.

### 🟡 R17-02 — void-invoice Phase 2 failure + auto-email attachment ordering hazard

**Severity**: 🟡 Warning. Gated in production by `FEATURE_F4_VOID_ATTACHMENT=false` (PG-2 DPA pending), so no live impact today. Remains a correctness issue the moment the flag flips.

**File**: `src/modules/invoicing/application/use-cases/void-invoice.ts:260-275` + `src/app/api/cron/outbox-dispatch/route.ts:149-208`

**The hazard**:

`voidInvoice` uses a two-phase commit (explicitly documented, `void-invoice.ts:36-46`):

- **Phase 1** (atomic DB tx): lock → render new VOID-stamped bytes in memory → `applyVoid` UPDATE (status='void', does NOT touch `pdf_sha256`) → emit `invoice_voided` audit carrying `new_pdf_sha256: rendered.sha256` (`void-invoice.ts:251`) → enqueue outbox row with `pdfBlobKey: loaded.pdf.blobKey` (`void-invoice.ts:265`) → commit.
- **Phase 2** (best-effort, post-commit): `blob.uploadPdf` (overwrite at same content-address key) → `applyInvoicePdfRegeneration` UPDATE of `pdf_sha256`.

If Phase 2 fails (Blob outage, cold-start timeout), the DB is already committed:
- Audit log says `new_pdf_sha256=X` — but no Blob bytes with sha X exist.
- Invoice row still carries the ORIGINAL `pdf_sha256` (Phase 1 doesn't touch it).
- Blob holds the ORIGINAL un-stamped invoice bytes at that key.
- **Outbox row sits in `pending`, waiting for the cron to drain it.**

When the cron fires (~60s later), `dispatchOne` (`outbox-dispatch/route.ts:269`) calls `vercelBlobAdapter.downloadBytes(pdfBlobKey)` — which returns **the ORIGINAL un-stamped invoice PDF**. The payload builder at line 186-197 then ships that attached as `{document_number}-VOID.pdf` in an email that says "This document is no longer payable." A bookkeeper receives what looks like a perfectly valid tax invoice attached to a cancellation notice.

The dispatcher has no integrity check against the audit log's recorded `new_pdf_sha256` or against the outbox row's expected bytes.

**Why R15/R16 didn't catch it**: the inline comment at `void-invoice.ts:39-44` says Phase-2 failure leaves state "CONSISTENT (bytes match hash) but INCOMPLETE". That's true for the **DB-only** view. The comment does not mention the **outbox + email** view, which is what the attacker-shaped reality looks like to a member.

**Mitigation options** (none currently in place):

1. Enqueue the outbox row in **Phase 2** after the Blob upload succeeds, not in Phase 1. Breaks transactional-outbox semantics (email-intent could be lost on a Phase-1-commit + process-crash window). Not ideal.
2. Have `buildPayload` (`outbox-dispatch/route.ts:149-208`) verify `sha256(prefetchedBytes) === ctx.expected_sha256` and permanently-fail with `auto_email_delivery_failed` audit on mismatch. **This is the defensive fix.** The outbox row already carries `pdfBlobKey` + `pdfTemplateVersion`; add `expected_sha256` alongside and the check becomes cheap.
3. Use distinct content-addressed Blob keys for original + void-stamped (`…_v1.pdf` vs `…_v1_void.pdf`) — outbox points at the void-stamped key, which either exists (Phase 2 succeeded) or the prefetch returns null (graceful degradation to link-only email via `hasAttachment: false`).

**Recommendation**: option 2 — `expected_sha256` on the outbox context + dispatcher verification. Lowest blast radius, keeps the two-phase commit pattern intact. Ship before flipping `FEATURE_F4_VOID_ATTACHMENT=true` post-DPA.

### 🟡 R17-03 — `record-payment.ts` reads settings INSIDE withTx; `issue-credit-note.ts` explicitly hoisted same call OUT

**Severity**: 🟡 Warning. Concurrent-payment pool-deadlock risk; may not manifest at SweCham's current scale but the fix is a 2-line move and the inconsistency suggests someone missed it.

**File**: `src/modules/invoicing/application/use-cases/record-payment.ts:167`

**Evidence** — compare with `issue-credit-note.ts:126-135`:

```ts
// issue-credit-note.ts:126–135 (OUTSIDE withTx, with detailed rationale)
// A. Load settings BEFORE the withTx. The settings repo opens its own
// `runInTenant` transaction under the hood; nesting that inside the
// outer withTx deadlocks the pool when two credit-note issues run
// concurrently (outer tx holds conn1, inner settings-read waits for
// conn2 which is held by the other concurrent caller, and vice
// versa). Settings are effectively immutable during an issue …
const settings = await deps.tenantSettingsRepo.getForIssue(input.tenantId);
```

```ts
// record-payment.ts:167 (INSIDE withTx — no guard rail)
const settings = await deps.tenantSettingsRepo.getForIssue(input.tenantId);
if (!settings) return err({ code: 'settings_missing' });
```

`void-invoice.ts:130-132` **did** get the fix. `record-payment.ts` did not. If `tenantSettingsRepo.getForIssue` opens a nested `runInTenant` (per the issue-credit-note comment, it does), concurrent record-payment calls on different invoices can deadlock on the 2-connection pool the same way.

**Why may-not-manifest**: SweCham concurrency is ≤ 5 admin staff and payment recording is a low-frequency action. The probability of two admins hitting `/pay` simultaneously is near zero. But the hazard is structural, not traffic-dependent — any pool-size regression (Neon transient connection limit drop, a different tenant scaling up) would surface this.

**Recommendation**: hoist `tenantSettingsRepo.getForIssue(input.tenantId)` above the `withTx` call in `record-payment.ts:117`, matching the `issue-credit-note.ts` + `void-invoice.ts` pattern. 2-line refactor. Add a note to `tenant-settings-repo.ts` JSDoc warning callers not to nest it.

### 🟢 R17-05 — Recipient email stored raw in long-retention audit payloads

**Severity**: 🟢 Suggestion. Current pattern is consistent across the module; flag for policy discussion.

**Files**: `resend-pdf.ts:228`, `:243`, `:321`; `issue-invoice.ts:351` (via outbox); `record-payment.ts:301` (via outbox).

Audit events `invoice_pdf_resent`, `receipt_pdf_resent`, `credit_note_pdf_resent` carry `recipient_email: recipientEmail` as a plain string in the audit payload. Audit has **10-year retention** (FR-029). Recipient email is not new exposure (it's already snapshotted on the invoice row's `member_identity_snapshot`), but the combination of (long retention × append-only × no forget-capability) makes this the worst cell in the matrix for PII.

Logger redaction (`src/lib/logger.ts:45` `REDACT_PATHS`) handles `*.*.recipient_email` for pino log lines — but that's runtime logs, not the persisted `audit_log.payload`.

**Recommendation**: two options:
1. Hash recipient email in the audit payload the way `record-payment.ts:273` hashes `paymentReference`. Pros: symmetric with existing pattern. Cons: breaks operator ability to correlate bounces with recipients during forensic review.
2. Accept the current pattern; document in `security.md § 4 Cat-B` as a deliberate retention trade-off for incident-response ergonomics.

Current code picks (2) implicitly. Make that explicit in `security.md` or flip to (1).

### 🟢 R17-08 — `issue-credit-note.ts` loads ALL CNs for annotation build without limit

**Severity**: 🟢 Suggestion. Bounded by business logic; unlimited query is unnecessary.

**File**: `src/modules/invoicing/application/use-cases/issue-credit-note.ts:405-420`

```ts
const allCreditNotes = await deps.creditNoteRepo.findByOriginalInvoiceInTx(
  tx,
  invoiceId,
  input.tenantId,
);
```

The remainder-guard caps total credit at `invoiceTotal`, so the list has ≤ N entries where N is the number of partial credits admins issued. In practice this is 1-3. No ORDER BY + LIMIT is present on the repo side either.

**Recommendation**: `ORDER BY sequence_number LIMIT 20` on the repo method. If the list ever exceeds 20, the rollup to `credited` has already happened and no further CNs could be issued. Defensive cap against a pathological data-state.
