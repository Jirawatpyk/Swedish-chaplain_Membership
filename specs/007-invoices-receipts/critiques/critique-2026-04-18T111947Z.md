## Executive Summary

The F4 spec + plan bundle is strong overall: 7 user stories with clear priorities, 33 well-scoped FRs, 11 measurable SCs, 5 resolved clarifications, full 10-principle Constitution Check, a 15-threat security snapshot, and a thoughtful data model with RLS + FORCE on every new table. The transactional sequence-number + PDF + Blob + audit pattern is the single most load-bearing design decision and it is specified to a level of rigour (8-scenario chaos test blueprint) that matches the Thai RD §87 compliance stakes.

Three 🎯 items block `/speckit.tasks` primarily because they'd cause material mid-implementation rework:

1. **No draft-PDF preview path** — the "issuance is a point of no return (seq consumed)" design has no safe dry-run for admins to verify layout/content before committing. Real-world admins will burn seq numbers on layout mistakes until this is closed.
2. **Tenant logo upload lacks validation rules in the contract** — plan.md §12 research mentions "PNG/JPEG, ≤ 1MB, ≤ 2000×500" but the `PATCH /api/tenant-invoice-settings` endpoint in contracts does not declare these as zod validation rules. SVG upload is an XSS/SSRF vector through `@react-pdf/renderer`'s image handling; this MUST be blocked before any logo endpoint exists.
3. **US4 (tenant config UI, P3) is a P1 dependency wearing P3 priority** — FR-010 refuses issuance without complete settings, and no tenant can bootstrap invoicing without those settings. The "seeded config" escape hatch in plan.md needs an explicit, listed seeder script + a test + quickstart instruction before tasks.md is written, otherwise the MVP slice has a hidden infra gap.

Nine recommendations improve robustness (automation of the bidirectional-dep invariant, numbering-format overflow docs, email PII audit, PDF-render benchmark ahead of budget commit, load test for concurrent issuance, template-version policy, dependency review, build-vs-buy rationale, baseline capture for SC-008). Two questions need your input on scope boundaries.

All 10 Constitution gates still pass; none of the 🎯 items require architectural rework, they are scope/gap additions. Offer of targeted edits to `spec.md` + `plan.md` + `contracts/invoicing-api.md` included below.

### Alternative Approaches

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| P4 | 🤔 | **Why not combined invoice+receipt (TH: ใบกำกับภาษี/ใบเสร็จรับเงิน) as the default?** Spec + data-model support both "separate" and "combined" modes via `tenant_invoice_settings.receipt_numbering_mode`, but defaults are not stated. Most Thai SMBs use **combined** (issues + receives payment in one document on payment date) because it simplifies bookkeeping. Separate mode requires two numbering streams and two PDFs per paid invoice. | Decide on default per SweCham's actual practice: (a) Default `combined` — simpler, Thai SMB norm. Invoice PDF acts as the tax invoice before payment; on payment, a new combined PDF is generated and replaces (supersedes) the invoice as the legal document. (b) Default `separate` — more traceable for audit; two PDFs. **Need your input**: which does SweCham's current Excel workflow use? This affects PDF template count (3 vs. 4) and numbering stream count. |

### Edge Cases & UX

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| P5 | 🎯 | **No PDF preview before issue**. FR-001 states "A draft has no sequential tax-invoice number, no rendered PDF". The admin clicks Issue → seq consumed → PDF rendered → too late to check layout. If the logo renders wrong, a line wraps weirdly, or the Thai font breaks on a specific company name (long Thai word, diacritic, etc.), the fix path is: void the issued invoice (which keeps a sequence hole), fix the draft, issue again. In the first month of production this will happen 3-5 times for SweCham's first real invoices — each one consumes a sequence number and an audit trail. | Add a "Preview PDF" action on the draft that renders a **watermarked draft PDF** via the same template pipeline, with a huge "DRAFT / ร่าง — NOT A TAX DOCUMENT" watermark, **without consuming a sequence number** and **without persisting to Blob** (render in-memory, stream to browser, discard). This is a minor addition: new `POST /api/invoices/[id]/preview` endpoint returning the PDF body directly (not a signed URL). Closes the UX gap without compromising FR-003. Add FR-001a covering the preview. |

## Engineering Lens Findings 🔬

### Architecture Soundness

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E1 | 💡 | **Bidirectional invoicing ↔ members read-only dependency has no automated enforcement**. Plan.md Complexity Tracking row acknowledges this and says enforcement is "code review + PR template checkbox". That is weaker than every other invariant in the codebase (all others are either TypeScript-compile-enforced or ESLint-enforced). A developer six months from now could write a `members/application/use-case.ts` that mutates invoicing state and CI would not catch it. | Add an ESLint rule (or a `tests/unit/architecture.test.ts`) that scans both `src/modules/invoicing/application/**/*.ts` and `src/modules/members/application/**/*.ts` for imports of each other's **port** types (`InvoiceRepo` / `CreditNoteRepo` from members-app, `MemberRepo` / `ContactRepo` from invoicing-app). Read-only consumption should only occur through the public barrel, not through port types. This is ~20 lines of architectural-test code. Alternatively, extend the existing `no-restricted-imports` ESLint rule with `{ paths: [{ name: "@/modules/invoicing/application/ports/*", importNames: ["InvoiceRepo", "CreditNoteRepo"], message: "Members module must only read via @/modules/invoicing barrel" }] }`. |

### Failure Mode Analysis

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E2 | 💡 | **Sequence-number format overflow**. `document_number` format is tenant-prefix + year + 6-digit zero-padded (`SC-2026-000042`). A tenant issuing >999,999 documents in one year (impossible for SweCham at 131 members/yr, but a growing SaaS tenant with auto-renewal + event fees + sponsorship could plausibly hit this) would break format. DB column is `integer` which supports ~2 billion, so the bug surfaces only in the PDF + UI render. | Add a Domain-level invariant: `document_number_format_fits(sequence_number) → Result<DocumentNumber, FormatOverflow>`; fail the issue transaction cleanly with an actionable error message ("Contact support — this tenant exceeded the 6-digit annual document limit; numbering width needs upgrade"). Document the overflow path + escalation in `docs/observability.md § F4 Invoicing` runbook. Cheap insurance; no impact on SweCham today. |
| E3 | 💡 | **No load test for concurrent issuance on year boundary**. The 8-scenario atomicity test in `plan.md § Testing` includes year-boundary crossover but only with 2 concurrent writers. Real-world stress (SweCham annual-renewal Monday: 100 members renewed by 3 admins in 30 minutes) could hit advisory-lock contention patterns not covered by 2-writer tests. | Extend `tests/integration/invoicing/seq-number-atomicity.test.ts` scenario (d) from "2 concurrent issues on same tenant" to **50 concurrent issues** via `Promise.all()` under a single tenant + fiscal year. Assert: (i) all 50 receive unique sequence numbers, (ii) numbers are contiguous 1..50 with no gaps, (iii) wall-clock < 30s (lock-queue throughput). Run under `RUN_PERF=1` gate. |

### Security & Privacy

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E4 | 🎯 | **Tenant logo upload lacks contract-level validation**. `research.md § 12` states "size-capped at 1 MB + dimensions capped at 2000×500 on upload" but `contracts/invoicing-api.md § 3.2` (`PATCH /api/tenant-invoice-settings`) does not declare these as zod validation rules and does not restrict MIME types. **SVG uploads are a real risk** — `@react-pdf/renderer` accepts SVG via `<Image src=…>`, which can contain `<script>` tags or external resource references (SSRF vector when the PDF engine fetches them). A malicious tenant admin uploading a crafted SVG can exfiltrate data via the server's outbound HTTP during PDF render. | Add to contracts: `logo_blob_key` can only be set via a separate `POST /api/tenant-invoice-settings/logo` endpoint that accepts multipart upload, validates MIME ∈ `{image/png, image/jpeg}` (NOT image/svg+xml), validates size ≤ 1MB, validates dimensions 200≤w≤2000 + 100≤h≤500, re-encodes via `sharp` to strip metadata + EXIF before storing. The subsequent settings PATCH can only reference the blob key returned by that upload endpoint. Add a `security.md § Logo upload` threat section. |
| E5 | 💡 | **Auto-email bounces may leak invoice PDFs into Resend's bounce logs**. FR-024 auto-emails carry invoice/receipt/credit-note PDFs as attachments. When Resend receives a bounce (SMTP 5xx), most email providers (Resend, SendGrid, etc.) store the full bounced message — including attachments — in bounce logs retained for 30-90 days. This creates an unmonitored PII replica outside Chamber-OS's retention policy (10-year legal-obligation retention vs. ~30-day provider retention mismatch is fine; but the *unmonitored* copy is the concern). | Two options: (a) Accept the risk; document it in `security.md` § Data sovereignty. Resend is a SOC2 + GDPR-DPA-signed vendor; tax documents + PII transit via email by design. (b) Switch auto-email to a **link-to-download** pattern: email body contains a time-limited signed download URL; no PDF attachment. Member clicks → authenticated → Blob download. More secure, but adds friction (auth redirect) for a bookkeeper who wants the PDF for filing. Recommend **(a)** with an explicit DPA review line added to `security.md`, because the link-to-download pattern materially worsens the member UX for a risk that is already regulated by Resend's compliance posture. |
