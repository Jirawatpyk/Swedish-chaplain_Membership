## Round-2 Findings

### Product Lens Findings 🎯

#### Edge Cases & UX

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| R2-P1 | 💡 | **Void does not auto-email a cancellation notice**. `plan.md § VIII Reliability` states "Void invoice — **No outbox row** — void is not auto-emailed (admin decides when to notify; resend available via FR-025)". But real-world CX: a member got invoice #42 in their inbox yesterday, today it's void, they have no idea. If they pay the voided invoice, confusion + bookkeeper rework. The "admin decides when to notify" design assumes admins will remember to send a void notification — they won't, especially in the first year. | Add **FR-036**: "When an issued invoice is voided, the system MUST automatically email the member's primary billing contact a bilingual (TH+EN) **cancellation notice** that references the original invoice number, includes the void reason, and explicitly states the document is no longer payable. This is a separate outbox row type (`invoice_voided_notice`) delivered via the same outbox pipeline as issue/pay auto-emails." Add matching US5 AS: "When admin voids an issued invoice, member receives a cancellation email within 1 minute." Same admin override flag (`auto_email_on_issue`) applies symmetrically — admin can suppress if they plan to notify manually. |
| R2-P2 | 💡 | **Draft visibility default in admin list is unspecified**. Spec + contracts list drafts as a first-class `status`, but neither specifies whether the admin `/admin/invoices` landing page shows drafts **by default**. If drafts are shown, the admin list conflates "working state" with "legal documents" — financial reporting screens get polluted. If drafts are hidden, admins have to remember to filter. | Decide + document: default filter `status ∈ {issued, paid, void, credited, partially_credited}` (exclude drafts); drafts accessible via a clearly-labelled "Drafts" tab or filter pill with a count badge. Add to contracts § 1.2 GET /api/invoices: "Default behaviour: if `status` query param absent, drafts are excluded; clients must pass `status=draft` (or `status=all`) to include drafts." Small UI change, big clarity improvement. |
| R2-P3 | 💡 | **Auto-email template content is unspecified**. FR-024 mandates bilingual auto-email with PDF attachment, but the spec + plan + contracts do not specify: (a) subject line format, (b) email body copy (greeting, summary, CTA), (c) which i18n keys hold these strings, (d) where the sender identity comes from (tenant legal name vs. platform generic). This is a real implementation blocker — the dev writing `dispatch-outbox.ts` will have to invent content. | Add a plan § "Auto-email templates" listing: (a) subject key namespace `admin.invoices.autoEmail.subject.{issued,paid,voided,creditNote}` — suggested defaults like `"Tax Invoice #{document_number} — {tenant_legal_name}"` (bilingual); (b) body template file locations `src/modules/invoicing/infrastructure/email/templates/{issued,paid,voided,credit-note}.tsx` using `@react-email/components` (already in stack); (c) `From:` uses tenant-configured "from name" with `billing@{tenant_subdomain}.zyncdata.app` fallback; (d) `Reply-To:` set to tenant admin email. Reuses F3's Resend wrapper. |

### Engineering Lens Findings 🔬

#### Failure Mode Analysis

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| R2-E1 | 💡 | **Credit-note concurrency — `SELECT FOR UPDATE` on parent invoice is mentioned in plan but not encoded in data-model invariant**. Plan § VIII Reliability states: "issue credit note → state guard (paid invoice only) → partial-accumulation guard → advisory lock on credit-note sequence …". The partial-accumulation invariant is in data-model.md § 2.3 as an Application-layer invariant: "before insert, `invoices.credited_total_satang + NEW.credit_amount_satang <= invoices.total_satang`. Enforced transactionally via `SELECT … FOR UPDATE` on the parent invoice row." But this is only in prose — no DB-level CHECK constraint and no explicit test scenario for concurrent partial-credit issuance on the same invoice. | Add to `tests/integration/invoicing/credit-note-partial-accumulation.test.ts` a new scenario: 2 admins issue partial credits of 60% each on the same paid invoice simultaneously via `Promise.all()` — assert exactly one succeeds and the other fails with a clean `conflict` error (the one that lost the FOR UPDATE lock gets "credit amount exceeds remainder"). This directly tests the race the plan describes. |
| R2-E2 | 💡 | **Concurrent member-archive during invoice issuance race**. Admin A begins issuing an invoice for member X; admin B archives member X at the same moment. If B's archive commits first, A's issue succeeds but the invoice's snapshotted member data is correct while the live member row is archived — consistent. But: the F3 archive use case revokes the member's portal session (FR-005 in F3) — an in-flight issue on an archived member is philosophically invalid. Does F4 issuance check member status? Spec doesn't say. | Add to `issue-invoice.ts` use case: at start, take `SELECT … FOR UPDATE` on the target member row + check `status = 'active'`; if archived, fail with `conflict` + clear message "Cannot issue invoice for an archived member — undelete first". This aligns with F3's FR-005 soft-delete semantics + prevents the edge case. Document in spec Edge Cases § and add a scenario to tenant-isolation test suite. |
| R2-E3 | 💡 | **Overdue emission duplication under concurrent reads**. `research.md § 5` states overdue is derived at read time, but `invoice_overdue_detected` is emitted "opportunistically on the first admin/member read per day" comparing against `last_overdue_emission_at`. If two admins read the same overdue invoice in the same second, both writes race; without a guard, the audit log gets two entries for one day. | Implement emission via an `INSERT INTO audit_log … ON CONFLICT (tenant_id, event_type, (payload->>'invoice_id'), DATE(timestamp)) DO NOTHING` pattern — requires a partial unique index. Document in `data-model.md § 4` and in `research.md § 5`. Alternative: use a `SELECT … FOR UPDATE SKIP LOCKED` pattern on the invoice row. Recommend the ON CONFLICT pattern — no extra lock contention, strict idempotency. |

#### Security & Privacy

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| R2-E4 | 🤔 | **Tax-ID snapshot at issue vs. payment — Thai RD expectation unclear**. FR-011 snapshots member identity (incl. tax_id) at issue time. If the member corrects their `tax_id` between issue and payment (a real edge case — new company registration, Thai RD re-issues the VAT registration certificate), the subsequent receipt renders the OLD (issue-time) tax_id, not the CURRENT one. Thai RD's position on which tax_id appears on a receipt for a payment received by a payer whose tax_id has since changed: probably the tax_id on the original invoice (legal continuity), but a bookkeeper may expect the current one. | **Question for you / Thai-tax reviewer**: Which tax_id should appear on the receipt when the member's tax_id has changed between invoice issue and payment? **Recommendation pending input**: use the invoice-time snapshot (legal continuity; matches how Thai RD audits documents as a pair — receipt references invoice). Add explicit spec Assumption: "Receipts render the tax_id and identity as snapshotted at invoice issue time, not at payment time. If a member's tax_id changes mid-cycle, admin must void + reissue or ask Thai RD for guidance." |

#### Operational Readiness

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| R2-E5 | 💡 | **Blob logo orphan cleanup on tenant-logo replacement**. When admin uploads a new logo, the old `logo_blob_key` becomes orphan (no DB references it any more, but the Blob object lingers). Over a year this is trivial (dozens of KB per tenant). But: historical invoices' `tenant_identity_snapshot.logo_blob_key` could reference the OLD logo — if we delete it, historical PDFs re-rendered (after Blob loss + re-render) would have a broken logo reference. | Two options: (a) **Never delete logos** — accept minor Blob growth as the cost of deterministic historical rendering. Add a metric `invoicing.logo_blob.count` per tenant to monitor; intervene only if growth becomes meaningful. (b) **Reference-count logos** — track which invoices reference which logo blob key; delete when count = 0. Complex. Recommend **(a)** with a cap audit: "tenant logo upload blocked if the tenant has >50 historical logos" (highly unusual; nudges admin to clean up manually via support). Document in `research.md § 12`. |
| R2-E6 | 💡 | **Multipart upload middleware support on Vercel Fluid Compute**. Round-1 added `POST /api/tenant-invoice-settings/logo` accepting multipart. Next.js App Router on Vercel Fluid Compute supports multipart via the standard `Request.formData()` API, but the body-size limit (4.5MB default on Vercel) must be confirmed above our 1MB logo cap. Also: the `Idempotency-Key` header is required on all mutations per contract shared-headers; does the logo endpoint honour it? | Confirm in contracts § 3.3 that the logo upload accepts `Idempotency-Key` (replays return the same `logo_blob_key` for 24h) AND that Vercel's multipart body-size default (4.5MB) comfortably exceeds our 1MB cap (true — no config needed). Add a one-line assertion in `logo-upload-security.test.ts`: "duplicate POST with same Idempotency-Key returns identical `logo_blob_key` without re-uploading to Blob." |
