# Feature Specification: F8 ‚Äî Renewal Tracking + Smart Reminders

**Feature Branch**: `011-renewal-reminders`
**Created**: 2026-05-03
**Status**: Draft
**Input**: User description: "F8 ‚Äî Renewal Tracking + Smart Reminders Full feature"

## Overview *(non-mandatory context)*

Chamber-OS members pay annual (or multi-year for partnership tiers) membership fees and receive a bundle of benefits in return. Today, two failure modes hurt the chamber:

1. **Silent churn** ‚Äî members drift away over the year (no events attended, no benefits used) and quietly fail to renew. Admin notices only after the relationship has ended.
2. **Generic, ignored reminders** ‚Äî a single one-size-fits-all "Your membership expires in 30 days" email is the same for a 1,000 THB Thai-Alumni member and a 200,000 THB Diamond Partnership. Both groups feel mis-served: small members feel pestered, big members feel under-cared-for.

F8 closes both gaps by combining three differentiator capabilities (catalogued in `docs/smart-chamber-features.md`):

- **Smart feature #3 ‚Äî Smart Renewal Reminders** ‚Äî tier-aware schedule with email cadence calibrated to member size + decision-making complexity.
- **Smart feature #2 ‚Äî At-Risk Member Detection** ‚Äî rule-based 0‚Äì100 score that flags churn risk 90+ days before expiry so admin can intervene early.
- **Smart feature #16 ‚Äî Auto Tier-Upgrade Suggestions** ‚Äî surfaces members whose declared turnover or paid-invoice history qualifies them for a higher tier.

It also delivers the **Renewal Pipeline Dashboard** that admins use as their daily heads-up display for the next 30/60/90 days.

F8 is **multi-tenant aware** (Constitution v1.4.0 Principle I) ‚Äî every entity carries `tenant_id`, every cron iterates tenants in isolation, every query is RLS-scoped, and a cross-tenant integration test is a Review-Gate blocker.

F8 is **PII-sensitive** (member contact emails, plan history, invoice history). It requires ‚â•2 reviewers at the Review gate.

F8 emails are **transactional** (renewal lifecycle communications) ‚Äî they reuse the F1+F4 Resend transactional surface, NOT the F7 Resend Broadcasts surface, and they are NOT subject to F7's marketing unsubscribe list. Each email still carries a "Manage renewal preferences" link to a tenant-scoped preference page (FR-016).

### Session 2026-05-03

- Q: Membership year anchor model ‚Äî per-member vs calendar-year vs per-tenant configurable? ‚Üí A: **Per-member anchored** (`expires_at = joined_at + plan.term_months` on first cycle; advances by `plan.term_months` on each renewal). Smooth year-round cron load; matches F4 invoicing model and SweCham 2026 Membership Package PDF.
- Q: Schedule-policy bucket level ‚Äî 5 fixed tier buckets vs per-F2-plan vs hybrid? ‚Üí A: **5 fixed tier buckets** (`thai_alumni` / `start_up` / `regular` / `premium` / `partnership`). F2 plan rows carry a `renewal_tier_bucket` enum field; one schedule policy per bucket per tenant. Admin maintains 5 schedules per tenant.
- Q: At-risk score behavior when F6 EventCreate data is unavailable (F8 implements before F6 ships in Phase 2)? ‚Üí A: **Skip the event-attendance factors when F6 module is not detected as active**. Score is computed from the remaining 6 factors; max possible score drops from 105 to 70; band thresholds are proportionally adjusted (warning ‚â•17, at-risk ‚â•35, critical ‚â•52). When F6 ships, the factors auto-activate without F8 code change ‚Äî detection via a feature-port `EventAttendeesPort.isAvailable()`. Combined with go-live gate (next bullet) the gap is internal-only.
- Q: F8 production go-live timing? ‚Üí A: **F8 ships dark in production (kill-switch `FEATURE_F8_RENEWALS=false`) until F9 (Admin Dashboard + Directory + Timeline + Audit Viewer) ships**. F8 backend, cron jobs, audit, and module are deployed and exercised in staging end-to-end before F9; the chamber begins using F8 only when F9 provides the surrounding admin shell that hosts the renewal pipeline + at-risk widget + tier-upgrade suggestions consistently. Because F6 ships before F9 in Phase 2 ordering (F5 ‚Üí F7 ‚Üí F8 ‚Üí F6 ‚Üí F9), the at-risk event-attendance factors are guaranteed live by the time F8 activates in production ‚Äî Option B above is purely a dev/staging/E2E concern.
- Q: Multi-year contract reminder schedule behavior (Partnership tier 3-year contracts)? ‚Üí A: **Email reminders fire only in the final year before `expires_at`; manual escalation tasks (quarterly review, benefit-fulfillment report) repeat every year of the multi-year cycle**. Year-1 and year-2 of a 3-year cycle generate annual escalation tasks rebased on a `year_in_cycle` index so the executive director keeps visibility into deliverables; only year-3 (the final year) generates the T-120/T-90/T-60/T-30/T-7 email payment-reminder cadence. Avoids spamming Diamond partners with "expires in 30 days" emails 24 months early while preserving internal accountability checks.
- Q: Auto-downgrade tier suggestions in MVP (Premium‚ÜíRegular when turnover drops)? ‚Üí A: **Upgrade-only in MVP**. Cron suggests Regular‚ÜíPremium / Start-up‚ÜíRegular / etc. only ‚Äî auto-downgrade is explicitly out of scope. The downgrade conversation is sensitive and admin should handle it manually with full member context. Auto-downgrade can ship as a follow-up if validated with chambers later.
- Q: Manager-role access to F8 pipeline + widgets (round 2)? ‚Üí A: **Manager = full read-only on all F8 admin surfaces** (pipeline, at-risk widget, tier-upgrade queue, escalation tasks). Mutating CTAs (Send reminder ¬∑ Snooze ¬∑ Contact ¬∑ Accept ¬∑ Dismiss ¬∑ Mark done ¬∑ Reassign ¬∑ Manual outreach record) MUST be hidden or disabled when `actor_role === 'manager'`. Matches F1+F2+F4 manager pattern (read-only finance/dashboards). Board treasurer + executive director get visibility without lateral risk.
- Q: Definition of "active member" for F8 cron eligibility (round 2)? ‚Üí A: **`members.status === 'active'` AND `renewal_cycle.status NOT IN ('lapsed', 'cancelled')`**. Filter applies to all 3 F8 cron jobs (daily reminder dispatch, weekly at-risk recompute, weekly tier-upgrade evaluate). `lapsed` members appear in the "Lapsed" tab of the pipeline dashboard but are NOT re-reminded or re-scored; `archived` (F3) and `cancelled` cycle members are excluded entirely from cron passes.
- Q: Lapsed-portal blocked-action scope (round 2)? ‚Üí A: **Read-own-data + renewal-payment only** ‚Äî lapsed members can sign in and access exactly: `/portal/profile` (read-only), `/portal/renewal/<member_id>` (renewal flow + payment), `/portal/billing` (read-only invoice history). Every other action (E-Blast submission F7, event registration F6, profile edit F3, contact edit F3, preferences edit) is blocked with 403 + redirect to `/portal/renewal` carrying a banner "Your membership has lapsed; renew to restore full access". Audit event `lapsed_member_action_blocked` carries the attempted route + action.
- Q: Email-bounce escalation threshold for `email_unverified` flag (round 2)? ‚Üí A: **Differentiated thresholds based on Resend bounce classification**: `1 hard-bounce` (permanent: mailbox not found, domain doesn't exist) OR `3 soft-bounces within the same reminder schedule cycle` (transient that became persistent for one cycle) OR `5 soft-bounces rolling 30 days` (transient pattern even across cycles) ‚Üí set `members.email_unverified = true` + transition cycle to `manual_outreach_required` + emit audit event `member_email_unverified_threshold_crossed` with `{trigger: 'hard_bounce' | 'soft_streak' | 'soft_rolling', bounce_count, classification}`. Reads bounce events from F1's `email_delivery_events` table.
- Q: Tier-upgrade acceptance scheduling when next renewal cycle is far away (round 2)? ‚Üí A: **`TierUpgradePending` state + immediate member email + mid-period reminder + apply at next renewal**. On admin "Accept": (1) suggestion transitions to status `accepted_pending_apply` with `accepted_at` + `accepted_by_user_id` + `target_apply_at_cycle_id` recorded; (2) member receives a single transactional email "Your upgrade to {plan} has been approved; it takes effect at your next renewal on {expires_at}"; (3) if `expires_at - today > 180 days`, an admin escalation task `verify_pending_tier_upgrade` is scheduled for T-180; (4) at next renewal cycle rollover, F4's `createMembershipInvoice` reads pending suggestions for the member and issues the invoice at the upgraded plan price. Audit events: `tier_upgrade_accepted` (existing) + `tier_upgrade_pending_member_notified` (new) + `tier_upgrade_applied_at_renewal` (new) + `tier_upgrade_pending_admin_verification_due` (new).

### Session 2026-05-03 (round 3)

- Q: Lapsed-member reactivation policy on payment success? ‚Üí A: **Auto-reactivate on payment success** ‚Äî when lapsed member completes self-service renewal payment, cycle transitions immediately to `renewed_post_lapse` and access is restored. **Admin override**: `members.blocked_from_auto_reactivation BOOLEAN NOT NULL DEFAULT FALSE` (set per-member by admin); when TRUE, payment-success transitions cycle to `pending_admin_reactivation` state instead, admin review required. Default behaviour applies industry-norm friction-free recovery; per-member block handles compliance edge cases without forcing the heavy default. Per-tenant "require admin re-approval after N days" model deferred to OOS-13 (post-MVP).
- Q: Plan price at renewal ‚Äî frozen-at-cycle-start vs live-at-confirm-time? ‚Üí A: **Frozen-at-cycle-start** ‚Äî `renewal_cycles` carry `frozen_plan_price_thb` + `frozen_plan_term_months` + `frozen_plan_currency` snapshotted at cycle creation. Reminder email, portal display, and F4 invoice all use the frozen price. F2 plan price changes during an open cycle do NOT affect that cycle's invoice (next cycle picks up the new price). Predictable invoicing + member trust + zero "email said X but invoice charged Y" complaints. Per-tenant configurability deferred to OOS-14.
- Q: F4 invoice-paid hook implementation choice for `markCycleCompleteFromInvoicePaid` integration? ‚Üí A: **Option A ‚Äî callback parameter on F4's `markPaidFromProcessor`**. F4's use-case accepts an optional `onPaidCallbacks: ((evt: F4InvoicePaidEvent) => Promise<void>)[]` parameter populated at composition-root wiring; F8 pushes the `markCycleCompleteFromInvoicePaid` callback into the array; F4 invokes each callback inside the same DB transaction that persists the invoice state change. **Atomic guarantee preserves FR-023 intent** (cycle complete + invoice paid + receipt email enqueue + reminder cancel all in one tx). Smallest F4 contract change (~10 lines + 1 contract test). F4 maintainer coordination locked in.
- Q: Renewal-link URL format / domain? ‚Üí A: **`resolveTenantFromRequest()` abstraction (F1's existing tenant-context resolver) ‚Äî compatible with both single-tenant MVP and future multi-tenant**. (Revised at /speckit.critique 2026-05-03 round 2 / M4.) **MVP era (single tenant SweCham)**: URL = `https://swecham.zyncdata.app/portal/renewal/<memberId>?token=<...>`; `resolveTenantFromRequest()` returns the constant `env.tenant.slug` ("swecham") per `src/lib/tenant-context.ts`; F8 verifier cross-checks `payload.tid === resolveTenantFromRequest()` (defence-in-depth via constant comparison). **Post-F10 era (multi-tenant SaaS)**: URL = `https://<tenant>.zyncdata.app/portal/renewal/<memberId>?token=<...>` OR custom-domain via F12; `resolveTenantFromRequest()` returns per-request resolved tenant from subdomain/host header; F8 verifier code UNCHANGED ‚Äî abstraction handles both eras. F8 does NOT extend F1's middleware (that's F10's job). Member-trust + CSP/cookie-scoping benefit applies post-F10.
- Q: At-risk score factor weights tunability? ‚Üí A: **Locked single canonical formula** ‚Äî 8-factor weights hardcoded in `src/modules/renewals/domain/at-risk-score.ts` per `smart-chamber-features.md` ¬ß 3 baseline. ALL tenants use the same formula. Re-calibration = code release (revisited if 6+ tenants surface a justified delta). Pros: Constitution Principle X (Simplicity/YAGNI), product team learns cross-tenant patterns simultaneously, single test fixture, no config UI surface. Per-tenant configurable weights deferred to OOS-15 (post-MVP if validated need from multi-tenant data).

## User Scenarios & Testing *(mandatory)*

### User Story 1 ‚Äî Renewal Pipeline Dashboard (Priority: P1)

Chamber admin opens `/admin/renewals` and immediately sees, for the current and next 90 days, every member by renewal-due date ‚Äî grouped and color-coded by tier and urgency. From this single view they can drill into any member, send/resend a reminder, mark a manual outreach as done, or open the member detail page.

**Why this priority**: This is the single screen that replaces the daily "scroll through the Excel sheet looking for upcoming expiries" workflow. It is the smallest unit of F8 that delivers immediate operational value with zero email automation ‚Äî admins can use it day one even if every other US ships later.

**Independent Test**: Seed 30 members with mixed tiers and expiry dates spread across the next 120 days. Open `/admin/renewals` as an admin user. Verify that (a) the dashboard renders in <500ms, (b) every member with `expires_at ‚â§ now() + 90 days` is shown, (c) tier color/grouping matches the catalog, (d) urgency labels (T-90/T-60/T-30/T-14/T-7/T-0/grace) match the computed date math, (e) no member from a different tenant appears.

**Acceptance Scenarios**:

1. **Given** the admin is signed in to tenant `swecham` and 5 members are within 30 days of expiry, **When** they open `/admin/renewals`, **Then** all 5 members appear with urgency `T-30` or shorter, sorted by expiry-date ascending, and each row shows tier badge, expiry date, last-reminder-sent timestamp, and quick actions (Send reminder ¬∑ Open member ¬∑ Mark contacted).
2. **Given** the admin filters by tier "Premium", **When** the filter applies, **Then** only Premium members appear in the table without a page reload, and the URL updates with a query parameter so the filter is shareable/bookmarkable.
3. **Given** a member is past their grace period, **When** the dashboard loads, **Then** the member is shown in a separate "Lapsed" tab with reason badges (`grace_expired`, `payment_failed`, etc.) and an "Archive" or "Reactivate" action.
4. **Given** an admin from tenant `tscc-test` opens the dashboard, **When** the page loads, **Then** zero members from tenant `swecham` appear, and a synthetic cross-tenant probe (URL hand-craft `?member_id=<swecham-id>`) returns 404 with audit event `renewal_cross_tenant_probe` recorded.
5. **Given** the dashboard is loaded for a tenant with 5,000 active members and 600 within the 90-day window, **When** the admin scrolls and filters, **Then** p95 render time stays below 500ms (FR-046 / SC-003) using server-side pagination + virtualization where rows exceed 100.

### User Story 2 ‚Äî Tier-Aware Smart Reminder Schedule (Priority: P1)

The system automatically dispatches scheduled renewal-reminder emails to each member according to a **tier-calibrated schedule**, in the member's preferred locale, with content tailored to their plan, benefit summary, and outstanding invoice. A daily cron job evaluates every active member and sends only the reminders that are due, idempotently.

**Why this priority**: This is the core anti-churn automation. Members no longer fall off the radar between events. The schedule is published from a tier-aware policy table so admins can see exactly when members will be contacted and adjust if a tier's cadence isn't working.

**Independent Test**: Seed 4 members on different tiers (Thai Alumni, Regular, Premium, Partnership) with `expires_at = today + 30 days`. Run the renewal-cron job once. Verify that (a) only the tiers whose schedule includes a `T-30` step receive an email, (b) each email is rendered in the recipient's `preferred_locale` (EN / TH / SV), (c) re-running the cron a second time on the same day sends ZERO additional emails (idempotency), (d) audit events `renewal_reminder_sent` are emitted for every dispatch with template-id, tier, offset-day metadata.

**Acceptance Scenarios**:

1. **Given** a Premium-tier member's `expires_at` is exactly 30 days away and their schedule includes a `T-30` step, **When** the daily cron runs, **Then** a reminder email is dispatched in the member's preferred locale, the `RenewalReminderEvent` row is persisted with status `sent`, and an audit event `renewal_reminder_sent` carries `{tier: 'premium', offset_day: -30, template_id: 'renewal.t-30.premium', delivery_id: 'rsd_‚Ä¶'}`.
2. **Given** the same member's reminder was already dispatched in a previous cron run, **When** today's cron re-evaluates them, **Then** the cron logs `idempotency_hit` (audit event `renewal_reminder_skipped` with `reason: 'already_sent'`) and does NOT call the email gateway again.
3. **Given** a Partnership-tier member is at `T-90`, **When** the cron runs, **Then** the system emits BOTH (a) a transactional email (per the Partnership schedule) AND (b) a manual escalation task entry (`renewal_escalation_task` of type `quarterly_review_meeting`) so the executive director sees it in their queue (US6).
4. **Given** a member's `preferred_locale` is `sv` and the renewal email template's `sv.json` translation is missing, **When** the cron renders the email, **Then** the system falls back to `en` per the standard i18n fallback rule, logs a non-fatal warning, and does NOT block dispatch.
5. **Given** a member's primary contact email has hard-bounced (recorded by F1+F4 transactional Resend webhook), **When** the cron evaluates them, **Then** the cron skips the email dispatch, emits audit event `renewal_reminder_skipped` with `reason: 'email_unverified'`, and creates a manual escalation task `manual_outreach_required` so an admin can reach the member another way.
6. **Given** the kill-switch `FEATURE_F8_RENEWALS=false`, **When** the cron triggers, **Then** the cron handler returns 200 with `{ skipped: true, reason: 'feature_flag_disabled' }`, does NOT enqueue any work, and emits NO audit events (no-op).
7. **Given** an admin manually triggers "Send reminder now" from the dashboard for a Regular-tier member at T-7, **When** the action submits, **Then** the email is dispatched immediately via the same use-case (single dispatch path), the dashboard shows a toast "Reminder sent to <email>", and the audit event records `actor_user_id` as the admin (NOT the cron service principal).

### User Story 3 ‚Äî Member Self-Service Renewal (Priority: P2)

A member receives a tier-aware reminder email, clicks the "Renew now" link, signs in to `/portal/renewal` (signed token enables sign-in if their session expired), reviews their benefit consumption summary for the closing year, optionally requests a tier change, confirms the renewal, is redirected to F4's invoice-issuance flow ‚Üí F5's online payment flow, and on payment success sees a confirmation page with their new `expires_at` and a download-receipt link.

**Why this priority**: Eliminates the admin-touch on every renewal. Most renewals are routine and members are happy to self-serve. Frees admin time to focus on the at-risk and high-value members who need human attention.

**Independent Test**: Seed a Regular-tier member with `expires_at = today + 14 days`. As that member, open the email link ‚Üí portal ‚Üí confirm renewal ‚Üí mock-pay via F5 test card. Verify that (a) F4 issues a renewal invoice with the correct period (`expires_at` to `expires_at + 1y`), correct amount per the member's current plan, correct VAT, (b) F5 captures the payment and the F4 invoice transitions to `paid`, (c) the member's `expires_at` advances by 1 year, (d) audit events are emitted at each transition (`renewal_self_service_initiated`, `renewal_invoice_created`, `renewal_completed`), (e) a tax receipt PDF is delivered via the F4 dispatcher.

**Acceptance Scenarios**:

1. **Given** a member receives the T-30 reminder email and clicks the "Renew now" link, **When** the link's signed renewal-token verifies successfully, **Then** they are auto-signed-in to a member session and redirected to `/portal/renewal/<member_id>` showing their current plan, expiry date, benefit-consumption summary, and a "Confirm renewal" button. **First-time renewers** (members who have never previously self-served a renewal) see a one-time onboarding banner explaining the 3-step flow (Confirm ‚Üí Pay ‚Üí Receipt) with a link to the chamber FAQ; banner dismissable but reappears the next year on first renewal of that cycle.
2. **Given** the member is on the renewal page and clicks "Confirm renewal", **When** the form submits, **Then** F4's `createMembershipInvoice` use-case is invoked with `period: { from: <expires_at>, to: <expires_at + 1y> }`, the member is redirected to F5's `/portal/billing/<invoice_id>/pay` page, and the audit event `renewal_invoice_created` is emitted with `{member_id, plan_id, invoice_id, period_from, period_to, amount, vat}`.
3. **Given** the member completes payment via F5 (Stripe Card or PromptPay), **When** F5 records `payment_succeeded` and F4 transitions the invoice to `paid`, **Then** F8 listens for the F4 `invoice_marked_paid` event, advances `members.expires_at`, transitions the renewal-cycle to `completed`, cancels any remaining reminder-schedule entries for that cycle, and dispatches a "Welcome back for <year>" confirmation email with a tax-receipt download link.
4. **Given** payment fails (F5 returns `payment_failed`), **When** the member is bounced back to the portal, **Then** the renewal cycle remains `awaiting_payment`, the renewal invoice remains `unpaid`, the reminder schedule resumes (member will get the next scheduled nudge), and an audit event `renewal_payment_failed` is emitted with the F5 failure reason.
5. **Given** the member opens the portal at the renewal page after `expires_at` has already passed, **When** the page renders, **Then** the page shows a "Your membership lapsed on <date>; renew now to restore access" message, the renewal flow still works, and on successful payment the cycle transitions from `lapsed` to `renewed_post_lapse` (audit event `renewal_completed_post_lapse`) **with full access auto-restored per FR-005b** (default policy). If `members.blocked_from_auto_reactivation === TRUE`, the cycle instead transitions to `pending_admin_reactivation` and an admin task `manual_admin_reactivation_review` is created; the member sees a "Payment received; awaiting admin review to restore access" page until admin acts.
6. **Given** the member wants to change their plan during renewal (upgrade or downgrade), **When** they click "Change plan" on the renewal page, **Then** they see a tier comparison rendered from F2's plan catalogue, can select a new plan, and on confirm the renewal invoice is issued at the NEW plan's price (with audit event `renewal_with_plan_change` carrying both old and new plan IDs).
7. **Given** the renewal token in the email has expired (>30 days old) or has been used once already, **When** the member clicks the link, **Then** they see a "This renewal link has expired or already been used; sign in normally to continue your renewal" page, and the audit event `renewal_token_invalid` is emitted with the failure reason (`expired` / `replay`) but does NOT leak whether the token was ever valid.

**AS-Note (1) ‚Äî `isFirstTimeRenewer` default is `false`** (Staff-Review-2026-05-09 SUG-8 closure). When the renewal-summary load encounters insufficient signal to definitively classify a member as a first-time renewer (e.g. F8 has not yet shipped the `consumed_link_tokens` historical lookup, or the cycle's member has never previously held an active membership cycle), the field defaults to `false`. Rationale: a false-negative on the onboarding banner is strictly preferable to a false-positive ‚Äî a returning member who already understands the flow seeing the banner anyway is a one-time visual annoyance, but a true first-time renewer who never sees the banner is left to guess at the 3-step flow with no FAQ link. The banner is dismissible (`localStorage` keyed on `cycle_id`), so re-renders are bounded. Tracked at `src/modules/renewals/application/use-cases/load-renewal-summary.ts:221`.

### User Story 4 ‚Äî At-Risk Member Detection (Priority: P2)

A weekly cron job recomputes a 0‚Äì100 risk score for every active member based on objective behavioural and financial signals (event attendance, benefit-quota consumption, overdue invoices, days-since-last-payment, contact freshness, recent tier downgrade). Admin sees an "At-Risk Members" widget on the dashboard sorted by score, can click "Contact" to open a context-prefilled email, can "Snooze" a member for N days to suppress repeat alerts, and the score history is auditable.

**Why this priority**: Early-warning is the moat ‚Äî by the time a member doesn't renew, they're already gone. Catching them at T-90 with a personalised outreach has measurably higher save rates than a generic T-30 reminder. Independent of US2/US3 because at-risk insights are useful even without the renewal pipeline.

**Independent Test**: Seed 10 members with synthetic engagement profiles (some perfect, some with overdue invoices, some with no event attendance for 6 months). Run the at-risk cron once. Verify that (a) every member has a `risk_score` in [0, 100] within 60 seconds of cron start (SC-005), (b) the score correctly weights each factor per the rule table (FR-029), (c) members above the 50-threshold appear in the dashboard widget sorted descending, (d) clicking "Snooze" hides the member for the configured period, (e) snooze auto-expires.

**Acceptance Scenarios**:

1. **Given** a member has 0 events attended in the last 12 months, 1 invoice overdue, and 280 days since last payment, **When** the at-risk cron runs, **Then** their score is computed as `25 + 25 + 10 = 60` (per the rule table FR-029) and audit event `at_risk_score_recomputed` is emitted with `{member_id, score, factors: [...], threshold_band: 'at-risk'}`.
2. **Given** a member's score crosses from 45 (warning) to 60 (at-risk) on this cron run, **When** the score is persisted, **Then** an additional audit event `at_risk_score_threshold_crossed` is emitted with `{from_band: 'warning', to_band: 'at-risk'}` so audit reports can flag the deterioration.
3. **Given** an admin clicks "Snooze" on a member with score 78, **When** they pick "30 days" and confirm, **Then** `members.risk_snoozed_until` is set to `today + 30d`, the member disappears from the at-risk widget for the next 30 days, audit event `at_risk_snoozed` is emitted, and on day 31 the member reappears (snooze auto-expires; cron re-evaluates).
4. **Given** an admin clicks "Contact" on an at-risk member, **When** the admin chooses "Email" and confirms a pre-filled draft, **Then** a tracked outreach record is created (`at_risk_outreach`) and the audit event `at_risk_outreach_recorded` carries `{member_id, channel: 'email', template_id: 'at_risk.outreach.event_drought'}`.
5. **Given** the at-risk widget is shown to a `member` role user (the at-risk member themselves), **When** the page loads, **Then** the widget is NOT rendered (admin-only by design ‚Äî at-risk visibility to the member would be demotivating and risks self-fulfilling churn). FR-035.
6. **Given** the at-risk cron encounters a transient DB error mid-tenant, **When** the error is raised, **Then** the cron logs the failure for that tenant, continues to the next tenant (per-tenant fault isolation), emits audit event `at_risk_compute_partial_failure`, and the failed tenant's scores remain at their previous values until the next successful run.

### User Story 5 ‚Äî Auto Tier-Upgrade Suggestions (Priority: P3)

A weekly cron job evaluates each active member's declared turnover and 12-month paid-invoice history against the eligibility thresholds of higher tiers (per F2's plan catalogue). When a member objectively qualifies for an upgrade (e.g., a Regular-tier member's declared turnover crossed 100M THB, or their 12-month invoice spend with the chamber crossed N√ótheir plan's annual fee), the system creates a `TierUpgradeSuggestion` row and surfaces it in the admin dashboard. Admin can accept (triggering the F2 plan-change flow), dismiss (suppressing for 90 days), or escalate (drafting an outreach email).

**Why this priority**: Revenue uplift, not churn prevention. Less critical than US2/US3/US4 because the chamber survives without it, but every accepted upgrade is direct revenue. P3 because the rule engine is small and can ship after the renewal core lands.

**Independent Test**: Seed a Regular-tier member with `declared_turnover_thb: 120_000_000` (above the Premium threshold of 100M). Run the tier-upgrade cron. Verify that (a) a `TierUpgradeSuggestion` row is created with `from_plan: regular, to_plan: premium, reason_code: 'declared_turnover_above_threshold', evidence: {turnover: 120000000}`, (b) the dashboard widget shows the suggestion, (c) admin "Accept" triggers F2's `changeMemberPlan` use-case + audit event, (d) admin "Dismiss" sets `suppressed_until: today + 90d`.

**Acceptance Scenarios**:

1. **Given** a Regular-tier member with `declared_turnover_thb` newly updated to 120M (above the Premium 100M threshold), **When** the upgrade cron runs, **Then** a `TierUpgradeSuggestion` is created and audit event `tier_upgrade_suggested` is emitted; the cron is idempotent so re-running on the same data does NOT create a duplicate.
2. **Given** an admin reviews a suggestion in the dashboard, **When** they click "Accept" with confirmation, **Then** F2's `changeMemberPlan` use-case is invoked with the suggested target plan, the member's plan transitions on the next renewal cycle (NOT immediately mid-year ‚Äî to avoid surprise mid-year invoicing), audit event `tier_upgrade_accepted` is emitted, and the member is notified by email.
3. **Given** an admin clicks "Dismiss" on a suggestion, **When** they confirm with an optional reason, **Then** the suggestion's status becomes `dismissed`, `suppressed_until` is set to `today + 90d`, audit event `tier_upgrade_dismissed` is emitted with the reason, and the cron will not re-suggest the same upgrade for that member for 90 days.
4. **Given** a member's plan has already been changed manually in F2 to the same target the cron would suggest, **When** the cron evaluates them, **Then** no suggestion is created (the cron checks current `plan_id` BEFORE inserting) and an audit event `tier_upgrade_already_at_target` is emitted with `level: debug`.
5. **Given** the cron runs against a tenant that has not configured turnover thresholds, **When** the cron evaluates members, **Then** the cron emits `tier_upgrade_skipped_no_thresholds_configured` per tenant once, completes successfully, and never crashes another tenant's pass.
6. **Given** the auto-upgrade feature is disabled for a tenant via `tenant_renewal_settings.auto_upgrade_enabled = false`, **When** the cron runs, **Then** that tenant is skipped entirely with audit event `tier_upgrade_tenant_disabled` emitted once per cron run.

### User Story 6 ‚Äî Manual Escalation Task Queue (Priority: P3)

For tier-specific manual touchpoints that are NOT email (phone calls from Executive Director for Premium members, in-person meeting for Partnership members, board-escalation for outstanding Diamond renewals), the system creates `RenewalEscalationTask` rows on the appropriate offset day, assigns them to the configured staff role (or to a specific user), and surfaces them in an admin task queue. Admin marks each task done, skipped, or reassigns ownership.

**Why this priority**: P3 because chambers can run manually for these high-value touchpoints in the short term. But shipping this avoids the embarrassment of "the system reminded me at T-30 but no one called the Diamond partner at T-7 and they walked".

**Independent Test**: Seed a Partnership-tier member with `expires_at = today + 60 days` and a tenant config that includes a "T-60: Send benefit-fulfillment report" task. Run the cron. Verify that (a) a `RenewalEscalationTask` row is created with `due_at: today, assigned_to_role: 'admin', task_type: 'benefit_fulfillment_report'`, (b) the admin task-queue page shows the task, (c) marking it "Done" emits audit event `escalation_task_completed`, (d) marking it "Skipped" requires a reason and emits `escalation_task_skipped`.

**Acceptance Scenarios**:

1. **Given** a Partnership-tier member's schedule includes a T-120 quarterly-review-meeting task, **When** the cron creates it, **Then** the task appears in `/admin/renewals/tasks` with the member's name, tier, expiry, suggested action, and links to the member detail page; status is `open`.
2. **Given** an admin completes the task, **When** they click "Done" and add an optional outcome note, **Then** the task transitions to `done`, the outcome note is persisted, and audit event `escalation_task_completed` is emitted with `{task_id, outcome_note?}`.
3. **Given** an admin reassigns a task to a colleague, **When** they pick a user from the assignee selector and confirm, **Then** the task's `assigned_to_user_id` is updated, audit event `escalation_task_reassigned` is emitted, and the new assignee sees it in their queue.
4. **Given** an escalation task is overdue (past `due_at` by ‚â•3 days), **When** the dashboard renders, **Then** the task is highlighted in red and a banner "X overdue tasks" appears at the top of the queue.

### Edge Cases

- **Member with no `expires_at`** (e.g., legacy data, manually created member without a paid invoice yet) ‚Äî they MUST NOT appear in the renewal pipeline; cron skips them with audit event `renewal_skipped_no_expiry`. Admin sees a "Members without renewal cycle: N" banner with a link to fix them.
- **Tenant with no membership-plan duration configured** (F2 plan rows missing `term_months`) ‚Äî the cron emits a per-tenant `renewal_tenant_misconfigured` warning once per run and skips reminder dispatch for that tenant.
- **Member's tier changes mid-reminder-cycle** ‚Äî when F2 issues a `member_plan_changed` event, F8 cancels any remaining reminders scheduled under the OLD tier and reschedules using the NEW tier's policy (audit event `renewal_schedule_rescheduled`). Already-sent reminders are NOT recalled.
- **Cron failure mid-run** ‚Äî the cron uses `SELECT ‚Ä¶ FOR UPDATE SKIP LOCKED` per-tenant and per-member to support safe retry; a failed tenant's pass does NOT block other tenants.
- **Two crons fire simultaneously** (operator triggers manual run during scheduled run) ‚Äî `pg_advisory_xact_lock(hashtextextended('renewals:dispatch:'||tenantId, 0))` per tenant ensures only one dispatch worker runs per tenant at a time; the second blocks until the first commits.
- **Member receives reminder, ignores it, then admin manually marks renewal "Already paid offline"** ‚Äî admin opens member, clicks "Mark renewal as paid offline" ‚Üí enters payment-method + reference ‚Üí F4 invoice issued + marked paid in one transaction ‚Üí all remaining scheduled reminders for that cycle are cancelled with audit event `renewal_cycle_completed_offline`.
- **Member opts out of renewal reminders specifically** (separate from F7 marketing unsubscribe) ‚Äî `members.renewal_reminders_opted_out = true` via the portal ‚Üí cron skips email dispatch but STILL surfaces them in the admin pipeline + escalation tasks (the chamber still has a relationship duty to communicate). FR-016.
- **Renewal pricing change between reminder generation and member click-through** ‚Äî the renewal page always renders the LATEST price from F2 at click-through time; the member sees the actual amount that will be invoiced, not a stale snapshot from the email.
- **Concurrent admin action** (admin manually sends "send reminder now" while cron is dispatching the same step) ‚Äî idempotency guard on `(member_id, cycle_id, schedule_step)` prevents double-send; the second attempt logs `idempotency_hit`.
- **At-risk score for a brand-new member** (joined <30 days ago) ‚Äî the cron suppresses score computation for members below a minimum tenure threshold (`MEMBER_MIN_TENURE_DAYS_FOR_AT_RISK`, default 30); audit event `at_risk_skipped_below_min_tenure`.
- **Bouncing email** ‚Äî per FR-012a (Clarification Q4 round 2): differentiated thresholds based on Resend bounce classification ‚Äî 1 hard-bounce (permanent) OR 3 soft-bounces within a single renewal cycle OR 5 soft-bounces rolling 30-day window ‚Üí `members.email_unverified = true` + cycle transitions to `manual_outreach_required` + escalation task created for admin. Member sees this in the at-risk widget.
- **Member has multiple primary contacts** (data anomaly from F3) ‚Äî F8 uses `members.primary_contact_email` (single field per F3 schema) and assumes F3's invariant holds; if NULL, escalation task `manual_outreach_required` is created.
- **READ_ONLY_MODE=true** ‚Äî all F8 cron handlers and member-facing portal mutating actions return 503; the next cron run after read-only mode is lifted catches up safely (no missed reminders are silently lost ‚Äî they shift one day later, with audit event `renewal_reminder_deferred_read_only`).
- **Tenant's preferred locale changes mid-cycle** ‚Äî emails always render at dispatch time using the recipient's `preferred_locale` at that moment (NOT cached at schedule-creation time).
- **Member is archived (F3) mid-renewal-cycle** ‚Äî F8 cancels the cycle, cancels remaining reminders, audit event `renewal_cycle_cancelled_member_archived`.
- **Member with `joined_at IS NULL` (data hygiene from F3 / Excel import)** ‚Äî System MUST NOT compute `expires_at` for the member; the member appears in the admin "Members without renewal cycle" tray (FR-001 edge-case) AND in a sub-tray "Members with missing join date" so admin can backfill. Cron jobs skip such members entirely. Audit event `renewal_skipped_no_joined_at`. Per /speckit.critique 2026-05-03 round 1 / P13.
- **Zero-tenant cron pass** (resolves checklist security gap CHK032) ‚Äî when a coordinator cron iterates and finds zero active tenants (`SELECT tenant_id FROM tenants WHERE active = TRUE` returns empty), the coordinator MUST return 200 with `{ skipped: false, tenants_enqueued: 0, tenants_succeeded: 0, tenants_failed: 0, duration_ms: <small> }` and emit a single `cron_dispatch_orchestrated` audit with `tenants_enqueued: 0`. NO error, NO alert (alerting is on cron-failure-to-run, not on legitimate empty result).
- **Concurrent admin actions on same cycle/member** (resolves checklist ux gap CHK034) ‚Äî when two admins simultaneously click "Send reminder now" on the same cycle: the FR-011 idempotency primitive (`renewal_reminder_events` unique on `(cycle_id, step_id, year_in_cycle)`) ensures the second action returns 409 with `{error: 'idempotency_hit', existing_event_id, dispatched_at, dispatched_by_actor}`. Frontend renders a toast "Reminder was already sent {timestamp} by {actor_name}" and refreshes the row state. Same idempotency guard applies to: snooze (last write wins per `risk_snoozed_until`), tier-upgrade accept (open suggestion exclusivity per partial unique index), escalation task done (status='done' is terminal). Document in `docs/ux-standards.md` ¬ß Concurrent-Action Handling for cross-feature consistency.
- **Token re-issuance when T-30 fires after T-90 link still valid** (resolves checklist security gap CHK033) ‚Äî each scheduled reminder step generates a FRESH token at dispatch time (NOT at cycle-creation time). Tokens are independent (each is a single-use HMAC blob with its own `iat`/`exp`). The T-90 token continues to verify until its 30-day TTL expires OR until consumed. The T-30 token is independently valid. If a member clicks T-90 token at T-60, completes renewal, the cycle transitions to `completed` ‚Üí all subsequent scheduled reminder steps for that cycle are CANCELLED (FR-023 cancel-remaining-reminders) ‚Üí T-30 token is never generated. If T-30 dispatch DOES fire (e.g., race condition) and member clicks the T-30 link AFTER the cycle is already `completed`, FR-027 verification step 7 (member-tenant ownership check passes) but the cycle's status check returns "already-completed" page (no-op renewal, no double invoice). Audit `renewal_token_clicked_on_completed_cycle` for forensic visibility.

### Functional Requirements

#### A. Renewal lifecycle state machine

- **FR-001**: System MUST compute and persist a per-member `expires_at` derived from the most recent paid membership invoice's billing period (or, if a member has no paid invoice yet, from `joined_at + plan.term_months`). The membership year anchor model is **per-member anchored** (NOT calendar-year) ‚Äî see Assumption A1. Confirmed at `/speckit.clarify` Session 2026-05-03 Q1.
- **FR-002**: System MUST track each member's renewal cycle through the states `upcoming` ‚Üí `reminded` ‚Üí `awaiting_payment` ‚Üí `completed` (terminal) OR `awaiting_payment` ‚Üí `lapsed` (terminal) OR any state ‚Üí `cancelled` (admin override).
- **FR-003**: System MUST honour a per-tenant `grace_period_days` (default 14, configurable per tenant in `tenant_renewal_settings`) ‚Äî during grace, the member retains access but `renewal_cycle.status = awaiting_payment` and they appear in the "Grace" tab of the pipeline dashboard.

- **FR-039**: Suggestions MUST be admin-actionable. **Per Clarification Q5 round 2**, "Accept" follows a `TierUpgradePending` flow (NOT immediate plan mutation):

  3.   4. ## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 95% of active members within 30 days of expiry receive at least one tier-appropriate renewal reminder email at the correct offset-day window, measured per cohort over 30 days.
- **SC-002**: 90% of members who click "Renew now" from a reminder email complete the self-service renewal flow (token verify ‚Üí confirm ‚Üí invoice ‚Üí payment success) in under 3 minutes (median; measured by elapsed time from token-verify span start to payment-success span close).
- **SC-003**: Renewal pipeline dashboard renders at p95 < 500ms for tenants with up to 5,000 active members and 600 members within the 90-day window (FR-046). **Measured 2026-05-04 against live Neon ap-southeast-1**: tenant A p50=258ms ¬∑ p95=293ms ¬∑ p99=310ms; tenant B (RLS-isolated) p95=276ms. Headroom ~41% under budget. Evidence: `tests/integration/perf/renewals-pipeline-perf.test.ts` (RUN_PERF=1 gated).
- **SC-004**: Renewal rate improves by at least 10 percentage points compared to the pre-F8 baseline within 90 days of launch, measured per tenant. **Formula** (per research.md R11): `renewal_rate = count(members whose RenewalCycle.status === 'completed' within expires_at + grace_period_days) / count(members whose previous-cycle expires_at fell within the measurement window)`. **Pre-F8 baseline**: computed from SweCham 2024-2025 admin records using the same formula and documented in `perf-benchmarks.md` before /speckit.implement starts. **Measurement window**: rolling 90-day window starting day +30 after F8 production go-live (warm-up period for first reminder cycle to complete). Confirmed at /speckit.critique 2026-05-03 round 1 / P1 + P14.
- **SC-005**: Daily renewal-dispatch cron, weekly at-risk recompute cron, and weekly tier-upgrade evaluate cron each complete a full pass within 60 seconds for a tenant with 5,000 active members.
- **SC-006**: Zero unauthorised cross-tenant data leaks ‚Äî verified by a Review-Gate-blocking integration test that probes every F8 endpoint with a foreign tenant context and expects 404 / empty result + cross-tenant-probe audit event.
- **SC-007**: 100% coverage of active members in the renewal pipeline ‚Äî every member with `expires_at` set appears in the right urgency bucket; members without `expires_at` are surfaced in a separate "Members without renewal cycle" admin tray (FR-001 edge-case).
- **SC-008**: Cron reminder dispatch is provably idempotent ‚Äî re-running the daily cron 3 times in a row on the same data produces zero duplicate `RenewalReminderEvent` rows and zero duplicate emails (verified by integration test).
- **SC-009**: Every successfully renewed member receives a Thai-tax-compliant invoice + receipt PDF within 10 seconds of payment success (delegates to F4 SLO).
- **SC-010**: Reminder email bounce rate stays below 2% over a rolling 30-day window (sender reputation health for the chamber's transactional domain).
- **SC-011**: At-risk widget admin engagement ‚Äî for tenants where the at-risk widget is enabled, ‚â•80% of weekly at-risk-band-crossing alerts result in either an outreach action or a snooze within 7 days (the widget is being acted on, not ignored).
- **SC-012**: Auto-tier-upgrade suggestions accepted within 90 days of creation: ‚â•30%, OR **fewer than 10 suggestions emitted in the measurement window** (in which case SC-012 is N/A and the chamber reviews tier eligibility thresholds with the F2 maintainer to verify F2 plan rows have the required `eligibility.declared_turnover_thb_min` + `eligibility.lifetime_invoice_thb_min` metadata populated). Prevents division-by-zero / high-variance pathology when historical tenant data is sparse. Confirmed at /speckit.critique 2026-05-03 round 1 / P15.

## Assumptions

- **A1**: Membership year is **per-member anchored** ‚Äî each member's `expires_at` is `joined_at + plan.term_months` for the first cycle and `previous_expires_at + plan.term_months` for renewals. Calendar-year (everyone expires Dec 31) is NOT used. This matches F4's invoicing model and the SweCham 2026 Membership Package PDF tier descriptions, where renewal is implicitly tied to the individual member's anniversary. **Confirmed at `/speckit.clarify` Session 2026-05-03 Q1**.
- **A2**: Auto tier-upgrade NEVER mutates `members.plan_id` automatically ‚Äî only admin approval ("Accept") triggers F2's `changeMemberPlan`. This is the safe default chosen to avoid surprise mid-year invoicing. The cron's job is to surface objective evidence; the chamber-admin's job is to make the call.
- **A3**: Auto-DOWNGRADE suggestions (e.g., a Premium member whose 12-month invoice spend dropped below the Premium floor ‚Üí suggest Regular) are explicitly OUT of scope for the MVP. The downgrade conversation is sensitive and admin should handle it manually with full member context. Auto-downgrade can ship as a follow-up if validated with chambers post-MVP. **Confirmed at `/speckit.clarify` Session 2026-05-03 Q5**.
- **A4**: At-risk score is **admin-only**. Members do not see their own risk score in the portal ‚Äî exposing it risks self-fulfilling churn ("I'm at risk so I'll just leave"). Chambers can surface a softer message ("We've noticed you haven't been to events lately ‚Äî here's what's coming up") via the existing benefit-usage dashboard, but the raw 0‚Äì100 score stays internal.
- **A5**: Renewal reminders are TRANSACTIONAL communications, NOT marketing. They reuse the F1+F4 transactional Resend surface and are NOT subject to F7 marketing unsubscribe. Members can still opt out of renewal reminders specifically (FR-016) but the chamber retains the right to communicate about the contractual relationship (renewal due date) until the relationship ends.
- **A6**: The cron-job.org pattern (Bearer-auth via `CRON_SECRET`, 5-min cadence on Hobby plan) is reused from F4/F5/F7. F8 cron jobs run daily for dispatch and weekly for at-risk + tier-upgrade. No new infrastructure is introduced. Migration to native Vercel Cron when the project moves off Hobby is a configuration change only.
- **A7**: F2 plan rows already carry (or will be extended in F8 to carry) `term_months`, `renewal_tier_bucket`, `eligibility.declared_turnover_thb_min`, and `eligibility.lifetime_invoice_thb_min` ‚Äî F8 does NOT redefine F2's plan model; it consumes existing columns and adds `renewal_tier_bucket` (5-value enum) as a coordinated migration with F2 maintainers. Existing SweCham plans are backfilled to the matching bucket in the same migration.
- **A8**: F4's `createMembershipInvoice` and F5's `createPaymentIntent` use-cases are stable public APIs from their respective module barrels and accept the inputs F8 needs to pass (member_id, plan_id, billing_period). F8 does NOT bypass these and write directly to F4/F5 tables.
- **A9**: F3's `members.primary_contact_email` is the canonical email for renewal communications. If a member has additional contacts (per F3 1:N model), F8 does NOT email them about renewal ‚Äî only the primary contact. This matches F7's primary-contact-only resolution policy.
- **A10**: Members can only have one active renewal cycle at a time. Multi-year contracts (e.g., a 3-year Diamond Partnership) are modelled as a single cycle with `period_to = period_from + 36 months` rather than three back-to-back annual cycles. **Reminder behavior inside a multi-year cycle**: email payment reminders (T-120 / T-90 / T-60 / T-30 / T-7) fire ONLY in the final 12 months before `expires_at`; manual escalation tasks (quarterly review meeting, benefit-fulfillment report, etc.) repeat every year of the multi-year cycle, rebased on a `year_in_cycle` index (1 / 2 / 3 ...) so the executive director maintains visibility into year-1 and year-2 deliverables. Confirmed at `/speckit.clarify` Session 2026-05-03 Q4.
- **A11**: Reminder template content for the SweCham launch is authored in collaboration with the chamber liaison; `docs/smart-chamber-features.md` ¬ß 4 provides the schedule shape but actual copy is content-team work delivered alongside spec implementation.
- **A12 (revised v3 ‚Äî clarified by maintainer 2026-05-03)**: F8 ships **complete in scope** within this branch ‚Äî every US (US1‚ÄìUS6), every FR, every UI surface (admin pipeline, at-risk widget, tier-upgrade queue, escalation task list, member self-service renewal flow, preferences page, schedule-policy editor, tenant settings), every backend use-case, every cron job, every audit event, every test ‚Äî all merged + verified in staging end-to-end before merge. **No scope cuts, no UI placeholders, no "ship pipeline-only and add the rest with F9" approach** (an earlier critique remediation suggested this; rejected by maintainer because Chamber-OS is not yet live for chamber adoption, so the staged-rollout reasoning does not apply). **Production go-live (= chamber starts using the system)** is a separate operational decision, NOT a scope question for F8: F8 ships dark in production (`FEATURE_F8_RENEWALS=false`) until the entire planned MVP scope is ready and the chamber is onboarded. At that point the kill-switch flips on (Vercel Rolling Releases 10% ‚Üí 50% ‚Üí 100% per F5 pattern) for ALL features simultaneously. F6 (EventCreate Integration) ships before MVP go-live so FR-029a F6-readiness fallback is a dev/staging/E2E concern only (chamber never sees the fallback bands).
- **A13**: Solo-maintainer substitute (Constitution Principle IX) applies to this F8 spec since the project is currently single-maintainer; the substitute's ‚â•3 `/speckit.review` rounds + ‚â•2 `/speckit.staff-review` rounds + DPIA + threat-model pre-flight + DB+App+Concurrency+Audit+Observability defence-in-depth is satisfied at ship-time the same way F7 was.
