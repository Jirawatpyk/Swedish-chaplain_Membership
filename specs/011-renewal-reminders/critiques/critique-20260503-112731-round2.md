### 🎯 M3 — `pending_admin_reactivation` has no exit timeout (Engineering / Failure Modes)

**Finding**: FR-005b introduces a new cycle state `pending_admin_reactivation` for the admin-override branch (when `members.blocked_from_auto_reactivation === TRUE` and a lapsed member completes payment). The spec describes the entry path (payment → pending_admin_reactivation + escalation_task) and admin's two exit actions (reactivate → completed, OR reject → cancelled + refund) but **no automatic exit policy**.

Operational concerns:
- If admin forgets the task, cycle sits in `pending_admin_reactivation` indefinitely
- Member has paid but cannot use the system; their payment is in limbo
- F4 invoice is `paid` but F8 cycle is neither completed nor cancelled
- Audit trail shows orphaned state; no SLA breach signal

The spec sets a 3-day overdue highlight for admin tasks (FR-045) but that's UX-only; cycle state itself has no automatic exit.

**Suggestion**: Add FR-005c specifying an exit timeout policy:

```
FR-005c: A cycle in status `pending_admin_reactivation` MUST exit within 30 days
or the system auto-cancels the cycle + auto-issues a refund + creates a
high-priority escalation task for admin notification. Daily cron
`/api/cron/renewals/reconcile-pending-reactivations` (added to coordinator
fan-out OR as a 6th cron-job.org job) scans for `pending_admin_reactivation`
cycles where `entered_pending_at < now() - 30 days` and:
  1. Auto-cancels the cycle (cycle.status → cancelled)
  2. Triggers F5 refund via the F5 barrel
  3. Triggers F4 credit-note creation via the F4 barrel
  4. Emits audit `lapsed_member_admin_reactivation_timed_out`
  5. Notifies the original blocked-by-admin user (audit log entry)
Reminder ladder before timeout: T-7 / T-3 / T-1 day admin email reminders.
```

Add corresponding audit event to taxonomy: `lapsed_member_admin_reactivation_timed_out`.

### 🎯 M4 — F1 middleware multi-tenant subdomain capability unverified (Engineering / Integration)

**Finding**: research.md R1 step 1 (revised at Q4 round 3) says:

> F1 middleware resolves `subdomain_tenant_id` from request host BEFORE the F8 route handler runs. Unknown subdomain → 404 (no F8-specific audit; F1 generic).

This is a **soft assumption** about F1's existing middleware. Per `docs/saas-architecture.md` § 4 the multi-tenant subdomain routing is *planned* design, not necessarily *shipped*. F1 today is deployed on a single subdomain (`swecham.zyncdata.app`); if F1's middleware is hard-coded for that single subdomain, F8's R1 step 1 fails silently OR routes everything to swecham regardless of token.

If F1 middleware is single-tenant today:
- Q4 URL pattern `https://<tenant>.zyncdata.app/...` may not actually route per-tenant
- Cross-tenant defence-in-depth (R1 step 5) becomes "trust the token only" — half-strength
- Custom-domain branch (post-MVP F12) is even further unsupported

**Suggestion**: Before /speckit.tasks closes, verify F1 middleware capability via 1-line spec investigation:

```bash
# Check F1's middleware for subdomain resolution:
grep -nE "subdomain|host\\.|tenants?" src/middleware.ts src/app/middleware.ts \
  src/lib/tenant-context.ts 2>/dev/null | head
```

Three possible outcomes + remediations:

- **A. F1 middleware already has subdomain resolution** (documented + tested): no change to F8 spec needed; just confirm in research.md R1.
- **B. F1 middleware is single-tenant hardcoded today, but extending it is small (~20 lines)**: add Complexity Tracking entry #5 "F8 requires F1 middleware extension for multi-tenant subdomain routing"; coordinate as part of F8 PR (mirrors F4 + F2 cross-module extensions).
- **C. F1 middleware is single-tenant AND not designed for multi-tenant routing**: revise Q4 decision — fall back to **path-prefix routing** instead (`https://swecham.zyncdata.app/t/<tenant_slug>/portal/renewal/<memberId>?token=<...>`) until F10 ships true multi-tenant routing. Reduces Q4 win but unblocks F8 ship.
