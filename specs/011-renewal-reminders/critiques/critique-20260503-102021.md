**Date**: 2026-05-03
**Feature**: [spec.md](../spec.md)
**Plan**: [plan.md](../plan.md)
**Research**: [research.md](../research.md)
**Data Model**: [data-model.md](../data-model.md)
**Verdict**: ⚠️ **PROCEED WITH UPDATES** (4 🎯 Must-Address + 14 💡 Recommendations + 5 🤔 Questions)

---

## Executive Summary

F8's spec + plan + research + data-model + contracts are **comprehensively scoped** — 10 clarifications closed across 2 sessions, Constitution Check GREEN on all 10 principles, 8 tables with RLS+FORCE, 43 audit events catalogued with payload schemas, three cron jobs with idempotency contracts, and a clean Clean Architecture mapping that matches the F1+F2+F3+F4+F5+F7 precedent. Zero new npm dependencies, transparent rule-based at-risk scoring, well-documented tier-upgrade pending lifecycle.

However, four serious concerns surface under stress and warrant resolution before `/speckit.tasks`:

1. **🎯 E1 (Architecture)**: research.md R8 mandates Postgres `LISTEN/NOTIFY` for the F1 bounce-event integration. **This does not work in Vercel Fluid Compute serverless** — function instances are short-lived and cannot maintain a persistent LISTEN connection. Needs a synchronous-call architecture (F1 webhook handler invokes F8 use-case via barrel) instead.

2. **🎯 E2 (Integration risk)**: FR-022/FR-023 + the `markCycleCompleteFromInvoicePaid` hook assume F4 emits a "invoice marked paid" event that F8 can listen to. **F4 currently does not expose this hook** — F4's `markPaidFromProcessor` is an internal mutation. F8 either needs F4 changes (coordinated extension with the F4 maintainer, similar to the F2 column extension in plan.md Complexity #2) or a polling shim. The spec is silent on this.

3. **🎯 E3 (F2 contract)**: The Q5 round-2 tier-upgrade pending flow assumes F2's `changeMemberPlan` accepts an "effective at next renewal cycle" semantic. **F2 in its current shipped form only supports immediate plan changes** — F8's `apply-pending-tier-upgrade.ts` use-case needs F2 to expose a new use-case or F8 must call `changeMemberPlan` synchronously inside F4's renewal-invoice-creation hook (which itself depends on E2 being resolved).

4. **🎯 X1 (Scale architecture)**: Vercel Fluid Compute functions have a **300s default execution timeout** (per the active session-start vercel knowledge update). The daily reminder dispatch cron iterates ALL active tenants of ALL active members. At SaaS scale (50 tenants × 5,000 members = 250,000 evaluations × ~10ms each = ~42 minutes), the cron handler **will exceed the 300s budget** and fail mid-pass. The 60s/tenant SLO (FR-017) guards per-tenant duration but doesn't cap the total. Needs a chunked/per-tenant-cron architecture.

Lesser concerns include unbounded growth of `consumed_link_tokens` (no pruning cron specified), missing baseline measurement methodology for SC-004 ("renewal rate improves by 10pp" against an undocumented baseline), absence of a "first-time renewer" portal microcopy nudge for member self-service, and a few questions about chamber operational practice (lapsed reactivation policy, communication strategy at go-live).

The Constitution Check passes 10/10 with 2 documented Complexity Tracking entries (solo-maintainer substitute + cross-module F2 column migration). Both are clean and follow the F1/F4/F5/F7 precedent.

### 4b. Failure Mode Analysis

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| **X1** | 🎯 | **Vercel function 300s timeout vs SaaS-scale cron pass.** Vercel Fluid Compute default function timeout is 300s (per active session-start vercel knowledge update). FR-017 claims "<60s for a tenant with 5,000 active members". At SaaS scale (50 tenants × 5k members = 250,000 evaluations × ~10ms each = ~42 minutes), the daily reminder dispatch cron handler **will hit the 300s cap** and the function will be terminated mid-pass. Survivors will retry on tomorrow's cron — which means tenants 41-50 may have skipped reminders for an entire day. Per-tenant SLO is fine; total-job SLO is broken at SaaS scale. | **Fan out per-tenant**: cron-job.org calls a coordinator endpoint `/api/cron/renewals/dispatch-coordinator` that itself enqueues per-tenant work (e.g., creates 50 separate cron-job.org jobs at provisioning time, OR uses Vercel Queues / a queue table for incremental processing). Each per-tenant invocation runs in its own 300s budget. Update FR-017 + plan.md SLO + cron-renewals-api.md + research.md as R14. |
| E6 | 💡 | **Resend API outage during dispatch window — recovery path unclear.** Spec mentions retry-with-backoff but doesn't specify: if the API is down for 6 hours during the daily 06:00 cron, do failed reminders retry on tomorrow's cron pass, or are they silently lost? The `renewal_reminder_events` table has `status='failed'` rows but no FR mandates re-attempt. | Add FR-010a: "Failed reminder events with `failure_reason ∈ ('upstream_unavailable', 'rate_limited', 'transient')` are eligible for retry on the next cron pass within `retry_until = original_dispatch_at + 24h`. Beyond 24h, transitioning to permanent failure with admin escalation task `manual_outreach_required`. Audit event `renewal_reminder_retried`." |
| E7 | 💡 | **`consumed_link_tokens` unbounded growth.** quickstart.md mentions "TTL cleanup (consumed tokens >60d old can be pruned weekly)" but no cron job is defined and no migration ships the cleanup. Table grows ~5,000 rows/year/tenant = 250k rows at SaaS scale × 5 years = 1.25M rows. Not catastrophic but unnecessary. | Add a 4th cron `/api/cron/renewals/prune-consumed-tokens` (weekly) deleting rows where `consumed_at < now() - interval '60 days'`. Trivial 5-line use-case. Add to contracts/cron-renewals-api.md. |
| E8 | 💡 | **Member primary_contact_email NULL crash path.** FR-014 requires reminder emails go to `members.primary_contact_email`. If null (data hygiene issue), the dispatch use-case must NOT crash — needs a graceful path.
