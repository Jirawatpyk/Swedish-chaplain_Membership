## Findings

### 🟡 F9-QA-01 — Member timeline logs `MISSING_MESSAGE` for uncatalogued audit event types (US3)

- **Symptom (browser console, observed on `/admin/members/[id]` + timeline)**: `MISSING_MESSAGE: Could not resolve audit.eventType.invoice_pdf_downloaded in messages for locale en`.
- **Root cause**: `src/components/members/timeline-event-item.tsx:165` calls `tAuditEvent(eventType)` (namespace `audit.eventType`) directly inside a `try/catch`. The catch prevents a UI crash (falls back to the row summary / source label), but next-intl's `onError` reporter logs the `MISSING_MESSAGE` **before** the throw is caught — so the console is spammed and the member sees a degraded (non-localized) label.
- **Scale**: the `audit_event_type` enum has **~181** values (migration `ADD VALUE`s) but the `audit.eventType` i18n catalogue has only **99 keys** — **~140 missing** (most F7 `broadcast_*`, F6 `attendee_*`/`csv_import_*`, F4 `*_pdf_downloaded`/`credit_note_*`). Any of these appearing on a member timeline or in the audit viewer triggers the console error.
- **Why CI missed it**: `check:i18n` only verifies EN↔TH/SV **parity**, not `enum ↔ catalogue` **completeness**; the f9-timeline e2e has no console-error listener so it passed.
- **Recommended fix (1 line, kills all ~140 at once)**: use the `.has` guard (the shared `resolveEventLabel` helper in `src/lib/audit-event-label.ts` already does this) — `tAuditEvent.has(eventType) ? tAuditEvent(eventType) : <fallback>` — so next-intl's `onError` never fires. Optionally add the high-frequency missing keys (e.g. `invoice_pdf_downloaded`, common `broadcast_*`) to the catalogue in EN/TH/SV for proper localization, and consider a `check:audit-event-labels` gate asserting enum↔catalogue coverage.
- **Severity**: quality / observability (no crash, no data issue) — but member-facing console spam + degraded labels; fix before ship.

### 🟢 F9-QA-02 — Pre-existing base-ui hydration mismatch on member detail page

- **Symptom**: React hydration warning on `/admin/members/[id]` — `data-slot="button"` (client) vs `data-slot="dialog-trigger"` (server) on a base-ui `<DialogTrigger render={<Button>}>`.
- **Assessment**: **not an F9 remediation regression** — the diff `6750e0fc..HEAD` touches no `members/[memberId]` file except `member-invoices-skeleton.tsx` (no Dialog). Known base-ui `render`-prop `data-slot` composition quirk; cosmetic (`data-slot` is a styling/test hook), "won't be patched up", non-breaking.
- **Recommendation**: track as a base-ui composition cleanup (align `data-slot` on the trigger+button merge); not a blocker.

### 🟢 F9-QA-03 — 2 flaky a11y axe scans (benefit view)

- `f9-a11y.spec.ts:76` (`/admin/members/[id]/benefits`) + `:93` (`/portal/benefits`) on chromium **failed once, passed on retry** (exit 0). Likely an axe-scan timing race against a still-settling render (same class as the member-invoices skeleton issue fixed earlier this session).
