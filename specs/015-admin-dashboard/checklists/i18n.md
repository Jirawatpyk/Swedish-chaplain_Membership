# i18n (EN/TH/SV) Requirements Quality Checklist: F9

**Purpose**: Validate that F9's **internationalization & localization requirements** are
complete, clear, and consistent across all surfaces — *before* implementation. Tests the
requirements, not the rendered strings.
**Created**: 2026-05-25
**Feature**: [spec.md](../spec.md) · constitution v1.4.2 (Principle V) · `docs/ux-standards.md`
**Depth**: Formal release gate · **Audience**: reviewer

> "Is localisation specified for X?" — not "Is the Thai string correct?"

## Locale Coverage

- [ ] CHK001 Is the EN/TH/SV requirement stated for **every** F9 surface (dashboard, audit, directory, timeline, benefits, GDPR export UI)? [Completeness, Spec §FR-034]
- [ ] CHK002 Is the EN-canonical / TH+SV fallback-with-CI-block policy reaffirmed for new F9 keys? [Consistency, Constitution V, CLAUDE.md]
- [ ] CHK003 Is a key-coverage verification requirement (`pnpm check:i18n`) stated for F9 keys? [Measurability, quickstart §5]

## Calendar / Number / Currency Formatting

- [ ] CHK004 Is the Buddhist-Era **display-only** rule (storage stays Gregorian UTC) restated for all F9 date displays (timeline, audit timestamps, dashboard "as of", export dates)? [Clarity, Spec §FR-034, Constitution V]
- [ ] CHK005 Is locale-aware number/currency formatting required for dashboard revenue (THB primary) and benefit counts? [Completeness, Spec §FR-034]
- [ ] CHK006 Is the audit export's dual timestamp requirement (UTC + human-readable local rendering) specified unambiguously, and is the local-rendering locale defined? [Clarity, Spec §FR-012]

## Timeline Multi-Source Localisation (the hard one)

- [ ] CHK007 Is it specified that the timeline view returns `source` + `payload` and the **application layer** maps to i18n keys (not SQL-synthesised summaries)? [Clarity, data-model §5, research R4/R2-E3]
- [ ] CHK008 Is an i18n key + interpolation requirement defined for **each** of the six timeline sources (audit/invoice/payment/event/broadcast/renewal)? [Completeness, Spec §FR-014/034, Gap]
- [ ] CHK009 Is the handling of legacy `audit_log.summary` strings (likely English) defined so the timeline is consistently localised, not a mix of keyed + raw text? [Consistency, research R4/R2-E3]

## Artefact Localisation

- [ ] CHK010 Is the language of the Directory E-Book specified (per-tenant locale? bilingual TH/EN like F4 tax docs? member's chosen locale)? [Ambiguity, Spec §FR-026, Gap]
- [ ] CHK011 Is the GDPR export README/manifest localisation requirement defined (which locale)? [Completeness, Spec §FR-029, Gap]
- [ ] CHK012 Are smart-insight strings and the under-use warning required to be localised (not just English templates)? [Coverage, Spec §FR-004, §FR-021]

## Consistency & Layout

- [ ] CHK013 Is terminology consistency required between F9 and existing modules (e.g. "member", "tier", "E-Blast", "invoice") across all three locales? [Consistency, Gap]
- [ ] CHK014 Is content-length variance (TH/SV vs EN) accommodation required for the data-dense dashboard cards, audit columns, and directory listings (no truncation/broken wrap)? [Coverage, Constitution V]
- [ ] CHK015 Are notification/email strings (export-ready, async-job notifications) required to be localised? [Completeness, Spec §FR-030, Gap]

## Measurability

- [ ] CHK016 Can the "fully renders in EN/TH/SV with no missing strings" success criterion be objectively verified for F9? [Measurability, Spec §SC-010]

## Notes
