## Product Lens Findings

### 3b. User Value
- **R2-P1 (🎯 "membership year" undefined)**: FR-019/023 and the benefit dashboard scope
  consumption to "the current membership year", but the term is **never defined**. Three
  plausible meanings give *different* quota counts: (a) **calendar year** (the
  smart-features doc's materialized view used `EXTRACT(YEAR FROM NOW())`); (b) the
  member's **join/renewal anniversary** year (F8 renewals model membership periods);
  (c) a **tenant fiscal year** (F4 uses Asia/Bangkok fiscal boundaries for numbering).
  This decides which broadcasts/tickets count and directly drives SC-006. *Suggestion*:
  pick one explicitly (recommend **calendar year in the tenant timezone** for v1 —
  matches the existing benefit-usage doc + simplest to reconcile; revisit if plans
  become anniversary-based). Add as an FR + a clarification.
- **R2-P2 (🎯 aggregate "consumed %" method undefined)**: FR-021's warning compares
  "elapsed-year %" to "consumed-benefit %", and the mockup says "used only 33% of their
  benefits" — an **aggregate across benefit types**, but the aggregation is unspecified.
  Mean of per-benefit ratios? Weighted? Does it include/exclude **unlimited** benefits
  (which have no ratio)? Each choice changes when the 25-pt warning fires. *Suggestion*:
  define it — recommend **the simple mean of the consumption ratio of each *quantifiable*
  benefit (E-Blasts, cultural tickets), excluding unlimited/active-only benefits**; state
  it in FR-021 so SC-006's reconciliation test is deterministic.

### 3d. Edge Cases & UX
- **R2-P3 (💡 directory logo = new write/upload surface)**: FR-025 + `directory_listings.logo_blob_key`
  imply members can upload a logo, re-encoded/EXIF-stripped "like F4" — i.e. a **new
  image-upload pipeline** (sharp re-encode, MIME/dimension validation, Blob write,
  malware consideration). That is non-trivial scope hiding inside a P4 field toggle, and
  F3 members may not already have a logo asset to reuse. *Suggestion*: either (a) confirm
  logo upload is in P4 scope and add explicit FRs (size/type limits, re-encode, audit),
  or (b) **defer logo to a fast-follow** and ship the directory with text fields + an
  existing-asset reference only. Avoids an unscoped upload surface in the first cut.
- **R2-P4 (💡 portal IA / discoverability)**: the member surfaces (own benefits, own
  timeline, data export, directory visibility) land on 4 portal routes; the spec doesn't
  say how a member *finds* directory-visibility settings vs data-export. *Suggestion*:
  specify the portal IA (e.g. directory visibility under `/portal/profile`, data export
  under `/portal/account`), and add nav entries, so these don't become orphaned pages.

## Engineering Lens Findings

### 4a. Architecture Soundness
- **R2-E1 (💡 event-triggered snapshot refresh mechanism + Principle III)**: research R1
  says the snapshot is "refreshed on key events (payment, approval, status change)" but
  not **how**. If the payments/broadcasts/members use-cases call into `insights` to mark
  the cache stale, that is **cross-module write coupling** (those modules would depend on
  `insights`, or `insights` would need hooks in them) — a Principle III smell.
  *Suggestion*: drive staleness from a **DB trigger on `audit_log`** (those events all
  write audit rows already) that flips `dashboard_metrics_cache.stale = true` for the
  tenant. This keeps modules decoupled (no app-layer cross-calls), is one trigger, and
  the coordinator already prioritises `stale` rows. Document it in R1/data-model §1.

### 4b. Failure Mode Analysis
- **R2-E2 (💡 large audit export sync→async threshold)**: FR-012 + research R5 keep audit
  exports **synchronous** ("small"), but an admin can filter to a wide date range and
  export tens of thousands of rows → request timeout / memory pressure (the very scale
  the audit indexes exist for). *Suggestion*: cap synchronous audit export at a row
  threshold (e.g. ≤ 10k); above it, route through the existing `export_jobs` async path
  (kind `audit_export`). Reuses infrastructure already built for E-Book/GDPR.

### 4c. Security & Privacy Review
- **R2-E3 (💡 timeline view `summary_key` / i18n shape)**: data-model §5 has the SQL view
  emit a `summary_key`, but the six sources are heterogeneous: `audit_log` already stores
  a plain `summary` string (likely English, not an i18n key), while invoices/payments/etc.
  have only raw columns. A SQL view cannot cleanly synthesise localized, interpolated
  summaries. *Suggestion*: the view should return **`source` + a structured `payload`
  only**; the **application/presentation layer maps `(source, payload)` → an i18n key +
  interpolation values** (FR-034). Clarify that `summary_key` is not produced in SQL, and
  note how legacy `audit_log.summary` strings are handled (render raw vs back-fill keys)
  so the timeline is consistently localised across all six sources.

### 4d. Performance & Scalability
- **R2-E4 (💡 activity-feed index)**: the activity feed is now a **live** query of the
  last N audit events per tenant (round-1 P2), but the planned audit indexes (0190) are
  `(tenant_id, event_type, timestamp)` and `(tenant_id, actor, timestamp)` — neither is
  the ideal index for an *unfiltered* "latest N for this tenant" scan. *Suggestion*: add
  `audit_log(tenant_id, timestamp DESC)` to the 0190 migration so the per-load activity
  feed is an index scan.

### 4e. Testing Strategy
- **R2-E5 (💡 benefit-usage determinism tests)**: contingent on R2-P1/P2 — once
  "membership year" and the aggregation method are defined, add explicit unit/property
  tests: (a) year-boundary (a broadcast sent on Dec-31 vs Jan-1 counts in the right
  year, tenant-tz aware); (b) aggregation excludes unlimited benefits; (c) the 25-pt
  warning fires exactly at the boundary. Ties SC-006 to deterministic fixtures.
