## Executive Summary

The spec and plan are **strong and unusually well-grounded** — six independently
shippable user stories, 9 resolved clarifications, a full Constitution Check with 4
honest Complexity-Tracking entries, and a design that reuses F1–F8 infrastructure
(F8 risk → engagement, F4 react-pdf → E-Book, F6/F8 cron → snapshot/export). Tenant
isolation and GDPR redaction — the two highest-risk areas for an all-PII feature —
are addressed at the design level (RLS+FORCE, `security_invoker` view, private-Blob
proxy, mandatory cross-tenant test).

The critique surfaces **2 must-address** items (both are *verification/closure* gaps,
not design flaws) and **8 recommendations** that will materially de-risk
implementation. None require re-running `/speckit.specify` or re-architecting. The
dominant themes are: (a) verify the `@vercel/blob` private-storage assumption before
committing to R6; (b) close two reliability/freshness gaps (stuck-`processing`
reclaim; activity-feed vs 5-min snapshot staleness); (c) make the load-bearing
cross-module **read surface** and **index set** explicit so `/speckit.tasks` doesn't
under-scope them.

**Verdict: ⚠️ PROCEED WITH UPDATES** — apply the small `plan.md`/`research.md` edits
below, then `/speckit.tasks`.

## Product Lens Findings

### 3a. Problem Validation
- **Strong**: each pillar maps to a documented pain in `smart-chamber-features.md`
  (unused benefits, un-pullable reports, non-queryable audit) and to the live F1
  `/admin` placeholder. No "solution looking for a problem."
- **P1 (💡 scope breadth)**: F9 bundles 6 stories across 2 portals on one branch — the
  phases plan itself flagged a possible split. The spec mitigates this by making each
  story independently shippable, but a single long-lived all-PII branch is a large,
  hard-to-review surface. *Suggestion*: commit to a **delivery slicing** in tasks —
  ship P1 (dashboard) + P2 (audit) + P3 (timeline/benefits) first; treat P4 (directory
  + GDPR) as a second mergeable slice, with the option to split the branch if velocity
  slips.

### 3b. User Value
- **Strong**: outcome-focused acceptance criteria; clear MVP core (US1–US4).
- **P2 (💡 activity-feed freshness)**: `research.md` R1 folds the activity-feed head
  into the 5-minute snapshot, but the spec's intent (and the smart-features mockup)
  shows "2 min ago" items. A ≤5-min-stale feed will feel wrong on the most-watched
  screen and could fail the spirit of US1 AS-3. *Suggestion*: serve the activity feed
  from a **lightweight live query** (last N audit events for the tenant) separate from
  the cached KPI snapshot; keep only the *counts* in the snapshot.

### 3c. Alternative Approaches
- **Strong**: engagement = inverse-of-F8 (no reinvented scorer); no charting dep;
  audit viewer reuses TanStack. All good Simplicity calls.
- **P3 (🤔 GDPR self-service necessity)**: the spec admits GDPR-request volume is low,
  yet ships a member-facing self-service export *and* admin-on-behalf. *Question for
  stakeholder*: is member self-service needed for launch, or does admin-on-behalf
  satisfy the Art. 20/PDPA obligation while removing a PII-archive surface from the
  member portal? (You explicitly chose "include member surfaces" in clarify — this is a
  re-confirm, not a reversal.)

### 3d. Edge Cases & UX
- **Strong**: empty/large/stale/redaction/isolation/i18n/archived all covered.
- **P4 (💡 archived/erased subjects + GDPR)**: edge cases cover archived members for
  timeline/audit/directory but **not for GDPR export**. Data-subject rights persist
  after archival; an erasure may also conflict with audit-retention. *Suggestion*: add
  an edge case + FR clarifying that an archived (not erased) member can still export;
  and how erasure interacts with the ≥5-year audit retention (retain pseudonymised
  audit, drop the rest).

### 3e. Success Measurement
- **Strong**: SC-001–011 measurable; SC-002 has a p95 target.
- **P5 (💡 product KPIs + rollback trigger)**: success criteria are mostly capability
  checks; there is no *adoption/value* metric (e.g., % members viewing own benefits,
  # insights acted on) and no explicit **rollback trigger** beyond the kill-switch
  existing. *Suggestion*: add 1–2 product KPIs to track post-launch and a one-line
  rollback trigger (e.g., "flip `FEATURE_F9_DASHBOARD` off if dashboard error-rate
  >X% or snapshot age p95 >Y min").

## Engineering Lens Findings

### 4a. Architecture Soundness
- **Strong**: new `insights` module + barrel + ESLint rule; timeline stays in
  `members`; audit-read added to `auth` (owns the table). Clean.
- **E1 (💡 cross-module SQL coupling)**: `member_timeline_v` UNIONs six module-owned
  tables directly in SQL, bypassing the TS barrels (Principle III is about imports, but
  this is the same coupling at the DB layer). A renamed column in any source module
  breaks the view silently. *Suggestion*: add a `check-f9-schema` CI assertion that the
  view exists + is `security_invoker`, plus a per-source integration test so a source
  schema change fails loudly. (Partially planned — make it explicit.)

### 4b. Failure Mode Analysis
- **E2 (🎯 stuck-`processing` export reclaim)**: the `export_jobs` state machine has a
  TTL sweep (`ready|delivered→expired`) but **no reclaim for jobs stuck in
  `processing`** after a worker crash — they'd never complete or fail. F7 hit exactly
  this and shipped `reconcile-stuck-sending`. *Suggestion*: add a `processing→failed`
  (or `→requested`) reclaim after a timeout to the worker sweep; emit a metric.
- **E3 (💡 dashboard cold-start)**: before the first cron run (new tenant), there is no
  `dashboard_metrics_cache` row. FR-006 requires a friendly empty state, not an error,
  but `DashboardError.snapshot_unavailable` implies an error path. *Suggestion*: define
  cold-start = lazy compute-on-first-read (then cache) **or** seed an empty snapshot at
  tenant creation; never surface a raw error.

### 4c. Security & Privacy Review
- **Strong**: RLS+FORCE, `security_invoker`, private-Blob proxy, signed token,
  role redaction, PII-read auditing, documented lawful basis (R10).
- **E4 (💡 download-token reuse window)**: the proxy validates a token + expiry and
  moves `ready→delivered` on first download, but doesn't state whether the link is
  **single-use** or replayable within TTL. A leaked link to a PII archive is
  replayable until expiry. *Suggestion*: make the token **single-use** (invalidate on
  `delivered`; require a fresh authenticated request + new token to re-download) and
  keep the TTL short.
- **E5 (🤔 audit export actor identities)**: a manager exporting the audit log obtains
  staff actor identities. Payload redaction is specified, but actor identity exposure to
  the manager role isn't. *Question*: is staff-actor identity acceptable to the manager
  role, or should it be hashed/limited for non-admins?

### 4d. Performance & Scalability
- **E6 (💡 explicit index set — load-bearing)**: SC-005 (full timeline) and the audit
  viewer at "tens of thousands of events" depend entirely on indexes. `member_timeline_v`
  keyset pagination over a 6-way `UNION ALL` requires a composite index
  `(tenant_id, member_id, <timestamp> DESC)` on **every** source table, not just where
  "missing" (data-model 0189 is vague; 0190 covers only audit). *Suggestion*: enumerate
  the exact per-source indexes in `data-model.md §9` and add an `EXPLAIN`-backed perf
  test for both the timeline view and the audit query.
- Note: at the current single-tenant ~131-member scale this is comfortable; the concern
  is the 5,000-member SC-002 target and 10x headroom.

### 4e. Testing Strategy
- **Strong**: TDD-first, cross-tenant isolation test (Principle I blocker), fast-check
  invariants, live-Neon integration, coverage targets stated.
- **E7 (💡 union cursor-correctness test)**: the multi-source timeline needs an explicit
  integration test for **keyset cursor correctness across source boundaries** (two
  events with identical `occurred_at` from different sources must paginate without
  loss/duplication via the `ref_id` tiebreak). Call it out as a named test.

### 4f. Operational Readiness
- **Strong**: kill-switch, cron coordinators, SLOs, runbook append all planned.
- **E8 (💡 raw-SQL rollback)**: the `security_invoker` view + RLS policies are
  hand-authored SQL outside Drizzle's generator; document the **down/rollback** SQL
  (DROP VIEW / DROP POLICY) since drizzle-kit is forward-only. Minor but avoids a
  scramble during a bad deploy.

### 4g. Dependencies & Integration Risks
- **E9 (🎯 verify `@vercel/blob` private support)**: R6 + the plan's "no new npm
  dependencies" claim both hinge on the installed **`@vercel/blob@2.3.3`** supporting
  **private/non-public** storage. The F4 adapter uses `access:'public'`, and v2's
  `put()` has historically typed `access` as the literal `'public'`. If 2.3.3 cannot
  store private objects, R6's delivery model needs either a **version bump** (so "no new
  deps" becomes "no new deps; one version bump") or an alternative (random-suffix public
  key + strict proxy is **not** acceptable for a PII archive — security by obscurity).
  *Suggestion*: verify against current `@vercel/blob` docs **before `/speckit.tasks`**;
  record the confirmed API + minimum version in `research.md` R6.
- **E10 (💡 null engagement behaviour)**: engagement = `100 − riskScore`, but riskScore
  is null until F8's at-risk cron runs for a tenant. The member-list column is
  "sortable/filterable" — define **null handling** (sort nulls last; display "—"; filter
  excludes nulls) so an un-scored tenant doesn't get a broken sort.

## Cross-Lens Insights (convergence — highest leverage)

- **X1 (💡 Scope × Risk)** — P1 + the all-PII review surface: slice delivery (P1–P3
  first, P4 second) reduces *both* product time-to-value and the engineering review
  blast radius for a security-sensitive branch.
- **X2 (💡 Freshness × Architecture)** — P2 + E3: splitting the **live activity feed**
  from the **cached KPI snapshot** fixes the staleness UX *and* removes activity-feed
  churn from the snapshot job (smaller, faster refresh).
- **X3 (🤔 Scope × Surface)** — P3 + E4: deferring member-facing GDPR self-export (keep
  admin-on-behalf) would remove a replayable-PII-archive surface from the member portal
  while still meeting the legal obligation. Re-confirm the clarify decision with this
  trade-off explicit.
