# Feature Specification: F7 — Email Broadcast (E-Blast)

**Feature Branch**: `010-email-broadcast`
**Created**: 2026-04-29
**Status**: Draft
**Input**: User description: "F7 — Email Broadcast (E-Blast) per docs/email-broadcast-analysis.md"

## Context

F7 is the Phase 2 deliverable that closes the largest **paid-benefit gap** in
Chamber-OS today. Six of the nine tiers in the SweCham 2026 Membership Package
PDF (Premium/Large/Regular Corporate + Diamond/Platinum/Gold Partnership)
contractually purchase a fixed annual quota of **E-Blasts** (1–15 per year
depending on tier — see `docs/membership-benefits-analysis.md` § 3.4) — a
member-authored marketing email broadcast to a curated chamber recipient list.
Until F7 ships, the chamber cannot deliver this benefit at all. F7 turns the
tier promise into a working self-service flow: members compose and submit, an
admin reviews, the system delivers to a tenant-scoped recipient list, and the
quota counter on the Smart Feature #1 Benefit Dashboard moves from "promised"
to "used".

Tenant-scoped per SaaS architecture (`docs/saas-architecture.md`). Phase 2 is
still single-tenant (SweCham / TSCC) so the deployed `tenant_id` defaults to
`'swecham'`; the data model is multi-tenant-aware (`tenant_id`-scoped
broadcasts, deliveries, unsubscribes, segment definitions) so future tenants
provisioned via F10 inherit isolated quotas without schema migration.
Sensitivity: **⚠ PII** (recipient lists derived from member contact data) plus
**⚠ Marketing-consent** (PDPA Section 24 + GDPR Article 21 + ePrivacy unsubscribe
obligation). Review gate requires **≥2 reviewers** (or the Constitution § IX.5-stack
solo-maintainer substitute when no second human reviewer is available).

**Why built on Resend Broadcasts** (decision pre-locked in
`docs/email-broadcast-analysis.md` § 3): Resend is already the transactional
provider from F1, so we inherit verified sender domains, SPF/DKIM/DMARC,
deliverability reputation, React Email template engine, and SOC 2 / GDPR
compliance posture without onboarding a second vendor. Resend's Broadcasts
product (GA 2024) is purpose-built for marketing/bulk email — separate audience
+ broadcast resources, separate webhook stream, separate suppression list — so
broadcast volume cannot pollute transactional reputation. Mailchimp / Brevo
were rejected because they double the integration surface (two providers, two
sender identities, two webhook conventions, per-tenant API-key sprawl) for no
deliverability or feature benefit at SweCham's scale.

**Why a member-author + admin-approver flow** (not member-direct-send): an
E-Blast is sent in the chamber's name to a chamber-curated list — a misfire
(typo, off-tone, off-topic) damages the chamber's brand and other members'
inboxes, not just the originating member's. The admin gate is a cheap human
control that prevents reputational and deliverability incidents at MVP scale.
Auto-approval for high-tier members can be revisited post-MVP if admin queue
load proves unsustainable.

**Scope boundary — F7 vs. F1 vs. F8 vs. F11**:

- **F7 (this spec)** — *member-authored marketing broadcasts* sent through
  Resend Broadcasts on behalf of a paying member, consuming that member's
  annual quota, with admin approval, recipient segmentation, and per-recipient
  delivery tracking.
- **F1 (Auth & RBAC, shipped)** — owns transactional email (password reset,
  invitation, idle-warning) via Resend's transactional API. F7 does NOT
  duplicate transactional templates; admin/member notifications **about** an
  E-Blast (submission, approval, rejection, delivery summary) reuse F1's
  transactional path.
- **F8 (Renewal Tracking)** — sends renewal reminder emails, also via
  Resend transactional (NOT Broadcasts), because reminders are 1:1 personalised
  with deep-links and not subject to marketing-consent unsubscribe rules.
  F8 does not consume F7 quota.
- **F11 (SaaS Billing)** — charges tenants for the Chamber-OS subscription;
  F11 does NOT charge per-broadcast. The annual E-Blast quota is part of
  the membership-tier price (F4 invoice line), not a metered F11 line.
  Pay-per-send beyond quota is OUT of scope for F7 MVP.

## Clarifications

### Session 2026-04-29

- Q1 **(scope — quota consumption point)**: At which lifecycle transition is a member's annual quota slot permanently consumed? → **A: Reserve at `submitted`, consume at `sent`.** Rationale: most member-fair model — rejection, cancellation, and `failed_to_dispatch` all release the slot so admin error or our outage never silently costs a member their paid benefit. Adds a "used" + "reserved" two-state quota math (FR-006 + FR-008) and requires reliable release logic on every off-happy-path transition (FR-003 + FR-004). Trade-off accepted: member-facing quota display is slightly more verbose ("3 remaining + 1 reserved" vs simple "4 remaining") but the dispute-resistance value outweighs the wording cost.
- Q2 **(UX — admin approval SLA + escalation)**: Should F7 MVP enforce or escalate on a stale submission, or just surface the target SLA as text? → **A: Display 48-hour target on member confirmation + admin queue UI; ship NO automated escalation, auto-cancel, or auto-notify in MVP.** Rationale: a chronically-overflowing admin queue IS the signal that ops scale needs revisiting — automating around the bottleneck papers over the underlying capacity issue. Auto-cancel risks killing time-sensitive submissions (event invitations, news with a date). Manager escalation requires a notification channel that does not yet exist (F1 ships transactional only — no Slack/webhook integration). Members are not punished for admin slowness because Q1's reservation model holds the quota slot until decision. Post-MVP: revisit if SC-002 (median ≤24h, p95 ≤48h) is missed in two consecutive monthly windows.
- Q3 **(state machine — edit-after-submit policy)**: Can a member edit a `submitted` broadcast before admin acts, and can an admin edit content during review? → **A: Immutable after submit.** Once `status` leaves `draft` the content (subject, body, segment, scheduled_for) is frozen; the only mutating actions on a `submitted`/`approved` row are the system-driven state transitions defined in FR-004 (admin approve/reject/schedule, member or admin cancel) and the system-driven dispatch/webhook lifecycle. To change content the member MUST cancel (which releases the reservation per Q1) and create a new draft + submit; this consumes a fresh reservation. Admins MUST NOT inline-edit content during review — typos or link issues route through Reject-with-reason → member-recompose, preserving audit integrity ("what version did the admin see?" is unambiguous: exactly the version stored on the broadcast row at the time of the audit event) and avoiding a collaborative-editing surface that has no MVP user research backing.
- Q4 **(security + data model — HTML sanitisation + size limits)**: What sanitisation rules apply to member-authored subject + body content, and what are the size caps? → **B: Strict allowlist sanitisation + hard size caps.** Server-side sanitiser (e.g., DOMPurify-equivalent or rehype-sanitize on the Application layer) MUST strip everything outside an allowlist before persisting `body_html`. Allowed tags: `p`, `br`, `strong`, `em`, `u`, `a[href]`, `ul`, `ol`, `li`, `h1`–`h4`, `blockquote`, `hr`. **(Critique 2026-04-29 E9/X3 removed `<img>` from the MVP allowlist** — member-authored image embedding allows third-party tracking pixels that bypass our SC-005 privacy-OFF default, and admin review is bypass-prone for this class. Members use external `<a href="…">` links to images instead. Image embedding with mandatory source allowlist enforcement is F7.1 scope.) Explicitly forbidden (stripped without warning): `script`, `style`, `iframe`, `<form>`, all event-handler attributes (`on*`), inline `style="..."` attributes, `<link>`, `<meta>`, `<base>`, `<object>`, `<embed>`, `<svg>` (script vector), `<img>` (tracking-pixel vector), and external stylesheet imports. `a[href]` is permitted but URLs MUST be `http://`, `https://`, or `mailto:` schemes; `javascript:` and `data:` are rejected. Size caps: subject ≤ 200 chars (UI surfaces a soft warning at >60 chars matching email-client preview-pane truncation), body ≤ 200 KB rendered HTML (matches Resend Broadcasts payload limit with ~10% safety margin). Submissions exceeding caps OR containing forbidden constructs after sanitisation diff > 0 are rejected at submit boundary with a bilingual error code (`broadcast_subject_too_long`, `broadcast_body_too_large`, `broadcast_body_unsafe_html`); rejection is a 422 with the offending element listed for the editor to highlight. Sanitisation runs at the **Application layer** (security boundary — Tiptap client-side filtering is best-effort, not the source of truth) and the unsanitised raw body is NEVER persisted. Rationale: a member's body lands in 100+ recipient inboxes; without strict sanitisation a single compromised or malicious member account can phish the entire chamber audience. Editor-side filtering alone is bypassable via direct API call. Image-source allowlisting (Option D) was originally deferred to F7.1 with `<img>` open in MVP, but Critique 2026-04-29 E9/X3 surfaced that admin review is bypass-prone for this class (a malicious or compromised member account could embed `<img src="https://attacker.com/track.gif?recipient=X">` to track recipient opens, contradicting our SC-005 privacy-OFF default). **MVP scope therefore drops `<img>` entirely** — members link to externally hosted images via `<a href="https://example.com/image.png">View image</a>` instead. Image embedding with mandatory same-origin / Resend-CDN source allowlist returns in F7.1.
- Q5 **(scope — F6 EventCreate dependency for `event_attendees_last_90d` segment)**: F6 (EventCreate Zapier import — order #8 in Phase 2) is downstream of F7 (order #6) in `docs/phases-plan.md`, but the project ships Phase 2 as a single batch release. Should F7 keep the `event_attendees_last_90d` segment in MVP, drop it, gate it behind a flag, or stub it? → **D: Stub data source — F7 builds the segment now against a thin `event_attendees` interface (Application-layer port) that returns an empty list until F6's Zapier importer populates the source table.** Implementation contract: F7's segment-resolver depends on a `EventAttendeesRepository.findRecentAttendeeEmails(tenantId, sinceDays = 90)` port that F7 ships with a stub implementation returning `[]`; F6's `/speckit.implement` swaps in the real Drizzle-backed implementation that reads from the `event_attendees` table. At big-bang release time both features land together — the segment "just works" with real data without requiring a spec amendment, schema migration, or feature flag flip. Trade-off accepted: during F7-only development a member picking the segment will hit the empty-segment refusal of FR-002c — this is acceptable because (a) F7 is not deployed in isolation per the batch-release plan, (b) the alternative (feature flag) adds permanent maintenance overhead for a flag that will be removed at ship, (c) the alternative (drop and re-add) requires a spec amendment + re-validation cycle right before release. Rationale: keep the spec describing **end-state truth** (what ships to production) rather than per-feature-step intermediate truth.
- Q6 **(NFR — performance budgets)**: What p95 latency budgets does F7 commit to per surface, so the `/speckit.verify` perf gate can assert against concrete numbers? → **B: Explicit per-surface p95 budgets matching F4/F5 convention.** Compose page TTFB **p95 < 600ms**; submit endpoint (sanitiser + segment resolution + reservation insert) **p95 < 1.2s** (mirrors F5 initiate budget); admin queue list page (server-rendered, 1k pending broadcasts max) **p95 < 500ms** (mirrors F4 invoice-list budget); admin approve & send-now endpoint (state transition + Resend Broadcasts API call) **p95 < 1.5s** (Resend round-trip dominates); webhook handler (signature verify + delivery row upsert) **p95 < 250ms**; public unsubscribe page TTFB **p95 < 400ms** (server-rendered, no auth). Budgets apply at production scale (~131 members for SweCham; future tenants ≤5k members) and are enforced via CI synthetic load + Vercel Speed Insights real-user monitoring. SC-010 captures the aggregate budget invariant. Rationale: qualitative phrases ("within 30 seconds") cannot fail a verify gate; concrete budgets give load tests a pass criterion and regressions a tripwire. Numbers chosen from analogous F4/F5 measurements rather than independent benchmark since F7 surfaces are similar weight (compose ≈ F3 member-create; admin queue ≈ F4 invoice-list; webhook ≈ F4/F5 webhook).
- Q7 **(scale — recipient list maximum per broadcast)**: What is the maximum recipient-list size F7 supports per single broadcast, and what happens when a segment resolves above the cap? → **B: Hard cap of 5,000 recipients per single broadcast at the dispatch boundary.** Submissions whose resolved+suppression-filtered recipient list exceeds 5,000 entries are rejected at submit time AND at dispatch time (defence in depth) with a bilingual `broadcast_audience_too_large` error code; the audit log records the breached count; the member sees "Audience exceeds the 5,000-recipient maximum — please choose a narrower segment". Cap rationale: (a) accommodates SweCham's ~131 members with 30+× headroom, (b) accommodates a typical mid-size chamber (100–500 members) with 10× headroom, (c) stays well under Resend Broadcasts' documented 10,000 hard limit so MVP NEVER paginates a broadcast across multiple Resend audiences, (d) caps cost-per-broadcast at a predictable ceiling for budget forecasting, (e) keeps the dispatch + audit + delivery flow single-batch (one Resend audience id, one broadcast id, one set of delivery events) — pagination logic is the kind of edge case that bug-farms in MVP. Pagination across multiple Resend audiences for >5k recipients is OUT-OF-SCOPE F7 MVP and tracked as F7.1 (re-add the existing >10k edge-case note as the F7.1 design starting point).
- Q8 **(domain — recipient resolution per member, primary contact vs all contacts)**: When a segment like `all_members` or `tier:premium` resolves, does the broadcast target each member's primary contact only, every contact attached to the member (F3's 1:N contacts), or a separate per-contact opt-in surface? → **B: Primary contact only.** For every member-based segment (`all_members`, `tier:<code>`), the recipient list is the SET of `primary_contact_email` values across in-segment members (deduplicated by email — the same person who is primary contact for two member companies receives ONE broadcast). Members without a `primary_contact_email` are EXCLUDED from the resolved list AND surfaced via a `member_missing_primary_contact` audit signal at resolve time so admin can backfill. The `event_attendees_last_90d` segment continues to resolve to the (already person-level) attendee email list from F6. The `custom` segment is a member-supplied flat email list and is unaffected by member↔contact mapping. Rationale: (a) recipient-count math stays intuitive ("131 members in segment → 131 emails"), (b) mirrors F4 invoicing and F1 transactional which already address members via `primary_contact_email`, (c) preserves the 5k cap (FR-016a) headroom, (d) matches member intuition that "an E-Blast to all members" sends one message per company, (e) avoids inventing a per-contact opt-in surface in MVP (Option C/D) which is significant new UX. Trade-off: secondary contacts at a member company do NOT receive broadcasts in MVP — if a chamber wants every named contact to receive E-Blasts, that is F7.1 scope (add a `receive_broadcasts` flag to F3 contacts + UX for managing it).
- Q9 **(security — custom segment validation policy)**: The `custom` segment lets a member paste up to 100 email addresses. What constraints apply to entries to prevent the chamber's verified Resend sender identity being used for arbitrary external blasts? → **B: Restricted to tenant's known emails.** Each entry in the custom list MUST resolve at submit time to one of: (a) any `members.primary_contact_email` in the tenant, (b) any `contacts.email` row in the tenant (F3 secondary contacts ARE allowed in the custom list, even though they are NOT auto-included by member-based segments per Q8), or (c) any `event_attendees.email` from F6 (with the FR-015a stub returning empty during F7-only development). Entries that do NOT resolve are rejected at submit boundary with a `broadcast_custom_recipient_unknown` error code listing each unresolved address (case-insensitive, trim-normalised) so the member can remove them. Admin review remains mandatory regardless. Rationale: (a) preserves "broadcasts go to people the chamber has a relationship with" invariant — chamber sender identity never reaches arbitrary inboxes, (b) keeps GDPR/PDPA lawful-basis defensible (existing relationship is the lawful-basis), (c) keeps suppression + unsubscribe mechanics meaningful (every recipient is identifiable in the tenant graph), (d) makes admin review tractable (admin recognises every address). External-invitee broadcasts (e.g., a partner-event invitation to a curated outside list) are F7.1 scope and require their own consent/lawful-basis design.
- Q10 **(state machine — cancellation cutoff)**: Until which state can a broadcast be cancelled, and by whom? → **C: Cancellable until `approved` (with or without `scheduled_for`); rejected once `sending`.** Both the originating member and any admin (and `manager` is read-only — no cancel authority) MAY cancel a broadcast in `submitted` or `approved` state; cancellation transitions the row to `cancelled`, releases the quota reservation per FR-003, writes a `broadcast_cancelled` audit event with actor + cancellation reason (free-text optional, ≤500 chars; admin-initiated cancels MUST include a reason for member transparency, member-initiated cancels MAY omit), and sends a transactional notification to the originating member (admin-cancel) or admin queue (member-cancel). Cancellation requests against rows already in `sending`, `sent`, `cancelled`, `rejected`, or `failed_to_dispatch` MUST be rejected with 409 and a bilingual `broadcast_cancel_too_late` error code; an audit event of the same name is written so the attempt is traceable. The point-of-no-return is the `approved → sending` transition (i.e., the moment the Resend Broadcasts API is called and acknowledged) — once Resend has accepted the dispatch, recipient inboxes are committed and only the natural delivery lifecycle (US5 webhook → `sent`) can complete the broadcast. This mirrors F5 payment-cancel semantics (cancel before processor accepts; refund after, no mid-flight cancel) and gives both actors a clear, predictable cutoff tied to an irreversible external action.
- Q11 **(submit precondition — reply-to fallback when submitting member has no primary contact email)**: A member of a quota-paying tier whose company record has `primary_contact_email = NULL` would dispatch a broadcast with no valid reply-to. Should the system block, fallback to the submitting user's login email, fallback to tenant support, or allow no Reply-To header? → **A: Block submission with deep-link to F3 profile edit.** Add precondition `j` to FR-002 — submission requires the originating member's `primary_contact_email` to be non-null. Rejection emits a bilingual `broadcast_member_missing_primary_contact_email` error code carrying a deep-link to the F3 self-service profile edit screen so the member fills it in and resubmits. Rationale: (a) preserves the invariant "every broadcast has a valid reply-to that goes to the right person", (b) reuses existing F3 self-service profile UX with zero new screens, (c) gives the member a clear next-action ("update your company profile, then come back to compose"), (d) keeps the dispatch path single-branch (no fallback chain to maintain), (e) makes the data-quality nudge member-visible — which is healthier than silently falling back to a less-correct address. Trade-off: a member with a missing primary contact cannot self-service E-Blasts until they update their profile; admin can still see the member's broadcast attempts in the audit trail and proactively help.
- Q12 **(authorisation — admin-on-behalf-of-member submission)**: Can an admin submit a broadcast on behalf of a specific member (real-world chamber workflow: a member calls the office and says "send this for me"), and how should the actor identity + quota be tracked? → **B: Admin proxy with dual-actor tracking.** Admin queue includes a "Submit on behalf of `<member>`" surface that opens the standard compose flow with the originating member explicitly selected. The broadcast row carries TWO actor fields: `requested_by_member_id` (the member whose quota slot is reserved/consumed and whose `primary_contact_email` populates the reply-to) AND `submitted_by_user_id` (the user who actually clicked Submit — the admin in this case, the member themselves in the self-service path). Both fields are NOT NULL. When the two correspond to different users, the audit log records `broadcast_submitted` with `actor_role = 'admin_proxy'` plus both ids; quota counts against the member regardless. Rationale: (a) preserves quota fairness — the chamber never gets free broadcasts via admin proxy, (b) makes the operational reality (admin assistance) first-class without inventing a new role, (c) preserves audit accountability — it is unambiguous which admin proxied and on whose behalf, (d) the originating member still receives the rejection email if applicable + appears as the row owner on the member portal so they can see their broadcast history, (e) avoids the security anti-pattern of admin "becoming" the member with no trail. The proxied submission STILL goes through the standard admin-review queue (Q10 + US2) — admins do NOT auto-approve their own proxied submissions; an admin approving their own proxied submission is permitted (single-admin chambers exist) but the audit row records both the proxy-submit AND the approve as the same actor for transparency. Self-service member submission is unchanged (`requested_by_member_id` and `submitted_by_user_id` resolve to the same user identity).
- Q13 **(strategic framing — build vs buy positioning, Round 1 critique P8/X5)**: Is F7 primarily a **ROI play for SweCham alone** or a **SaaS-foundation investment** for F11 multi-tenant billing + future tenant onboarding? → **C: SaaS-foundation investment.** F7 ships in Phase 2 as one of the foundation features for the multi-tenant Chamber-OS platform; SweCham single-tenant utilization is a bonus, not the primary success metric. Implications: (a) **SC-001 lowered from "80% of paid quota slots delivered within same calendar quarter" to "≥50% year 1, ramping to 65% year 2, 75% year 3"** (industry benchmark for marketing-quota utilization is 50-70%; an 80% bar in year 1 sets up false-failure signal); (b) a **new strategic-value success criterion SC-011 added: "F7 ready to onboard a 2nd tenant within 6 months of F11 ship — verified by an end-to-end provisioning dry-run on a JCC-test tenant fixture"**; (c) the 18-22 day build cost is framed as multi-tenant-platform spend, not single-tenant-feature spend; (d) Mailchimp / Brevo "buy" alternatives remain rejected (per `docs/email-broadcast-analysis.md` § 3) on vendor-sprawl + per-tenant-API-key + integration-surface grounds — at SaaS scale these costs compound across tenants, so build-now is genuinely cheaper than retrofit-buy-then-build-later. Rationale: this framing prevents stakeholder disappointment if year-1 chamber adoption is modest (typical for new self-service surfaces), aligns the budget conversation with strategic intent, and makes /speckit.tasks effort allocation honest about what F7 actually is.
- Q14 **(success criteria — SC-005 averaging window, Round 1 critique P14)**: Should SC-005 (≤2% bounce + complaint combined) add a per-broadcast threshold so single-broadcast spikes are caught immediately, or stay rolling-30d-only? → **B: Two-layer — rolling 30d + per-broadcast >5% complaint threshold with auto-halt.** SC-005 splits into two independent invariants: (a) **rolling 30d**: combined hard-bounce + complaint rate ≤2% across all broadcasts in the tenant in a rolling 30-day window — same as before; tracks tenant-level deliverability trend, (b) **per-broadcast spike**: any single broadcast whose complaint rate at any point during its delivery exceeds **5%** triggers an immediate `broadcast_complaint_rate_per_broadcast_breach` audit event (high severity), pages on-call admin via the deliverability runbook, AND **auto-halts further dispatches from the offending member until an admin reviews + clears the hold** (`members.broadcasts_halted_until_admin_review = true` flag, manually cleared via admin action; new audit event `broadcast_member_dispatch_resumed` on clear). The halt prevents the same member from queuing a 2nd bad broadcast 5 minutes after the 1st. Rationale: aligns with industry SES / Mailchimp / Resend best-practice (complaint-rate alerting is per-campaign, not per-account-rolling); catches single-broadcast disasters within minutes not 30 days; creates an automated safety net before the rolling window flags it. The two layers are independent — a tenant could pass the rolling 30d invariant while still having one bad broadcast trigger the per-broadcast halt. Adds 2 new audit event types (`broadcast_complaint_rate_per_broadcast_breach` + `broadcast_member_dispatch_resumed`) at session-5 close — extended further by Critique Round 3 to add `broadcast_member_halted_pending_review` (R3-NEW-1) bringing the catalogue eventually to 37 events. Adds 1 new column to the `members` table: `broadcasts_halted_until_admin_review boolean default false`. **Clear-halt UI** (Critique Round 3 R3-NEW-3 — single source of truth): the admin clear-halt action lives on the **F7 admin queue page** (`/admin/broadcasts`) as a top-of-page banner whenever ≥1 member in the tenant is in halt state. Each halted member appears with a "Review + Clear halt" button that opens a confirmation dialog (typed-phrase pattern matching F4 destructive-action convention). Clearing emits `broadcast_member_dispatch_resumed` audit. The F3 admin members list (`/admin/members`) shows a small "Broadcasts halted" badge per affected member for affordance discoverability but does NOT host the clear action — clicking the badge navigates to the F7 admin queue's clear-halt surface. `manager`-role users see the badge + queue but no clear-action button (read-only per FR-014). FR-002 precondition `k` (R3-NEW-1) adds the matching submit-time check.
- Q15 **(compliance trail — opt-in trail strengthening, Round 1 critique P9)**: Should F7 record per-member acknowledgement of marketing broadcasts to strengthen the GDPR Art. 7 "demonstrable consent" defence, beyond the existing PDPA §24 + GDPR Art. 6(1)(b) "contract performance" lawful basis? → **B: Add `members.broadcasts_acknowledged_at timestamp` column + one-time portal banner.** The banner ("Your tier includes marketing broadcasts from chamber members. You may unsubscribe at any time.") appears on first sign-in post-F7-launch AND on the F7 benefits-page first-visit per member, until acknowledged. Acknowledgement records the timestamp; emits a new audit event `member_acknowledged_broadcasts_terms` (36th in F7 catalogue). **The acknowledgement is NOT a precondition for receiving broadcasts** — the lawful basis remains "contract performance" (existing membership relationship), so a member who hasn't yet acknowledged is still in-scope for the "All members" segment per FR-015c. The banner + column are evidence-strengthening, not gating. The banner UX is bilingual EN/TH/SV per FR-039. Manager-role + admin-role users do NOT see the banner (only `member`-role). The column populates an audit-log query for compliance officer requests ("show me when each member acknowledged the marketing-broadcasts notice"). Adds 1 audit event + 1 column on F3 `members` table; 1 new migration (0071). Rationale: cheap to implement (~1 day), zero ongoing operational cost, materially strengthens the GDPR Art. 7 defence ("we can show you exactly when this member saw + acknowledged the marketing-broadcasts notice"), and creates a forward-compat hook for F12 white-label tenants that may want to customize the banner copy. **Banner trigger conditions** (Critique Round 3 R3-NEW-2 refinement): banner appears IF AND ONLY IF (i) signed-in user has `member` role, (ii) member's tenant has `FEATURE_F7_BROADCASTS=true`, (iii) member's `broadcasts_acknowledged_at IS NULL`, (iv) member's plan has `eblast_per_year > 0` OR member's record is `is_active = true` (member may receive broadcasts as recipient even without sender quota). Banner is **server-driven** — `broadcasts_acknowledged_at` is the sole source-of-truth, no device-local cookie. Banner shows on **every sign-in** until acknowledged (not just first sign-in). Member can dismiss with "Remind me later" link → records no audit event + no column update → banner shows again next sign-in. **Per-tenant scope** (Critique Round 3 R3-NEW-7): `broadcasts_acknowledged_at` is scoped to `(tenant_id, member_id)` per FR-018 tenant isolation — a user who is a member of multiple tenants sees an independent banner per tenant context, and acknowledging tenant A does NOT acknowledge tenant B.
- Q16 **(recipient resolution — self-targeting case, Round 1 critique P7)**: When a paying-tier member submits a member-based segment ("All members" or `tier:<code>`), should the submitter's own primary contact email be included in the resolved recipient list (they receive their own broadcast)? → **B: Auto-exclude the submitter from member-based segments.** FR-015c resolver query is extended to `SELECT DISTINCT primary_contact_email FROM members WHERE tenant_id = $1 AND <segment_predicate> AND primary_contact_email IS NOT NULL AND member_id != $2 /* requested_by_member_id */`. The compose-page surfaces a bilingual microcopy hint ("You won't receive your own broadcast — your primary contact email is automatically excluded from member-based segments."). The `estimated_recipient_count` reflects the exclusion (count is `total - 1` for the submitter when the submitter is themselves a member of the segment). The `custom` segment is **unaffected** (member explicitly chose those addresses, including possibly themselves — explicit choice overrides the auto-exclude rule). The `event_attendees_last_90d` segment is **unaffected** (the submitter being an event attendee is independent of being the broadcast originator; if they attended the event, they remain in the segment — their primary-contact email is in the F6-populated `event_attendees` table, not derived from `members`). Admin-proxy submissions (Q12) auto-exclude the **proxied member** (not the admin), since `requested_by_member_id` identifies the proxied member, not the acting admin. Rationale: matches member intuition ("most marketing tools exclude the sender by default"), prevents the confusing UX of submitter wondering if their own broadcast actually went out when they see it in their inbox, costs nothing (one extra WHERE clause), and aligns with chamber operations expectation. Trade-off accepted: a member who wants to "see what their broadcast looks like" loses a free QA check — but the compose-page preview pane already shows the rendered email, so the QA need is covered upstream.
- Q17 **(actor scope — events / sponsorship team submission, Round 1 critique P2)**: Do chamber events / sponsorship teams need a first-class submission UX in F7 MVP, or is the existing admin-proxy mechanism (Q12) sufficient? → **A: Admin-proxy (Q12) is sufficient for F7 MVP — defer dedicated events/sponsorship UX to post-MVP after user research.** F7 actor model remains the F1-inherited 3-role set: `member` (compose own broadcasts) + `admin` (review/approve/proxy + manage settings) + `manager` (read-only). Chamber operational broadcasts (event reminders from the events team, partner announcements from the sponsorship team, chamber-wide news) route via the existing admin-proxy flow per Clarifications Q12 — the admin uses the "Submit on behalf of `<member>`" surface targeting either (a) the chamber's own admin-managed member entity if the chamber maintains one, or (b) a designated point-person's member entity. **No new RBAC role, no new auto-approve bypass, no new audit category.** Rationale: (a) avoids inventing a new role + permission scheme just for an edge use case that admin-proxy already supports, (b) keeps the F1-aligned actor model simple, (c) Option B (auto-approve `chamber_operations` role) would bypass brand-safety review for the very content class most likely to need it (mass-event invitations), (d) Option C (pre-seed "Chamber Communications" pseudo-member at F10 onboarding) is a reasonable post-MVP enhancement once we have data on operational broadcast frequency — but premature for MVP, (e) we haven't done user research with chamber events / sponsorship teams; building a UX without that research is YAGNI per Constitution Principle X. **Post-MVP follow-up**: tracked as F7.2 in roadmap notes — if admin queue analytics show >30% of submissions originate as admin-proxy in the first 6 months post-launch, promote events/sponsorship dedicated UX out of the backlog with user research as the input.
- Q18 **(success criteria — SC-011 timing model, Critique Round 3 R3-NEW-6)**: SC-011 was originally scoped as "F7 ready to onboard 2nd tenant within 6 months of F11 ship", but F11 (SaaS Billing) is in Phase 4 of phases-plan — possibly 1+ year out. Should SC-011 stay a conditional 6-month clock that starts at F11, or convert to a per-release invariant? → **B: Per-release invariant — multi-tenant readiness MUST be demonstrated at every F7 release.** Rewording: "F7's multi-tenant readiness MUST be demonstrated at every release via the JCC-test tenant fixture: (a) cross-tenant integration test green (Constitution v1.4.0 Principle I clause 3 Review-Gate blocker — already required); (b) provisioning dry-run scripted in CI nightly job that creates a fresh test-tenant, seeds default segments per migration 0068, configures a Resend test-mode account stub, submits + approves + dispatches a synthetic broadcast, verifies cross-tenant suppression isolation, and tears down — completing in <5 minutes; (c) data-only configuration: no F7 code change required to onboard a new tenant. Failure of any of these = F7 ship blocker. F11's onboarding window (when it ships) inherits this readiness — the original 6-month clock becomes irrelevant because readiness is continuous." Rationale: (a) removes the F11-deferred-clock ambiguity (R3-NEW-6 finding), (b) makes readiness a continuous CI-verified property rather than a one-time milestone, (c) prevents drift in the F7-ship-to-F11-ship interim, (d) aligns with how other F7 SCs work (continuous invariants, not milestones), (e) the JCC-test tenant fixture is concrete + automated rather than aspirational. The "6 months of F11 ship" phrasing in plan.md narrative + spec § Q13 retained for strategic-framing context but SC-011 itself is now per-release.
- Q19 **(compliance scope — banner per-tenant vs platform-global, Critique Round 3 R3-NEW-7 formal confirmation)**: Is the GDPR Art. 7 acknowledgement banner (Q15) per-tenant or platform-global? → **A: Per-tenant scope (confirmed).** A user who is a member of multiple tenants sees an independent banner per tenant context — `broadcasts_acknowledged_at` lives on the tenant-scoped `members` row, so the data model enforces per-tenant naturally. Acknowledging tenant A does NOT acknowledge tenant B. Rationale: (a) FR-018 tenant-isolation invariant — "a SweCham unsubscribe MUST NOT also unsubscribe the same person from JCC" generalises to "a SweCham acknowledgement MUST NOT auto-acknowledge JCC", (b) GDPR Art. 7 "demonstrable consent" is per-controller — each tenant is its own data controller in the SaaS model, so consent is per-tenant by regulatory design, (c) tenant marketing voice / terms may differ — JCC's banner copy may differ from SweCham's at F12 white-label time, and per-tenant acknowledgement is the only model that allows independent re-acknowledgement when tenant terms change, (d) Option B (platform-global on F1 `users` row) would create cross-tenant data leakage since F1 `users` is intentionally cross-tenant per `docs/saas-architecture.md` § 4. This Q19 is a **formal confirmation** of the per-tenant default already integrated by Round 3 remediation into the Q15 narrative, US3 AS9, and data-model § 1.3a — no additional spec changes required beyond this Clarifications record.

### User Story 5 — Per-recipient delivery tracking via Resend webhook (Priority: P2)

When Resend sends a broadcast, it emits per-recipient delivery events
(`email.sent`, `email.delivered`, `email.bounced`, `email.complained`) to a
webhook endpoint. F7 ingests those events idempotently, records each event
on a `broadcast_deliveries` row, updates the broadcast aggregate counts, and
when all expected events for a broadcast are received (or a 24-hour timeout
elapses), transitions the broadcast to `sent` and **consumes the quota**
(this is the only state transition that consumes a member's annual quota
counter — drafts, submits, approvals, schedules, and even partial deliveries
do NOT consume it).

**Why this priority**: Without delivery tracking the chamber cannot tell a
member "your E-Blast went out", cannot remediate bounces, cannot detect
deliverability problems, and cannot defend the quota counter against
disputes. P2 — the broadcast technically delivers without it, but the
benefit is unobservable.

**Independent Test**: With one `sending` broadcast (from US2), simulate a
Resend webhook sequence (`email.sent` for 100 recipients, then
`email.delivered` for 95, `email.bounced` for 3, `email.complained` for 2)
delivered to `/api/webhooks/resend-broadcasts` with valid signatures. Verify
(a) 100 `broadcast_deliveries` rows are created keyed by `(tenant_id,
delivery_id)` with deduplication on Resend's event id, (b) the broadcast
aggregate counts (`delivered_count`, `bounced_count`, `complained_count`)
match the events, (c) the broadcast transitions `sending → sent` after the
last expected event, (d) the quota counter (`quota_year_consumed = 2026`)
is incremented at this transition, (e) re-delivering the same webhook
payload is idempotent (no duplicate rows, no duplicate quota consumption,
no duplicate audit event).

**Acceptance Scenarios**:

1. **Given** a `sending` broadcast for 100 recipients, **When** Resend posts `email.delivered` events one by one over 5 minutes, **Then** each event creates exactly one `broadcast_deliveries` row keyed by Resend's `message_id` + event-type, and re-posts of the same event are no-ops (idempotency).
2. **Given** a `sending` broadcast and a webhook payload whose HMAC signature does NOT match the configured Resend signing secret, **When** the webhook handler validates the signature, **Then** the request is rejected with 401, the payload body is NOT read or persisted (only the failure is audited as `broadcast_webhook_signature_rejected`), and Resend will retry per its own retry policy.
3. **Given** all expected delivery events have been received for a broadcast, **When** the last event lands, **Then** the broadcast transitions `sending → sent`, `quota_consumed_at` is stamped with the transition timestamp, `quota_year_consumed = 2026` (the calendar year of the transition, not of submission — see Assumptions), the originating member receives a transactional "delivery summary" email with delivered/bounced/complained counts, and the audit log contains `broadcast_sent` (with counts) and `broadcast_quota_consumed` (with member id + year + count = 1).
4. **Given** a broadcast where Resend has sent only 97 of the 100 expected events after 24 hours, **When** the timeout elapses, **Then** the broadcast transitions to `sent` anyway (timeout-completion path) with the partial counts persisted, the quota IS consumed (the broadcast was successfully dispatched even if a few recipient mailboxes were slow to confirm), and a `broadcast_send_timeout_completed` audit event is written so admin observability sees the partial.
5. **Given** a bounce event with reason `hard_bounce` for a recipient email, **When** the event is recorded, **Then** in addition to the delivery row, the recipient email is auto-added to the tenant's suppression list with `reason = 'hard_bounce'` so future broadcasts in that tenant skip it (Resend itself does this internally, but we mirror it locally for our own segmentation).
6. **Given** a complaint event (recipient marked as spam in their mail client), **When** the event is recorded, **Then** the recipient email is auto-added to suppression with `reason = 'complaint'`, an admin-only alert is raised on the tenant's dashboard (complaint rate is a deliverability KPI), and the audit log contains `broadcast_complaint_received` with email hash (NOT raw email).

### User Story 6 — Scheduled future-dated send via cron handler (Priority: P3)

A member submitting an E-Blast may pick a future send date/time (e.g., "send
on Tuesday morning so it lands in inboxes during peak open-rate"). After
admin approval, the broadcast remains in `approved` status with
`scheduled_for` set; a cron handler runs every 5 minutes, picks up
`approved` broadcasts whose `scheduled_for ≤ now()`, calls the Resend
Broadcasts API, and transitions them to `sending`. From there, US5 takes
over.

**Why this priority**: Send-time control is a recognised marketing-delivery
best practice and is requested by sophisticated members. P3 because
"approve & send now" (US2) covers the majority case; future-dated send is
a value-add that can ship after the MVP slice is validated.

**Independent Test**: Submit an E-Blast as a member with `scheduled_for =
now() + 1h`, approve as admin. Wait until the cron handler fires after
`scheduled_for` has elapsed (or trigger it manually in test). Verify the
Resend API was called within 5 minutes of `scheduled_for`, the broadcast
transitions `approved → sending` exactly once (no duplicate trigger if cron
fires twice while the call is in-flight), and the delivery flow continues
per US5.

**Acceptance Scenarios**:

1. **Given** an `approved` broadcast with `scheduled_for = T0`, **When** the cron handler runs at `T0 + 2min`, **Then** the Resend Broadcasts API is called exactly once, `status = approved → sending`, and the audit log contains `broadcast_send_started` with `scheduled_for = T0`, `actual_send_at = T0+2min`, and `delay_seconds`.
2. **Given** an `approved` broadcast with `scheduled_for = T0` but the cron handler fails to dispatch (Resend API outage), **When** the next cron run happens at `T0 + 5min`, **Then** the broadcast is retried (still in `approved`); after 1 hour of retries, `status = approved → failed_to_dispatch`, an admin alert is raised, the originating member is notified via transactional email that their scheduled send did not go out, and the quota reservation remains held (NOT released — admin can manually re-trigger or rescheduled).
3. **Given** two cron worker instances run concurrently, **When** both attempt to dispatch the same `approved` broadcast in the same window, **Then** only one Resend API call is made (idempotent dispatch via row-level `SELECT … FOR UPDATE` or equivalent transactional guard scoped per `(tenant_id, broadcast_id)`), and no duplicate `sending` transition occurs.
4. **Given** a member or admin cancels a scheduled `approved` broadcast before `scheduled_for`, **When** the cancellation is committed, **Then** `status = approved → cancelled` per FR-004a, the cron handler skips that row on its next run, the quota reservation is released, the audit log contains `broadcast_cancelled` with actor + actor_role + optional cancellation reason, and the broadcast detail page surfaces "Cancelled before send".
6. **Given** a `sending` broadcast (Resend dispatch already acknowledged), **When** a member or admin attempts to cancel via UI or API, **Then** the request is rejected with HTTP 409 `broadcast_cancel_too_late`, the broadcast continues its natural delivery lifecycle, no state change occurs, and an audit event of the same name is written with the attempting actor id + current state.
5. **Given** an `approved` broadcast was scheduled for a date when the originating member's plan would have expired, **When** the cron handler picks up the broadcast and the member is no longer in an entitled tier, **Then** the broadcast is still dispatched (the entitlement was confirmed at submit + approve time, and the tenant has already accepted the obligation) but a `broadcast_sent_with_expired_member_plan` audit event is written for admin observability.

### Edge Cases

- **Webhook replay / out-of-order delivery**: the system MUST be idempotent on Resend's unique event id; a second delivery of the same event is a no-op. Out-of-order events (e.g., `delivered` before the matching `sent`) MUST not create inconsistent aggregate counts — counts are computed from the union of received event ids, not from a state machine of "which event came first".
- **Recipient list cap exceeded** (Clarifications Q7, B + FR-016a): a broadcast whose resolved+suppression-filtered list is >5,000 is rejected at submit AND dispatch with `broadcast_audience_too_large`; pagination across multiple Resend audiences is OUT-OF-SCOPE F7 MVP (deferred to F7.1). SweCham's ~131 members is far below the cap; the cap protects against unbounded cost + Resend account-level rate-limit incidents at future-tenant scale.
- **Member missing primary contact email** (Clarifications Q8, B + FR-015c): a member is in-segment but has `primary_contact_email IS NULL` (e.g., admin created the member row before the contact was filled in). The resolver MUST exclude that member from the recipient list AND emit a `member_missing_primary_contact` audit signal so admin can backfill. The member is NOT counted toward `estimated_recipient_count`. Excluded members do NOT cause submission rejection unless ALL members in the segment are excluded (in which case FR-002c empty-segment rule fires).
- **Same person primary-contact for two member companies**: deduplication is on `LOWER(TRIM(primary_contact_email))`. One email is sent regardless of how many member companies that person is primary-contact for; the `broadcast_deliveries` row records the delivery once with `recipient_member_id = NULL` (ambiguous) OR the first member encountered (deterministic; documented at plan time).
- **Custom-segment entry not in tenant graph** (Clarifications Q9, B + FR-015d): a member pastes an email that does NOT match any `members.primary_contact_email`, `contacts.email`, or `event_attendees.email` in the tenant. The submission is rejected at submit boundary with `broadcast_custom_recipient_unknown`; the response lists each unresolved address verbatim so the editor can highlight them for removal; an audit event of the same name is written carrying the count of unresolved entries (NOT the addresses). This prevents the chamber's verified Resend sender identity being used for arbitrary external blasts.
- **Member tier downgrade between submit and send**: the entitlement check at SUBMIT time is the source of truth — once an admin has approved the submission, the broadcast WILL send even if the member's tier changes before `scheduled_for` (the tenant has accepted the obligation). The quota counter for the consumption year remains tied to the member's plan-at-send-time `eblast_per_year` for **subsequent** quota math, but the already-approved broadcast counts as 1 consumed regardless.
- **Quota year boundary** (1 January at 00:00 in the tenant's configured timezone — Asia/Bangkok for SweCham): a broadcast submitted on 2026-12-31 23:59 ICT, approved on 2027-01-01 00:01 ICT, sent at 2027-01-01 00:02 ICT consumes the **2027** quota (consumption is keyed to the timestamp at the `sending → sent` transition, not at submission). Members are warned on the compose screen near the year boundary: "If your broadcast is approved after midnight, it will consume your 2027 quota."
- **Tenant disables F7 mid-flight**: an admin may toggle off the "Online E-Blast" feature flag globally for the tenant. Existing `submitted` and `approved` broadcasts MUST be visible to admins for completion/rejection (no orphaning), but new submissions are blocked at the compose surface with a bilingual "E-Blast service is temporarily unavailable" message.
- **Body content with sensitive data leaks**: members might paste a customer list or PII into the body. The system MUST NOT scan content for PII (out of scope for MVP) but MUST display a microcopy warning on compose ("This email goes to all selected recipients — do not include private contact data") and the admin reviewer is the human gate.
- **Stripe / payment / billing data in body**: same warning as above. Admin rejects if observed. Future automated content-policy scanning is out of MVP.
- **Recipient address with mixed case or trailing whitespace**: suppression-list matching MUST normalise (lowercase + trim) before comparing to incoming Resend events.
- **Same recipient on multiple tenants**: suppression is **tenant-scoped** — unsubscribing from SweCham broadcasts MUST NOT unsubscribe the same person from JCC broadcasts (they may not even be on JCC). This is a Principle I tenant-isolation guarantee enforced via `(tenant_id, email_lower)` composite suppression key.
- **Sender identity / from-name conflict**: the from-name is constructed by the system as `<member display name> via <tenant display name>` (e.g., "Fogmaker International AB via SweCham"). The reply-to is the originating member's `primary_contact_email` so that recipient replies route correctly; submissions where this field is null are blocked at FR-002 precondition `j` (Clarifications Q11, A) — there is no fallback chain (login email / tenant support / no-reply-to) so every dispatched broadcast has a known-correct reply-to. Both from-name and reply-to are non-editable in MVP to prevent impersonation.
- **Subject line containing Stripe-decline-code-style noise**: not in scope — admin reviews subject; spam-score analysis is out of MVP.
- **Reduced motion / keyboard-only / screen reader**: the compose editor (rich text), preview pane, segment picker, scheduling date-picker, admin queue, and unsubscribe page MUST respect `prefers-reduced-motion`, be fully keyboard-navigable, and announce status transitions via ARIA-live regions per `docs/ux-standards.md`. The rich-text editor specifically MUST expose all formatting commands via keyboard (Ctrl+B etc.) and announce focus changes.
- **i18n coverage**: every member-facing + admin-facing + recipient-facing string (including the unsubscribe public page) MUST exist in EN + TH + SV. Locale resolution rules: signed-in users → user's selected locale; recipient on the unsubscribe page → `lang` query param (signed) → Accept-Language → tenant-default → EN.
- **Sensitive-data leakage in logs**: full email recipient lists, raw email body content, raw subject lines, and Resend signing secrets MUST NEVER appear in logs. Audit payloads carry hashes (sha256) of recipient emails and rejection reasons, NOT raw content. Body content is stored in `broadcasts.body_html` only; admin-review audit rows reference it by id, not by inline value.
- **Concurrent admin approval race** (two admins click Approve at the same time): per US2 AS6, the second action returns 409 and is audited; only one Resend dispatch happens.
- **Edit attempt on a submitted/approved broadcast** (member or admin tries to PATCH the row, e.g., via direct API call or a stale browser tab): per FR-004 + Clarifications Q3, the request is rejected with 409 `broadcast_immutable_after_submit` at both application and database layers; no fields are mutated; an audit event of the same name is written. The member-facing flow is "Cancel + recompose" — there is no in-place edit surface for non-draft broadcasts. Admin-side: typo / link fixes go through Reject-with-reason so the member sees the issue and resubmits the corrected version.
- **Member archived / GDPR-deleted between submit and send**: their `submitted` or `approved` broadcasts are auto-cancelled with `cancellation_reason = 'originator_member_deleted'`, the quota reservation is released (the deleted member no longer has a quota counter to compare against), and the audit log records the cascade.
- **`broadcasts_halted_until_admin_review` flag lifecycle on member state changes** (Privacy checklist CHK047): the halt flag must behave consistently across F3 member lifecycle events: (a) **Member archived** — the halt flag is **preserved** as TRUE (suppression intent persists; if the member is reactivated, the historical halt remains in effect until cleared). (b) **Member reactivated** — the halt flag stays as it was at archive time (admin must explicitly clear via the F7 admin queue Clear-halt action; reactivation does NOT auto-clear). (c) **Plan downgrade** (paying tier → non-paying tier) — the halt flag is preserved (downgraded member with halt history retains the halt for the audit + behavioural-record reason; broadcasts are already blocked by FR-002 precondition `a` for non-paying tiers anyway). (d) **Plan upgrade** (non-paying → paying) — the halt flag is preserved (upgrade does NOT clear historical halts; admin reviews + clears explicitly). (e) **GDPR Art. 17 erasure** — the halt flag is deleted alongside the member row via the data-model.md GDPR Art. 17 erasure cascade. (f) **Member primary contact email change** — the halt flag is preserved (halt is per-member, not per-email; the new contact email inherits the halt state). All state transitions are audit-logged via the existing `member_status_changed` event from F3 (no new F7 audit event needed).
- **Resend account-level outage**: the cron handler degrades gracefully — `approved` broadcasts remain `approved`, retries continue, after 1h sustained outage admin is paged via the same alert-rule mechanism F1 + F4 use, and the originating member is informed by transactional email that their send is delayed.

## Requirements *(mandatory)*

### Functional Requirements

#### Broadcast lifecycle

- **FR-001**: System MUST allow any signed-in `member` user (in a tenant where F7 is enabled) to create a draft broadcast at any time, regardless of remaining quota; drafts do NOT consume or reserve quota. A draft is private to the originating member. System MUST ALSO allow any `admin` user to create a draft on behalf of a specific member via the admin "Submit on behalf of `<member>`" surface (Clarifications Q12, B); the resulting draft is owned by the proxied member (visible on the member portal as their own broadcast) but the `submitted_by_user_id` actor field on the row records the acting admin.
- **FR-001a**: Drafts MUST be retained for **30 days** from `updated_at` (US1 AS3 draft restoration window — Ambiguity A1 resolved 2026-04-29 post-/speckit.analyze). The 30-day window restarts on every draft edit (`updated_at` reset on any field mutation — N5 clarification post-/speckit.analyze re-verify). Drafts older than 30 days with `status='draft'` MUST be auto-deleted by a nightly cleanup cron (`/api/cron/broadcasts/prune-expired-drafts`, Bearer auth via `CRON_SECRET`, daily cadence, separate from the 5-min dispatch cron). Pruning emits NO audit event (drafts are user-controlled scratch space — preserving the FR-001 "drafts do NOT consume or reserve quota" invariant). The 30-day window is a UX promise (members can return to a half-finished compose within a month) rather than a regulatory requirement; future tweaks remain spec-amendable. Members are NOT notified of impending draft expiry in MVP (a "your draft will expire in N days" toast is F7.1 scope per Constitution Principle X).
- **FR-002**: System MUST allow a member to submit a draft for admin review IF AND ONLY IF (a) the member's plan has `eblast_per_year > 0`, (b) the member has at least 1 quota slot not already used or reserved in the current quota year, (c) the resolved recipient segment has ≥1 non-suppressed recipient, (d) the member has not exceeded the 10-submissions-per-rolling-24h rate limit, (e) the tenant has F7 enabled, (f) the subject is 1–200 characters and the body is 1 character–200 KB rendered HTML after sanitisation, (g) the body content passes strict-allowlist sanitisation per FR-002a with zero forbidden constructs (i.e., the diff between submitted HTML and sanitised HTML is empty), (h) the resolved+suppression-filtered recipient list is ≤ 5,000 entries (FR-016a), (i) for `custom` segments only, every entry resolves to a known email in the tenant graph per FR-015d, (j) the originating member's `primary_contact_email` is non-null (Clarifications Q11, A — required so the broadcast has a valid reply-to address; rejection deep-links to the F3 self-service profile edit screen), and **(k) the originating member's `broadcasts_halted_until_admin_review` flag is `false`** (Clarifications Q14, B + Critique Round 3 R3-NEW-1 — set automatically by F7 webhook handler when a single broadcast's complaint rate exceeds 5%; cleared manually by admin via the F7 admin queue clear-halt action per R3-NEW-3; `manager`-role users cannot clear). Each precondition is enforced both client-side (UX) and server-side (security boundary); server-side rejection emits the corresponding bilingual error code (`broadcast_quota_blocked`, `broadcast_empty_segment_blocked`, `broadcast_rate_limit_exceeded`, `broadcast_not_in_plan`, `broadcast_subject_too_long`, `broadcast_body_too_large`, `broadcast_body_unsafe_html`, `broadcast_audience_too_large`, `broadcast_custom_recipient_unknown`, `broadcast_member_missing_primary_contact_email`, **`broadcast_member_halted_pending_review`**).
- **FR-002a**: System MUST sanitise member-authored body HTML server-side (Application layer — Tiptap client-side filtering is best-effort, not the source of truth) before persisting to `body_html`, using a strict allowlist (Clarifications Q4, B; Critique 2026-04-29 finding E9/X3 removed `<img>` from MVP allowlist to close the tracking-pixel privacy bypass). **Allowed tags**: `p`, `br`, `strong`, `em`, `u`, `a[href]`, `ul`, `ol`, `li`, `h1`, `h2`, `h3`, `h4`, `blockquote`, `hr`. **`<img>` is OUT-OF-SCOPE for F7 MVP** — members link to externally hosted images (e.g., `<a href="https://fogmaker.com/product.png">View product image</a>`) instead of embedding them. Image embedding (with mandatory same-origin / Resend-CDN source allowlist enforcement) is F7.1 scope. **Forbidden tags** (stripped silently): `script`, `style`, `iframe`, `form`, `link`, `meta`, `base`, `object`, `embed`, `svg`, **`img`** (member-authored embedded images bypass our SC-005 privacy-OFF default by allowing third-party tracking pixels — admin review is bypass-prone for this class). **Forbidden attributes** (stripped silently): all `on*` event handlers, inline `style="..."`, all `data-*` except a documented allowlist (`data-broadcast-id`, `data-tenant-id`), and any attribute containing `javascript:` or `data:` URI scheme. **`a[href]` URL schemes**: only `http://`, `https://`, `mailto:` are permitted; other schemes (including `javascript:`, `data:`, `file:`, `vbscript:`) cause the link to be stripped (text content preserved). The unsanitised raw body MUST NEVER be persisted; only the sanitised output reaches `body_html`. The sanitiser MUST be deterministic — identical input produces identical output across runs (snapshot-tested in unit tests).

  **Editor sanitiser-strip-warn UX** (Critique 2026-04-29 R2-NEW-2): the editor SHOULD provide real-time feedback when forbidden constructs are present in the member's input — paste handlers MUST run the same sanitiser eagerly and surface a tooltip / inline highlight ("This will be removed at submit. Use [allowlisted alternative].") on each forbidden tag or attribute. The editor's preview pane MUST reflect the post-sanitisation state, not the raw input. Submit-boundary rejection (FR-002 precondition `g`) remains the security guarantee; the editor-side warning is the UX path that prevents the surprise-at-submit pattern. Verified by `tests/e2e/broadcast-compose-and-submit.spec.ts`: paste a body containing forbidden constructs (e.g., `<img>`, inline `style="..."`) → preview shows the construct stripped + a warning toast / inline indicator appears + the Submit button shows a "Content will be modified" microcopy → on Submit, the server-side sanitiser is the source of truth (defence in depth: editor warning is best-effort UX, not a security boundary).
- **FR-003**: A submission MUST reserve exactly 1 quota slot for the originating member in the current quota year (Clarifications Q1, A). The reservation MUST be released on `rejected`, `cancelled`, or `failed_to_dispatch` transitions and consumed on the `sending → sent` transition. The member-facing quota display surfaces both states ("`<used>` sent · `<reserved>` reserved · `<remaining>` of `<eblast_per_year>`"). The "reserved" state MUST never be reachable except via a `submitted` or `approved` broadcast row — it is a derived, not stored, value (queried as `COUNT(broadcasts WHERE status IN ('submitted','approved') AND requested_by_member_id = M AND submission_year = current_year)`).
- **FR-004**: System MUST transition a broadcast through the state machine `draft → submitted → approved → sending → sent` (happy path) with side branches `submitted → rejected`, `approved → scheduled_for_future → sending`, `submitted/approved → cancelled` (per FR-004a), `approved → failed_to_dispatch`. Each transition MUST be atomic, audited, and idempotent; replaying a transition is a no-op. **Content fields (subject, body_html, body_source, segment_type, segment_filter, custom_recipient_emails, scheduled_for) are mutable ONLY while `status = 'draft'`** (Clarifications Q3, A). Any mutation attempt on a row whose `status ≠ 'draft'` MUST be rejected at the application layer (use-case precondition) AND at the database layer (row-level CHECK or trigger), with the request returning 409 `broadcast_immutable_after_submit` and an audit event of the same name. To change content the member cancels (which releases the reservation per FR-003) and creates a new draft.
- **FR-004a**: Cancellation is permitted ONLY while `status IN ('submitted', 'approved')` (Clarifications Q10, C). Both the originating `member` user and any `tenant_admin` user MAY cancel; the `manager` role MUST NOT cancel. Cancellation transitions the row `submitted|approved → cancelled` atomically, releases the quota reservation per FR-003, writes a `broadcast_cancelled` audit event with actor id + actor role + optional `cancellation_reason` (≤500 chars; required for admin-initiated, optional for member-initiated), and sends a transactional notification (admin-cancel → originating member, member-cancel → admin queue badge). Cancellation requests against rows already in `sending`, `sent`, `cancelled`, `rejected`, or `failed_to_dispatch` MUST be rejected with HTTP 409 and a bilingual `broadcast_cancel_too_late` error code, with an audit event of the same name carrying actor id + current state. The point-of-no-return is the `approved → sending` transition — the moment the Resend Broadcasts API call is issued and acknowledged. Once `sending`, recipient inboxes are committed; only the natural delivery lifecycle (FR-028 webhook completion) can advance the row.
- **FR-005**: Every broadcast row MUST carry **two distinct actor fields** snapshotted at submit time (Clarifications Q12, B): `requested_by_member_id` (the member whose quota is reserved/consumed and whose `primary_contact_email` populates reply-to) AND `submitted_by_user_id` (the user who clicked Submit — equal to the member's own portal user in the self-service path; equal to the acting admin's user id in the admin-proxy path). Both NOT NULL at submit time. The row MUST also carry tenant id + plan id at submit time (snapshot — not a foreign key reference) so future changes to plan or member state do not retroactively alter historical broadcast records.

  **`actor_role` enum** (`broadcast_actor_role`) carries 3 values reflecting the lifecycle path: (a) **`'member_self_service'`** — member submitted their own broadcast (default Q12 path); (b) **`'admin_proxy'`** — admin submitted on behalf of a member via "Submit on behalf of `<member>`" surface (Clarifications Q12); (c) **`'system'`** — system-initiated state mutation (currently only auto-cancel cascade from F3 archival/erasure per FR-001a + T178a; added post-/speckit.analyze N1 remediation 2026-04-29). When `actor_role = 'system'`, `submitted_by_user_id` may be a stale snapshot from the original submit, but `cancelled_by_user_id` MUST be `NULL` (no human cancelled — the cascade is system-driven); the audit event payload carries `actor_role = 'system'` to distinguish from human admin cancels.

#### Quota accounting

- **FR-006**: The quota year MUST be the calendar year (1 Jan 00:00 to 31 Dec 23:59:59) in the tenant's configured timezone (`Asia/Bangkok` for SweCham). The quota counter MUST be `quota_consumed = COUNT(broadcasts WHERE status='sent' AND quota_year_consumed = current_year AND requested_by_member_id = M)` and MUST NOT carry over unused quota into the next year.
- **FR-007**: The quota year that a `sent` broadcast is charged to MUST be the **calendar year of the `sending → sent` transition timestamp** (not the submission timestamp). This means a year-boundary submission that gets approved/sent in the new year consumes the new year's quota. UI MUST warn members near the boundary.
- **FR-008**: The quota counter MUST never go negative. Any attempt to send a broadcast that would push consumption above `eblast_per_year` MUST be refused at the submit boundary AND at the dispatch boundary (defence in depth — protects against state corruption from concurrent submissions).
- **FR-009**: Members on a plan where `eblast_per_year = 0` MUST NOT see the compose surface; the page MUST surface an upgrade explainer instead. Direct API submissions return 403.

#### Admin approval

- **FR-010**: The admin review queue MUST surface every `submitted` broadcast across the tenant, sorted oldest-first by submission timestamp by default, filterable by member + segment + date range, paginated.
- **FR-011**: Admin approval action MUST require either (a) `Approve & send now` → transitions to `approved` and immediately to `sending`, OR (b) `Approve & schedule` with a future timestamp → transitions to `approved` with `scheduled_for` set and remains `approved` until the cron handler picks it up.
- **FR-012**: Admin rejection MUST require a free-text reason of at least 1 non-whitespace character; the reason MUST be sent verbatim to the member in the rejection email; the audit row stores `sha256(reason)` only.
- **FR-013**: System MUST surface the **48-hour target review SLA** on the member submission confirmation screen and on the admin queue page header (Clarifications Q2, A). The SLA is informational only — F7 MVP ships **NO** automated escalation, auto-cancel, auto-notify, or manager re-routing for stale `submitted` broadcasts. Members retain their reserved quota slot until an admin decision is taken, regardless of how long the queue takes. SC-002 (median ≤24h, p95 ≤48h) is the operational measurement; missing it in two consecutive monthly windows triggers a post-MVP design pass on automated escalation rather than a hot-patch under load.
- **FR-014**: `manager` role MUST be able to read the review queue + every broadcast detail (read-only) but MUST NOT see Approve / Reject / Schedule controls; direct API attempts return 403.

#### Recipient targeting

- **FR-015**: System MUST support the following segment types in MVP: `all_members` (every active member of the tenant — resolves per FR-015c to each member's primary contact email), `tier:<tier_code>` (e.g., `tier:premium`, `tier:large` — multi-select allowed; same FR-015c primary-contact resolution), `event_attendees_last_90d` (attendees imported via F6 — already person-level emails, see FR-015a for stub-pattern dependency contract), and `custom` (a member-supplied list of email addresses, capped at 100 entries per submission, validated per FR-015d, must be admin-reviewed).
- **FR-015a**: The `event_attendees_last_90d` segment MUST resolve via an Application-layer port `EventAttendeesRepository.findRecentAttendeeEmails(tenantId, sinceDays = 90): Promise<string[]>` (Clarifications Q5, D). F7 ships a stub implementation returning `[]` so the segment-resolver code path is exercised end-to-end during F7-only development; F6 `/speckit.implement` later provides the real Drizzle-backed implementation reading from F6's `event_attendees` table. Both features release together (batch ship per project workflow), so production never sees the stub. The port contract MUST be defined in F7's Domain layer (per Principle III) and MUST NOT leak F6's schema into F7. During F7-only development, members selecting this segment will encounter the empty-segment refusal of FR-002c — this is documented behaviour, not a defect, and resolves automatically when F6 ships.
- **FR-015c**: For every member-based segment (`all_members`, `tier:<tier_code>`), the resolver MUST produce the recipient list as `SELECT DISTINCT primary_contact_email FROM members WHERE tenant_id = $1 AND <segment_predicate> AND primary_contact_email IS NOT NULL AND member_id != $2 /* requested_by_member_id — Clarifications Q16, B self-exclusion */` (Clarifications Q8, B + Q16, B). One email per member (deduplicated globally — the same person serving as primary contact for two member companies receives ONE broadcast). Members WITHOUT a `primary_contact_email` MUST be excluded from the resolved list AND MUST emit a `member_missing_primary_contact` audit signal at resolution time (one signal per excluded member per broadcast, batched as a single audit row carrying a count + an array of excluded member ids). The originating member (per `requested_by_member_id`, NOT `submitted_by_user_id` — admin-proxy submissions still auto-exclude the proxied member, not the admin) is excluded from member-based segments per Clarifications Q16; the `estimated_recipient_count` reflects this. The compose-page UI surfaces a bilingual microcopy hint ("You won't receive your own broadcast — your primary contact email is automatically excluded from member-based segments."). The `custom` segment is unaffected (explicit member choice overrides auto-exclude — if the member adds their own email to the custom list, they will receive it). The `event_attendees_last_90d` segment is unaffected (event attendance is independent of broadcast originator). Secondary contacts attached to F3's `contacts` table are NOT recipients in MVP — that surface is deferred to F7.1.
- **FR-015d**: For the `custom` segment, every entry in the member-supplied list MUST resolve at submit time to a known email in the tenant graph (Clarifications Q9, B). Resolution checks (case-insensitive, trim-normalised) MUST query in the following order, with a match in any source counting as resolved: (a) `SELECT 1 FROM members WHERE tenant_id = $1 AND LOWER(TRIM(primary_contact_email)) = $2`, (b) `SELECT 1 FROM contacts WHERE tenant_id = $1 AND LOWER(TRIM(email)) = $2`, (c) `SELECT 1 FROM event_attendees WHERE tenant_id = $1 AND LOWER(TRIM(email)) = $2` (F6 source — returns empty during F7-only development per FR-015a stub). Entries that match NONE of the three sources MUST cause submit-boundary rejection with `broadcast_custom_recipient_unknown` error code; the response MUST list each unresolved address verbatim so the member's editor can highlight them for removal. The 100-entry cap is checked BEFORE the per-entry resolution to keep the validation cost bounded. The `custom` list MUST be persisted (post-validation) on the broadcast row as a flat array; the suppression filter (FR-017) still applies at dispatch time. Custom-list entries are admin-visible in the review queue with a per-entry resolved-source badge so admins can verify the relationship at a glance.
- **FR-016**: Segment resolution to a recipient list MUST happen at submit time (for the `estimated_recipient_count` display + the empty-segment refusal of FR-002c) AND at dispatch time (the actual list passed to Resend). The two MAY differ if members were added/removed between submit and dispatch — this is acceptable; the dispatch-time list is the source of truth for delivery + audit.
- **FR-016a**: The resolved + suppression-filtered recipient list MUST be ≤ **5,000 entries** for any single broadcast (Clarifications Q7, B). The cap MUST be enforced at BOTH the submit boundary (FR-002h — early reject so the member sees the error before reservation) AND the dispatch boundary (defence in depth — protects against late-membership-growth between submit and send pushing the count over). Breaches MUST be rejected with a bilingual `broadcast_audience_too_large` error code, audited as `broadcast_audience_too_large` with the breached count + segment id (NOT the recipient list), and surfaced to the member as "Audience exceeds the 5,000-recipient maximum — please choose a narrower segment." Pagination across multiple Resend audiences for >5k recipients is explicitly OUT of MVP scope (deferred to F7.1).
- **FR-017**: Resolution MUST exclude suppressed emails (the tenant's `marketing_unsubscribes` list) at both the submit-time count AND the dispatch-time list — there is no scenario where a known-suppressed email is sent a broadcast.
- **FR-018**: Segment definitions MUST be tenant-scoped — a SweCham `tier:premium` segment is not visible to JCC and vice versa (Principle I tenant isolation).

#### Delivery (Resend Broadcasts)

- **FR-019**: System MUST call the Resend Broadcasts API to dispatch every approved broadcast. F7 MUST NOT bypass Resend (no direct SMTP), MUST NOT mix transactional (Resend transactional API) with broadcast (Resend Broadcasts API) — they are separate API products with separate suppression lists and separate sending IPs.
- **FR-020**: Each Resend dispatch MUST attach a **stable** `Idempotency-Key` of the form `broadcast-{tenantId}-{broadcastId}` (no attempt counter — Critique 2026-04-29 finding E2/X2). Cross-attempt retries — whether SDK auto-retries within a single tick OR cron-tick re-dispatches after a failed transaction (where the Resend call succeeded but the row update aborted) — MUST reuse the same key so Resend's idempotency primitive returns the SAME existing broadcast resource. This guarantees zero duplicate Resend broadcasts in the account and zero duplicate recipient deliveries even under transient DB or network failure during the dispatch transaction.
- **FR-021**: Dispatch failures (Resend 5xx, network outage, timeout) MUST be retried with exponential backoff for up to 1 hour from `scheduled_for` (or from `approved` time for "send now"); after 1 hour of sustained failure, transition to `failed_to_dispatch` with admin alert + member transactional notification.

  **Implementation note (Phase 8 Slice D, 2026-05-02; corrected verify-fix R3, 2026-05-02)**: The 1-hour retry budget is enforced at the cron-tick boundary, not via in-process exponential backoff. The dispatch use-case (`dispatch-scheduled-broadcast.ts`) computes `pastBudget = (now - scheduled_for) > 1h` **lazily inside the `gateway_retryable` catch branch** (i.e., only after a Resend call returns a retryable error — NOT eagerly at use-case entry). When `pastBudget` is true, the row transitions to `failed_to_dispatch` + emits `broadcast_failed_to_dispatch` audit + emits `broadcasts.dispatch_budget_exhausted` metric (E2 alert pipeline) + enqueues the FR-021 / AS2 transactional notification email (Slice E). Within budget the original behaviour holds: the row stays `approved` for the next 5-min cron tick. This cron-tick re-attempt model replaces the spec-implied in-process exponential backoff (1/2/4/8/16s = 31s) because (a) cron cadence already provides natural retry, (b) a 31s in-process wait inside a serverless function adds wall-clock cost without recovering retryable failures faster than the next 5-min tick, and (c) all observability (logs, audit, metrics) is per-tick, not per-attempt — keeps the dashboards interpretable.

  **Implementation note (Phase 8 Slice E, 2026-05-02)**: Member transactional notification on `failed_to_dispatch` uses `notification_type='broadcast_failed_to_dispatch_notification'` (migration 0080 — idempotent ADD VALUE mirroring 0079). The dispatch use-case enqueues into `notifications_outbox` via `EmailTransactionalPort.sendMemberEmail` (best-effort: lookup failures + enqueue throws are logged but do NOT block the terminal-fail transition + audit). The F4 cron outbox dispatcher renders the bilingual member-friendly email via `buildBroadcastFailedToDispatchEmail` (chamber branding, scheduled-for + reason fields, "your quota was preserved" reassurance, admin-rescheduling deep-link). Email copy lives in `src/i18n/messages/{en,th,sv}.json` under `email.broadcastFailedToDispatch.*` (9 keys × 3 locales = 27 entries).
- **FR-022**: Dispatch failures (Resend 4xx — invalid audience, invalid template, signature error) MUST NOT be retried; transition immediately to `failed_to_dispatch` with the Resend error code persisted (NOT the audience or content) and admin alert.
- **FR-023**: The Resend account API key MUST be a per-tenant configuration (so future SaaS tenants can BYOK without code change), but in single-tenant SweCham deployment the env var `RESEND_BROADCASTS_API_KEY` is the source. (This MAY be the same key as F1 transactional — Resend supports both API products on one key — but it MUST NOT be hardcoded; rotation is an env-var change only.)

#### Webhook ingest

- **FR-024**: The Resend webhook endpoint (`/api/webhooks/resend-broadcasts`) MUST verify the request signature against the configured `RESEND_BROADCASTS_WEBHOOK_SECRET` before reading any payload field; signature failures return 401 + audit + no state change.
- **FR-025**: The webhook handler MUST be idempotent on Resend's unique `event_id` — replays of the same event are no-ops at the database level (`INSERT ... ON CONFLICT DO NOTHING` on `(tenant_id, resend_event_id)`).
- **FR-026**: Each `email.delivered` / `email.bounced` / `email.complained` event MUST persist a `broadcast_deliveries` row keyed by `(tenant_id, delivery_id)` with `recipient_email_lower`, `recipient_member_id` (resolved by lookup, nullable for non-member recipients), `status`, `event_timestamp`, `resend_message_id`, and (for bounces/complaints) `error_message`.
- **FR-027**: Hard-bounce events MUST auto-add the recipient email to the tenant's `marketing_unsubscribes` with `reason = 'hard_bounce'`. Complaint events MUST do the same with `reason = 'complaint'` AND raise an admin dashboard alert. **Implementation note (verify-gate finding E1, 2026-05-01)**: the per-broadcast >5% complaint-rate auto-halt invariant from SC-005 (b) carries a **20-event small-N noise floor** at the application layer — i.e. the halt fires when complaint count ≥1 AND terminal-event count ≥20 AND `complained / terminalCount > 0.05`. Below the floor, the suppression upsert + `broadcast_complaint_received` audit + `broadcast_suppression_applied` audit still fire (the spec invariants are preserved); only the member-halt cascade is deferred to subsequent events. Rationale: 1-out-of-1 complaint = 100% rate but is not statistically distinguishable from noise; halting on it would create false-positive halts for 3-recipient broadcasts where the first event is a spam-marker. This is an implementation refinement, not a relaxation — the halt still fires within seconds of the threshold being crossed on real-world (≥20 recipients) broadcasts.
- **FR-028**: When all expected delivery events for a broadcast have arrived (counted against `estimated_recipient_count`) OR the 24-hour timeout fires, the broadcast MUST transition `sending → sent`, `quota_consumed_at` is stamped, the originating member receives a transactional summary email, and `broadcast_sent` + `broadcast_quota_consumed` audit events are written. Subsequent late events still create delivery rows but do not double-consume quota.

#### Unsubscribe (public)

- **FR-029**: Every broadcast email MUST include a one-click unsubscribe affordance that, when clicked, results in a tenant-scoped suppression record protecting the recipient on subsequent dispatches. The signed-token URL pattern is `https://<tenant-host>/unsubscribe/{token}` where `{token}` is an HMAC-SHA256-signed payload of `(tenant_id, broadcast_id, recipient_email_lower, [optional locale_hint])` using a **dedicated `UNSUBSCRIBE_TOKEN_SECRET`** (resolved 2026-04-29 per research.md § 4 — Underspecification U1 closed at /speckit.analyze). Rationale for dedicated secret (NOT reusing `AUTH_COOKIE_SIGNING_SECRET`): (a) rotating session cookies must NOT invalidate every outstanding unsubscribe link in members' inboxes — months-old unsubscribes must still work, (b) compromise of one secret does NOT expose the other, (c) different rotation lifetimes (session secrets quarterly; unsubscribe tokens valid forever per FR-030 idempotency).
  
  **Implementation note (verify-fix C1, 2026-05-01)**: Resend Broadcasts API ships ONE shared HTML body for the entire audience and does not accept per-contact arbitrary merge fields or custom HTTP headers (`broadcasts.create` parameters: `audienceId`/`from`/`subject`/`html`/`name`/`replyTo`/`previewText` only). Per-recipient signed URL embedding via `emails.send` would mix marketing volume into the F1 transactional surface, violating FR-019 (separate Resend product / suppression list / reputation pool). Two convergent unsubscribe surfaces deliver this requirement together:
  
  1. **Body-CTA (per-broadcast)**: Resend's built-in `{{{RESEND_UNSUBSCRIBE_URL}}}` merge tag is rendered inside the chamber-branded footer (T147). At send time Resend substitutes per-recipient with their hosted unsubscribe page; on click Resend marks the audience contact `unsubscribed=true` so subsequent broadcasts to the same audience auto-filter that recipient at Resend's edge — preserving the FR-017/SC-004 invariant ("zero marketing emails delivered to a suppressed recipient"). The Resend audience filter applies BEFORE our `marketing_unsubscribes` lookup, so no leak path exists.
  
  2. **Direct-link `/unsubscribe/{token}` route (signed)**: implemented at `src/app/unsubscribe/[token]/page.tsx`. Used by (a) the RFC 8058 `List-Unsubscribe` header generator (`buildListUnsubscribeHeaders`) — Mail clients (Gmail / Apple Mail / Outlook) honour this header to surface a one-click "Unsubscribe" button OUTSIDE the body chrome; (b) admin "share unsubscribe link" affordance for direct GDPR Art. 21 requests; (c) future per-recipient send paths once Resend Broadcasts API gains per-contact merge field support. This route writes directly into our `marketing_unsubscribes` table (SOT-of-record).
  
  **Convergence**: both surfaces prevent re-delivery — Resend's audience filter (path 1) AND our `marketing_unsubscribes` lookup at dispatch (`resolve-segment-recipients.ts` calls `marketingUnsubscribes.lookupBatch`, path 2). Our DB SOT may drift from Resend's audience-suppression state (Resend does NOT fire a webhook event for hosted-page unsubscribes), but SC-004 holds in both directions — Resend filters its-DB suppressed contacts; we filter our-DB suppressed contacts. A periodic sync from Resend's `unsubscribed=true` audience contacts → our `marketing_unsubscribes` is tracked as a future enhancement (post-MVP F7 enhancement, not F7.1) once Resend exposes audience-contact pagination at scale.
  
  **Audit trail completeness**: clicks via path 2 audit-emit `broadcast_unsubscribed` + `broadcast_suppression_applied` (T142). Clicks via path 1 do NOT emit a tenant-scoped audit (Resend hosted page → no webhook), but are observable via Resend's dashboard + the next-broadcast `broadcast_suppression_applied` event when `lookupBatch` filters Resend's mirror.
- **FR-030**: The `/unsubscribe/{token}` route MUST be public (no authentication required), MUST be served fully server-rendered (no client-side JS required to complete the unsubscribe — recipients on basic mail clients with image proxies can still unsubscribe), MUST be idempotent (double-click is fine), and MUST work in EN + TH + SV.
- **FR-031**: A successful unsubscribe MUST insert/upsert `(tenant_id, email_lower)` into `marketing_unsubscribes` with `unsubscribed_at = now()`, `member_id` (nullable, resolved from email if it matches an active member), `source_token = sha256(token)`, optional `reason` if the recipient typed feedback. The audit log records `broadcast_unsubscribed` with broadcast id + email hash (NOT raw email).
- **FR-032**: Token verification failure (bad signature, expired token if expiry is enforced, malformed) MUST render a bilingual fallback page with a "contact `<tenant_support_email>`" instruction — NOT silently fail and NOT silently succeed; failures MUST be audited as `broadcast_unsubscribe_token_invalid`.

- **FR-036**: Every broadcast / delivery / suppression / segment / quota query MUST be scoped by `tenant_id` injected via middleware from the resolved tenant context, AND the underlying tables MUST have Postgres RLS policies enforcing `tenant_id = current_setting('app.current_tenant')`. Both layers are mandatory; either layer alone is non-compliant.
- **FR-037**: A `swecham` member MUST NOT be able to read, update, cancel, approve, or unsubscribe a `jcc` broadcast (or any other tenant's), and vice versa. Cross-tenant probe attempts MUST return 404 (NOT 403 to avoid leaking existence) and MUST be audited as `broadcast_cross_tenant_probe`.

#### Forbidden in logs

- **FR-042**: pino logs MUST NEVER contain raw recipient email addresses, raw email body content, raw subject lines (the body of the broadcast itself), Resend API keys or webhook secrets, unsubscribe token plaintext, or session-bearing cookies. All of these have correlation alternatives (member id, broadcast id, sha256 hash) for debugging.

### Key Entities

- **Broadcast** — A single member-authored E-Blast request with its full lifecycle (draft → … → sent or terminal). Carries the snapshot of the originating member + plan + segment at submit time. Tenant-scoped; Principle I.
- **Broadcast Delivery** — One row per Resend per-recipient event. The granular delivery record used for "did N reach their inbox" analytics. Tenant-scoped; recipient email is stored lower-cased.
- **Marketing Unsubscribe** — Tenant-scoped suppression entry. A member-of-tenant-A unsubscribed from broadcasts MUST still be deliverable in tenant B if they are also a member there.
- **Recipient Segment Definition** — A named, parameterised filter (e.g., `tier:premium` with `tier_codes = ['premium','large']`). MVP ships fixed presets; future tenants may add custom-named segments via admin UI (post-MVP).
- **Quota Counter** — A computed view (NOT a stored aggregate) over `members.plan_id × membership_plans.eblast_per_year × broadcasts WHERE status='sent' AND quota_year_consumed = Y`. Smart Feature #1 Benefit Dashboard reads this view.
- **Approval Decision** — The persisted record of an admin's approve/reject/schedule action on a submission, including actor + timestamp + (for rejections) reason. Append-only — once written, never edited.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: **Paid quota slots delivered to inboxes within the same calendar quarter as submission** ramps over the first 3 years (Clarifications Q13, C — industry benchmark for marketing-quota utilization is 50-70%; Round 1 critique P12/X1 + the SaaS-foundation framing): **≥50% year 1, ≥65% year 2, ≥75% year 3.** Measured at quarter-end across all tenants in production (e.g., Q3 2026 quota slots have a Q3-ending delivery rate ≥50%). Below the year's bar indicates either the admin queue is bottlenecking OR member adoption is below expectations — both warrant investigation but are NOT a sign of F7 failing technically.
- **SC-002**: **Admin queue median time-to-decision (submit → approve OR reject) is ≤ 24 hours**, p95 ≤ 48 hours, measured over rolling 30-day windows. (Service expectation surfaced to members on the confirmation screen.)
- **SC-003**: **Quota dispute rate is ≤ 1 dispute per tenant per year** (a dispute is a member helpdesk contact alleging "I paid for X E-Blasts and got Y"). Measured by support-ticket tag.
- **SC-004**: **Zero marketing emails delivered to a recipient email present in `marketing_unsubscribes` at the moment of dispatch**, verified by an automated e2e test running before every release that creates a suppression then attempts to send to that address (the assertion is that the address never reaches Resend's audience).
- **SC-005** (two-layer per Clarifications Q14, B):
  - **(a) Rolling 30d invariant**: hard-bounce + complaint rate combined is ≤ **2%** across all broadcasts in a tenant in a rolling 30-day window. Above this rate, deliverability is at risk and the next release MUST gate on remediation.
  - **(b) Per-broadcast spike invariant**: any single broadcast whose complaint rate exceeds **5%** at any point during its delivery MUST trigger an immediate audit event (`broadcast_complaint_rate_per_broadcast_breach`), page on-call admin per the deliverability runbook, AND auto-halt further dispatches from the offending member until an admin manually clears the hold (`members.broadcasts_halted_until_admin_review` flag flipped to `false` via admin action — emits `broadcast_member_dispatch_resumed`). Verified by a dedicated integration test that simulates 5% complaint rate on a 100-recipient broadcast and asserts the halt + audit + alert all fire.
- **SC-006**: **i18n coverage is 100%** in EN + TH + SV for every member-facing, admin-facing, and recipient-facing string at release; verified by `pnpm check:i18n` in CI.
- **SC-007**: **WCAG 2.1 AA conformance is ≥ 99%** on the compose surface, admin queue, broadcast detail, and public unsubscribe page (axe-core automated + 1 manual SR pass per release).
- **SC-008**: **At least 95% of submissions whose admin decision is "Approve & send now" reach the recipient inbox within 30 minutes** of approval, measured by the gap between `broadcast_approved` audit timestamp and the median `email.delivered` event timestamp.
- **SC-009**: **Cross-tenant isolation tests pass at every release** (Principle I Review-Gate blocker — 0 cross-tenant reads or writes across all F7 aggregates).
- **SC-010**: **Per-surface p95 latency budgets are met in production** at every release (Clarifications Q6, B; verified via CI synthetic load + Vercel Speed Insights RUM): compose page TTFB p95 < 600ms · submit endpoint p95 < 1.2s · admin queue list p95 < 500ms (@ 1k pending broadcasts) · admin approve & send-now endpoint p95 < 1.5s · webhook handler p95 < 250ms · public unsubscribe page TTFB p95 < 400ms. Any budget breach in two consecutive monthly RUM windows blocks the next release until remediation lands.
- **SC-011** (Clarifications Q13, C + Q18, B per-release invariant — Critique Round 3 R3-NEW-6 reword): **F7's multi-tenant readiness MUST be demonstrated at every release** via the JCC-test tenant fixture. The fixture is a CI-nightly job that:
  - (a) **Cross-tenant integration test passes** (Constitution v1.4.0 Principle I clause 3 — already a Review-Gate blocker; SC-009 covers this independently),
  - (b) **Provisioning dry-run scripted in CI**: creates a fresh test-tenant ("JCC-test"), seeds default segments via migration 0068, configures a Resend test-mode account stub, submits + approves + dispatches a synthetic broadcast (single test recipient), verifies cross-tenant suppression isolation, verifies tenant-scoped audit log + metrics, tears down — total runtime budget < 5 minutes,
  - (c) **Data-only configuration**: no F7 code change required to onboard the test-tenant — verified by ensuring zero diff in `src/modules/broadcasts/**` between baseline and post-fixture-run.
  
  Failure of any sub-criterion = F7 ship blocker. The original strategic framing ("ready to onboard 2nd tenant within 6 months of F11 SaaS-billing ship" per Q13) is **retained as context** but the operational measurement is now per-release continuous (Q18). This prevents the F7-ship-to-F11-ship interim drift that R3-NEW-6 surfaced. F11 (when it ships, whenever that is) inherits this readiness automatically.
