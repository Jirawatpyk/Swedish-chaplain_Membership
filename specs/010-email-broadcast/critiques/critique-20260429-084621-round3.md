## Recommended remediation

Five surgical edits (one is the must-address):

### Edit 1 (🎯 Must-Address): Spec FR-002 — add precondition `k` + error code

**Current** (spec.md FR-002, end of preconditions):

> ... (j) the originating member's `primary_contact_email` is non-null (Clarifications Q11, A — required so the broadcast has a valid reply-to address; rejection deep-links to the F3 self-service profile edit screen). Each precondition is enforced both client-side (UX) and server-side (security boundary); server-side rejection emits the corresponding bilingual error code (`broadcast_quota_blocked`, ..., `broadcast_member_missing_primary_contact_email`).

**Proposed**:

> ... (j) the originating member's `primary_contact_email` is non-null (Clarifications Q11, A — required so the broadcast has a valid reply-to address; rejection deep-links to the F3 self-service profile edit screen), and **(k) the originating member's `broadcasts_halted_until_admin_review` flag is `false` (Clarifications Q14, B — set automatically by F7 webhook handler when a single broadcast's complaint rate exceeds 5%; cleared manually by admin via the F7 admin queue clear-halt action per R3-NEW-3)**. Each precondition is enforced both client-side (UX) and server-side (security boundary); server-side rejection emits the corresponding bilingual error code (`broadcast_quota_blocked`, ..., `broadcast_member_missing_primary_contact_email`, **`broadcast_member_halted_pending_review`**).

Also update data-model.md § 6 validation map: add row "Originating member's halt flag is false | FR-002 precondition `k` | Application | `MembersBridgePort.getMember(ctx, requested_by_member_id).broadcasts_halted_until_admin_review === false`".

### Edit 2 (💡): Spec Q15 banner trigger refinement

Append to Q15 narrative (or as new sub-bullet):

> **Banner trigger conditions (R3-NEW-2 refinement)**: banner appears IF AND ONLY IF (i) signed-in user has `member` role, (ii) member's tenant has F7 enabled (`FEATURE_F7_BROADCASTS=true`), (iii) member's `broadcasts_acknowledged_at IS NULL`, (iv) member's plan has `eblast_per_year > 0` OR member is in any chamber-managed segment that may receive broadcasts (i.e., `members.is_active = true`). Banner is **server-driven** — `broadcasts_acknowledged_at` is the sole source of truth (no device-local cookie). Banner shows on **every sign-in** until acknowledged (not just first). Member can dismiss with "Remind me later" link → records no audit, banner shows again next sign-in. **Per-tenant scope** (R3-NEW-7): a user who is a member of multiple tenants sees a separate banner per tenant context — each tenant's `broadcasts_acknowledged_at` is independent, aligning with FR-018 tenant isolation.

### Edit 3 (💡): Spec Q14 cross-feature flag-clearing UI

Append to Q14 narrative:

> **Clear-halt UI (R3-NEW-3 — single source of truth)**: the admin clear-halt action lives on the **F7 admin queue page** (`/admin/broadcasts`) as a top-of-page banner whenever ≥1 member in the tenant is in halt state. Each halted member appears as a row with a "Review + Clear halt" button that opens a confirmation dialog (typed-phrase pattern matching F4 destructive-action convention). Clearing emits `broadcast_member_dispatch_resumed` audit. The F3 admin members list (`/admin/members`) shows a small "Broadcasts halted" badge per affected member for affordance discoverability but does NOT host the clear action — clicking the badge navigates to the F7 admin queue's clear-halt surface. Manager-role users see the badge + queue but no clear-action button (read-only per FR-014).

### Edit 4 (💡): Plan.md § VII — alert list extension

Update plan.md § VII alerts list:

- Add alert **#11**: `broadcast_complaint_rate_per_broadcast_breach` audit event fires (high severity — page on-call admin via deliverability runbook; member is auto-halted per Q14)
- Reword alert **#7**: "complaint rate > 0.5% over 1h (alarm — early-warning before either the per-broadcast spike #11 or the rolling-30d alarm #8 fires)"

### Edit 5 (💡): Spec US3 — add banner acceptance scenarios

Add to US3 acceptance scenarios (currently 1-5):

> 6. **Given** a paying-tier member who has not yet acknowledged the marketing-broadcasts terms (`broadcasts_acknowledged_at IS NULL`), **When** they sign in to the portal, **Then** a bilingual banner appears with the FR-039 EN/TH/SV notice ("Your tier includes marketing broadcasts from chamber members. You may unsubscribe at any time.") + an "Acknowledge" CTA + a "Remind me later" link.
> 7. **Given** the same member clicks "Acknowledge", **Then** `broadcasts_acknowledged_at` is set to `now()` (Asia/Bangkok timezone for SweCham), the audit log records `member_acknowledged_broadcasts_terms` with `member_id + user_id + acknowledged_at + banner_locale`, the banner is dismissed permanently for that member-tenant pair, and the member proceeds to their intended portal destination (no redirect interruption).
> 8. **Given** the same member clicks "Remind me later" instead, **Then** no audit event is recorded, no column update occurs, and the banner appears again on next sign-in.
> 9. **Given** a member who is in multiple tenants (cross-tenant user identity per F1), **When** they sign in to tenant B for the first time, **Then** they see tenant B's banner regardless of having acknowledged tenant A's banner — each tenant's acknowledgement is independent (FR-018 + Q15 per-tenant scope).
