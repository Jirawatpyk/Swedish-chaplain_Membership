## Executive Summary

Round 1 remediations (3 edits applied via "all" approval) **landed correctly** — FR-020 stable idempotency-key fix is consistent across spec, plan, research. FR-002a `<img>` removal is consistent in spec.md, research.md DOMPurify config, plan.md, and CLAUDE.md.

However, Round 2 surfaced **one fix-induced regression** that Round 1 missed: **research.md § 2 (Tiptap editor config) still enables the `Image` extension** with `inline: false, allowBase64: false`. This directly contradicts the new FR-002a `<img>` forbidden rule. When a member uses Tiptap's "Insert image" button, the editor will emit `<img>` → submit → sanitiser strips → submission rejected with `broadcast_body_unsafe_html`. The member sees an unexplained rejection of a UI affordance that the editor's own toolbar exposes. This is a **product gap** + **spec inconsistency** rising to 🎯 Must-Address.

Two new 💡 Recommendations also surfaced:
1. **Sanitiser-strip-warn UX** — currently the editor silently allows forbidden constructs and the sanitiser strips them at submit boundary. Members are surprised at submission. Editor should highlight forbidden constructs in real-time with a "will be removed at submit" indicator.
2. **Stale-`sending` reconciliation** — orphan-webhook-event handling exists, but no symmetric handling for "broadcast in `sending` state forever because Resend events never arrive (e.g., Resend dashboard manual-delete of the broadcast)". Add a periodic reconciliation query.

The 8 🤔 Strategic Questions from Round 1 (P2, P7, P8, P9, P14, E11, E15, E23) remain **all open** — none silently resolved by Round 1 edits, and Round 2 didn't introduce new ones.

The plan is one fix away from tasks-ready. The fix is small (3 lines of research.md § 2 + matching note in plan.md).

## Recommended remediation

Three surgical edits:

### Edit 1 (🎯 Must-Address): research.md § 2 — disable Tiptap Image extension

**Current** (research.md line 45):

> **Extensions enabled**: `Document`, `Paragraph`, `Text`, `Bold`, `Italic`, `Underline`, `Strike` (NOT in FR-002a allowlist — disabled), `Heading` (levels 1-4 only — `levels: [1, 2, 3, 4]`), `BulletList`, `OrderedList`, `ListItem`, `Blockquote`, `HardBreak`, `HorizontalRule`, `Link` (with `protocols: ['http', 'https', 'mailto']`, `linkOnPaste: true`, `openOnClick: false`), `Image` (with `inline: false`, `allowBase64: false`).

**Proposed**:

> **Extensions enabled**: `Document`, `Paragraph`, `Text`, `Bold`, `Italic`, `Underline`, `Heading` (levels 1-4 only — `levels: [1, 2, 3, 4]`), `BulletList`, `OrderedList`, `ListItem`, `Blockquote`, `HardBreak`, `HorizontalRule`, `Link` (with `protocols: ['http', 'https', 'mailto']`, `linkOnPaste: true`, `openOnClick: false`).
>
> **Extensions explicitly disabled** (would emit tags outside the FR-002a allowlist): `CodeBlock`, `Code` (inline), `Strike`, **`Image`** (Critique 2026-04-29 R2-NEW-1 — disabled to match the FR-002a `<img>` forbidden rule applied in Round 1; the "Insert image" toolbar button is omitted; members link to externally hosted images via the `Link` extension instead). `History` (undo/redo) is enabled — does not affect output HTML.

Also update the toolbar enumeration in § 2:

**Current**:

> **Toolbar**: minimal — Bold, Italic, Underline, Heading dropdown (H1–H4), Bullet list, Ordered list, Blockquote, Link, Hr. No image-upload button in MVP (attachments deferred F7.1); paste-image disabled.

**Proposed**:

> **Toolbar**: minimal — Bold, Italic, Underline, Heading dropdown (H1–H4), Bullet list, Ordered list, Blockquote, Link, Hr. **No image-upload button + no paste-image** in MVP — `<img>` is forbidden by FR-002a (R2-NEW-1). Image embedding (with mandatory source allowlist) returns in F7.1.

### Edit 2 (💡 Recommendation): spec.md FR-002a — sanitiser-strip-warn UX

Add a new sub-bullet to FR-002a:

> Editor SHOULD provide real-time feedback when forbidden constructs are present: paste handlers MUST run the same sanitiser eagerly and surface a tooltip / inline highlight ("This will be removed at submit. Use [allowlisted alternative].") on each forbidden tag or attribute. The editor's preview pane MUST reflect the post-sanitisation state, not the raw input. Submit-boundary rejection (FR-002 precondition `g`) is the security guarantee; editor-side warning is the UX path that prevents the surprise-at-submit pattern. Verified by `tests/e2e/broadcast-compose-and-submit.spec.ts` AS9: paste a body containing `<img>` → preview pane shows the image stripped + a warning toast appears + Submit button shows "Content will be modified" indicator.

### Edit 3 (💡 Recommendation): plan.md § Reliability — stuck-`sending` reconciliation

Add a sub-bullet under § 4d Failure Mode Analysis (Reliability):

> **Stuck-`sending` reconciliation** (R2-NEW-3 — defends against Resend operational errors): when the 24h timeout fires (FR-028) on a `sending` broadcast, BEFORE consuming quota, the timeout handler MUST query `BroadcastsGateway.retrieveBroadcast({ id: resend_broadcast_id })`. If Resend returns 404 (broadcast was manually deleted via the Resend dashboard or never properly dispatched), transition to `failed_to_dispatch` instead of `sent`, release quota reservation per FR-003, audit a new event `broadcast_resend_resource_missing` (33rd entry in catalogue), and alert admin. If Resend returns the broadcast, proceed with the normal `sent` transition (the events were just slow). This adds ~200ms to the 24h-timeout path (acceptable — 24h timeout is rare) and prevents the rare "quota consumed but recipients never received" failure mode.

This adds a 33rd audit event type. Update spec FR-033 + data-model.md § 5 + CLAUDE.md to "33 audit event types".
