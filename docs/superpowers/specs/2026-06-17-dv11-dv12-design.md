# Design: DV-11 + DV-12 — wire existing backends to UI

**Date:** 2026-06-17
**Source backlog:** `docs/Bug/spec-code-divergence-2026-06-16.md` (the 2 HIGH items)
**Decisions:** build the UI for both · DV-11 = Approach B (visible-gate) · 2 separate PRs
**Spec review:** revised after a 5-specialist review (architect / spec-compliance / qa / security / a11y-ux) — 0 blockers, 11 HIGH; all incorporated below (load-bearing wiring now PINNED, not deferred).

## Context

Two HIGH spec↔code divergences: a backend exists (route + use-case + audit + most i18n) but has **no UI control**.

- **DV-11 (F3):** `POST /api/members/[memberId]/contacts/[contactId]/resend-verification` + `resendVerificationEmail` use-case + audit `email_verification_resent` exist; no button.
- **DV-12 (F7):** member route `/api/broadcasts/[id]/cancel` + admin route `/api/admin/broadcasts/[id]/cancel` + `cancel-broadcast` use-case + `admin.broadcasts.cancelDialog.*` i18n exist; no cancel control (`review-actions.tsx` is Approve/Reject only).

## Goals

- Admin re-sends a portal-linked contact's email verification from the member-detail page, **shown only when that contact's linked user email is unverified** (Approach B), **rate-limited**.
- Admin cancels a `submitted` **or** `approved` broadcast (reason **required**) — reachable for **both** statuses.
- Member cancels their **own** `submitted`/`approved` broadcast (reason **optional**) from the portal broadcast detail.

## Non-goals / scope notes

- **One backend change only:** add a rate-limiter to the resend-verification route (security requirement, finding S-1). No other backend changes — all routes/use-cases/audit exist and are tested.
- No new cancellable statuses (reuse `submitted`|`approved`; `sending`-with-batches stays out of the button gate, handled by the route's 409).
- Not addressing the other DV items.

## DV-11 — F3 resend-verification (Approach B, visible-gate) — PINNED

**Backend (existing):** `POST …/resend-verification` (admin-only) → `200 {outbox_row_id, invalidated_prior}`; `409 {error:'not_eligible', reason:'no_linked_user'|'email_verified'}`; `404 {error:'not_found'}` (flat, use-case) **and** `404 {error:{code:'not_found',message}}` (nested, param-zod-fail); `500 {error:'server_error'}`. Use-case `resend-verification-email.ts`; audit `email_verification_resent`. Verification state is on the F1 user, read via `UserEmailPort.isEmailVerified(userId)` (single-user; `users` has no `tenant_id`).

**Wiring (PINNED — do NOT defer):**
1. **Do NOT modify `getMember`** — it returns domain `Contact[]`, has 9 call sites, and one uses the lean `buildMemberProbeDeps` (no `userEmails`). Changing its shape/deps is a large blast radius and pushes F1-user state into the domain entity (Principle III breach).
2. **New application loader** `src/modules/members/application/use-cases/_lib/resolve-contact-verification.ts` mirroring `_lib/resolve-contact-subscriptions.ts`: given the linked contacts + `UserEmailPort`, returns a `Set<contactId>` (or map) of **verification-pending** contacts. It owns the per-contact loop over the single-user `isEmailVerified` (N small reads — members have few contacts; **no batch port method** added → keeps backend untouched). **Best-effort/degraded:** if `isEmailVerified` returns `repo.not_found`/`repo.unexpected`, default that contact to **not-pending** (hide the button when state is unknown — safer than a possibly no-op resend).
3. **Page** (`…/[memberId]/page.tsx`) calls `getMember` (unchanged) then this loader (in the existing `Promise.all`, alongside `resolveContactSubscriptions`), threading the `UserEmailPort` that `buildMembersDeps` already provides. Presentation calls the **application loader**, never `isEmailVerified` directly (Principle III).
4. **Component:** new client `ResendVerificationButton` mirroring `resend-bounced-invite-button.tsx` (`variant="outline" size="sm"` — keep sm for ContactBlock cluster consistency; `useState` submitting; `fetch` POST; toast; `router.refresh()`). Icon `aria-hidden`; visible label carries the action.
5. **Render gate** (in `ContactBlock`): `canWrite && contact.linkedUserId && contact.emailVerificationPending`.
6. **Rate-limit (backend, finding S-1):** add a per-`(actorUserId, contactId)` Upstash limiter (≈3/hour, 10/day) at the route, **fail-soft** on Upstash quota (mirror sign-in/reset limiters), returning 429; the button maps 429 → a "too many requests" toast.
7. **Error handling:** 409 `email_verified`/`no_linked_user` → specific toast; 404 (flat use-case shape) → notFound toast; 429 → rate-limit toast; else → generic.
8. **i18n (en/th/sv):** `admin.members.detail.resendVerification.{resendLabel, resendSubmitting, resendSuccess, errors.notFound, errors.noLinkedUser, errors.emailVerified, errors.rateLimited, errors.serverError}`.
9. **Tests (TDD):**
   - component: success · 409 email_verified · 409 no_linked_user · 404 (flat) · 429 · 500.
   - loader **unit** test (mocked `UserEmailPort` — **not** live-Neon): linked+unverified→pending · linked+verified→not-pending · no linkedUserId→not-pending · `isEmailVerified` error→not-pending (safe default).
   - route rate-limiter: a test that the 4th call in an hour returns 429.

## DV-12 — F7 cancel-broadcast (admin + member) — PINNED

**Backend (existing):** admin `POST …/cancel` body `{cancellationReason}` **required** (1–500); member `POST …/cancel` body `{cancellationReason}` **optional** (≤500). Envelope on error: `{error:{code,message,messageThai,details?},correlationId}`. Codes: `broadcast_not_found`(404), `broadcast_cancel_too_late`(409), `broadcast_concurrent_action_blocked`(409); over-length reason → `invalid_body`(400) at the zod layer (the use-case's `broadcast_cancel_reason_too_long` is unreachable from these routes). Use-case `cancel-broadcast.ts`; cancellable `submitted`|`approved`; audit `broadcast_cancelled`. Member route rejects cross-member as `broadcast_not_found` (no existence leak) + no audit.

**Admin placement (PINNED — page-level, finding A-1/SC-1/QA-1/SEC-2/UX-1):**
- The admin detail page (`…/admin/broadcasts/[id]/page.tsx:243`) currently mounts `ReviewActions` only when `status === 'submitted'`. **Widen** that gate to `(status === 'submitted' || status === 'approved') && !isReadOnlyManager`.
- **Thread `broadcast.status`** into `ReviewActions`: render Approve/Reject only for `submitted`; render **Cancel for `submitted` and `approved`**. Cancel hidden for the read-only manager (the `managerReadonlyBanner` already promises this).
- New client `CancelDialog` (admin variant) mirroring `reject-dialog.tsx`: `AlertDialog` + `Textarea` reason **required** + char counter capped **500** (not reject's 2000); body field name `cancellationReason`; `useTransition` + fetch + toast + `router.refresh()`.

**Member placement (PINNED):**
- New client `BroadcastDetailActions` on `…/(member)/portal/broadcasts/[id]/page.tsx`, rendered only when `status ∈ {submitted, approved}` (page already loads the owned broadcast via `findOwnedByMember` → ownership enforced at read layer).
- `CancelDialog` (member variant): reason **optional** (confirm enabled with empty reason); cap 500.

**409 discrimination (PINNED — finding A-3/QA-3):** the dialog reads `body.error.code` (not just `res.status`): `broadcast_cancel_too_late` → "too late" toast; `broadcast_concurrent_action_blocked` → "concurrent" toast. Fetch mocks in tests return the `{error:{code}}` envelope with 409.

**Focus/a11y (PINNED — finding UX-2/UX-3):**
- Member `CancelDialog` (optional reason): **initial focus on Cancel** via `initialFocus={cancelRef}` (ux-standards §6.2). Admin (required reason) may keep textarea-focus like reject-dialog — stated explicitly.
- **finalFocus chain** on the success-path unmount (actions disappear when status leaves submitted/approved): `triggerRef ?? #main-content`, mirroring reject/approve dialogs.
- Add `tabIndex={-1}` to the member portal `#main-content` (`portal/layout.tsx:100`) so the fallback focus + skip-link target works.
- Cancel **trigger** = `variant="destructive-outline"` (matches Reject); dialog **confirm** = opaque destructive style (reject-dialog's). Colour never sole indicator — explicit "Cancel broadcast" label.

**i18n (en/th/sv — finding QA-2/UX-4/UX-5):**
- admin: reuse `admin.broadcasts.cancelDialog.{title,description,reasonLabel,reasonPlaceholder,reasonHelp,confirm,cancel}` (exist) + add `admin.broadcasts.cancelDialog.errors.{reasonRequired,reasonTooLong}` + `admin.broadcasts.toast.{cancelled,cancelError,cancelTooLate}`.
- member: add `portal.broadcasts.detail.cancelDialog.{title,description,reasonLabel,reasonPlaceholder,confirm,cancel,errors.reasonTooLong}` + a button label + member toast equivalents.

**Tests (TDD):** admin component — Cancel renders for `submitted` **and** `approved`, hidden for terminal statuses **and** for read-only manager; success; 409 too-late; 409 concurrent; over-cap (>500) shows `errors.reasonTooLong`. Member component — renders for owned submitted/approved, success with empty reason, 409, focus-lands-on-Cancel. Use the real `NextIntlClientProvider` + real `en.json` (catches MISSING_MESSAGE — every error branch must be rendered with a text-level assertion). Keep the existing route/use-case **cross-member + cross-tenant** regression tests in the net.

## Cross-cutting

- **TDD (NON-NEGOTIABLE):** failing test → commit red → implement → commit green; `tasks.md` (writing-plans) enumerates each failing test with its exact assertion.
- **Gates per PR:** `pnpm lint` · true `typecheck` (temp tsconfig excl `.next`, non-incremental) as the FINAL gate after the last edit · `check:i18n` (en/th/sv parity) · relevant `vitest` (component + loader + route) · `check:layout` if a `page.tsx` container changes · `e2e` if a journey covers it. Standalone `test:coverage` exit-1 on the ~22 pre-existing per-file thresholds is expected — verify only that touched files meet their own tier (Application 80/80; these are not on the security-critical-100%-branch list).
- **Isolation:** each PR in its own worktree off `main`, done **sequentially** (DV-11 fully shipped before DV-12) — avoids git-index races with the active COMP-1 session and between PRs.
- **Constitution:** F3 (member-contact PII) + F7 (broadcasts) are security-sensitive → Review gate normally ≥2 reviewers; **solo-maintainer substitute (Principle IX)** applies. DV-11 adds a security-relevant rate-limiter → the security checklist must be (self-)signed.

## Risks

- **R1 — members loader collision (DV-11):** the new `resolve-contact-verification` loader + the page `Promise.all` edit touch the `members` module, where the active COMP-1 session may have in-flight work → possible rebase at their merge. Mitigation: build off current `main` (COMP-1 merged), minimal footprint, rebase if needed.
- **R2 — strict TS:** `emailVerificationPending` flows as a derived page-level `Set<contactId>` (not a domain/Contact field) — thread under `exactOptionalPropertyTypes`.
- **R3 — i18n runtime:** component tests use the REAL next-intl provider, so every new key (incl. the error subtrees) must exist + be rendered by a test; `check:i18n` is parity-only.
- **R4 — rate-limiter:** reuse the existing Upstash limiter pattern; fail-soft so an Upstash outage never blocks a legitimate resend (and rate-limit tests can hit Upstash quota — see the known flake note).
