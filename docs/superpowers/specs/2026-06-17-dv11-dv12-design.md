# Design: DV-11 + DV-12 — wire existing backends to UI

**Date:** 2026-06-17
**Source backlog:** `docs/Bug/spec-code-divergence-2026-06-16.md` (the 2 HIGH items)
**Decisions:** build the UI for both · DV-11 = Approach B (visible-gate) · 2 separate PRs

## Context

Two HIGH spec↔code divergences: a feature users may expect exists in the backend (route + use-case + audit + most i18n) but has **no UI control**.

- **DV-11 (F3):** "Re-send verification email" — `POST /api/members/[memberId]/contacts/[contactId]/resend-verification` + `resendVerificationEmail` use-case + audit `email_verification_resent` exist; no button renders it.
- **DV-12 (F7):** "Cancel broadcast" — member route `/api/broadcasts/[id]/cancel` + admin route `/api/admin/broadcasts/[id]/cancel` + `cancel-broadcast` use-case + `admin.broadcasts.cancelDialog.*` i18n exist; no component renders a cancel control (`review-actions.tsx` is Approve/Reject only).

## Goals

- Admin can re-send a portal-linked contact's email verification from the member-detail page, **shown only when that contact's linked user email is unverified** (Approach B).
- Admin can cancel a `submitted`/`approved` broadcast (reason **required**) from the review surface.
- Member can cancel their **own** `submitted`/`approved` broadcast (reason **optional**) from the portal broadcast detail.

## Non-goals

- **No backend changes** — routes, use-cases, audit events already exist and are tested.
- No new cancellable statuses (reuse policy: `submitted` | `approved`; `sending`-with-batches handled by the route's 409, not surfaced as a button state).
- Not addressing the other DV items.

## DV-11 — F3 resend-verification (Approach B, visible-gate)

**Backend (existing, unchanged):** `POST …/resend-verification` → `200 {outbox_row_id, invalidated_prior}`; `409 {error:'not_eligible', reason:'no_linked_user'|'email_verified'}`; `404 {error:'not_found'}`; `500`. Use-case `src/modules/members/application/use-cases/resend-verification-email.ts`; audit `email_verification_resent`.

**Verification status source:** not on `Contact` — it lives on the F1 user, read via the `isEmailVerified(userId)` port (`UserEmailPort`). The member-detail view does not currently load it.

**Design:**
1. **Data-load (the B addition):** extend the member-detail view assembly so each contact with a `linkedUserId` carries `emailVerificationPending: boolean` (true when the linked user's email is NOT verified). Threaded via the existing `UserEmailPort` through the application use-case / loader that builds the member-detail view. **Exact wiring pinned during writing-plans**; keep footprint minimal (extend the existing contact view type, do not restructure the loader).
2. **Component:** new client `ResendVerificationButton` mirroring `src/components/members/resend-bounced-invite-button.tsx` (variant=outline, size=sm, `useState` submitting, `fetch` POST, `toast` + `router.refresh()`).
3. **Render gate (in `ContactBlock`, `…/[memberId]/page.tsx`):** show when `canWrite && contact.linkedUserId && contact.emailVerificationPending`.
4. **Error handling:** 409 `email_verified` / `no_linked_user` → specific toast; 404; 500 → generic toast.
5. **i18n (en/th/sv):** add `admin.members.detail.resendVerification.{resendLabel, resendSubmitting, resendSuccess, errors.notFound, errors.noLinkedUser, errors.emailVerified, errors.serverError}`.
6. **Tests (TDD):**
   - component test: success + each error path (404/409×2/500).
   - loader test: the member-detail view sets `emailVerificationPending` correctly for verified vs unverified linked contacts (unit or live-Neon integration depending on the wiring chosen in the plan).
   - route + use-case already covered.

## DV-12 — F7 cancel-broadcast (admin + member surfaces)

**Backend (existing, unchanged):** admin `POST …/cancel` (reason **required**, 1–500); member `POST …/cancel` (reason **optional**, ≤500). Use-case `cancel-broadcast.ts`; cancellable `submitted`|`approved`. Errors: `404 broadcast_not_found`; `409 broadcast_cancel_too_late` / `broadcast_concurrent_action_blocked`; `400 reason_too_long`; `500`. Audit `broadcast_cancelled` / `broadcast_cancel_too_late`.

**Admin design:**
1. New client `CancelDialog` mirroring `src/components/broadcast/admin/reject-dialog.tsx` (AlertDialog + Textarea reason **required** + char counter + `useTransition` + fetch + toast + `router.refresh()`).
2. Add a **Cancel** button to `review-actions.tsx`, rendered only when status ∈ {`submitted`, `approved`}.
3. i18n: reuse existing `admin.broadcasts.cancelDialog.*`; add `admin.broadcasts.toast.{cancelled, cancelError}` (en/th/sv).
4. 409 too-late / concurrent → toast + refresh.

**Member design:**
1. New client actions component on `src/app/(member)/portal/broadcasts/[id]/page.tsx` (e.g. `BroadcastDetailActions`), rendered only when status ∈ {`submitted`, `approved`}.
2. `CancelDialog` (member variant, reason **optional**).
3. i18n: add `portal.broadcasts.detail.cancelDialog.{title, description, reasonLabel, reasonPlaceholder, confirm, cancel}` + a button label (en/th/sv).
4. 409 too-late / concurrent → toast + refresh.

**Tests (TDD):** component tests for the admin Cancel button + dialog (success + 409 too-late + 409 concurrent) and the member actions (success + 409). Route/use-case already covered.

## Cross-cutting

- **TDD (NON-NEGOTIABLE):** failing test → commit red → implement → commit green.
- **Gates per PR:** `pnpm lint` · `typecheck` (temp tsconfig excl `.next` for a true check) · `check:i18n` (en/th/sv parity) · relevant `vitest` (component/contract) · `check:layout` if a `page.tsx` container changes · `e2e` if a journey covers it.
- **Isolation:** each PR in its own worktree off `main`, done **sequentially** (DV-11 fully shipped before DV-12 starts) — avoids git-index races with the active COMP-1 session and between the two PRs (no concurrent committers).
- **Constitution:** F3 (member-contact PII) + F7 (broadcasts) are security-sensitive → Review gate is normally ≥2 reviewers; the **solo-maintainer substitute (Principle IX)** applies. Both PRs are UI wiring onto already-reviewed, already-tested backends.

## PR structure

- **PR-A:** DV-11 (F3) on `075-dv11-resend-verification`.
- **PR-B:** DV-12 (F7) on a separate branch off `main` (started after PR-A merges).

## Rollout

After each PR merges, update `docs/Bug/spec-code-divergence-2026-06-16.md`: set the merged DV item's status → `fixed`.

## Risks

- **R1 — members data-load collision (DV-11):** the visible-gate adds a field to the member-detail view assembly in the `members` module, where the active COMP-1 session may have in-flight work → possible rebase conflict at their merge. Mitigation: build off current `main` (COMP-1 already merged), keep the change minimal, rebase if needed.
- **R2 — strict TS:** `emailVerificationPending` is an optional view field; thread it carefully under `exactOptionalPropertyTypes`.
- **R3 — i18n runtime:** new keys must resolve in en/th/sv; `check:i18n` is parity-only, so verify the keys render (unit tests mock next-intl).
