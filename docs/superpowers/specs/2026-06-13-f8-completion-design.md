# F8-completion ŌĆö renewal-flow wiring ŌĆö design (v2)

## Problem (the unwired boundaries)

F8 Renewal Tracking shipped the cycle-tracking schema, the SAFE/CAS transition primitive, the reminder dispatcher, the F4ŌåöF8 invoice bridge, the F5 paidŌåÆcomplete callback, and three race-safe admin use-cases ŌĆö but several F4ŌåöF8 boundaries were never connected, so the renewal pipeline is **inert end-to-end**. Confirmed by code: **(B2)** no production writer ever calls `cyclesRepo.insert` (`drizzle-renewal-cycle-repo.ts:367`) ŌĆö zero `upcoming` cycles exist; **(B1)** no writer transitions a cycle to `awaiting_payment` (`grep "to: 'awaiting_payment'"` = 0 hits), yet `confirm-renewal.ts:182` and `mark-cycle-complete-from-invoice-paid.ts:136` both *require* that status, so self-service confirm + invoice-paid-completion are dead on arrival; **(B3)** `adminReactivateLapsedCycle` / `adminRejectReactivation` are implemented, race-safe, and barrel-exported (`index.ts:557,565`) but have no API route + no UI; **(G4)** `/portal/renewal/[memberId]/page.tsx:218` renders `<RenewalConfirmFlow>` for any active cycle with no `summary.status` payability gate (pure UX dead-end ŌĆö server gate at `confirm-renewal.ts:182` is intact); **(G5)** `transitionStatus` (`drizzle-renewal-cycle-repo.ts:768`) enforces only the optimistic `WHERE status = from` CAS, never `canTransition`, and the domain `TRANSITIONS` map is missing two real edges; **(G6)** a dead `'grace'` literal in the dispatch-candidate status filter (`drizzle-dispatch-candidate-repo.ts:258`) that no DB row can ever match. Separately, a **tax-compliance contradiction**: `confirm-renewal` bills F4 from the *live* F2 catalogue price (`create-invoice-draft.ts:145`), not the cycle's frozen price, violating FR-022 ŌĆö the portal shows the frozen price but the ┬¦86/4 invoice can bill a different live price. **None of these are live production bugs ŌĆö the system is pre-launch; these are completeness gaps that must close before F8 can function.**

## Goals

0. **PRODUCTION-READY for the SweCham launch.** The renewal system must be **fully usable end-to-end in production for the SweCham go-live** ŌĆö a real member can renew, a lapsed member can come back, and an operator can run the launch sequence to completion ŌĆö not a partial gap-close. This is the headline acceptance bar; every slice below is in service of it. The launch scope is **12-month THB cycles for the ~131 known SweCham members** (see Non-goals for what is deferred without blocking this bar).
1. A production writer **creates** an `upcoming` renewal cycle (steady-state on prior-cycle-paid; first-load via the member import; post-launch onboarding for new members) with a **frozen** price snapshot.
2. A production writer **advances** `upcoming|reminded ŌåÆ awaiting_payment` so the confirm + paid-completion paths become reachable.
3. Admin **reactivate / reject** for `pending_admin_reactivation` cycles are reachable: API routes + cycle-detail UI + a discovery surface. *(money-hold variant DEFERRED post-launch ŌĆö see Resolved #6; Slice 3 ships the reachable fresh-cycle lapsed-comeback path.)*
4. The portal renewal page **gates** the Confirm CTA on payability (no enabled-button-that-always-409s).
5. The cycle state machine is **authoritative**: `transitionStatus` enforces `canTransition`, the `TRANSITIONS` map declares every real edge, and the dead `'grace'` arm is removed.
6. The ┬¦86/4 renewal invoice **bills the frozen price** the member was shown (FR-022), not the live catalogue.
7. Every new audit event lands in all its required sites; every new cross-tenant + tax invariant is covered by an integration test against live Neon; the real renew journey passes a **production E2E smoke** post-implementation.

## Non-goals (YAGNI)

- **Multi-year (>12mo) renewal invoicing** and **non-THB renewal cycles** ŌĆö F4 is THB-only and `plan-lookup-for-renewal-drizzle.ts` hardcodes `termMonths:12`/`currency:'THB'`. Constrain renewals to 12-month THB cycles for launch; multi-year is a separate spec (see Resolved decisions). **Does NOT block production-readiness** ŌĆö every known SweCham member is a 12-month THB cycle.
- **Converging the `pending_admin_reactivation` terminal states** (rejectŌåÆ`cancelled` vs timeoutŌåÆ`lapsed`). Recommend **document the split as intentional**, do not change member-visible lifecycle/funnel behaviour (changes at-risk + lapsed-tab bucketing).
- **A `blockAutoReactivation` admin UI** + the **`pending_admin_reactivation` money-hold reactivate/reject routes** ŌĆö sibling gap to B3; scope it explicitly but it is **DEFERRED post-launch** (Resolved #6). The race-safe use-cases already exist and can be wired later. **Does NOT block production-readiness** ŌĆö the reachable lapsed-comeback path (Slice 3 fresh cycle) covers the common case.
- **Invoice-at-cycle-creation** ŌĆö tax-dangerous (phantom ┬¦86/4 + ┬¦87-number / credit-note churn). Keep issuance at confirm; only snapshot price at creation.
- **Reworking the reminder dispatcher to write `reminded`** ŌĆö out of scope unless the maintainer wants `reminded` in the active lifecycle (resolved: do NOT write `reminded`); the new writers treat `reminded` as a valid `from` either way.
- Pro-rate / registration-fee redesign beyond suppressing them for renewals.

## Architecture

### A. Renewal-cycle creation (F4 hook ŌåÆ `upcoming`)

**Steady-state (on prior-cycle-paid):** add a 3rd callback to the array returned by `f8OnPaidCallbacks` at `renewals-deps.ts:469` (currently `[markCycleCompleteŌĆ” , makeApplyTierUpgradeOnPaidCallback]`). New use-case `create-next-cycle-on-paid.ts` ŌĆö a thin wrapper over the shared `createCycleInTx` helper (see ┬¦Shared helper). It:

- Resolves the just-paid cycle: `cyclesRepo.findByInvoiceIdInTx(tx, tenantId, evt.invoiceId)` (`drizzle-renewal-cycle-repo.ts:450`). No-op if absent.
- **Idempotency guard (must run AFTER callback[0] flips priorŌåÆ`completed`):** call the NEW **`findActiveForMemberInTx(tx, tenantId, memberId)`** (NOT the connection-fresh `findActiveForMember`) ŌĆö see callback discipline below for why. If a non-terminal cycle already exists for the member, **no-op** (do not insert-and-catch; the partial-unique index `renewal_cycles_active_member_uniq`, migration `0087:170` would otherwise throw and roll back the payment).
- Derives `periodFrom = prior.periodTo` (gapless ŌĆö recommended), `periodTo = periodFrom + prior.cycleLengthMonths`.
- Resolves the effective plan: `getEffectivePlanForRenewal` (`plans/application/get-effective-plan-for-renewal.ts:41`, docstring: "Called from F4's renewal-invoice-creation hook at cycle-creation time") ŌåÆ `makeDrizzlePlanLookupForRenewal.loadPlanFrozenFields` (`plan-lookup-for-renewal-drizzle.ts:49`) returns `{tierBucket, priceTHB, termMonths, currency}` ŌĆö the exact `NewRenewalCycleInput` frozen columns.
- `cyclesRepo.insert(tx, tenantId, NewRenewalCycleInput)` (`drizzle-renewal-cycle-repo.ts:367`; input shape `renewal-cycle-repo.ts:22-36`).
- Emits `renewal_cycle_created` (payload shape already defined `renewal-audit-emitter.ts:261`).

**Callback[2] discipline ŌĆö THROW, do not swallow (corrects v1).** Mirror callback[0]'s in-tx state-work discipline: **THROW on any insert / plan-resolution failure** ŌĆö this rolls back the just-completed payment's F4 tx, and the next Stripe at-least-once webhook retry heals via the idempotency guard (completion is idempotent; `findActiveForMemberInTx` makes creation idempotent). **DELETE** the v1 claim "a creation bug must not roll back the member's payment" ŌĆö it misread the precedent: `apply-tier-upgrade-on-paid-callback.ts:337` *does* `throw e` on its in-tx state work; only the POST-commit F2 finaliser swallows. **Error-swallowing applies ONLY to the option-D post-commit `createMember` bootstrap listener** (which has no tx to roll back and no webhook retry to lean on ŌĆö it logs + counts instead; see ┬¦Observability). Atomic-throw here is strictly safer than swallow: a swallow would commit the payment while the member silently drops out of the pipeline with no retry trigger (the paid webhook won't re-fire a fresh creation).

**Why `findActiveForMemberInTx`, not `findActiveForMember` (corrects a happy-path-DEAD bug in v1).** F4 fires the `f8OnPaidCallbacks` **BEFORE** `withTx` commits, so callback[0]'s prior-cycle `ŌåÆcompleted` flip is still uncommitted when callback[2] runs. A connection-fresh `findActiveForMember` (which opens its own `runInTenant`/connection) cannot see that uncommitted flip under READ COMMITTED ŌåÆ it still sees the prior cycle as active ŌåÆ callback[2] no-ops ŌåÆ **the next cycle is NEVER created**. Threading F4's `tx` via a new `findActiveForMemberInTx(tx, tenantId, memberId)` (add to the repo port + drizzle adapter, mirroring `findByIdInTx`, **NO `runInTenant`** ŌĆö the caller already holds the tenant-scoped tx) lets the guard see the in-flight completion, so the prior cycle is correctly excluded and the new cycle is created on the **first (non-retry) delivery**. The integration test MUST assert the new cycle IS created on first delivery (not only "no constraint violation on retry").

**Reusable to call:** `createCycleInTx` (the shared helper) ┬Ę `cyclesRepo.insert` ┬Ę `findByInvoiceIdInTx` ┬Ę `findActiveForMemberInTx` (NEW) ┬Ę `getEffectivePlanForRenewal` ┬Ę `makeDrizzlePlanLookupForRenewal` ┬Ę `markCycleCompleteInTx` (`mark-cycle-complete-from-invoice-paid.ts:118`) as the InTx-callback precedent.

**Ship `renewal_cycle_created` ŌĆö it is a whitelist MOVE, NOT a new migration (corrects v1).** This value is **already in the pgEnum** (added by migration `0109`, currently parked in `_F8_ENUM_DEFERRED`). Shipping it is a **5-place whitelist MOVE** (deferred ŌåÆ shipped tuple: move the literal from `_F8_ENUM_DEFERRED` into `F8_ENUM_SHIPPED_TUPLE` at `drizzle-renewal-audit-emitter.ts:67`) **+ the 2 count-assertion files** (`audit-event.test.ts` + `completeness.test.ts`) **+ the emit site**. **NO `ALTER TYPE ŌĆ” ADD VALUE` migration** ŌĆö the enum value already exists in the DB. (Contrast `renewal_entered_awaiting_payment` below, which *is* a full new-event with a migration.)

**First-load / initial cohort (import-integrated, NOT a standalone backfill):** the 131 SweCham members are **not yet in the DB** ŌĆö they are loaded by the one-time bulk import (`scripts/import-members.ts`, spec `docs/member-import-spec.md`). The initial-cohort renewal cycle is created **BY / alongside that import**: the import script (or a coordinated post-import step) calls the shared `createCycleInTx` per imported member, **inside the same `runInTenant` tx** as the member + contact rows. Period anchor is **resolved by data** ŌĆö `period_from = registration_date`, `period_to = registration_date + 12 months`, frozen at the resolved `plan_id` price ŌĆö **no operator-supplied date, no uniform "go-live + 12mo" guess** (that v1 language is removed). The reviewer safety template (per-row `tx` from `runInTenant`, **never the global `db`** ŌĆö CLAUDE.md RLS gotcha; `--dry-run` counters; server-validated anchor; **no member PII in stdout** ŌĆö uuid/row-index identifiers only; double-run idempotency via `findActiveForMemberInTx` no-op; cross-tenant integration test) applies to **this import-integrated cycle creation**, coordinated explicitly with `docs/member-import-spec.md` (its ┬¦5 execution model + ┬¦8 test plan must grow the cycle-creation step + its tests). A **RoPA / runbook note** records the bulk cold-start as a processing activity.

**Post-launch onboarding (option D, NEW members only):** post-commit best-effort listener on `create-member.ts:144`. **`createMember` has NO listener mechanism today** (unlike `change-plan.ts:63-89`'s `manualPlanChangeListeners`) ŌĆö adding one, mirroring that precedent, is in scope. It calls `createCycleInTx` for the freshly-onboarded member with `period_from = registration_date`. This covers the **normal post-launch onboarding flow ONLY** ŌĆö it is **NOT** the initial cohort (that's the import). Best-effort + error-swallowing (the only swallow site): log + OTel-count on failure (see ┬¦Observability), never throw (no tx to roll back, no webhook retry).

## Frozen-price billing (decision #2 ŌĆö the ┬¦86/4 tax fix)

The issued ┬¦86/4 (Ó╣āÓĖÜÓĖüÓĖ│ÓĖüÓĖ▒ÓĖÜÓĖĀÓĖ▓ÓĖ®ÓĖĄ) MUST bill the cycle's **frozen** price (FR-022), not the live F2 catalogue. Today `confirm-renewal` bills live (`create-invoice-draft.ts:145` reads `getAnnualFeeSatang` from the live plan) ŌåÆ a tenant editing a plan mid-cycle creates a tax-document amount diverging from what the member agreed to (a ┬¦86/10 credit-note correction problem). Closing it requires all three of the following:

**(a) THB-string ŌåÆ satang parse (precision-sensitive).** The cycle stores `frozenPlanPriceThb` as a **decimal THB string** (e.g. `"15000.00"`, `"50000.50"`); the F4 path is **satang minor units**. Convert at the bridge boundary with an **integer-only** parse ŌĆö **split on `.`**, compute `baht * 100 + satang` (zero-pad / validate the 2-digit satang group), and wrap with `Money.fromSatangUnsafe`. **NO float, NO `parseFloat(x) * 100`** (re-introduces a binary-float rounding bug ŌĆö e.g. `parseFloat("50000.50")*100` is not guaranteed `5000050`). Both the frozen path and the live path ultimately derive from the same `annualFeeMinorUnits`; the only risk is a re-introduced float at this conversion.

**(b) VAT-basis WARNING ŌĆö the membership override is VAT-EXCLUSIVE.** The membership `unitPriceSatang` override is the **VAT-exclusive** unit price (VAT 7% is added on top at issue, `issue-invoice.ts:280,426`). This is **opposite** to the event-path `amountOverride`, which is **VAT-INCLUSIVE**. **Copying the event-path semantics produces a grand total wrong by the 7% factor.** The membership line subtotal must equal `frozen ├Ś 100` exactly (VAT-exclusive); the issued grand total must equal `frozen ├Ś 100 ├Ś 1.07`.

**(c) Server-side-only sourcing (price-tampering surface).** Read `cycle.frozenPlanPriceThb` **in the already-open tenant tx** (the confirm-renewal Step-1 `runInTenant` tx), **NEVER from a request body / HTTP boundary**. A client-supplied frozen price is a price-tampering surface on a tax document ŌĆö the value must come from the server-side cycle row, full stop.

**`createInvoiceDraft` input contract ŌĆö a renewal signal (input-CONTRACT change, not pure data-threading).** Thread a NEW **renewal signal** parameter from the F4 bridge into `createInvoiceDraft`. It is an input-contract change because it changes the function's behaviour, not just its data: when set it **forces `proRate = 1.0`** (no re-derivation ŌĆö `create-invoice-draft.ts:161-170`), **suppresses the one-off `registration_fee` re-bill** (`create-invoice-draft.ts:205`), and **supplies the frozen `unitPriceSatang`** (VAT-exclusive) for the membership line. Call this out explicitly in the plan as a contract change to `createInvoiceDraft`, reviewed as such.

**┬¦87 / discriminator / retention untouched:**
- **Issue at confirm, never at cycle-creation.** A ┬¦86/4 with an allocated ┬¦87 number must not exist before the member commits ŌĆö early issuance creates phantom tax documents + gapless-stream pollution for every member who lapses instead of renewing (┬¦87 RD audit liability). Snapshot price at creation; consume a ┬¦87 number only at confirm.
- **┬¦87 gapless numbering untouched.** The advisory-locked allocator (`postgres-sequence-allocator.ts`) and `issue-invoice.ts:400` ┬¦87 allocation are reused as-is ŌĆö the frozen-price override only sets the membership line's unit price *before* VAT is computed.
- **┬¦86/4 vs ┬¦105 discriminator untouched.** Membership always resolves `pdfDocKind='invoice'`, TIN line omitted when buyer has no TIN (066 ruling, `document-kind.ts`) ŌĆö reuse verbatim.
- **VAT 7% + buyer snapshot** pinned at issue (`issue-invoice.ts:280,426`) ŌĆö unchanged.
- **Retention.** The renewal invoice **INHERITS F4's 10-year retention** (Thai RD ┬¦87/3 + GDPR Art. 6(1)(c)) ŌĆö do **NOT** downgrade it to the 5-year audit default.
- **Atomicity residual (accepted, pre-existing):** F4 issuance runs outside the F8 tx (`confirm-renewal.ts:273`); the frozen-price rework must not widen the issueŌåölink orphan window.
- **THB-only, 12-month** for launch (term/currency hardcoded in `plan-lookup-for-renewal-drizzle.ts`). Multi-year / non-THB have no correct ┬¦86/4 total path today ŌĆö scoped out (Resolved #4).
